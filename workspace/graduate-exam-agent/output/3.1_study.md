其中，预处理、编译、汇编的过程也可被通称为“编译”。  
因此，要在系统中运行用户程序，一般要经过以下3个步骤，分别是编译、链接、装入，下面通过一个具体的例子讲解整个过程。

【例】假设使用如下的用户程序文件完成一个加法计算的任务，代码如下所示。

```c
#ifndef ADD_H
#define ADD_H

int add(int a, int b);

#endif
```

add.h

```c
#include "add.h"

int add(int a, int b) {
    return a + b;
}
```

add.c

```c
#include <stdio.h>
#include "add.h"

int main() {
    int num1 = 5;
    int num2 = 3;
    int result = add(num1, num2);
    printf("%d + %d = %d\n", num1, num2, result);
    return 0;
}
```

main.c

步骤 1：编译（Compiler）  
由编译程序将用户源程序 add.c 和 main.c 分别编译成目标模块。

```bash
leon@leon-virtual-machine:~/OS/example$ gcc -c add.c -o add.o
leon@leon-virtual-machine:~/OS/example$ gcc -c main.c -o main.o
```

[图片：add.c 编译为 add.o，main.c 编译为 main.o 的示意图]  
不妨使用 nm 指令查看两个目标模块的构造。

```bash
leon@leon-virtual-machine:~/OS/example$ nm add.o
0000000000000000 T add
```

T 表示 Text section，即代码段，这说明 add.o 模块包含了 add 函数，其起始地址为 0000000000000000H。

```bash
leon@leon-virtual-machine:~/OS/example$ nm main.o
                 U add
0000000000000000 T main
                 U printf
```

U 表示 Undefined reference，这说明 main.o 中引用了 add 函数以及 printf 函数，但这两个函数未在 main.o 中定义。main 函数的起始地址为 0000000000000000H。  
步骤 2：链接（Linker）  
显然，main.o 模块中并没有加法操作以及输出操作的具体实现，因此需要使用链接程序将若干个目标模块以及它们所需要的库函数链接在一起，形成一个完整的装入模块。

```bash
leon@leon-virtual-machine:~/OS/example$ gcc add.o main.o -o add_program
```

[图片：add.o, main.o 和 printf 库链接生成 add_program 的示意图]  
步骤 3：装入（Loader）和运行  
接着由装入程序将装入模块装入内存，设置好程序的执行环境，随后执行该程序。

```bash
leon@leon-virtual-machine:~/OS/example$ ./add_program
5 + 3 = 8
```

### 第四题：综合理解

结合我们之前讨论过的内容，在“链接”这一步，链接程序除了把各个目标模块拼在一起，还需要完成一件非常关键的事，这件事是什么？（提示：和地址有关）

**第四题更准确的答案应该是：修改模块间的相对地址（或称为"符号解析/符号重定位"）。**

具体来说：

- 在链接前，`main.o`​ 里有一条 `CALL add`​ 指令，但 `add` 此时只是一个符号，链接器不知道它到底在哪个地址。
- 链接时，链接器把 `add.o`​ 拼接到 `main.o`​ 后面，此时 `add`​ 函数就有了一个确定的​**相对地址**​（比如 `L`）。
- 链接器会把 `CALL add`​ 这条指令修改为 `JSR "L"`，让调用指令指向正确的地址。

**记忆要点：**

|过程|主要修改的地址|目的|
| :-----| :-------------------------------| :-----------------------|
|**链接**|修改**模块间调用**的符号地址（如把`CALL add`​改成`JSR "L"`）|让分散的模块能互相找到|
|**装入/重定位**|修改**逻辑地址到物理地址**的映射（如把`[200]`​改成`[1200]`）|让程序在内存中正确定位|

所以第四题问的是链接阶段做的事，答"把模块拼接起来，并修改模块间的调用地址（符号解析）"是最贴切的。如果说"地址重定位"容易和装入阶段的逻辑地址→物理地址的转换搞混。

式，分别是静态重定位和动态重定位。  
（一）静态重定位  
静态重定位也叫做静态地址映射。  
基本思路：当用户程序被装入内存时，直接对指令代码进行修改，一次性实现逻辑地址到物理地址的转换，以后不再转换。  
【例】如果采用静态地址映射方式，那么在把该程序装入到内存后，如下图所示。

[图片：静态重定位示意图。左侧为“可执行文件”，包含地址0, 100, 200, 204, 300及对应指令和数据（如str 5, [200]等）。中间箭头指向右侧“内存”，标注“装入分区”。右侧内存起始地址为1000，对应指令中的地址变为物理地址（如str 5, [1200]）。]

具体来说，由于内存分区的起始地址为 1000，因此，每一个逻辑地址只要加上 1000，就得到了相应的物理地址。经过这样的修改以后，所有的逻辑地址都转换成了物理地址。  
由于指令中使用的地址都是物理地址，因此不能把这段程序再移动到内存的其他地方，否则会造成内存访问错误。  
优点：实现起来很简单，不需要任何硬件方面的支持。  
缺点：程序一旦装入到内存以后，就不能再移动了。  
（二）动态重定位  
动态重定位也叫动态地址映射。其基本思路为，当用户程序被装入到内存时，不对指令代码做任何的修改，而是在程序的运行过程中，当它需要访问内存单元时，再来进行地址转换。为了提高效率，通常设置一个基地址寄存器（重定位寄存器）来实现动态重定位。  
【例】如果采用动态地址映射方式，那么在把该程序装入到内存后如下图所示，指令中的每一个逻辑地址都原封不动，不做任何修改。

[图片：动态重定位示意图。左侧为“可执行文件”，指令保持原样（如str 5, [200]）。中间箭头指向右侧“内存”，标注“装入”。右侧内存中指令地址未变。右上方有一个“基地址寄存器”值为1000。图示展示了相对地址200与基地址1000相加得到物理地址1200的过程。]

程序执行时，操作系统会把起始地址 1000 装入到基地址寄存器中。  
当这个程序运行时，如果在指令执行过程中需要访问内存，硬件装置就会自动地把其中的相对地址取出来，如 200，然后把它和基地址寄存器做一个加法，从而得到实际的物理地址，即 1200，然后再用这个新的地址来访问内存单元。

Q：问题来了，这个时候只有多个目标模块，它们需要经过链接和装入后才可以放入内存中，确定物理地址。然而，各个模块内的指令和数据都需要地址，否则就没法说清楚各个指令之间的相对位置，也没法根据某个地址去访问某个数据，或者实现程序跳转。该怎么办呢？

## 五、程序的链接

# 五、程序的链接

五、程序的链接（理解并记忆）  
（一）静态链接方式  
前面已经学习了静态链接方式，它指将若干个目标模块在装入前链接成一个完整的装配模块后，再将其装入内存。

```text
0       模块A
L-1     CALL B;
        Return;

0       模块B
M-1     CALL C;
        Return;

0       模块C
N-1     Return;

目标模块
```

[图片：展示了静态链接过程。左侧为三个独立的目标模块（模块A、B、C），中间通过“链接”箭头指向“装入模块”。装入模块中，模块A的CALL B变为JSR "L"，模块B的CALL C变为JSR "L+M"，各模块按顺序拼接，地址从0到L+M+N-1。最后通过“装入”箭头指向“内存”。]

（二）装入时动态链接  
装入时动态链接是指，在装入一个目标模块时，若发生一个外部模块调用事件，则将引起装入程序找出相应的外部目标模块，并将它装入内存，修改目标模块中的相对地址。

```text
0       模块A
L-1     CALL B;
        Return;

0       模块B
M-1     CALL C;
        Return;

0       模块C
N-1     Return;
```

[图片：展示了边装入边链接的过程。左侧为目标模块。中间箭头指向右侧“内存”。内存中依次装入了模块A（JSR "L"）、模块B（JSR "L+M"）、模块C。]

装入时动态链接方式有以下优点。

1. 便于修改和更新  
   对于经静态链接装配在一起的装入模块，如果要修改或更新其中的某个目标模块，则要求重新打开装入模块。这不仅是低效的，而且有时是不可能的。若采用动态链接方式，则由于各目标模块是分开存放的，因此要修改或更新各目标模块是一件非常容易的事。
2. 便于实现对目标模块的共享  
   在采用静态链接方式时，每个应用模块都必须含有其目标模块的复制版本，而无法实现对目标模块的共享。

```c
// 目标模块
模块A
CALL B;
Return;

模块B
CALL C;
Return;

模块C
Return;

模块D
CALL E;
Return;

模块E
CALL C;
Return;
```

```c
// 应用模块1
模块A
JSR "L"
Return;

模块B
JSR "L+M"
Return;

模块C
Return;
```

```c
// 应用模块2
模块D
JSR "L"
Return;

模块E
JSR "L+M"
Return;

模块C
Return;
```

但采用装入时动态链接方式时，OS 很容易将一个目标模块链接到几个应用模块上，实现多个应用程序对该目标模块的共享。

从上面的内容来看，动态链接主要就是为了**克服静态链接的短板**。换句话说，**不是因为静态链接本身不好，而是在需要灵活更新、节省空间和模块共享的时候，它就不够用了。**

下面我们结合上面的文档，用具体的例子来说明为什么要动态链接，主要分三种情况：

---

### 1. 为了便于修改和更新（对比静态链接的死板）

**场景**：假设你开发了一个大型软件，比如一个办公套件，里面有一个“拼写检查”模块B。

- **静态链接的做法**：  
  开发完成后，模块B和主程序模块A、以及其他模块被提前打包成一个巨大的可执行文件。如果后来你发现拼写检查有bug，需要更新模块B，你就必须重新编译、重新链接整个办公套件，然后让用户重新下载整个巨大的安装包。文档里提到这“不仅是低效的，而且有时是不可能的”。
- **动态链接的好处**：  
  如果采用动态链接，各个模块（A、B、C...）是分开存放的独立文件。当需要更新拼写检查模块B时，你只需要单独开发、测试这个模块，然后发布一个小补丁替换掉旧的那个模块B文件就行。程序下次启动时，在需要用到它的那一刻，操作系统会动态地把新的模块B装入内存并链接上。

**举例**：就像你的手机App更新，动态链接允许你只更新App的某个功能组件；而静态链接则要求你每次更新哪怕一行代码，都得把整个App重新下载安装一遍。

---

### 2. 为了实现对目标模块的共享（对比静态链接的浪费）

**场景**：假设你同时打开了两个应用——一个文字处理软件（应用1）和一个演示文稿软件（应用2）。这两个软件都需要用到同一个“打印模块C”。

- **静态链接的做法**：  
  在静态链接下，文字处理软件和演示文稿软件在打包时，都各自把“打印模块C”的代码复制了一份，链接到自己的可执行文件里。当你同时运行这两个软件时，**内存里就会存在两份完全相同的打印模块C的代码**，造成了内存的浪费。文档里说“每个应用模块都必须含有其目标模块的复制版本”，就是指这个。
- **动态链接的好处**：  
  OS 很聪明，在采用装入时动态链接时，当第一个应用要调用打印模块C时，OS把它装入内存并链接上。当第二个应用也要调用打印模块C时，OS发现内存里已经有了，**就不再装入一份新的**，而是直接让第二个应用模块也链接到内存里已经存在的那份打印模块C上。

**举例**：Windows系统里的`DLL`​（动态链接库）文件就是这个原理。多个程序可以共享一个`DLL`文件，既节省了磁盘空间，也节省了运行时的内存空间。

---

### 3. 为了节省内存空间，按需加载（运行时动态链接的改进）

**场景**：你的程序有一个模块C，它的功能是负责“严重错误恢复和数据上报”。在日常使用中，程序99%的时间都运行良好，这个模块根本用不上。

- **静态/装入时动态链接的做法**：  
  无论你用不用得上，**因为事先不知道会不会出错，所以这个负责错误处理的模块C都会被装入内存并链接好**。这部分内存就被白白占用了。文档里说这就是低效的。
- **运行时动态链接的好处**：  
  这是一种更“聪明”的动态链接。程序启动时只装入主模块A。当程序运行过程中，**只有当真的调用了那个执行特定功能的模块、且发现它还没被装入时**，OS 才会立刻去找这个模块C，把它装入内存并链接起来。如果整个运行过程都一帆风顺，这个模块C就永远不会被装入内存。

**举例**：想象一个游戏，里面有上百个角色的技能模块。你在一场比赛中只会遇到几个特定角色，用运行时动态链接，游戏就只会把你遇到的这些角色的技能模块加载到内存，而不会一开始就把所有角色的技能都加载进来，这大大节省了内存，让游戏启动也更快。

---

总结一下，**动态链接的核心好处就是让程序变得更灵活、更省资源**：

- **更新变简单** → 像换积木，而不是重新造房子。
- **内存省下来** → 同一个东西可以共享，没用的可以不装。

（三）运行时动态链接  
应用程序在运行时，每次要运行的模块可能是不相同的。但由于事先无法知道本次要运行哪些模块，故只能将所有可能要运行的模块全部装入内存，并在装入时全部链接在一起。  
【例】假设模块 C 负责错误处理，如果程序在整个运行过程中都不出现错误，则不会用到该模块。

它们（装入时 vs 运行时动态链接）最核心的区别在于**触发链接的时刻不同**，这个时间点的差异直接导致了**加载模块的范围**和**内存使用的效率**完全不同。

结合文档内容和之前的讨论，可以这样来区分和理解：

---

### 1. 触发时刻不同：“装模块时” vs “程序运行时”

- **装入时动态链接**  
  它的触发时刻是在  **“装入一个模块”的过程中**。  
  当装入程序把模块A放进内存时，发现它调用了模块B，于是装完A后立刻把B也装进来；看到B又调用了C，接着再把C装进来。这是一个**边装入、边解析、边链接**的连续过程，在程序正式开始运行前，所有被直接或间接调用的模块都已经被链接好了。
- **运行时动态链接**  
  它的触发时刻是在  **“程序执行一条调用指令”的那一刻**。  
  程序先被装入内存并开始运行。当CPU真正执行到 `CALL C`​ 这条指令时，才会停下来检查模块C在不在内存。如果不在，此时OS才会去找模块C、装入并链接它。这是一个**在执行中按需中断、加载、链接**的过程。

---

### 2. 加载范围不同：“全量加载” vs “按需加载”

这是触发时刻不同带来的直接结果。

- **装入时动态链接：用到用不到，全部都得装。**   
  由于它在装入阶段就“递归”地追踪并加载所有被调用模块，所以**只要在调用链上的模块，无论最终会不会被执行到，都会被装入内存**。  
  比如，文档里的模块C负责“错误处理”。如果程序运行一切正常，根本没有错误发生，这个模块C的代码就白占了一份内存空间。文档评价这“显然是低效的”。
- **运行时动态链接：真正用到时，才去装。**   
  这是对前者的改进。**程序运行中用不到的模块，根本不会被装入内存**。  
  还是那个“错误处理模块C”的例子，采用运行时动态链接，只要程序运行全程无错，`CALL C`这条指令就永远不会被执行，模块C就永远不会被装入内存。这样不仅节省了内存，也让程序的初始装入和启动过程更快了。

---

### 一个具体的执行流程对比

假设现在有模块A调用B，B调用C。但根据程序逻辑，只有当用户点击了某个特殊按钮时，才会执行到 `CALL C` 这条指令。现在程序运行，用户没有点击按钮。

- **如果是装入时动态链接**：

  1. 装入程序加载模块A。
  2. 发现A调用了B，立刻去找B并装入。
  3. 发现B调用了C，立刻去找C并装入。
  4. 装入完成，程序开始运行。  
     **结果：**  模块C虽然没被用到，但在一开始就已经在内存里了。
- **如果是运行时动态链接**：

  1. 操作系统加载并启动模块A。
  2. 程序执行，当执行到A里的 `CALL B` 时，发现B不在内存，OS暂停程序，装入B并链接，然后继续执行。
  3. 程序正常执行B的代码，因为用户没点按钮，`CALL C`​ 未被执行，程序流畅运行直到结束。  
     **结果：**  模块C自始至终都没有被装入内存。

所以，可以这么总结它们的进化关系，方便记忆：

**装入时动态链接** → 解决了 **共享** 和 **更新** 的难题。  
**运行时动态链接** → 进一步解决了 **内存浪费** 的问题，做到了真正的  **“懒加载”**  或  **“按需加载”** 。

这是一个非常容易混淆，但又是考试重点的问题。

你的困惑很精准：**装入过程**和**链接过程**里确实有地址修改，但它们被严格地称为“重定位”，而不叫“**动态**重定位”。这道题考的是狭义的、专指**运行时**才进行的地址变换技术。

我们可以通过对比三个层次的“地址转换”来彻底分清：

### 1. 装入时（B）和链接时（C）做的，叫“静态或装入时重定位”

无论是在链接时把模块拼好，还是在装入时修改地址，它们都有一个共同点：**一旦程序被装入内存，地址就固定下来了。**

- **链接过程（C）** ：把模块A、B、C拼成一个完整的程序，把 `CALL B`​ 改成 `JSR "L"`​。地址在**运行前**就定死了。
- **装入过程（B）** ：如果采用“静态重定位”，装入时一次性把所有逻辑地址换成内存的物理地址。一旦装入完成，这个程序在内存里再也不能移动，地址也**定死了**。

如果在装入过程中，程序需要被移动（例如从内存一块区域搬到另一块），因为地址已经“写死”，它就无法运行了。

### 2. 执行时（D）做的，才叫“动态重定位”

**动态重定位**是一个专有名词，特指  **“地址的转换被推迟到程序真正执行的那一刻，而不是在装入时就完成”** 。

它的核心机制是：程序装入时，指令里的地址**根本不改，保留原来的逻辑地址**。直到CPU真正执行这条指令时，才由硬件（重定位寄存器）把逻辑地址加上基址，计算出最终的物理地址。

**这给系统带来了巨大的灵活性：**

- **程序可以在内存中移动**：因为指令里的地址没写死，物理地址是执行时动态算出来的。OS只需修改重定位寄存器的值，程序就能在新位置正确运行。
- **内存管理更灵活**：这是实现虚拟存储器、程序在内存和外存之间换入换出的基础。

### 为什么不选B和C？（针对你的困惑）

- **为什么不选“装入过程”（B）**   
  教材里，装入时一次性把所有地址都改好的方式，专门称为  **“静态重定位”**  。它是一次性搞定，而非“运行中动态进行”。所以“装入过程”里发生的是“静态重定位”，不是这题问的“动态重定位”。
- **为什么不选“链接过程”（C）**   
  链接时虽然也叫“动态链接”，但它解决的核心问题是 **“目标模块的按需链接”** ，即把不同模块的调用符号（如 `CALL B`​）解析为相对地址（如 `JSR "L"`），而不是把逻辑地址最终转换为物理地址，并且它也不是指令执行时每条指令实时进行的地址转换。
- **那“运行时动态链接”里的地址修改呢？**   
  你说得对，运行时链接确实在执行时修改了地址。但这题考的是“**动态重定位**”这个术语，它特指 **“逻辑地址到物理地址的转换”** 延迟到执行时才进行，目的是解决 **“程序在内存中移动”** 的问题。而运行时动态链接修改的是 **“模块间的符号引用”** ，目的是解决 **“按需加载”** 的问题。

### 总结

可以这样记结论：

- **修改模块间的调用地址**（把`CALL B`​改成某个入口地址） → 发生在**链接过程**
- **一次性把所有逻辑地址改成物理地址** → 发生在**装入过程**，叫**静态重定位**
- **每条指令执行时才把逻辑地址转换成物理地址** → 发生在**执行过程**，这才是**动态重定位**

所以题目考的“动态重定位”，专指执行时才通过硬件进行的地址转换，答案是**D. 执行过程**。

# 六、内存保护

六、内存保护（理解即可）  
在多道程序并发运行的环境下，操作系统位于内存中，多个用户程序也位于内存中。为了防止一个用户程序去访问其他用户程序的内存区域，保护操作系统免受破坏，就必须进行内存保护。

加载基地址寄存器和界限寄存器时必须使用特权指令，由于特权指令只能在内核态下执行，因此只有 OS 内核才可以加载基地址寄存器和界限寄存器。

## 四、程序的装入

# 四、程序的装入

四、程序的装入（理解并记忆）  
在将一个装入模块装入内存时，可以有如下3种装入方式。  
（一）绝对装入方式  
当计算机系统很小，仅能运行单道程序时，完全有可能知道程序将驻留在内存的什么位置。此时可以采用绝对装入方式。

[图片：应用程序装入内存示意图，左侧为应用程序，右侧为内存（包含系统区和用户区）]

用户程序经编译后，将产生绝对地址（物理地址）的目标代码。

```
1000    ...             1000    ...
1100    str 5, [1200]   1100    str 5, [1200]
        ldr R1, [1200]          ldr R1, [1200]
        add R2, R1,3            add R2, R1,3
        str R2, [1204]          str R2, [1204]
1200    X               1200    X
1204    y               1204    y
1300    ...             1300    ...
      可执行文件                内存
```

绝对装入程序便可按照装入模块（可执行文件）中的地址，将程序和数据装入内存。装入模块被装入内存后，由于程序中的逻辑地址与内存地址完全相同，故无须对程序和数据的地址进行修改。  
（二）可重定位装入方式（静态重定位）  
在多道程序环境下，编译程序不可能预知装入模块应放在内存的何处，因此采用可重定位装入方式。  
若当用户程序被装入内存时，直接对指令代码进行修改，一次性实现逻辑地址到物理地址的转换，以后不再转换，该方式则称为静态重定位。（刚刚已经讲过静态重定位了，只是从装入的角度来解释）

```
0       ...             1000    ...
100     str 5, [200]    1100    str 5, [1200]
        ldr R1, [200]           ldr R1, [1200]
        add R2, R1,3            add R2, R1,3
        str R2, [204]           str R2, [1204]
200     X               1200    X
204     y               1204    y
300     ...             1300    ...
      可执行文件                内存
```

可重定位装入方式克服了绝对装入方式的缺点，不再需要预先知道装入位置，但该方式不允许程序在运行时移动位置。  
（三）动态运行时装入方式（动态重定位）  
当用户程序被装入到内存时，若不对指令代码做任何的修改，而是在程序的运行过程中，当它需要访问内存单元时，再来进行地址转换，则称为动态重定位。  
为了提高效率，通常会设置一个基地址寄存器（重定位寄存器），即物理地址=基址寄存器中的程序起始地址+逻辑地址。装入模块在装入后有可能还会调动位置，动态运行装入方式可以很好地应对这一情况。

下面按照虚拟地址递增顺序依次进行介绍。  
（一）只读代码段  
存储可执行代码的指令。只读，通常不可修改。  
（二）数据段  
存储全局变量和静态变量。  
（三）运行时堆  
由进程在运行时动态创建，该区域可以动态增长和收缩，例如 C 语言的 malloc()可以申请该区域的内存，free()可以释放该区域的内存。

该区域用来存放局部变量、返回地址，函数调用参数等信息，可以利用该区域实现函数调用。和堆一样，该区域也可以动态增长和收缩。  
（六）操作系统内核区  
操作系统内核使用的地址空间，包括内核程序、内核数据等信息，这些信息对用户程序不可见。  
一个进程映像（实体）主要包括三个部分：程序段、相关的数据段和 PCB。  
【例】有代码如下所示。

```c
......
char *ptr;
void main() {
    int length;
    ptr=(char*)malloc(10);
    length=sizeof(ptr);
    printf("%d",length);
    free(ptr);
}
```

(1) 代码执行时，PCB 位于哪个区域？执行 scanf()函数时，进程处于什么状态？  
(2) main()函数代码、ptr 指针、length 变量、malloc()函数以及 ptr 执行的字符串各位于哪个区域？  
【答案】(1) PCB 位于内核区，执行 scanf()时，进程处于阻塞态。  
(2) main()函数的代码位于只读代码段，ptr 位于读/写数据段，length 位于用户栈，malloc()函数的代码位于共享库，ptr 指向的字符串位于运行时堆。

【补充】地址空间实例（理解并记忆）  
linux 系统下，进程执行时的地址空间如下所示：

[图片：Linux进程地址空间示意图，从低地址0x0000 0000到高地址0xFFFF FFFF，依次划分为未使用区、只读代码段、读/写数据段、动态生成的堆（用户空间）、共享库、用户栈、操作系统内核区（内核空间）]

下面按照虚拟地址递增顺序依次进行介绍。  
（一）只读代码段  
存储可执行代码的指令。只读，通常不可修改。  
（二）数据段  
存储全局变量和静态变量。  
（三）运行时堆  
由进程在运行时动态创建，该区域可以动态增长和收缩，例如 C 语言的 malloc()可以申请该区域的内存，free()可以释放该区域的内存。  
（四）共享库  
该区域用于存放公共的共享库代码和数据，例如 C 语言的标准库函数。  
（五）用户栈  
该区域用来存放局部变量、返回地址，函数调用参数等信息，可以利用该区域实现函数调用。和堆一样，该区域也可以动态增长和收缩。  
（六）操作系统内核区  
操作系统内核使用的地址空间，包括内核程序、内核数据等信息，这些信息对用户程序不可见。  
一个进程映像（实体）主要包括三个部分：程序段、相关的数据段和 PCB。  
【例】有代码如下所示。

```c
......
char *ptr;
void main() {
    int length;
    ptr=(char*)malloc(10);
    length=sizeof(ptr);
    printf("%d",length);
    free(ptr);
}
```

(1) 代码执行时，PCB 位于哪个区域？执行 scanf()函数时，进程处于什么状态？  
(2) main()函数代码、ptr 指针、length 变量、malloc()函数以及 ptr 执行的字符串各位于哪个区域？  
【答案】(1) PCB 位于内核区，执行 scanf()时，进程处于阻塞态。  
(2) main()函数的代码位于只读代码段，ptr 位于读/写数据段，length 位于用户栈，malloc()函数的代码位于共享库，ptr 指向的字符串位于运行时堆。
