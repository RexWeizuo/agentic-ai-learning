一、外存空闲空间管理方法（理解并记忆）
在给文件分配盘块时，必须知道磁盘上哪些盘块是可分配的。因此，系统还应为可分配存储空间设置相应的数据结构。此外，还应提供对盘块进行分配和回收的手段。不论哪种分配和回收方式，存储空间的基本分配单位都是盘块。下面介绍几种常用的文件存储空间管理方法。

（一）空闲区表法
1. 数据结构
空闲区表法适用于为每个文件分配连续的存储块。具体来说，系统为辅存上的所有空闲区建立一张空闲表，每个空闲区对应于一个空闲表项，其中包括起始空闲块号、连续的空闲盘块个数等信息，将所有空闲区按它们的起始盘块号递增的次序排列，即可形成空闲盘块表。

| 序号 | 起始空闲块号 | 空闲块个数 |
| :--- | :--- | :--- |
| 1 | 2 | 4 |
| 2 | 8 | 6 |
| 3 | 17 | 2 |
| ... | ... | ... |

2. 存储空间的分配与回收
空闲区的分配与内存的动态分配类似，可采用首次适应算法、循环首次适应算法等。

[图片：分配后的磁盘块状态示意图及对应的空闲区表]

| 序号 | 起始空闲块号 | 空闲块个数 |
| :--- | :--- | :--- |
| 1 | 2 | 4 |
| 2 | 8 | 6 |
| 3 | 17 | 3 |

在系统为某新创建的文件分配空闲盘块时：
① 先顺序检索空闲表的各表项，直至找到第一个大小能满足要求的空闲区。
② 将该盘区分配给用户（进程），同时修改空闲表。
系统在对用户所释放的存储空间进行回收时，也采取类似于内存回收的方法，需要考虑回收区是否与空闲表中插入点的前区和后区相邻接，对相邻接者应予以合并。
由于这种组织方式具有较高的分配速度，可减少访问磁盘的 I/O 频率，因此其在诸多组织方式中仍占有一席之地。例如，对换空间一般都采用连续组织方式。

（二）空闲链表法
空闲链表法是将所有空闲盘区拉成一条空闲链。根据构成空闲链所用的基本元素的不同，可把链表分成两种形式：空闲盘块链和空闲盘区链。
1. 空闲盘块链
[图片：磁盘块示意图，包含数字0-19，以及count, f, tr等标记]
将磁盘上的所有空闲空间以盘块为单位拉成一条链，其中的每一个盘块都有指向后继盘块的指针。
[图片：空闲块首指针 -> 空闲块2 -> 空闲块3 -> 空闲块4 ......]
当用户因创建文件而请求分配存储空间时，系统便从链首开始，依次摘下适当数目的空闲盘块分配给用户。当用户因删除文件而释放存储空间时，系统便将回收的盘块依次挂在空闲盘块链的末尾。
这种方法的优点是用于分配和回收一个盘块的过程非常简单。缺点是以盘块为单位，所以空闲盘块链会很长，分配和回收的效率较低。
2. 空闲盘区链
这是将磁盘上的所有空闲盘区（每个盘区可包含若干个盘块）拉成一条链。在每个盘区上除了含有用于指示下一个空闲盘区的指针外，还应含有能指明本盘区大小（所含盘块数）的信息。
[图片：磁盘块示意图，包含数字0-19，箭头连接不同区域]
分配盘区的方法与内存分区的动态分配方法类似，通常采用首次适应算法。在回收盘区时，同样也要将回收区与相邻接的空闲盘区合并。
这种方法的优缺点刚好与空闲链表法的优缺点相反，即分配与回收的过程比较复杂，但分配与回收的效率较高，每次会为文件分配多个连续的盘块，且空闲盘区链较短。

**1. 盘块的分配**
在位示图方法中，盘块分配分为三步：
(1) 顺序扫描位示图，从中找出一个或一组值为“0”的二进制位（“0”表示空闲）。
(2) 将所找到的一个或一组二进制位转换成与之相对应的盘块号。假定找到值为“0”的二进制位处在位示图的第 i 行、第 j 列，则其对应的盘块号应按下式进行计算：（不建议死记公式，做题以推理为主）
b=n×(i-1)+j,
(3) 修改位示图，令 map[i, j]=1

**2. 盘块的回收**
盘块的回收可分两步进行。
(1) 将回收盘块的盘块号转换成位示图中的行号和列号，转换公式为：
i=(b-1) DIV n+1,
j=(b-1) MOD n+1。
(2) 修改位示图，令 map[i, j]=0。

位示图法的主要优点是，从位示图中很容易找到一个或一组相邻接的空闲盘块。例如，当需要找到 6 个相邻接的空闲盘块时，只需在位示图中找出值连续为“0”的 6 个位即可。此外，由于位示图很小，占用空间少，因此可将它保存在内存中，这使得在每次进行盘块分配时无须首先把空闲盘块表读入内存，从而节省了许多磁盘启动操作。因此，位示图常用于微机和小型计算机中。

FAT管理的磁盘卷的结构

由 FAT 管理的磁盘卷被分成 5 个区域，分别是引导区、文件分配表 1、文件分配表 2、根目录区和数据存储区，说明如下。
① 引导区存放引导程序以及有关该卷的信息（如扇区数、扇区大小、FAT 的大小、簇的大小等）。
② FAT1 和 FAT2 是两个内容完全相同的文件分配表，其中一个是另一个的副本，以便必要时恢复。
③ 根目录区用于存放该卷根目录下各个文件或目录文件的目录项内容。
④ 数据存储区是具体存放文件以及子目录内容的区域。
随着磁盘容量的增大，在进行盘块分配时，不再以盘块而是以“簇”为基本单位，簇的大小一般为 $2^n$ 个盘块。在为文件分配存储空间时以“簇”为单位分配，一个磁盘上有多少簇，文件分配表就有多少个表项。
在文件的目录中只要保存分配给它的第一个簇的簇号，通过文件分配表的文件分配链，就能够得到该文件在数据存储区中占用的所有簇的簇号。

FAT 根据根目录表 FDT（File Directory Table）来完成对簇的管理。FDT 中的每个目录项用来记录一个文件或目录文件 FCB 的内容。
假设要访问根目录下的文件 keys，过程如下。
① 首先在根目录区找到文件 keys 的 FDT 项，得到文件 keys 的起始簇号为 2。
② 然后到 FAT 表中找到 2 号表项，其中的值说明下一个簇是 4。
③ 用 4 去查找分配表中对应于簇号为 4 的表项，里面存放的是 6。
④ 查找簇号为 6 的表项，6 号表项的值为 7，7 号表项里面存放的是 NULL，意味着文件 keys 的分配链到此结束。于是，得到文件 keys 在磁盘上占用的簇链及其簇的排列顺序为 2、4、6、7。

五、Linux 文件系统的结构（理解即可）
Linux 能支持多种类型的文件系统，如 ext2/3/4、FAT 等。每种文件系统都有自己的组织结构和文件操作函数，彼此差别很大。因此，Linux 通过虚拟文件系统 VFS（Virtual File System）来管理各种文件系统。
VFS 向用户提供一个统一的文件系统接口，屏蔽了各类文件系统的各种差异。在 Linux 系统中，用户程序在需要访问文件时，首先会调用文件系统提供的系统调用，如 open()、read()、write()、close()等，接着由 VFS 调用具体的文件系统。下图所示为在 VFS 中访问文件的过程。

[图片：VFS 访问文件的过程架构图。从上到下依次为：用户空间（应用程序）、系统调用、VFS、内核空间（包含 ext2, ext3, FAT, ..., NFS）、通用设备块、设备驱动程序、硬件（本地硬盘，网络存储）。]

Linux VFS 主要包含 4 种数据结构：（了解即可）
① 超级块对象（superblock object），表示整个文件系统。它包含管理文件系统所需的信息，包括文件系统名称（如 ext2）、文件系统的大小等。为了便于大家理解，这里学长给出一个高度简化的示例。
```c
// 超级块结构体
struct super_block {
    char fs_type[16];      // 文件系统类型（如"ext4"）
    int block_size;        // 块大小（如 4096 字节）
    ......
};
```
注：实际的操作系统非常复杂，为了便于理解，学长做了简化，仅供参考
② 索引节点对象（inode object），表示一个单独的文件。文件系统处理文件所需要的所有信息都保存在索引节点中，包括索引节点编号、文件大小、访问权限、修改日期、数据位置等。
③ 文件对象（file object），表示一个已打开的文件。
```c
// 文件结构体（表示进程打开的文件上下文）
struct file {
    struct inode *inode;   // 关联的 inode
    ......
    // 当前读写位置
    struct file_operations *f_op; // 文件操作函数表
};
```

④ 目录条目对象（dentry object），表示单个目录条目。

六、文件系统的注册与挂载（理解即可）
（一）文件系统的注册
Linux 在使用一个文件系统前，必须先对这个文件系统进行注册和安装。只有完成了该文件系统的注册，内核才能使用该文件系统的各种功能。
每个文件系统都有一个初始化例程，它的作用就是在 VFS 中注册，即填写一个 filesystem_type 数据结构，该数据结构定义如下：
```c
struct file_system_type{
    const char * name;          /* 文件系统的类型名，如 ext2 */
    int fs_flags;               /* 文件系统类型标志 */
    ......
};
```
（二）文件系统的挂载
在安装 Linux 时，磁盘上已经有一个分区安装了 ext2 文件系统，该系统被作为根文件系统在启动时自动安装。要使用一个其他的文件系统，除注册外，还需要安装这个文件系统。用户可以使用 mount 命令将需要的文件系统安装到整个文件系统树的某一个目录节点上。
```bash
mount -t iso9660 /dev/cdrom /mnt/cdrom
```
这个命令传递给内核三个部分的信息：文件系统的类型（ISO9660）；文件系统的物理块设备（/dev/cdrom）；文件系统要挂载在现存文件系统的目的地（/mnt/cdrom）。

七、外核（理解即可）
操作系统内核在硬件管理方面的一个重要功能是资源抽象。例如，传统的内核设计都对硬件做了抽象，把硬件资源或设备驱动程序都隐藏在硬件抽象层下。比方说，在这些系统中，如果分配一段物理存储，应用程序并不知道它的实际位置。

[图片：逻辑地址空间（页面0-3）通过MMU映射到物理内存（页框0-3）的示意图]

然而，操作系统所提供的硬件资源抽象是针对所有应用的通用抽象，这些抽象对一些具体的应用（如数据库、Web服务器等）来说往往不是最优的选择，在许多场景中，应用比操作系统更了解该如何去抽象和使用硬件资源。
为此，有研究者于1995年提出了外核（Exokernel）架构。外核本身非常的小，它通常只负责系统保护和系统资源复用相关的服务，允许用户程序的设计者决定硬件接口的设计。比如说，外核允许应用程序直接请求一块特定的物理空间，一块特定的磁盘块等等。系统本身只保证被请求的资源当前是空闲的，应用程序就允许直接访问它。
既然外核系统只提供了比较低级的硬件操作，而没有像其他系统一样提供高级的硬件抽象，那么就需要增加额外的运行库支持。这些运行库运行在外核之上，给用户程序提供了完整的功能。
下图为外核操作系统的架构。按照不同应用的要求，提供了不同的运行库，进而获得更高的性能。

[图片：外核操作系统架构图，上方为播放器（POSIX/IPC/VM）和Web服务器（POSIX/网络），中间为外核，下方为TLB、内存、存储、网络]

然而，外核架构的劣势在于：库通常是为某种应用定制的，缺乏跨场景的通用性。
