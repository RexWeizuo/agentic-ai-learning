（五）检索目录
当用户要访问一个已保存的文件时，系统首先会利用用户提供的文件名对目录进行查询，找出该文件的 FCB 或对应的索引节点。目前，对目录进行查询的方法主要有两种：线性检索法和 Hash 方法。
1.线性检索法
线性检索法又称为顺序检索法。在单级文件目录中，基于用户提供的文件名，可以利用顺序查找法直接从文件目录中找到指定文件的目录项。在树形目录中，用户提供的文件名是由多个文件分量名所组成的路径名，此时需对多级文件目录进行查找。
【例】假设用户指定的文件路径名是/usr/ast/mbox，则查找/usr/ast/mbox 的过程如下图所示。
[图片：展示根目录、132号盘块（/usr的目录）、496号盘块（/usr/ast的目录）的目录项表格]
上述查找过程具体说明如下。
首先，系统应读入第一个文件分量名 usr，用它与根目录文件（或当前目录文件）中各目录项中的文件名依次进行比较，从中找出匹配项，并得到匹配项的索引节点编号为 6；再从 6 号索引节点中得知 usr 目录文件放在 132 号盘块中，将该盘块内容读入内存。然后，系统读入路径名中的第二个分量名 ast，用它与放在 132 号盘块中的目录文件中各目录项的文件名依次进行比较，从中找到匹配项，并得知 ast 的目录文件放在 26 号索引节点中，再从 26 号索引节点中得知/usr/ast 存放在 496 号盘块中，将该盘块的内容读入内存。
最后，系统读入该文件的第三个分量名 mbox，用它与目录文件/usr/ast 中各目录项中的文件名依次进行比较，得知/usr/ast/mbox 的索引节点编号为 60，即在 60 号索引节点中存放了指定文件的物理地址。目录查询操作到此结束。如果在顺序查询过程中发现有一个文件分量名不能被找到，则应停止查询，并返回“文件未找到”信息。
2.Hash 方法
如果我们建立了一张 Hash 索引文件目录，则可利用 Hash 方法进行查询，即系统将用户提供的文件名变换为文件目录的索引值，再利用该索引值到目录中去查询，这样会显著提高检索速度。
注意，在现代 OS 中，通常会提供模式匹配功能，即在文件名中使用了通配符，如“*”“？”等。
```bash
leon@leon-virtual-machine:~/example_dir$ touch test1.txt test2.txt testA.log demo.txt
secret.doc
leon@leon-virtual-machine:~/example_dir$ ls test*
test1.txt test2.txt testA.log
leon@leon-virtual-machine:~/example_dir$ ls *.txt
demo.txt test1.txt test2.txt
```
对于使用了通配符的文件名，系统无法利用 Hash 方法查询目录，此时，系统还是需要利用线性查找法来查询目录。

四、内存映射文件（理解即可）
若采用 read()和 write()来读取磁盘文件，则每次文件访问都需要系统调用，read()系统调用的大致过程如下。（注：实际的 OS 非常复杂，为了便于理解，作了简化，仅供参考）
①用户程序调用 read()函数，CPU 从“用户态”切换到“内核态”，把控制权交给操作系统。
②操作系统先在内核缓冲区（页缓存，Page Cache）查找目标数据。如果缓冲区里没有，就指挥硬盘控制器（通常通过 DMA 机制）把数据从硬盘拷贝到内核缓冲区。
③此时数据还在内核中，用户程序无法直接访问。CPU 需要把数据从内核缓冲区拷贝用户缓冲区。拷贝完成后，read()函数执行完毕，CPU 从“内核态”切换回“用户态”，程序继续往下执行。

在上述过程中，第一次拷贝（磁盘→内核）由DMAC主导完成，CPU仅在传输初始化和结束处理时接入；第二次拷贝（内核→用户）需要CPU参与数据的搬运，开销过大。
倘若在读/写文件前，允许将文件映射到一段进程虚拟地址空间，然后就可以直接读/写该段虚地址进行文件访问，将文件I/O作为常规内存访问，不必通过文件I/O操作，这种方法称为内存映射（memory mapping）文件。其大致流程如下。
①用户程序调用mmap()函数。操作系统在虚拟地址空间中划分出一块连续区域，并记录下这块虚拟地址和磁盘文件之间的对应关系。此时，没有发生任何数据拷贝。程序拿到了一个指向这段虚拟地址的指针。
②用户程序像操作普通数组一样，通过指针去读取这块内存中的数据。当发现这块虚拟地址背后并没有对应的物理内存时，触发中断，陷入内核态。操作系统指挥硬盘控制器（DMA方式）把数据从硬盘读入到内核缓冲区中。
③操作系统将发生缺页的虚拟地址直接映射到内核缓冲区的物理地址上。缺页中断处理完毕，回到用户态，程序顺利读到了数据。

比起普通的文件读写操作，内存映射文件减少了数据拷贝，数据从硬盘加载到内核缓冲区后，用户程序就直接通过虚拟地址访问这块内存了，省去了从内核缓冲区复制到用户缓冲区的开销。此外，mmap将文件I/O转化为内存访问，减少了显式的read/write系统调用，简化了文件的访问和使用。

5.3 目录和内存映射文件

(1) MAP_SHARED (共享映射)
该方式下，多个进程映射同一文件时，一个进程的修改对其它进程可见，适用于进程间通信（IPC）。
(2) MAP_PRIVATE (私有映射)
该方式下，当进程首次写入某一页时，内核会复制一页副本给该进程，后续修改均在副本上进行，原文件和其他进程的映射不受影响。多个进程映射同一文件时，修改互不可见，各自独立。适用于需要私有工作区的场景。
因此，内存映射文件带来的好处主要是：
①使程序员的编程更简单，已建立映射的文件，只需按访问内存的方式进行读写。
②多个进程可以共享同一个映射区域，方便进程间通信。

五、管道（理解即可）
（一）基本原理
管道（Pipe）是两个进程间的一条通道，一端负责投递，另一端负责接收。
管道的行为和 FIFO 队列非常像。最早传入的数据会被最先读出来。当一个进程输入数据后，另一个进程可以通过管道读到数据。

[图片：管道示意图，左侧为写进程通过pipe[1]写入，中间为管道（包含数据3、数据2、数据1），右侧为读进程通过pipe[0]读取]

管道在 UNIX 系列的系统中会被当作一个文件，它的创建会返回一组（两个）文件描述符，例如 pipe[0] 和 pipe[1]。向管道（共享文件）提供输入的发送进程（即写进程），会以字节流形式将大量的数据送入管道；而接收管道输出的接收进程（即读进程），则会从管道中接收（读）数据。
PS：不妨回忆下文件的打开系统调用，fd=open(文件路径名，打开方式);

[图片：进程和系统打开文件表示意图。左侧展示了进程P1和进程P2的打开文件表，包含打开方式、读写指针和系统表指针。右侧展示了系统打开文件表，包含共享计数和FCB。箭头表示进程表项指向系统表项]

打开成功后，会返回一个 fd（文件描述符），指向进程打开文件表当中的相应表项，fd 可以是一个整数。
管道的目的在于通信，为了保证数据传输的高性能，实际上管道并不会使用存储设备，而是使用内存作为数据的一个缓冲区。
为了协调双方的通信，管道机制必须提供以下 3 方面的协调能力。
① 互斥，即当一个进程正在对管道执行读/写操作时，其他（另一）进程必须等待。
② 同步，即当写（输入）进程把一定数量（如 4KB）的数据写入管道后，便去睡眠（等待），直到读（输出）进程取走数据后，再把它唤醒。当读进程读一空管道时，也应睡眠（等待），直至写进程将数据写入管道后，再把它唤醒。
③ 确定对方是否存在，只有确定了对方已存在时，才能进行通信。

下面分别探讨普通管道和命名管道。
（二）普通管道（了解即可）
普通管道允许两个进程按标准的生产者-消费者方式进行通信：生产者向管道的一端（写入端）写，消费者从管道的另一端（读出端）读。
因此，普通管道是单向的，只允许单向通信。如果需要双向通信，那么就要采用两个管道，而每个管道向不同方向发送数据。
在 UNIX 系统上，普通管道的创建采用函数：pipe(int fd[])。这个函数创建一个管道，以便通过文件描述符 intfd[]来访问：fd[0]为管道的读出端，而 fd[1]为管道的写入端。
UNIX 将管道作为一种特殊类型的文件。因此，访问管道可以采用普通的系统调用 read()和 write()。
[图片：父进程通过fd[1]向管道写入，子进程通过fd[0]从管道读取]
注意，对于 UNIX 和 Windows 系统，采用普通管道的进程通信需要有父子关系。普通管道提供了一个简单机制，允许一对进程通信。然而，只有当进程相互通信时，普通管道才存在。
（三）命名管道（了解即可）
命名管道提供了一个更强大的通信工具。通信可以是双向的，并且父子关系不是必需的。当建立了一个命名管道后，多个进程都可用它通信。在一个典型的场景中，一个命名管道有几个写者。
当通信进程完成后，命名管道继续存在。虽然 UNIX 和 Windows 系统都支持命名管道，但是实现细节有很大不同。对于 UNIX，命名管道为 FIFO。虽然允许双向通信，只允许半双工传输。如果数据要在两个方向上传输，那么通常使用两个 FIFO。与 UNIX 系统相比，Windows 系统的命名管道允许全双工通信。
