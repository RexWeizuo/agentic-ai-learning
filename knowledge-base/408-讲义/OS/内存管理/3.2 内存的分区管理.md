# 3.2 内存的分区管理

为了能将用户程序装入内存，必须为它分配一定大小的内存空间。

[图片：磁盘文件指向内存，内存中标注有“空闲分区”]

连续分配存储管理方式（简称连续分配方式），是最早出现的一种存储器分配方式，该分配方式为一个用户程序分配一个连续的内存空间。  
连续分配方式可分为4类：单一连续分配、固定分区分配、动态分区分配、动态重定位分区分配。

一、单一连续分配（理解即可）  
在单道程序环境下，早期的存储器管理方式是把内存分为系统区和用户区两部分。

[图片：应用程序指向内存，内存分为“系统区”和“用户区”，标注有地址0和1000]

系统区仅供 OS 使用，它通常放在内存的低址部分。  
而在用户区中，仅装有一道用户程序，即整个内存的用户区由该程序独占。这样的存储器分配方式被称为单一连续分配。  
该方式的优点如下：  
①内存的分配简单，当运行一个用户程序时，从磁盘读入该程序，然后把它装入用户区即可。  
②内存的回收简单，当一个程序运行完之后，直接用新的程序去覆盖它即可，无需额外的管理开销。  
③不需要去记录在每一个时刻，内存空间的使用情况。  
这种方法比较适合单用户、单任务的操作系统。

二、固定分区分配（理解即可）  
20 世纪 60 年代出现了多道程序系统，为了能在内存中装入多道程序，且使这些程序之间不会发生相互干扰，于是将整个用户空间划分为若干个固定大小的区域（称为分区），并在每个分区中只装入一道作业，这样就形成了最早的，也是最简单的一种可运行多道程序的分区式存储管理方式。  
【例】如果在内存中有 4 个用户分区，便可以装入 4 道作业。

[图片：后备队列中的三个文件指向内存，内存分为“系统区”、“用户区1”、“用户区2”、“用户区3”、“用户区4”]

当有1个空闲分区时，便可从外存的后备作业队列中选择1个适当大小的作业装入该分区。当该作业结束时，又可从后备作业队列中选择另一作业装入该分区。  
倘若各个用户区的位置和大小一旦确定后就固定不变、不能再修改了，则这样的存储器分配方式被称为固定分区分配。  
【例】在系统刚刚启动时，由管理员手工划分出若干个分区，并确定每个分区的起始位置和大小等参数。然后，在系统的整个运行期间，这些参数就固定下来，不再改变。  
[图片：内存垂直分区示意图，从下至上依次为：操作系统（0-100K）、分区1（100K-200K）、分区2（200K-400K）、分区3（400K-700K）、分区4（700K-800K-1）]  
当某个分区空闲时，可以将其分配给某个作业。  
（一）分区划分  
在划分分区时，有如下两种不同的方法：  
1.分区大小相等  
所有用户分区大小相等，这种方式灵活性较差，程序较小则会产生内部碎片，程序较大则无法装入。所谓内部碎片，就是在进程所占用的分区内部，未被利用的空间。  
【例】假设操作系统的大小空间为100KB，其他4个分区的大小均为200KB。若某个作业仅需100KB的内存空间，则产生了一个100KB的内部碎片。  
[图片：内存垂直分区示意图，从下至上依次为：操作系统、分区1、分区2（内部包含作业2和内部碎片）、分区3、分区4]  
2.分区大小不等  
为了增加存储器分配的灵活性，还可将存储器划分为若干个大小不等的分区。  
最好能对常在该系统中运行的作业大小进行调查，根据用户的需要来划分。通常，可把内存划分成含有多个小分区、适量的中等分区及少量的大分区。这样，便可根据程序的大小，为之分配适当的分区。  
【例】假设操作系统的大小空间为100KB，分区1、分区2、分区3的大小均为100KB，分区4、5的大小为200KB，分区6的大小为300KB。  
[图片：内存水平分区示意图，从左至右依次为：OS、分区1、分区2、分区3、分区4、分区5、分区6，起始位置为0]  
（二）内存分配需要的数据结构  
为了便于内存分配，通常将分区按其大小进行排队，并为之建立一张分区使用表，其中包括每个分区的起始地址、大小及状态（是否已分配），如下图所示。

当有一用户程序要装入时，由内存分配程序依据用户程序的大小检索该表，从中找出一个能满足要求的、尚未分配的分区，将之分配给该程序，然后将该表项中的状态置为“已分配”。若未找到大小足够的分区，则拒绝为该用户程序分配内存。

固定分区分配，是最早出现的、可用于多道程序系统的存储管理方式，由于每个分区的大小固定，必然会造成存储空间的浪费，因而现在已很少将它用于通用的 OS 中。

三、动态分区分配（理解并记忆）

为了更好地解决固定分区存在的问题，人们又提出了动态分区分配存储管理技术。其基本思路为，分区不是预先划分好的固定区域，而是动态创建的，即装入一个程序时，系统将根据它的需求和内存空间的使用情况来决定是否分配。

【例】假设整个内存的大小为 1024KB，按字节编址。  
① 初始时，操作系统占用了内存地址最低端的一部分区域，剩下的空间成为一个完整的大空闲区。  
[图片：内存示意图，地址从 0 到 1024K-1。底部 0-128K 为操作系统，上方为空闲区]  
② 进程 1 进入内存，其大小为 320KB。因此就紧挨着操作系统，给它分配了一块大小为 320KB 的内存，即创建了一个新的用户分区。  
[图片：内存示意图，地址从 0 到 1024K-1。底部 0-128K 为操作系统，中间 128K-448K 为进程1，上方为空闲区]  
③ 随后进程 2 和进程 3 先后到来，同样给它们各分配一块内存空间，因此又创建了两个用户分区。

如果此时有一个大小为250KB的程序要进来，虽然总的空闲空间是够的，但每一个空闲区的容量都不够，因此就无法装入这个程序。  
若接下来的程序大小均大于64KB，则大小为64KB的空闲区始终无法利用，从而成为一个外碎片。  
外碎片是在各个被占用的分区之间，难以利用的一些空闲分区，这通常是一些比较小的空闲分区。  
因此，在动态分区中，由于每一个分区都是按需分配的，分区的大小正好等于程序的大小，避免了在固定分区当中由于分区的大小不当所造成的内碎片。  
在实现动态分区分配时，需要解决动态分区分配中所用的数据结构、动态分区分配算法以及分区的分配与回收操作这3方面的问题。

（二）动态分区分配算法  
为了把一个新作业装入内存，须按照一定的分配算法从空闲分区表或空闲分区链中选出一分区分配给该作业。常用的分区分配算法按分区检索方式可分为顺序分配算法和索引分配算法。  
1.基于顺序搜索的动态分区分配算法（顺序分配算法）

首次适应算法倾向于优先利用内存中低址部分的空闲分区，从而保留了高址部分的大空闲分区。这为以后到达的大作业分配大的内存空间创造了条件。其缺点是低址部分不断被划分，进而会留下许多难以利用的、很小的外碎片；此外，每次查找又都是从低址部分开始的，增加了查找可用空闲分区时的开销。

② 循环首次适应算法  
为了避免低址部分留下许多很小的空闲分区，减少查找空闲分区时的开销，提出了循环首次适应算法。  
循环首次适应算法在为作业分配内存空间时，不再是每次都从链首开始查找，而是从上次找到的空闲分区的下一个空闲分区开始查找，直至找到一个能满足要求的空闲分区，然后从中划出一块与请求的大小相等的内存空间分配给作业。

③ 最佳适应算法  
最佳适应算法是指将空闲分区按大小升序排列，依次查找，选择第一个可以容纳程序的空间分配出去。该方法选取最小的可以容纳程序的空闲分区进行分配。

优点：能留下更多大分区以满足大进程需求。  
缺点：  
(a) 该方式容易留下许多小空间分区，其外部碎片是这四种方法中最多的。  
(b) 该算法每次分配和回收后需要对空闲分区按分区大小重新排序，开销大。

④ 最坏适应算法  
最坏适应算法将空闲分区按从大到小排序，每次分配时，总是将最大的那个空闲区切去一部分，分配给请求者。

2. 基于索引搜索的动态分区分配算法（索引分配算法）  
   当系统很大时，系统中的内存分区可能会很多，空闲分区链就可能很长，这时采用顺序搜索分区方法可能会很慢。为了提高搜索空闲分区的速度，在大、中型系统中往往会采用基于索引搜索的动态分区分配算法，目前常用的有快速适应算法、伙伴系统和哈希算法。  
   这些算法的基本思想都是将空闲分区根据分区大小进行分类，对于每类（具有相同大小的）空闲分区，单独设立一个空闲分区链表，并设置一张索引表来管理这些空闲分区链表。  
   空闲分区的分类是根据进程常用的空间大小进行划分的，如 2KB、4KB、8KB 等，对于其他大小的分区，如 7KB 这样的空闲区，既可以放在 8KB 的链表中，也可以放在一个特殊的空闲区链表中。

① 快速适应算法（理解即可）  
快速适应算法在搜索可分配的空闲分区时分为两步  
(a) 根据进程的长度，从索引表中去寻找到能容纳它的最小空闲区链表  
(b) 从链表中取下第一块进行分配即可。  
该算法在进行空闲分区分配时，不会对任何分区产生分割，因此能保留大的分区，满足对大空间的需求，查找效率高。该算法在分配空闲分区时以进程为单位，一个分区只属于一个进程，因此在为进程所分配的一个分区中，或多或少地存在一定的浪费。  
② 伙伴系统

因此，伙伴算法的分配原则为，若某个进程申请大小为 XB 的内存区域，则首先计算一个 $i$ 的值，使得 ($2^{i-1} < X \le 2^i$)。接着，会在大小为 $2^i$ 的空闲分区链中查找。

1.分配内存

**四、动态重定位分区分配（理解即可）**

当一台计算机运行了一段时间后，它的内存空间将会被分割成许多小分区，而缺乏大的空闲空间。由于这些分区不相邻接，无法把大程序装入内存。

【例】下图给出的内存中有四个互不邻接的小分区，它们的容量分别为 10KB、30KB、14KB 和 26KB，总容量为 80KB。

[图片：内存分配示意图。从上到下依次为：os、用户程序1（10KB）、空闲区、用户程序3（30KB）、空闲区、用户程序6（14KB）、空闲区、用户程序9（26KB）。]

假设现在有一作业到达，要求获得 40KB 的内存空间，由于必须为它分配一连续空间，故此作业无法装入。若想把大作业装入内存，可采用的一种方法是：将内存中的所有作业进行移动，使它们全都相邻接。这样即可把原来分散的多个小分区拼接成一个大分区，这时就可以把一个作业装入该分区了。

[图片：内存紧凑后的示意图。从上到下依次为：os、用户程序1、用户程序3、用户程序6、用户程序9，下方是一个大的空闲区（80KB）。]

这种通过移动内存中作业的位置，把原来分散的多个小分区拼接成一个大分区的方法，称为“紧凑”，亦称为“紧缩”或“拼接”。但这种技术有两个问题：  
① 需要把所有的进程都移动一定的距离，这就需要大量的 CPU 时间。  
② 当各个进程被移动之后，还需要解决其程序里面的地址重定位问题，因为它们在内存中的地址都已经发生了变化，需要对移动了的程序或数据的地址进行修改，这样会大大影响系统的效率。因此，需要使用动态重定位解决此问题。

|**单一连续**|概述|系统将用户区整体分配给一个进程单独使用。|

||优点|根据进程的实际需要，动态地为之分配内存空间，灵活性高。|
||缺点|使用紧凑方式处理外部碎片需要较大的系统开销。|
