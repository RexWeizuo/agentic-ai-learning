# 第三章 内存管理 — 复习笔记

## 3.1 程序装入的基本概念

### 一、程序生命周期的三个阶段

```
源代码 → 编译 → 目标模块(.o) → 链接 → 装入模块 → 装入 → 内存中运行
```

**编译**：源程序 → 目标模块（机器码，但地址从 0 开始，未解析外部引用）
**链接**：多个目标模块 + 库 → 一个完整的装入模块，重新统一编址
**装入**：装入模块 → 内存，必要时进行地址转换

### 二、链接的三种方式

| 方式 | 时机 | 优点 | 缺点 |
|---|---|---|---|
| **静态链接** | 运行前 | 简单、运行时无额外开销 | 各进程各自拥有一份库副本，浪费内存；更新库需重新链接 |
| **装入时动态链接** | 装入内存时 | 便于修改更新、便于模块共享 | 边装入边链接，速度较慢 |
| **运行时动态链接** | 执行中用到时 | 节省内存（未用到的模块不装入）、共享最彻底 | 实现复杂，需要 OS 支持 |

关键对比：装入时 vs 运行时 —
- 装入时：全部模块在装入时完成链接（可能装入但从不运行的模块也被链接了）
- 运行时：用到哪个模块才链接哪个（如 Windows DLL、Linux .so）

### 三、地址转换（重定位）的三种方式

| 方式 | 时机 | 硬件支持 | 程序可移动 | 内部碎片 |
|---|---|---|---|---|
| **绝对装入** | 编译/装入时 | 不需要 | 不可移动 | 无 |
| **静态重定位** | 装入时一次性 | 不需要 | **不可移动**（地址已写死） | 无 |
| **动态重定位** | **执行时**逐条指令 | 需要（重定位寄存器） | **可移动** | 无 |

核心要点：
- 静态重定位：装入时把指令中所有逻辑地址改为物理地址，改完后程序不能再移动
- 动态重定位：指令中保持逻辑地址，CPU 执行时通过重定位寄存器动态转换，程序可在内存中移动（支持紧凑）

### 四、内存保护

目的：防止进程访问不属于自己的内存区域。
机制：**基址寄存器** + **限长寄存器**（或**上界寄存器** + **下界寄存器**）。
每次访存时，硬件自动检查地址是否越界，越界则触发**地址越界中断**。

## 3.2 内存的分区管理（连续分配）

### 一、分配方式演进

```
单一连续分配 → 固定分区分配 → 动态分区分配 → 伙伴系统 → 动态重定位分配
```

### 二、四种分配方式对比

| 方式 | 内部碎片 | 外部碎片 | 说明 |
|---|---|---|---|
| 单一连续分配 | 有 | 无 | 整个用户区给一个进程 |
| 固定分区分配 | **有** | 无 | 分区大小预先固定，进程很难刚好填满 |
| 动态分区分配 | 无 | **有** | 按需分配，回收后产生不连续空洞 |
| 伙伴系统 | **较多** | 较少 | 只能分配 2^k 大小，最坏浪费近一半 |

### 三、动态分区分配算法

| 算法 | 策略 | 特点 |
|---|---|---|
| 首次适应(FF) | 第一个够用的空闲区 | 简单快速，低地址端易产生小碎片 |
| 循环首次适应(NF) | 从上次位置开始找 | 使空闲区分布更均匀 |
| 最佳适应(BF) | 最接近请求大小的 | **最容易产生碎片**（切剩的太小，几乎无法再用） |
| 最坏适应(WF) | 最大的空闲区 | 剩余部分还较大，可继续利用 |

### 四、碎片问题

**内部碎片**：已分配给进程但进程用不到的部分（"给多了"）
**外部碎片**：空闲总量够但因不连续无法分配（"零钱够但凑不出整钱"）

**紧凑（Compaction）**：将已分配分区移动到一起，使分散的空闲区合并。需要动态重定位支持。

### 五、伙伴系统

- 分配单位必须是 2^k
- 合并时只能合并互为"伙伴"且**大小相等**的两个块
- 内部碎片较多（请求 70KB，实际分配 128KB）

## 3.3 内存的基本分页-分段管理（非连续分配）

### 一、分页 vs 分段

| | 分页 | 分段 |
|---|---|---|
| 用户可见 | **不可见**（系统行为） | **可见**（程序员按逻辑组织） |
| 地址空间 | **一维** | **二维**（段名+段内偏移） |
| 大小 | 固定（由系统决定） | 可变（由程序逻辑决定） |
| 碎片 | 内部碎片（最后一页） | 外部碎片 |
| 共享/保护 | 按页（不直观） | 按段（自然，如代码段只读） |

### 二、分页地址变换

虚拟地址 = 页号 | 页内偏移
物理地址 = 页框号 × 页面大小 + 页内偏移

**页表**：虚拟页号 → 物理页框号的映射表，每个进程一张。

### 三、快表（TLB / 联想存储器）

- 位于 CPU 内部的高速缓存，存放最近使用的页表项
- 命中时**一次访存**即可完成地址变换（不用查页表）
- 未命中时需访存查页表（多级页表需多次访存），然后更新 TLB

### 四、多级页表

- 目的：减少页表占用的连续内存
- 页目录（一级）常驻内存，二级页表按需分配
- 32 位系统、4KB 页面、两级页表：页目录 4KB，完整二级页表 4MB
- 实际进程只用一小部分，比单级页表（固定 4MB）节省大量内存

### 五、段页式

- 先分段（逻辑），段内再分页（物理）
- 地址变换：段号→查段表得页表地址→页号→查页表得页框号→拼接偏移
- 优点：保留段的逻辑意义 + 消除外部碎片
- 缺点：一次访存需 **3 次**访问内存（段表+页表+数据）

## 3.4 虚拟分页管理

### 一、局部性原理（虚拟存储的理论基础）

| 类型 | 含义 | 例子 |
|---|---|---|
| **时间局部性** | 刚访问过的数据不久可能再次访问 | 循环变量、函数递归 |
| **空间局部性** | 刚访问过的数据附近的数据可能被访问 | 数组遍历、顺序执行 |

### 二、虚拟存储的基本概念

- **逻辑上**扩充内存（不是物理上）
- 进程的部分页面在内存，其余在磁盘交换区
- 访问的页不在内存 → **缺页中断** → 从磁盘调入

### 三、请求分页 vs 预调页

- 请求分页：用到哪页才调入哪页
- 预调页：预计将要使用的页提前调入（利用空间局部性）

### 四、缺页处理流程

1. 陷入缺页中断
2. 检查有无空闲页框
3. 若有 → 直接调入缺失页
4. 若无 → 选 Victim 淘汰 → 若被修改过则写回磁盘 → 调入缺失页
5. 更新页表，置存在位=1
6. 重新执行被中断的指令

### 五、页面分配与置换策略

| 策略组合 | 页框数 | 置换范围 | 是否可行 |
|---|---|---|---|
| 固定分配 + 局部置换 | 固定 | 仅本进程 | ✓ |
| 可变分配 + 全局置换 | 可变 | 所有进程 | ✓ |
| 可变分配 + 局部置换 | 可变 | 仅本进程 | ✓ |
| 固定分配 + 全局置换 | — | — | **✗ 矛盾** |

### 六、写时复制（Copy-on-Write）

`fork()` 后父子进程共享物理页，标记只读。任一进程写时触发保护 → OS 复制一份独立的物理页。`fork()+exec()` 几乎不需要拷贝。

## 3.5 页面置换算法

### 一、算法对比

| 算法 | 策略 | Belady异常 | 实现 |
|---|---|---|---|
| OPT | 淘汰未来最远使用的页 | 无 | **不可实现**（需预知未来） |
| FIFO | 淘汰最先装入的页 | **有** | 简单（队列） |
| LRU | 淘汰最久未使用的页 | 无 | 需硬件支持（栈/寄存器） |
| CLOCK (NRU) | 循环扫描，A=0淘汰，A=1清0 | 无 | 较简单 |
| 改进CLOCK | 优先淘汰(0,0)，其次(0,1) | 无 | 较简单 |
| LFU | 淘汰访问次数最少的 | 无 | 需计数器 |

### 二、改进型 CLOCK 淘汰次序

```
第一轮：(0,0) — 既未访问也未修改，直接淘汰
第二轮：(0,1) — 未访问但被修改，淘汰前需写回
第三轮：(1,0) — 被访问但未修改
第四轮：(1,1) — 被访问且被修改
```

### 三、Belady 异常

- 现象：物理块增多 → 缺页次数反而增加
- 只有 **FIFO 及其变体**存在此异常
- LRU 和 OPT 是**栈算法**，不会出现

### 四、抖动（Thrashing）

- 原因：多道程序度过高，每个进程分配的页框数**小于工作集大小**
- 表现：系统大部分时间在换入换出，CPU 利用率急剧下降
- 解决：降低多道程序度，或增加物理内存

### 五、工作集模型

- 工作集 = 进程最近 Δ 次访问中使用的页面集合
- OS 确保每个进程分配的页框数 ≥ 工作集大小
- Δ 太小 → 工作集不完整，频繁缺页
- Δ 太大 → 工作集过大，降低多道程序度

### 六、页缓冲队列

- 存放被淘汰页的缓冲
- 被淘汰页先入缓冲，系统空闲时批量写回
- 若在缓冲中被再次访问，可直接恢复（无需从磁盘重读）
