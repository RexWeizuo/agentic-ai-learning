# 2.2 线程与进程通信

2.2 线程与进程间通信
一、进程间通信（理解即可）
（一）基本概念
假设内存中有 QQ 进程和 WPS 进程在并发执行，QQ 进程有些数据想通过 WPS 进程处理，需要将这些数据传递给 WPS 进程。
QQ 进程和 WPS 进程之间的信息交换就是一种进程间通信（Inter-Process Communication，IPC）。

[图片：内存空间示意图，展示了QQ进程和WPS进程在内存中的隔离，包含栈、数据段、代码段以及PCB，中间有红色叉号表示不可直接访问]

Q：QQ 进程能把数据直接给 WPS 进程吗？
R：Of course not! QQ 进程是不可以直接访问其他进程的内存空间的。如果 QQ 进程可以直接访问 WPS 进程的地盘，在里面使坏怎么办！当然，具体的原理要更复杂一些，学长放到了“补充”板块。
既然如此，就得想些方法，让 QQ 进程能把数据送给 WPS 进程，即实现两者之间的通信。
根据两者通信量的大小不同，又可以分为低级通信和高级通信。低级通信暂且理解为进程之间只能传递很少量的控制信息，比如几个字节或者一个数组。高级通信暂且理解为进程之间可以传送大量数据，比如一个文件。
聪明的计算机科学家们想到了很多种方法来解决通信问题，不过 408 考纲仅要求掌握共享内存、消息传递、管道、信号四种方式，下面分别介绍说明。

常见的数据结构包括数组、队列等，例如，如果采用共享数组（共享数据结构）的方式，则两个进程只能按照数组格式交换数据，即进程1把数据保存在一个数组中，进程2再去访问这个数组。另外，程序员还需要考虑通信过程中的一些细节问题，比如公用的数据结构该怎么设置。显然，这种传输方式限制多，传输效率低，属于低级通信。
如果采用共享存储区的方式，操作系统就直接在内存里划出一块大空地，这块空地可以被多个进程访问，并且想怎么访问怎么访问，不必非得使用某个数据结构（如数组），因此通信效率高，属于高级通信方式。当然，共享内存的实现原理比我刚刚讲的复杂得多，学长放到“补充”板块了。

（五）信号
上述的几种通信方式旨在实现进程间数据的传输。
然而有些时候，我们并不需要进行数据传输，而是需要通过一些简单的信号来告知进程某个事件。
信号在 Linux 系统中的使用非常广泛，一个简单的例子是用 Ctrl+C 在终端中终止一个执行中的进程。

[图片：终端运行截图，显示程序输出Hello, World!并被Ctrl+C中断]

这背后其实是内核给前台进程（当前正在占用终端、与用户交互的进程）发送了一个 SIGINT 信号，告知了“键盘输入 Ctrl+C”这个事件，随后根据信号类型进行相应处理。

[图片：内核发送SIGINT信号给前台进程示意图]

任何进程都有一个 PCB，PCB 往往用一个结构体类型实现，PCB 中的不同字段描述了不同的状态信息，其中就包括了信号相关的字段。

二、线程（理解并记忆）
（一）初识线程
静态的程序代码无法描述动态的程序执行过程，因此产生了进程的概念。既然已经有了进程，为何还要引入线程这个概念呢？

[图片：Tell me why 表情包]

我想，聪明的宝子们肯定已经猜到了。进程虽然解决了一些问题，但是仍有一些缺点，无法满足我们的某些需求，所以才要提出“线程”的概念来补充和完善。
那进程到底哪里做得不好，以至于要创造一个新的概念呢，下面举例说明。
【例】假设某个进程 A 的程序代码如下所示：
```c
void main(){
    while (true)
    {
        Read();//负责读取外存数据
        Compute();//负责计算
    }
}
```
Read()函数负责不断读取外存数据，Compute()函数负责不断计算，Read()函数的代码逻辑如下所示：
```c
Read(){
    //设置读取外存数据的相关参数，包括外存地址等
    //申请读取文件的系统调用
    //检查读取结果是否正确
}
```
由于 Read 函数需要进行输入输出操作，因此每次执行 Read()函数时，进程 A 都会被设置为阻塞态，切换其他进程上 CPU 执行，这会导致不少时间浪费在进程切换上！

[图片：进程A在主存中的布局示意图，展示了外存、主存（包含进程A代码段、PCB）、CPU之间的关系]

因此，希望 Read 函数申请 I/O 操作时，能不能不进行进程切换，而是直接跳转到 Compute 函数上执行，这样，就可以在不发生进程切换的情况下，使得 CPU 和外存并行工作，即让一个进程的各个函数之间也可以并发执行！

然而，在以进程作为基本运行单位的操作系统当中，单个进程只能串行执行它的程序代码，没法实现各个函数的并发执行！

Q：有的宝子可能会想，学长，进程之间是可以并发运行的，可以分别编写两个程序，第一个程序用来读取数据，第二个程序用来计算，然后启动这两个程序，每个程序对应于一个进程，这样行不？

R：达咩达咩！这样处理的话，虽然可以利用进程间并发运行的特点，让读取操作和计算操作并行，但是频繁的进程切换可能使得计算机花费过多的时间在进程切换上，导致整体效率变低！

[图片：表情包“达咩”]

因此，“进程”这个概念已经不够用了！需要有一种新事物，这种事物必须能让同一个进程的各个函数之间并发运行，就像各个进程之间并发运行一样。这个新事物是谁呢，就是大名鼎鼎的“线程”啦～

线程可以简单理解为进程的一条执行流程，例如，进程 A 的两个函数可以看作是两条执行流程，即两个线程。可是，为什么这两个函数无法并发执行，一定要使用线程呢？别着急客官，咱们往下看～

显然，当 Read 函数执行了申请读取文件的系统调用后，会触发内核程序接管 CPU，不管是切换到其他进程还是切换到 Compute()，都需要保存 Read()函数的运行上下文，例如 CPU 中的通用寄存器的值。如果是进程切换，可以把运行上下文保存到进程 A 的 PCB 中。

Q：可是如果不进行进程切换，那这些信息应该保存在哪呢？

有些聪明的宝子也许想到，学长，可以保存在进程 A 的栈里～达咩达咩！进程 A 的栈可以保存它自己的不同子程序（函数）之间的调用信息（包括局部变量、调用参数、返回地址等），但是现在 Read()函数并没有调用 Compute()函数～

下面详细解释线程的作用以及定义。

我们通过 PCB 创建了进程，PCB 保存了进程执行过程中的所有状态信息。现在也可以给每一个线程创建一个线程控制块（Thread Control Block, TCB），用于控制和管理线程的状态、保存线程的信息。

TCB 的内容主要包括两个部分：
① 线程自身的管理信息，如线程标识符（通常是一个整数，它唯一地标明了某个线程）、线程状态、调度信息等。
② 线程所独享的资源，包括 CPU 寄存器的值和栈指针。

【例】可以用一个名为 thread_struct 的结构体类型来描述 TCB。

```c
struct thread_struct
{
    int tid;      //线程标识符(TID)。每个线程的唯一标识符
```

不过用户线程的实现机制有一个比较大的缺点：对于一个进程中的各个线程而言，如果其中有一个线程发起系统调用而引起阻塞，那么整个进程都无法运行。因为在操作系统眼中，调度的基本单位是进程，操作系统只知道进程，而不知道进程内部有多少个线程。

这实际上是一个比较严重的问题，因为引入线程的目的就是增加并发性，使得各个线程可以在同一个资源平台上并发地运行。而如果像现在这样，一个线程被阻塞了，就会影响到其他的线程，使别的线程也无法运行，这样就使线程之间的并发性打了一个很大的折扣。因此，现在大部分的系统都不采用这种方式。

2.执行信号的默认处理程序
系统给每个信号定义了默认操作，例如，进程收到 SIGINT 信号后会终止。

```c
int count = 0;// 用于记录程序运行的时间
// SIGINT 信号处理函数
void handle(int sigNum) {
    // 注意：虽然处理了 SIGINT 信号，但程序不会终止
    printf("cann't stop! count: %d, sigNum: %d\n", count, sigNum);

int main() {
    // 注册信号处理：将 SIGINT(中断信号)与 handle 函数绑定
    signal(SIGINT, handle);
    // 主循环：每秒增加计数器值
    while (1) {
        count++;          // 计数器自增
        sleep(1);         // 休眠 1 秒
    }
    return 0;
}
```

1.通过按键（键盘）发送信号，例如，按下 Ctrl+C 后会导致内核发送一个 SIGINT 信号给前台进程。
2.程序异常时发送信号终止进程，例如，执行非法指令时会发送 SIGILL 信号。
