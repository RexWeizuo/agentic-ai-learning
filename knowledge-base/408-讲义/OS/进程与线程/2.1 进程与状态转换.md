2.1 进程与状态转换

（一）动态性
进程是一个正在运行的程序，在程序的运行过程中，它的状态是在不断变化的。例如，一个程序在运行过程中，每执行一条指令，CPU 中那些通用寄存器的值都会发生变化，程序计数器的值也在变化，每次都会指向下一条即将执行的指令。

[图片：CPU寄存器示意图，包含AX 999, CX 888, BX 666, DX 000, PC 850, PSW ZF=0...]
CPU

（二）独立性
在传统的 OS 中，独立性是指进程是一个能独立运行、独立获得资源和独立接受调度的基本单位。凡是未建立 PCB 的程序都不能作为一个独立的单位参与运行。

（三）并发性
并发性指多个进程同存于内存中，且能在一段时间内交替运行。

（四）异步性
异步性是指进程是按异步方式运行的，即按各自独立的、不可预知的速度向前推进（1.3 讲过），会产生其结果的不可再现性。
【例】进程 1 和 2 都访问某个某个共享变量 a，倘若进程 1 最后执行 a=5，进程 2 最后执行 a=3，则 a 的最后结果取决于谁最后执行完。
为了使进程在并发运行时虽有异步性，但结果仍是可再现的，在 OS 中配置了同步机制（后面深入学习）。

【练习】下列关于并发进程特性的叙述中，正确的是（ ）。
A.进程是一个动态过程，其生命周期是连续的
B.并发进程执行完毕后，一定能够得到相同的结果
C.并发进程对共享变量的操作结果与执行速度无关
D.并发进程的运行结果具有不可再现性
答案：D

当然，并不是所有的程序都能全身而退，有的程序在运行时，可能会发现严重的错误。
例如，该进程在运行时需要用到某个文件，但是当它试图打开这个文件时，发现该文件已经被删掉了，无法打开，这时，这个进程就无法再继续执行下去，只好退出。
2.致命错误：在进程的执行过程中，由于程序设计的缺陷，造成了一些致命的错误。

这种情形下，进程自己并不知道犯了错误，所以也就不会主动退出，而是由操作系统发现了错误以后，把它强制性退出。

四、进程的状态与转换（理解并记忆）
（一）进程的状态
多道程序下，一个进程并不总是一直占用 CPU 运行，有时它也可能会待在旁边休息。那么如何来描述一个进程的当前状态呢？
一般来说，当一个进程被创建时，它的状态为创建态；当一个进程被创建完毕，它的状态为就绪态；当一个进程使用 CPU 执行时，它的状态为执行态；当一个进程因为使用打印机等 I/O 设备而暂停执行时，它的状态为阻塞态；当一个进程运行结束，它的状态为终止态。

运行状态，是指进程占有 CPU，并且在 CPU 上运行。显然，处于这种状态的进程数目必须小于或等于 CPU 的数目。如果在计算机系统中只有一个 CPU，那么在任何时刻，最多只能有一个进程处于运行状态。

[图片：主存、CPU 与 I/O 设备关系示意图]

就绪状态，是指进程已经具备了运行的条件，但是由于 CPU 正忙，正在运行其他的进程，所以暂时不能运行。不过，只要把 CPU 分给它，它就能够立刻执行，即“万事俱备，只欠 CPU”。

阻塞状态，也称为等待状态，是指进程因为等待某种事件的发生而暂时不能运行的状态，例如，它正在等待某个输入输出操作的完成。在这种情形下，即使 CPU 已经空闲下来了，这个进程也还是不能运行。

【例】若进程 1 的程序代码如下所示，进程 1 在执行时需要用户输入一个整数，显然 CPU 是不可能浪费时间等待用户输入的，因此在通过 scanf()进行系统调用的过程中，进程 1 会被设置为阻塞态。

```c
#include <stdio.h>
int main() {
  int number;
  int square;
  // 提示用户输入一个整数
  printf("请输入一个整数: ");
  scanf("%d", &number); // 读取用户输入的整数
  // 计算平方
  square = number * number;
  // 输出结果
  printf("%d 的平方是: %d\n", number, square);
  return 0;
}
```

**2.转换 2：从运行态转换成就绪态**

在系统中有很多进程，大家轮流去执行，而这项工作主要是由 CPU 的调度程序来完成的。

【例】当一个进程在 CPU 上正常运行时，如果操作系统的调度程序认为它已经连续运行了足够长的时间，就会暂停它的运行，把它从运行状态变为就绪状态，然后再调度其他进程来运行。这样每个进程都能得到运行的机会，因此比较公平。

**3.转换 3：从就绪态转换成运行态**

这个转换也是由 CPU 调度程序来完成的，即如果一个进程已经在就绪状态等待了足够长的时间，而别的进程都已经轮流使用过 CPU 了，那么调度程序就会选中它，重新占用 CPU 去运行。

**4.转换 4：从阻塞态转换成就绪态**

一个进程处于阻塞状态的原因是它正在等待某个事件的发生，如等待数据的输入，那么当这个事件发生以后，该进程就已经具备了继续运行的条件，所以操作系统就会把它从阻塞状态转换成就绪状态。

如果进程被转换成就绪态后没有其他就绪态进程，那么该进程就会立即占用 CPU 去运行，即又从就绪状态转换成运行状态。如果当前还有其他优先级更高的就绪态进程，那么该进程就需要在就绪状

五、状态队列（理解即可）
有些进程处于就绪状态，有些进程还处于阻塞状态，它们阻塞的原因可能各不相同。
为了有效地管理进程，有些 OS 会把各个进程的 PCB 按照它们的状态组织成一些状态队列。具体来说，由操作系统来维护一组队列，用来表示系统中所有进程的当前状态，不同的状态分别用不同的队列来表示。
【例】下图展示了单 CPU 下系统状态队列。

[图片：展示了就绪队列、磁带阻塞队列、磁盘阻塞队列、信号量S阻塞队列的结构。就绪队列包含PCB4->PCB2->NULL；磁带阻塞队列为NULL；磁盘阻塞队列包含PCB3->PCB1->PCB6->NULL；信号量S阻塞队列包含PCB5->NULL]

运行队列由处于运行状态的进程构成，单 CPU 下，任何一个时刻最多只能有一个进程在 CPU 中运行，可以不设置运行队列。
就绪队列由处于就绪状态的进程构成。
阻塞队列由处于阻塞状态的进程构成，根据它们阻塞的原因可以进一步细分成不同阻塞队列。
当一个进程的状态发生变化时，例如，从运行状态变成就绪状态，或者从阻塞状态变成就绪状态，就把它的 PCB 从一个状态队列中脱离出来，加入另外一个队列中。
例如，假设在某个时刻，进程 PCB3 对磁盘的输入输出操作已经完成，那么它的状态就要发生变化，从阻塞状态转换成就绪状态。所以要把它的 PCB 从阻塞队列中摘下来，挂到就绪队列中去。

六、进程控制块（理解即可）
前面对进程控制块已经做了简单的介绍，这里再来细化下。系统为每一个进程都建立了一个 PCB，用来保存与该进程有关的各种状态信息。不同 OS 的叫法有所不同，例如，在 Linux 系统中，它叫做“任务结构体（task_struct）”。在一个典型的操作系统中，一个 PCB 主要包含以下四个方面的信息（不唯一）：
（一）进程标识符
进程标识符用于唯一地标识一个进程。一个进程通常有两种标识符：
1. 外部标识符
为了方便用户（进程）对进程的访问，须为每一个进程设置一个外部标识符。它是由创建者提供的，通常由字母、数字组成。
2. 内部标识符
为了方便系统对进程的使用，在 OS 中又为进程设置了内部标识符即赋予每一个进程一个唯一的数字标识符，它通常是一个进程的序号。例如，可以使用如下的命令来查看当前系统的进程信息。

```bash

root         1  0.0  0.2 166608 1...
root         2  0.0  0.0      0
```

（二）处理机状态
处理机状态信息也称为处理机的上下文，主要是由处理机的各种寄存器中的内容组成的，包括通用寄存器、指令计数器、程序状态字 PSW 等。

（三）进程调度信息
在 OS 进行调度时，必须了解进程的状态及有关进程调度的信息，这些信息包括：
① 进程状态，指明进程的当前状态。
 进程优先级，是描述进程使用处理机的优先级别的一个整数，优先级高的进程应优先获得处理机。
【例】若进程 1 和进程 2 的优先级分别为 5 和 3，数字越大优先级越高，则进程 1 的优先级高于进程 3。
③ 进程调度所需的其他信息，比如，进程已等待 CPU 的时间总和、进程已执行的时间总和等（后面学习）。
④ 事件，是指进程由执行状态转变为阻塞状态所等待发生的事件，即阻塞原因。
（四）进程控制信息
进程控制信息指用于进程控制所必需的信息，它包括：
① 程序（代码段）和数据（数据段）的地址，以便再调度到该进程执行时，能从 PCB 中找到其程序和数据。

七、使用原语控制进程（理解即可）
OS 对进程的具体控制往往用原语完成。
原语指由若干条指令组成的，用于完成一定功能的一个过程（函数）。
[图片：指令序列示意图，包含指令1到指令4]
原语的主要特点为“原子操作”，即一个操作中的所有动作要么全做，要么全不做。
因此，原语在执行过程中不允许被中断。原子操作在系统态下执行，常驻内存。
（一）进程的创建
在系统中每当出现了创建新进程的请求后，OS 便调用进程创建原语 Creat 按下述步骤创建一个新进程：
① 申请空白 PCB，为新进程申请获得唯一的数字标识符。
② 为该新进程分配所需资源，包括各类资源，如内存、文件、I/O 设备等。
③ 初始化 PCB 上的内容，包括进程标识符、父进程标识符、处理器状态信息和处理器控制信息。
④ 若上述过程顺利，将该新进程状态设置为就绪态，放入就绪队列，等待调度；若不顺利，例如资源不足时，则创建该新进程的父进程置为阻塞态，新进程仍处于创建状态。

**（二）进程的终止**
如果系统中发生了要求终止进程的某事件，OS 便调用进程终止原语，按下述过程去终止指定的进程：
① 根据进程的进程号（PID），找到进程对应的 PCB，获取进程的状态。
② 修改该进程的状态，置为终止态。
③ 若该进程还有子孙进程，还应将其所有子孙进程也都予以终止，以防它们成为不可控的进程（有些 OS 无此要求）。
④ 将被终止进程所拥有的全部资源或者归还给其父进程，或者归还给系统。
⑤ 将被终止进程（PCB）从所在队列中移出，等待其他程序来搜集信息。

**（三）进程的阻塞和唤醒**
**1. 阻塞**
当进程请求资源失败（资源不足、资源未到达等）或等待某个事件发生时，该进程无法继续执行，便会调用阻塞原语，让出处理器资源，变为阻塞态。

正在执行的进程，如果发生了上述某事件，进程便通过调用阻塞原语 block 将自己阻塞。
【例】进程 B 使申请打印机时，打印机在为 A 服务，因此进程 B 调用 block 原语将自己阻塞。
可见，阻塞是进程自身的一种主动行为，即由进程自己调用阻塞原语来自我阻塞，因此进程只能从运行态转为阻塞态，不能由就绪态转为阻塞态。
进入 block 过程后，由于该进程还处于执行状态，所以应先立即停止执行，把进程控制块中的现行状态由“执行”改为阻塞，并将 PCB 插入相应事件的阻塞队列。

最后，转调度程序进行重新调度，将处理机分配给另一就绪进程。

当被阻塞进程所期待的事件发生时，比如它所启动的 I/O 操作已完成，或其申请的资源已经释放，则由有关进程调用唤醒原语 wakeup，将等待该事件的进程唤醒。

估计很多宝子都感觉困惑，学长，进程1的地址空间里，有自己的代码和数据不就行了，为什么要装操作系统内核的代码和数据？这个问题确实比较难解释，下面举例说明。
假设进程1在执行的过程中需要进行系统调用，即通过陷入指令跳转到中断服务程序，再去调用相关的系统服务。那在中断响应的过程中就需要修改PC的值，让它指向中断服务程序。注意，这里PC给出的地址都是虚拟地址，即进程1的虚拟地址空间中的地址。既然如此，只有中断服务程序也在这个虚拟地址空间中，才可以给出它对应的虚拟地址。
另外，也有一个常见的误区，就是误认为操作系统是一个独立的进程，其实不是的！系统内核并不一定以独立进程的形式存在！每一个进程的地址空间中其实都包含了内核，当发生中断或系统调用时，就会跳转到内核程序执行！
例如，Linux32位操作系统中，每个进程的地址范围从0x0000 0000到0xFFFF FFFF，按字节编址下，总共4GB的地址空间。

[图片：Linux 32位系统地址空间划分，分为1GB内核空间和3GB用户空间]

地址空间会被划分为两部分，一部分为内核空间，一部分为用户空间。
其中，最高的1G字节（从地址0xC000 0000到0xFFFF FFFF）供内核使用，存放内核的代码和数据，称为内核空间。较低的3G字节（从地址0x0000 0000到0xBFFF FFFF）存放用户的代码和数据，称为用户空间。
任何一个进程在执行自己的用户程序时（位于用户空间），无法直接访问内核空间的内容，但可以通过系统调用的方式间接访问内核空间，进而保护了内核的代码以及数据。
PS：有些宝子可能会想，学长，通过系统调用的方式不还是可以访问内核数据吗，这怎么保护内核？
注意，内核程序是OS工程师们写好了的，用户进程只能调用固定好的程序，这些程序的输入参数、代码逻辑和返回参数都是确定了的，即使你想偷偷使坏，也只能伪装成正常的参数提交给内核，不过大部分情况下内核都可以检测出来，然后击毙你！
另外，内核空间里有大量代码和数据是被所有进程所共享的。

【补充】进程切换模型（理解并记忆）
进程切换，指操作系统内核暂停当前运行的进程，调度某个就绪进程占用处理器运行。
下面通过一个简单的例子讲解进程切换的大体过程。
【例】假设有两个进程P0和P1。
① 在刚开始的时候，进程P0处于运行状态，占用了CPU。
② 当运行到某个时刻，出现了一个中断或系统调用。
例如，进程P0需要对输入输出设备进行读写，或者是它的时间片用完了，产生了一个时钟中断。
③ 这时，进程P0就不能再运行下去，而是把控制权交给了操作系统。
④ 操作系统内核会根据中断类型或系统调用参数调用相关的服务，并判断是否该让P0继续执行。
例如，如果这是一个时钟中断，说明该进程的时间片已经用完，因此应将P0的状态就从运行变为

--- Page 62 ---
就绪。如果这是一个输入输出操作，那么进程的状态就从运行变成阻塞。
⑤ 若内核认为 P0 不应该继续占用 CPU，则会把它的状态信息保存到 PCB0 中，将其状态修改为就绪或阻塞，并将 PCB0 插入到相应队列中。

[图片：进程切换流程图。左侧为进程P0，从“运行”状态经过“中断或系统调用”进入“特殊的运行状态”，最终变为“就绪或阻塞”。中间为操作系统，处理“中断或系统调用”，执行“中断处理或系统服务”，接着“修改P0状态，状态信息保存到PCB0”，然后“调度程序选中P1”，最后“从PCB1读入状态信息，修改P1状态”。右侧为进程P1，从“就绪”状态变为“运行”状态。]

⑥ 接着，内核会调用它的 CPU 调度程序，从所有处于就绪态的进程中选择一个去运行。
假设选中了进程 P1，操作系统就会从进程 P1 的 PCB 中读取它的所有状态信息，随后将这些信息装入到相应的硬件环境中，即恢复进程 P1 在上次运行中止时的现场，然后让它开始运行。
⑦ 这样，进程 P1 就从原来的就绪态转换成运行态。
因此，在多道程序系统中，系统内核也在内存中，它时不时会出来运行一下，它上处理机的方法就是中断，进程之间的切换由系统内核来完成，所以进程切换必然发生在内核态而非用户态。
当一个进程在运行时，如果由于中断等原因跳转到系统内核的代码，此时可能会发生进程的切换，也可能不发生。例如，若 P0 申请的系统调用服务执行时间很短，则内核程序执行完会继续执行进程 P0，不进行进程切换，毕竟进程切换的开销还是挺大的！
