    # 一、基本概念

    一、基本概念（理解即可）
    死锁指一组并发进程中的每一个进程都等待其余进程中的某个进程拥有的资源，且这些并发进程在得到自己需要的资源之前不会释放自己拥有的资源，从而造成所有并发进程都想得到资源而又都得不到资源，均不能继续向前推进的状态。

    [图片：两只猫分别拿着筷子和碗，对话框显示“我要吃饭！”]

    死锁、饥饿、死循环都是进程无法顺利向前推进的现象，但有以下区别：
    ① 死锁是 n 个进程 (n≥2) 循环等待对方手里的资源，导致 n 个进程都被阻塞，均无法向前推进的现象。
    ② 饥饿是进程由于长期未获得想要的资源，从而无法向前推进的现象。
    “资源”可以是 CPU——饥饿进程长期处于就绪态，也可以是 I/O 设备——饥饿进程长期处于阻塞态。
    ③ 死循环是进程在执行过程中，一直进行某个循环，而不跳出的现象。
    ④ 死锁和饥饿是由于资源的不合理分配引起，而死循环是由于程序的设计错误引起。

    [图片：内存分配示意图，显示主存中有不同进程占用不同内存块，旁边有CPU]

    # 四、死锁预防

    （一）死锁预防
    死锁预防是一种静态的死锁解决方法。在产生死锁的四个必要条件均满足时，系统才可能进入死锁状态，因此可以通过破坏四个必要条件中的一个或多个来防止死锁的发生。
    1.破坏互斥条件

    ### 🧠 深度理解：SPOOLing 如何破坏互斥条件（来自 2.5 DS）

    > **Q：**  多个进程共享临界资源必须保证互斥访问，反过来说如果只有一个进程访问资源就没必要保证互斥访问。SPOOLing 将打印机变成逻辑共享设备是怎么做到破坏互斥的？
    >
    > **A：**  安排一个唯一的"打印管理进程"当代理，其他进程不直接碰打印机，只把数据写到磁盘缓冲区。真正访问打印机的只有那个代理，只有一个进程在用，互斥条件形同虚设。这是通过**绕开互斥**来预防死锁——不是去掉互斥，而是让互斥场景根本不存在。
    >

    > **Q：**  说白了就是加了一个代理协调，代理给打印机请求排队是吗？
    >
    > **A：**  对。不加代理时多进程直接抢打印机→拿到就用、拿不到就阻塞。加代理后进程只管把打印任务塞进磁盘队列，打印管理进程按顺序逐个打印。变成了"排队叫号"模式，进程提交任务后继续干别的事，不阻塞。
    >

    多个进程共享临界资源必须保证互斥访问，反过来说，如果只有一个进程访问临界资源就没必要保证互斥访问。对于某些资源，可以为其建立一个专门的进程，用来代理其他进程对这种资源的访问。其他进程想要访问这种资源，就必须告知代理进程。访问这类资源的进程只有代理进程一个，自然不会产生死锁。例如，SPOOLing 技术就能将打印机改为逻辑上的共享设备，达到死锁预防的效果。

    [图片：P1、P2请求打印机，以及通过代理进程请求逻辑打印机的流程图]

    这一方式的缺点在于条件严苛，只有少数资源（如打印机）适合这一方式。因此，预防死锁主要是破坏产生死锁的后三个条件。
    2.破坏请求和保持条件

    ### 🧠 深度理解：第二种协议如何破坏请求与保持条件（来自 2.5 DS）

    > **Q：**  第二种协议中"进程只须请求磁带机、磁盘即可运行，数据复制完释放掉再去请求打印机"——复制数据不还需要磁带机吗，怎么做到提前释放的？
    >
    > **A：**  请求磁带机拿到控制权后，CPU 把磁带上的数据搬到磁盘，这个"复制"用的是 **CPU** 而不是磁带机（磁带机只负责输出数据流）。复制完成后磁带机就空闲了，可以释放它再去请求打印机。关键是把"持有一资源等另一资源"的长周期拆成了短周期操作，减少死锁窗口。
    >

    > **Q：**  所以进程并不是在等，而是在执行 CPU 的复制逻辑？
    >
    > **A：**  对。请求资源只是拿控制权（你拿到车的钥匙），使用资源才是真正干活（你开车上路）。拿到磁带机控制权后 CPU 执行复制指令，进程确实在用 CPU 干活，不是空等。只有请求不到资源时才被阻塞等待。
    >

    > **Q：**  请求阶段可能等（磁盘不空闲），数据复制时进程也在等（等 I/O 完成），两个阶段不都是在等不是在干活吗？
    >
    > **A：**  两个"等"本质不同：请求阶段等的是**资源被别人占用**——你手里没货，不释放你就进不去，这叫阻塞等待；复制阶段等的是 **I/O 设备完成传输**——你手里已经拿着资源了，只是活还没干完，这叫忙等。会导致死锁的是"攥着资源等别人手里的资源"，第二种协议让你先放手再等，避免"手里攥着不放"的局面。
    >

    为了能够破坏“请求和保持”条件，系统必须保证做到：当一个进程在请求资源时，它不能持有不可抢占资源。该保证可通过以下两个不同的协议实现。
    （1）第一种协议 （开始之前就申请好全部资源 一种不满足就等待）
    该协议规定，所有进程在开始运行之前，必须一次性地申请其在整个运行过程中所需的全部资源。这样，该进程在整个运行期间便不会再提出资源要求，从而破坏了“请求”条件。
    系统在分配资源时，只要有一种资源不能满足进程的要求，则即使所需的其他资源都空闲也不分配给该进程，而是让其等待。由于该进程在等待期间未占有任何资源，从而破坏了“保持”条件。

    [图片：硬盘和打印机]

    第一种协议的优点是简单、易行、安全。但缺点也极其明显。
    ① 资源被严重浪费，严重地降低了资源利用率。进程在开始运行时，就会一次性地占用整个运行过程所需的全部资源，其中有些资源可能仅在运行初期或运行快结束时才会使用。
    ② 进程经常会发生饥饿现象。因为仅当进程在获得其所需的全部资源后才能开始运行，所以可能由于个别资源长期被其他进程占用，等待该资源的进程迟迟不能开始运行。
    (2) 第二种协议 （只申请初期需要资源 边运行边释放边继续申请）
    该协议是对第一种协议的改进，它允许一个进程只获得运行初期所需的资源后，便开始运行。进程运行过程中再逐步释放已分配给自己的且已用毕的全部资源，然后再请求新的所需资源。
    下面通过一个具体例子来说明第二种协议比第一种协议要好。
    【例】例如有一个进程，它所要完成的任务是，先将数据从磁带上复制到磁盘上，然后对数据进行处理，最后把结果打印出来。

    [图片：数据从磁带复制到磁盘，再复制到打印机]

    攥着资源等 vs 空手等

    ① 在采用第一种协议时，进程必须在开始时就请求磁带机、磁盘和打印机，然而打印机仅在最后才会用到，所以刚开始就请求占用打印机，既影响其利用率，还会影响其他进程的运行。此外，若刚开始时磁带机和磁盘空闲，但因打印机被很多进程所需而分配给了其他进程，则该进程还需要等待。
    ② 在采用第二种协议时，进程在开始时只须请求磁带机、磁盘，然后即可运行。等到磁带上的全部数据已复制到磁盘中并已处理好后，便可将磁带机和磁盘释放掉，再去请求打印机。这不仅能使进程更快地完成任务，提高设备的利用率，还可减少进程发生饥饿现象的概率。
    3.破坏不可抢占条件（资源申请失败，就释放已保持的左右资源）
    为了能破坏“不可抢占”条件，协议中规定，当一个已经保持了某些不可抢占资源的进程提出新的资源请求而不能得到满足时，它必须释放已经保持的所有资源，待以后需要时再重新申请。这意味着进程已占有的资源会被暂时释放，从而破坏了“不可抢占”条件。
    该方法实现起来比较复杂，且需要付出很大的代价。因为一个不可抢占的资源（如打印机、刻录机等）在使用一段时间后被抢占，可能会造成进程前一阶段工作的失效。

    [图片：光盘/刻录机]

    4.破坏循环等待条件
    一个能保证“循环等待”条件不成立的方法是，对系统的所有资源进行编号，并要求进程只能按资源编号严格递增或递减的顺序请求资源。
    例如，进程在申请资源时，如果一个资源的编号小于进程已获得的某个资源的编号，则进程不能请求该资源。
    【例】假设在系统中有 10 个资源，可以把它们编号为 R1,R2,...,R10。假设进程 P1 当前已经申请了资源 R1、R3 和 R6。此后，它就不能再申请 R6 之前的资源了，只能申请 R6 以后的资源。

    [图片：资源编号 R1 到 R10，箭头指向 P0]

    可以证明，该方案下，死锁无法发生。
    【例】假设 P0、P1、P2、P3 分别占用 R0、R1、R2、R3，P0、P1、P2 依次申请 R1、R2、R3。

    [图片：资源分配图，显示进程P0、P1、P2、P3与资源R0、R1、R2、R3之间的分配与请求关系]

    若要实现循环等待链，则 P3 必须申请 P0 占用的 R0，显然，由于 R0 编号小于 R3，因此 P3 不可能有申请 R0 的请求，如果其需要 R0，则必须在申请 R3 前就申请 R0。因此，无法形成循环等待链。
    这种预防死锁策略和前两种策略相比，其资源利用率和系统吞吐量都有比较明显的改善，但也存在下列问题（了解即可）。
    ① 为系统中各类资源规定的序号必须相对稳定，这限制了新类型设备的增加。
    ② 尽管在为资源的类型分配序号时已经考虑到了大多数作业在实际使用这些资源时的顺序，但也经常会发生这种情况：作业使用各类资源的顺序与系统规定的顺序不同，造成对资源的浪费。
    第三，为了方便用户，系统对用户在编程时所施加的限制条件应尽量少，然而这种按规定次序申请资源的方法必然会限制用户进行简单、自主的编程。

    【练习】一次分配所有资源的方法可以预防死锁的发生，它破坏死锁 4 个必要条件中的（ ）。
    A.互斥
    B.占有并请求
    C.非剥夺
    D.循环等待
    答案：B 严格来说是请求

    5.哲学家进餐问题
    如图所示，有五个哲学家，他们共用一张圆桌，围坐在五把椅子上，桌上有五个碗和五只筷子，每根筷子只能互斥使用。

    [图片：哲学家进餐示意图，五只猫代表哲学家围坐在圆桌旁，桌上有碗和筷子]

    当哲学家饥饿的时候，他会尝试拿自己左右两边的筷子，只有当他手上有两根筷子的时候才能吃饭。饭后，他放下筷子，继续思考。为了实现对筷子的互斥使用，可以用 1 个信号量表示 1 根筷子，由这 5 根筷子对应的 5 个信号量构成信号量数组，描述如下。

    ```c
    semaphore chopstick[5]={1,1,1,1,1}; //初始化信号量
    //其中第i位哲学家的活动可描述如下：
    while (TRUE) {
        P(chopstick[i]);          //取左边筷子
        P(chopstick[(i+1) % 5]);  //取右边筷子
        ...
        吃饭;
        ...
        V(chopstick[i]);          //放下左边筷子
        V(chopstick[(i+1) % 5]);  //放下右边筷子
        ...
        思考:
        ...
    }
    ```
    在上述描述中，哲学家饥饿时总是会先去拿他左边的筷子，即执行 wait(chopstick[i])；成功后，再去拿他右边的筷子，即执行 wait(chopstick [(i+1)%5])；又成功后便可进餐。进餐完毕，先放下他左边的筷子，再放下他右边的筷子。

    上述解法有可能引起死锁。假如 5 位哲学家同时饥饿而各自拿起左边的筷子，这时就会使 5 个信号量 chopstick 均为 0；当他们再试图去拿右边的筷子时，都将会因无筷子可拿而进行无限期的等待。对于这样的死锁问题，可采取以下 3 种方法进行解决。
    （1）至多只允许有 4 位哲学家同时去拿左边的筷子，最终能保证至少有 1 位哲学家能够进餐，并在进餐毕时能释放出他用过的 2 根筷子，从而使更多的哲学家能够进餐。
    （2）仅当哲学家的左右两根筷子均可用时，才允许他拿起筷子进餐。
    （3）规定奇数号哲学家先拿他左边的筷子，然后再去拿他右边的筷子；而偶数号哲学家则相反。按此规定，5 位哲学家都先竞争奇数号筷子，获得后，再去竞争偶数号筷子，最后总会有一位哲学家能获得两根筷子而进餐。

    基于方法（1）的代码如下。

    ```c
    semaphore chopstick[5]={1,1,1,1,1}; //初始化信号量
    semaphore priority=4;               //设置允许拿筷子的哲学家数量
    //其中第 i 位哲学家的活动可描述如下：
    while (TRUE) {
        P(priority);
        P(chopstick[i]);                //取左边筷子
        P(chopstick[(i+1) % 5]);        //取右边筷子
        ...
        吃饭;
        ...
        V(chopstick[i]);                //放下左边筷子
        V(chopstick[(i+1) % 5]);        //放下右边筷子
        V(priority);
        ...
        思考:
        ...
    }
    ```
    【练习】在哲学家就餐问题中，若同时存在左撇子和右撇子（将先拿起左边筷子的人称为左撇子，而将先拿起右边筷子的人称为右撇子），则不会发生死锁，因为破坏了（ ）。
    A.互斥条件        B.请求与保持条件        C.不剥夺条件        D.循环等待条件
    答案：D

    ### 🧠 深度理解：哲学家进餐与破坏循环等待（来自 2.5 DS）

    > **Q：**  哲学家进餐用 `semaphore priority = 4` 限制，这个解法的逻辑是什么？
    >
    > **A：**  最多 4 人同时拿筷子，必然至少有 1 人能拿到两只筷子吃上饭。吃完释放筷子唤醒下一个。本质是破坏循环等待条件——限制并发数后，不可能出现"每个人都只拿了一只筷子"的循环等待局面。
    >

    > **Q：**  【练习】左撇子右撇子混合不会发生死锁，因为破坏了（ ）？
    >
    > **A：**  选 **C 破坏循环等待条件**。全左撇子（都先拿左边）时形成单向循环等待环。混合左右撇子后必然有人拿筷子顺序和别人相反，这个人打断了循环等待链——环被打破。
    >

    # 五、死锁避免

    （二）死锁避免（理解即可）
    1.基本概念
    死锁预防是静态的，在系统运行前限定了进程的运行规则、破坏了死锁产生的必要条件中的至少一个。<u>死锁避免是一种</u>​**运行时的检查方法**</u>​<u>，在进程请求资源时判断是否可能导致死锁；如果可能，就将其拒绝。</u>
    下面基于红绿灯和交警叔叔的例子说明死锁预防和死锁避免的区别。
    【例】假设有一个十字路口，如果要发生车辆相撞的话，显然，需要满足不同方向的车同时往十字路口中心开。
    [图片：十字路口交通示意图，带有红绿灯和箭头指示]

    死锁预防：设置红绿灯破坏发生相撞的条件，由于各个车辆都遵守“红灯停、绿灯行”的规则，因此，不可能出现两辆不同方向的车同时往十字路口中心开。

    [图片：左侧为十字路口示意图，中间为红绿灯，右侧为十字路口示意图]

    2.安全状态 （安全的运行序列）
    死锁主要是由不安全的进程推进顺序引起的，可以将系统的状态分为安全状态和不安全状态。安全状态指系统能按某种推进顺序为进程分配其所需资源，最终使每个进程都可顺利完成。如果系统无法找到这样一个安全序列，则称系统处于不安全状态。只要系统能一直保持在安全状态，就可以避免死锁发生。

    3.银行家算法（理解并记忆）（优先选择可满足其资源缺口的进程）
    最具代表性的死锁避免算法是银行家算法。该算法中，操作系统是银行，系统资源是资金，进程相当于向银行申请贷款的客户。不妨假装自己是一个银行家，如果有两个老板同时找你贷款，银行只能贷出 50w，而 A 老板需要 30w，B 老板需要 100w，那显然借给 A 老板更安全。因为借给 B 老板后，其资金缺口仍有 50w，万一他借不到更多的钱，项目就会烂尾。
    [图片：大楼]
    因此，操作系统分配资源时，优先选择可以满足其资源缺口的进程，进而保证系统不会进入不安全状态，以避免死锁。
    为实现银行家算法，每一个新进程在进入系统时，它必须申明在运行过程中，可能需要每种资源类型的最大单元数目，其数目不应超过系统所拥有的资源总量。
    当进程请求资源时，系统必须确定是否有足够的资源分配给该进程，以及进程申请资源数是否超过其所申报的数目。若资源足够且未超过申报数目，再计算在将这些资源分配给进程后，是否会使系统处于不安全状态。如果不会，才将资源分配给它，否则让进程等待。

    (3) 银行家算法
    理解了系统是如何进行安全性检测之后，再来学习一下银行家算法，整体流程如下。
    设 Requesti 为进程 Pi 的请求向量，如果 Requesti[j]=K，则说明进程 Pi 需要 K 个 Rj 类型的资源。
    ① 每次有进程申请资源时，都先判断
    a. 进程本次申请的资源数是否超过尚需的需求量，即是否满足 Requesti ≤ Need[i]。若不满足则请求失败，进程所需要的资源数量已经超过了它所申报的最大需求数量。
    b. 系统可用资源是否能满足申请的资源数目，即是否满足 Requesti ≤ Available，若不满足说明系统还没有足够的可用资源，进程 Pi 必须等待；

    ② 若以上判断都通过后，则系统会试探着为进程 Pi 分配资源，即更新以下数据结构。

    |P4|4 3 3|0 0 2|4 3 1||

    剩下步骤同理，最终可知，存在安全序列。

    【练习】某时刻进程的资源使用情况见下表，此时的安全序列是（ ）。
    A.P1,P2,P3,P4
    B.P1,P3,P2,P4
    C.P1,P4,P3,P2
    D.不存在

    |进程|已分配资源 (R1 R2 R3)|尚需分配 (R1 R2 R3)|可用资源 (R1 R2 R3)|
    | :---| :--------------------| :------------------| :------------------|
    |P1|2 0 0|0 0 1|0 2 1|
    |P2|1 2 0|1 3 2|P1 后 0 2 2|
    |P3|0 1 1|1 3 1|P2 - P4没有进程满足|
    |P4|0 0 1|2 0 0||

    答案：D

    如果在系统中，既不采取死锁预防措施，也未配有死锁避免算法，系统很可能会发生死锁。在这种情况下，系统应当提供两个算法：
    ① 死锁检测算法。该方法用于检测系统状态，以确定系统中是否发生了死锁。
    ② 死锁解除算法。当认定系统中已发生了死锁，利用该算法可将系统从死锁状态中解脱出来。

    ### 🧠 深度理解：银行家算法 vs 安全性算法（来自 2.5 DS）

    > **Q：**  银行家算法就是比安全性算法多加了一步试探回退的过程，这样理解对吗？
    >
    > **A：**  对。银行家 = **试探分配**（先假装把资源给你）→ **跑安全性算法**（检查分配后状态是否安全）→ 安全就真分，不安全就回退（拒绝请求）。安全性算法本身只判断"当前状态是否安全"，不管"如果分配了会怎样"。银行家多了这步"假装分配再检查"的试探过程。

    # 六、死锁检测与解除

    （三）死锁检测
    为了能对系统中是否已发生了死锁进行检测，在系统中必须：
    ① 保存有关资源的请求和分配信息
    ② 提供一种算法，它利用这些信息来检测系统是否已进入死锁状态。
    1.资源分配图
    可利用资源分配图来描述资源的请求和分配信息。

    [图片：资源分配图，包含进程P1、P2和资源R1、R2的节点及箭头]
    [1]

    如图所示，用圆圈代表一个进程，用方框代表一类资源。由于一种类型的资源可能有多个，用方框中的一个点代表一类资源中的一个资源。此时，请求边是由进程指向方框中的Rj，而分配边则应始于方框中的一个点。其中，P1 进程已经分得了两个 R1 资源，并又请求一个 R2 资源；P2 进程分得了一个 R1 和一个 R2 资源，并又请求 R1 资源。

    [1] 汤小丹,王红玲,姜华,汤子瀛.计算机操作系统（慕课版）[M].北京:人民邮电出版社,2021.

    2.死锁定理（简化资源分配图 -> 检测死锁）
    可以基于把资源分配图加以简化的方法，来检测系统是否为死锁状态。简化方法如下：
    ① 在资源分配图中，找出一个既不阻塞又非独立的进程结点 Pi。在顺利的情况下，Pi 可获得所需资源而继续运行，直至运行完毕，再释放其所占有的全部资源，这相当于消去 Pi 的请求边和分配边，使之成为孤立的结点。在图（a）中，将 P1 的两个分配边和一个请求边消去，便形成图（b）所示的情况。

    [图片：资源分配图简化过程(b)，显示P1, P2, R1, R2及其连接关系]

    ② P1 释放资源后，便可使 P2 获得资源而继续运行，直至 P2 完成后又释放出它所占有的全部资源，形成图（c）所示的情况，即将 P2 的两条请求边和一条分配边消去。

    [图片：资源分配图简化过程(c)，显示P1, P2, R1, R2及其连接关系]

    ③ 在进行一系列的简化后，若能消去图中所有的边，使所有的进程结点都成为孤立结点，则称该图是可完全简化的；若不能通过任何过程使该图完全简化，则称该图是不可完全简化的。若资源分配图是不可完全简化的，则发生了死锁。
    （四）死锁解除（理解即可）
    如果利用死锁检测算法检测出在系统中已发生了死锁，则应立即采取相应的处理措施来解除死锁。最简单的方法就是通过人工方法解除死锁。

    [图片：两只猫抢筷子和碗的图，配文“我要吃饭！”]

    另一种方法则是利用死锁解除算法，把系统从死锁状态中解脱出来。
    通常采用的解除死锁的两种方法是：
    1.抢占资源，从一个或多个进程中抢占足够数量的资源，然后将它们分配给死锁进程，以解除死锁状态。
    2.终止死锁进程，即终止系统中的一个或多个死锁进程，直至打破循环等待，使系统从死锁状态中解脱出来。终止死锁进程的方法有以下两种（了解即可）：
    （1）终止所有死锁进程
    终止所有死锁进程是一种最简单的方法，但由此付出的代价可能会很大，因为其中有些进程可能已经运行了很长时间并已接近结束。
    （2）逐个终止死锁进程
    相比于终止所有死锁进程，稍微温和一点的方法是：按照某种顺序逐个地终止死锁进程，直至有足够的资源来打破循环等待，把系统从死锁状态中解脱出来。
    但该方法所付出的代价也可能会很大，因为每终止一个进程，都需要用死锁检测算法来确定系统死锁是否已被解除，若未被解除，则需再终止另一个进程。

    ### 🧠 深度理解：资源图简化与死锁检测（来自 2.5 DS）

    > **Q：**  为什么资源图不可完全简化就是发生了死锁？能不能给个例子？
    >
    > **A：**  简化资源图 = 模拟进程执行顺序。找既不阻塞又非孤立的进程，假设它执行完释放资源，把资源分给在等的进程，消掉它的边。重复直到：
    >
    > - 所有边都能消掉 → **无死锁**
    > - 剩一些边消不掉 → **有死锁**
    >
    > 消不掉的边恰恰构成的就是循环等待环——每个进程都在等别人释放，但没人能释放。
    >

    > **Q：**  资源图简化不也类似银行家算法吗？能消掉所有边 = 能找到安全序列？
    >
    > **A：**  底层逻辑一致，但不是一回事：
    >
    > ||资源图简化（死锁检测）|安全性算法（死锁避免）|
    > | :-----------| :------------------------| :---------------------|
    > |时机|事后，检测是否已死锁|事前，分配前预防|
    > |过程|模拟释放+重新分配（消边）|计算 Need ≤ Available|
    > |可简化 = ?|无死锁|存在安全序列|
    > |不可简化 = ?|有死锁|拒绝分配|
    >
    > 一个**事后验尸**，一个**事前审批**，算的都是同一件事——"按这个顺序能不能走完"。
    >
