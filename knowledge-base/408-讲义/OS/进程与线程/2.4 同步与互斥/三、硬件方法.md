# 三、进程互斥的硬件实现方法

## （一）关中断

### 原理

在单处理器环境中，并发进程之间只能交替运行，当进程进入临界区之前**禁止中断**后，所有外部中断都被屏蔽，这样，进程在临界区执行期间，计算机系统不响应中断，从而不会引发调度，也就不会发生进程或线程切换。

```c
...
关中断;
临界区;
开中断;
...
```

### 三个缺点

#### ① 滥用关中断权

滥用关中断权可能会产生严重后果，不应将这一权限交给用户进程。

#### ② CPU 和 I/O 不能并行

并发环境中关中断会限制程序交替运行，导致 CPU 和 I/O 不能并行，影响系统执行效率。

**核心问题：**  关中断的本质是**禁止 CPU 响应外部中断**。在单 CPU 系统中，操作系统正是通过**时钟中断**来实现进程调度的——每次时钟中断触发，OS 才有机会切换进程。关中断后，时钟中断被屏蔽，CPU 就无法切换到其他进程。

**直观例子：两个进程交替运行**

假设有两个进程并发执行：

```
进程 P0:
    关中断;
    临界区操作（读写共享数据）;
    开中断;
    去访问磁盘（I/O操作，很慢）;

进程 P1:
    从磁盘读取数据（I/O操作）;
    关中断;
    临界区操作;
    开中断;
```

**正常情况下（不开中断）** ，并发执行的理想流程应该是：

```
时间线 →
P0: [临界区] → [读磁盘...等待...] → [继续运行]
P1:                     → [从磁盘读数据] → [临界区]
```

P0 在做磁盘 I/O（慢操作、不需要 CPU）时，CPU 可以切换给 P1 使用，**让 CPU 和 I/O 并行工作**。

**如果关中断的范围过大**（比如把临界区外的操作也包进关中断中）：

```
P0: [关中断] → [临界区] → [读磁盘...CPU空转等待...] → [开中断]
P1:                                                         → [从磁盘读数据] → [临界区]
```

由于 `关中断` → `开中断` 之间时钟中断被屏蔽，即使 P0 已经在等待慢速的磁盘 I/O（**不需要 CPU**），CPU 也无法切换给 P1。结果就是：

- **CPU 在空转等待**（本可以给 P1 用）
- **P1 在空闲等待**（本可以运行）
- **CPU 和 I/O 完全串行化**，浪费了系统资源

**更具体的场景：生产者和消费者**

```
生产者进程:        消费者进程:
关中断;            关中断;
while(缓冲区满);   while(缓冲区空);
放入数据;          取出数据;
开中断;            开中断;
```

如果生产者在关中断期间发现**缓冲区满了**，就会在 while 循环中空转等待。但消费者的临界区也需要开中断才能执行，而中断已被关闭——**生产者进不去临界区，消费者也进不去临界区，双方死等**（虽然不会死锁，但 CPU 被白白浪费）。

#### ③ 不适用于多 CPU 系统

关闭中断仅对执行该指令的处理器起作用，而其他处理器照常运行，也不能保证对临界资源的互斥使用。

## （二）TSL 指令

### 指令定义与原子性

TS/TSL (Test And Set Lock) 指令是一条特殊的机器指令，其特殊之处在于用一条指令连贯地执行了多个操作，即**原子操作**。

```c
bool TSL(bool *Plock) { // Plock 指向 lock 变量
    bool old = *Plock; // 记录锁的状态
    *Plock = TRUE;     // 上锁
    return old;
}
```

### 使用方式

lock 变量代表锁，其值为 FALSE 代表未上锁，其值为 TRUE 则对应上锁。

```c
While (TRUE) {
    ...
    while (TSL(&lock));  // 进入区
    临界区;
    Lock = FALSE;        // 退出区
    剩余区;
}
```

### 场景模拟：P0 和 P1 争用打印机

#### 初始状态

```
lock = FALSE    // 打印机空闲
```

#### Step 1：P0 想使用打印机

```
P0 执行: while (TSL(&lock));

TSL 干的事：
  ① old = *lock = FALSE    // 记录：锁是开的
  ② *lock = TRUE           // 立刻上锁（防止别人也进来）
  ③ return FALSE           // 返回 old 值

结果：
  while(FALSE) → 循环条件不成立 → P0 进入临界区，使用打印机！
  lock = TRUE               // 打印机已被占用
```

#### Step 2：P1 也想使用打印机（P0 正在用）

```
P1 执行: while (TSL(&lock));

TSL 干的事：
  ① old = *lock = TRUE     // 记录：锁是关的
  ② *lock = TRUE           // 上锁（保持上锁状态）
  ③ return TRUE            // 返回 old 值

结果：
  while(TRUE) → 循环条件成立 → P1 陷入自旋等待！
  lock = TRUE               // 依然上锁
```

#### Step 3：P0 用完打印机，释放锁

```
P0 执行: lock = FALSE;
结果: lock = FALSE  // 打印机空闲了！
```

#### Step 4：P1 再次尝试，成功

```
P1 再次执行: while (TSL(&lock));

TSL 干的事：
  ① old = *lock = FALSE    // 记录：锁开了！
  ② *lock = TRUE           // 上锁
  ③ return FALSE           // 返回 old

结果：
  while(FALSE) → P1 终于可以进入临界区了！
  lock = TRUE
```

### 优缺点分析

- 当临界区空闲时，lock 为 FALSE，TSL 指令会返回 FALSE，满足"空闲让进"
- 由于 TSL 是硬件原子指令，其执行操作一气呵成，不会有多个进程同时访问 lock 变量，只有一个进程可以顺利离开 while 循环体，满足"忙则等待"
- 被拦在临界区外的进程会不断测试 while 语句，该方法违反"让权等待"
- 相较于关中断方法，由于 lock 是共享的，TS 指令可以在多核处理器上使用

**关键**：TSL 指令就是"**先看看门有没有锁，同时把门锁上**"这个一气呵成的动作。

## （三）Swap 指令

### 指令定义

该指令又称 exchange 指令，用于交换两个变量的内容。

```c
void Swap (bool *a, bool *b) {
    bool temp = *a;
    *a = *b;
    *b = temp;
}
```

### 使用方式

Swap 指令为每个临界资源设置一个全局布尔变量 lock，其初值为 FALSE；在每个进程中还会设置一个局部布尔变量 key，其初值为 TRUE。

```c
while (TRUE) {
    key = TRUE;                        // 每次循环都把 key 重置为 TRUE
    while (key != FALSE) {             // 只要 key 不是 FALSE，就继续交换
        Swap(&lock, &key);             // 交换 lock 和 key
    }
    临界区;
    lock = FALSE; // 用完释放锁
    剩余区;
}
```

### 场景模拟：P0 和 P1 争夺打印机

#### Step 1：初始状态

```
lock = FALSE（打印机空闲）
key0 = TRUE（P0的钥匙）
key1 = TRUE（P1的钥匙）
```

#### Step 2：P0 先来，想用打印机

```
P0 执行 Swap(&lock, &key0)：

Swap 执行前：lock=FALSE,  key0=TRUE
                ↓
          temp = lock = FALSE
          lock = key0 = TRUE
          key0 = temp = FALSE
                ↓
Swap 执行后：lock=TRUE,   key0=FALSE

P0 检查 while (key0 != FALSE) → key0 = FALSE → 条件不成立，跳出循环，进入临界区！✅
```

#### Step 3：P1 也想用打印机（P0 正在用）

```
P1 执行 key1 = TRUE（重置钥匙），然后执行 Swap(&lock, &key1)：

Swap 执行前：lock=TRUE（P0锁着呢）,  key1=TRUE
                ↓
          temp = lock = TRUE
          lock = key1 = TRUE
          key1 = temp = TRUE
                ↓
Swap 执行后：lock=TRUE,   key1=TRUE

P1 检查 while (key1 != FALSE) → key1 = TRUE → 条件成立，继续循环，再次 Swap！

再次 Swap 执行前：lock=TRUE,  key1=TRUE
                ↓
          temp = lock = TRUE
          lock = key1 = TRUE
          key1 = temp = TRUE
                ↓
再次 Swap 执行后：lock=TRUE,   key1=TRUE  （一模一样，死循环了！）

P1 陷入死循环：不断 Swap，但 key 永远是 TRUE，永远进不去！⏳
```

#### Step 4：P0 用完打印机，释放锁

```c
lock = FALSE;   // P0 退出临界区，开锁
```

#### Step 5：P1 的下一次 Swap，终于成功

```
P1 的下一次 Swap(&lock, &key1)：

Swap 执行前：lock=FALSE（P0释放了）,  key1=TRUE
                ↓
          temp = lock = FALSE
          lock = key1 = TRUE
          key1 = temp = FALSE
                ↓
Swap 执行后：lock=TRUE,   key1=FALSE

P1 检查 while (key1 != FALSE) → key1 = FALSE → 跳出循环，进入临界区！✅
```

### 优缺点分析

- 能有效地实现进程互斥
- 当临界资源被访问时，其他访问进程必须不断地进行测试，即处于一种忙等状态，不符合"让权等待"的原则，会造成处理机时间的浪费
- 如果系统资源分配不当，将会出现若干个进程相互等待被对方占有的资源，而无法运行下去，系统产生**死锁**

## 🧠 深度理解

### 为什么 TSL 需要 old 变量？（来自 DS 问答）

你可能会问：为什么不直接 `*Plock = TRUE` 然后返回，还要先记录 old 值？

|情况|直接上锁返回|需要知道 old|
| :--------------------| :---------------------------------------------------------------------------| :----------------------------------|
|锁原来是开的（FALSE）|上锁后返回，但返回什么？返回 TRUE？那调用者以为原来就被锁着，会一直等下去 ❌|old=FALSE → 调用者知道抢到锁了 ✅|
|锁原来是关的（TRUE）|上锁后返回，但你怎么知道原来是不是被锁着？|old=TRUE → 调用者知道需要继续等 ✅|

**old 的作用就是：在执行 TSL 的瞬间，"快照"了锁的原始状态，让调用者知道自己是不是成功抢到了锁。**

### 为什么 TSL 要用硬件实现原子性？

「检查 lock → 上锁」这两个步骤如果分开，就可能出问题：

 **❌ 非原子操作（会出问题）**

```
P0: 检查 lock → 发现是 FALSE（空闲）→【被切换】→ P1: 检查 lock → 发现是 FALSE → 上锁 → 进入临界区
                                                                        ↑
                                                                   两个都进了！
P0: 上锁 → 进入临界区
```

 **✅ TSL 原子操作（正确）**

```
P0: TSL(&lock) → 检查+上锁一气呵成 → lock=TRUE → 进入临界区
P1: TSL(&lock) → 检查+上锁一气呵成 → lock=TRUE → while循环等待
```

### Swap 的类比：换牌游戏（来自 DS 问答）

可以把 Swap 想象成**换牌游戏**：

|元素|类比|
| :---| :------------------------------------------------------|
|`lock`|桌上唯一一张**通行证**（FALSE=通行证在桌上，TRUE=通行证被拿走了）|
|`key`|手里拿的**废牌**（TRUE）|
|`Swap`|**一手交牌，一手拿证**|

```
初始：桌上放着通行证(FALSE)，P0手里拿废牌(TRUE)

P0用废牌 Swap 桌上的通行证：
  → P0拿到通行证(FALSE) → 进门 ✅
  → 桌上只剩废牌(TRUE)

P1用废牌 Swap 桌上的废牌：
  → P1拿到废牌(TRUE) → 进不去，再换...
  → 桌上还是废牌(TRUE)  （怎么换都是废牌！）

P0出来，把通行证放回桌上(FALSE)

P1再 Swap → 终于拿到通行证 → 进门 ✅
```
