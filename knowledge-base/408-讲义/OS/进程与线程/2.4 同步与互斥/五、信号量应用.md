# 五、信号量的应用

## （一）利用信号量实现进程互斥

### 互斥信号量 mutex

用于实现进程互斥的信号量命名为 mutex（Mutual Exclusion）。当多个进程需要互斥访问某一临界资源时，每个进程在访问临界区前需对互斥量 mutex 执行 **P 操作**，在退出临界区后需对互斥量 mutex 执行 **V 操作**。

### 伪代码示例

```c
semaphore mutex = 1; //互斥量的初始值为1

Process_A {
    while(ture) {
        P(mutex);    //加锁
        访问临界区;
        V(mutex);    //解锁
        剩余区;
    }
}
Process_B {
    while(ture) {
        P(mutex); //加锁
        访问临界区;
        V(mutex); //解锁
        剩余区;
    }
}
```

### 要点

- 互斥信号量初始值为 1，表示该资源可用
- P/V 必须**成对出现**，否则会破坏互斥关系
- 每个访问临界资源的进程都需要执行相同的 P/V 操作

## （二）利用信号量实现进程同步

### 同步信号量

信号量可以实现两个进程 PA 和 PB 的同步关系，即 PA 和 PB 的**前后关系**。例如，PA 执行结束后才能执行 PB。

### 同步信号量初值为 0 的原因

同步信号量的初始值设为 0，表示该事件尚未发生（"还没做完"）。这与互斥信号量（初值为 1）有本质不同。

### 伪代码示例

```c
Semaphore S = 0; // 同步信号量的初始值为0

PA {
    修改文件;
    V(S);//表示修改完毕
}

PB {
    P(S); //判断是否修改完成
    修改文件;
}
```

### 同步 vs 互斥

- **同步是协作关系**：多个协作进程在工作中相互等待或通信的制约关系
- **互斥是竞争关系**：当一个进程进入临界区访问临界资源时，另一个进程必须等待，当访问临界资源的进程退出临界区后，另一个进程才被允许访问该临界资源

进程互斥是进程同步的一种特殊情况。

## （三）利用信号量实现多进程的前驱后继关系

### 场景描述

假设有 6 个进程，进程与进程之间存在着如图所示的前驱后继关系（前驱图）。图中两个节点间的箭头表示两个节点的前驱后继关系，例如 S1→S2 表示 S2 需要在 S1 进程之后运行。

```
S1 → S2, S3
S2 → S4, S5
S3 → S6
S4 → S6
S5 → S6
```

### 解题三部曲

#### ① 列出所有前驱后继关系

{S1→S2，S1→S3，S2→S4，S2→S5，S3→S6，S4→S6，S5→S6}

#### ② 为每个关系设置一个信号量，并初始化为 0

```c
semaphore s12 = s13 = s24 = s25 = s36 = s46 = s56 = 0;
```

#### ③ 编写伪代码

每个进程都需要**检查前驱进程是否运行完毕**（使用 P 操作），然后执行本进程的任务，最后**告知后继进程**本进程已运行完毕（使用 V 操作）。

```c
S1() {
    ...;
    V(s12);
    V(s13);
}
S2() {
    P(s12);    //检查前驱进程是否运行完毕
    ...;       //本进程工作
    V(s24);
    V(s25);    //告知直接后继进程已经运行完毕
}
S3() {
    P(s13);
    ...;
    V(s36);
}
S4() {
    P(s24);
    ...;
    V(s46);
}
S5() {
    P(s25);
    ...;
    V(s56);
}
S6() {
    P(s36);
    P(s46);
    P(s56);
    ...;
}
```

### 信号量取值练习

> 【练习】有三个进程共享同一程序段，而每次只允许两个进程进入该程序段，若用 PV 操作同步机制，则信号量 S 的取值范围是（ ）。
> A. 2,1,0,-1
> B. 3,2,1,0
> C. 2,1,0,-1,-2
> D. 1,0,-1,-2
> 答案：A

#### 分析思路

**关键点：信号量 S 的值 = 当前剩余的资源数**

**Step 1：确定初始值**

> 每次只允许 2 个进程进入

所以初始时，资源数是 **2**：

```c
semaphore S = 2;   // 一开始有 2 个"入场名额"
```

**Step 2：想象 3 个进程依次请求**

假设三个进程 P1、P2、P3 都想进入：

|操作|S 变化|说明|
| :-------| :-----| :-----------------------------------|
|初始状态|**S = 2**|两个名额空闲|
|P1 执行 `P(S)`|**S = 1**|P1 进去了，还剩 1 个名额|
|P2 执行 `P(S)`|**S = 0**|P2 进去了，名额刚好用完|
|P3 执行 `P(S)`|**S = -1**|⛔ 没名额了，P3 被阻塞，进入等待队列|

**S 的取值范围：2, 1, 0, -1**

**Step 3：检查极限情况**

|场景|S 的值|
| :------------------------------------| :-------|
|最多空闲时（没人进）|S = 2|
|有人进有人出，动态变化|S = 1, 0|
|最挤时（2 个名额已满，还有 1 个在等）|S = -1|

所以答案是 **A：2, 1, 0, -1** ✅
