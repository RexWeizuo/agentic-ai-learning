# 二、进程互斥的软件实现方法

## （一）临界区互斥的四个准则

```c
while (true) {
    进入区
    临界区
    退出区
    剩余区
}
```

四个准则：

|准则|含义|
| :---| :-----------------------------------------------------------------|
|**空闲让进**|当临界区空闲时，应允许一个进程进入|
|**忙则等待**|当已有进程在临界区时，其他想进入的进程必须等待|
|**有限等待**|等待进入临界区的进程不能无限期等待|
|**让权等待**|等待进入临界区的进程应主动释放 CPU（早期的软件方法均未满足此要求）|

## （二）单标志法

### 伪代码

```c
进程 P0:
while (turn!=0);        //进入区
critical section;       //临界区
turn=1;                 //退出区
remainder section;      //剩余区

进程 P1:
while (turn!=1);        //进入区
critical section;       //临界区
turn=0;                 //退出区
remainder section;      //剩余区
```

### 问题：违背"空闲让进"

该算法可以实现每次只允许一个进程进入临界区。

但两个进程必须交替进入临界区，若某个进程不再进入临界区，则另一个进程也将无法进入临界区（违背"空闲让进"准则）。这样很容易造成资源利用不充分。

例如，若 P0 顺利进入临界区并从临界区离开，则此时临界区是空闲的，但 P1 并没有进入临界区的打算，而 turn=1 一直成立，则 P0 就无法再次进入临界区。

## （三）双标志先检查法

### 伪代码

```c
进程 P0:
while(flag[1]);      (1)      //进入区
flag[0]=true;        (3)     //进入区
critical section;            //临界区
flag[0]=false;               //退出区
remainder section;           //剩余区

进程 P1:
while(flag[0]);      (2)      //进入区
flag[1]=true;        (5)      //进入区
critical section;            //临界区
flag[1]=false;               //退出区
remainder section;           //剩余区
```

优点：不用交替进入，可连续使用。

### 问题：违背"忙则等待"

按序列①②③④执行时，即检查对方标志后和设置自己的标志前可能发生进程切换，结果双方都检查通过，会同时进入临界区（违背"忙则等待"准则）。原因在于**检查和设置操作不是一气呵成的**（非原子操作）。

### 初始条件

- `flag[0]` 和 `flag[1]` 初始均为 `false`，表示两个进程都不在临界区。

### 问题场景（按序列①②③④执行）

1. **P0 执行 (1)** ：`while(flag[1]);`
   此时 `flag[1]` 为 `false`，循环条件不成立，P0 通过检查，**准备设置自己的标志**。
2. **P1 执行 (2)** ：`while(flag[0]);`
   此时 `flag[0]` 仍为 `false`（因为 P0 尚未执行 `flag[0]=true`），P1 也通过检查，**准备设置自己的标志**。
3. **P0 执行 (3)** ：`flag[0]=true;`
   P0 设置了自己的标志，表明它将要进入临界区。
4. **P1 执行 (5)** ：`flag[1]=true;`
   P1 设置了自己的标志，表明它将要进入临界区。

此时，两个进程都已经分别完成了"检查对方标志"和"设置自己标志"的操作，并且都认为对方不在临界区（因为检查时对方标志为 `false`）。因此，**两个进程会同时进入临界区**，违反了"忙则等待"准则。

### 根本原因

- **检查与设置不是原子操作**：每个进程的进入区实际上包含两个步骤：先检查对方标志，再设置自己的标志。这两个步骤之间可能被调度切换，导致竞争条件。
- **缺乏互斥机制**：该算法没有保证两个进程的"检查-设置"操作不会被交错执行，从而产生一个时间窗口，使得双方都以为对方不在临界区。

## （四）双标志后检查法

### 伪代码

```c
进程 P0:
flag[0]=true;        ①      //进入区
while(flag[1]);      ③      //进入区
critical section;            //临界区
flag[0]=false;               //退出区
remainder section;           //剩余区

进程 P1:
flag[1]=true;        ②      //进入区
while(flag[0]);      ④      //进入区
critical section;            //临界区
flag[1]=false;               //退出区
remainder section;           //剩余区
```

### 问题：违背"空闲让进"和"有限等待"

按序列①②③④执行时，即两个进程依次设置自己的标志，并依次检查对方的标志，发现对方也想进入临界区，双方都争着进入临界区，结果谁也进不了（违背"空闲让进"准则），于是因各个进程都长期无法访问临界区而导致**饥饿**现象（违背"有限等待"准则）。

## （五）Peterson 算法

### 伪代码

```c
进程 P0:
flag[0]=true;             //进入区
turn=1;                   //进入区
while(flag[1]&&turn==1);  //进入区
critical section;         //临界区
flag[0]=false;            //退出区
remainder section;        //剩余区

进程 P1:
flag[1]=true;             //进入区
turn=0;                   //进入区
while(flag[0]&&turn==0);  //进入区
critical section;         //临界区
flag[1]=false;            //退出区
remainder section;        //剩余区
```

### 正确性证明

若要证明 Peterson 解决方案是正确的，则须证明该方案满足解决临界区问题的 3 个准则：忙则等待、空闲让进、有限等待。

PS：在早期的解决方案中均未对"让权等待"做出要求，因为这样的做法虽会影响系统效率，但不影响临界区问题的解决。

#### 证明"空闲让进"

假设 P0 准备进入临界区，则先将 flag[0] 置为 true，表达自己的访问意愿，接着将 turn 置为 1，表示优先让对方 P1 进入临界区。之后，再不断检测 flag[1] 和 turn 的值。若 P1 的 flag 值为 false 或 turn==0，则 P0 不会陷入 while 循环，可以进入临界区，满足"空闲让进"准则。

#### 证明"忙则等待"

若在 P0 访问临界区的过程中发生进程切换，切换到进程 P1，由于 flag[0] 的值在 P0 访问临界资源结束前始终为 true，turn 的值在 P0 访问临界资源结束前始终为 0，因此 P1 始终陷入 while 循环，满足"忙则等待"准则。

#### 证明"有限等待"

只有当 P0 访问完临界资源，进入退出区，将 flag[0] 置为 false，P1 才可进入临界区，满足"有限等待"准则。

### 为什么违背"让权等待"？

Peterson 算法虽然很好地遵循了"空闲让进""忙则等待""有限等待"三个准则，但依然未遵循"让权等待"准则。因为进程即使进入不了临界区，也会一直运行 while 循环占用处理器资源（忙等 / busy waiting）。

#### 场景分析：P0 先进入临界区，P1 被阻塞

1. **P0 执行：**  `flag[0]=true;` → `turn=1;` → `while(flag[1]&&turn==1);` → 此时 flag[1] 为 false，条件不成立 → **P0 成功进入临界区** ✅
2. **发生进程切换** → CPU 切换到 P1
3. **P1 执行：**  `flag[1]=true;` → `turn=0;` → `while(flag[0]&&turn==0);` → 此时：
   - `flag[0]` = **true**（P0 尚未退出临界区）
   - `turn` = **0**（刚被 P1 设为 0）
   - 条件成立 → **P1 陷入 while 循环！**  ❌

#### 关键问题：P1 此时在做什么？

P1 陷入 `while(flag[0]&&turn==0);` 后，会**不断地、无休止地**进行条件测试：

```c
while(flag[0]&&turn==0);  // CPU不断执行这条语句
```

这被称为  **"忙等"（busy waiting）** ——进程虽然无法进入临界区，但依然**霸占着 CPU**，不断循环检查条件，不释放处理机。

**这就是违背"让权等待"准则的核心原因：**  进程即使进入不了临界区，也会一直运行 while 循环占用处理器资源，而不会主动让出 CPU 去睡眠/阻塞。

#### P0 退出后发生了什么？

当 P0 完成临界区操作，执行退出区 `flag[0]=false;`，此时 P1 的 while 条件 `flag[0]&&turn==0` 中的 `flag[0]` 变为 `false`，条件不成立，P1 跳出循环，成功进入临界区。但在此之前，P1 一直在"空转"浪费 CPU。

### 连续循环访问场景分析

【练习】若将 P0 进程和 P1 进程改造成不断循环访问临界区，则并发执行下，是否会出现饥饿现象？

答：不会，若 P0 先访问临界区，P1 卡在 while(flag[0]&&turn==0)时，进程 P1 无法再修改 flag[1] 以及 turn 的值，这就意味着他在表达自身想使用临界资源的基础上，又无法做到把资源让给进程 P0，因此一定会轮到 P1 执行。

```c
进程 P0:
While(true){
    flag[0]=true;
    turn=1;
    while(flag[1]&&turn==1);
    critical section;
    flag[0]=false;
    remainder section;
}

进程 P1:
While(true){
    flag[1]=true;
    turn=0;
    while(flag[0]&&turn==0);
    critical section;
    flag[1]=false;
    remainder section;
}
```

## （六）四种方法对照表

|方法|简述|典型缺点|
| :-------------| :-----------------------------------------| :-----------------------------------------|
|单标志法|引入 turn，循环检查直到 turn=本进程编号|只能交替进入，违背"空闲让进"|
|双标志先检查法|引入 flag[2]，先检查对方是否进入临界区|可能会同时进入，违背"忙则等待"|
|双标志后检查法|引入 flag[2]，先设置自身 flag，再检查|同时希望进入时，违背"空闲让进"、"有限等待"|
|Peterson 算法|引入 turn 和 flag[2]，进入临界区前检查两次|违背"让权等待"|

> 【练习】在 Peterson 算法中，turn 变量的作用是（ ）。
> A. 表示轮到哪个线程进入临界区
> B. 表示哪个线程先发出访问请求
> C. 表示哪个线程后发出访问请求
> C. 表示哪个线程已进入临界区
> 答案：A

## 🧠 深度理解：为什么说"不影响临界区问题的解决"？

Peterson 算法虽然违背了"让权等待"，但它**完美满足了其他三个准则**：

|准则|Peterson 满足情况|说明|
| :---| :----------------| :----------------------------------------------|
|✅ **空闲让进**|满足|当临界区空闲时，P0 或 P1 可以进入|
|✅ **忙则等待**|满足|P0 在临界区时，P1 被 while 拦住|
|✅ **有限等待**|满足|P0 退出后，P1 一定能进入（turn 机制保证了谦让）|
|❌ **让权等待**|**不满足**|P1 在 while 循环中忙等，占用 CPU 不释放|

### 对比：哪些方法满足了"让权等待"？

**信号量机制**（记录型信号量）中的 wait(S) 操作可以满足"让权等待"：

```c
wait(semaphore S) {
    S.s--;
    if(S.s < 0)
        阻塞该进程并将其放入阻塞队列中;  // ✨ 主动让出CPU，进入阻塞状态
}
```

当进程无法获取资源时，会被阻塞并放入等待队列，**主动让出CPU**，而不是像 Peterson 算法那样在 while 循环中空转。这就是从软件方法到信号量机制的演进逻辑。
