# 2.4 同步与互斥

2.4 同步与互斥
一、基本概念（理解即可）
在多道程序环境下，计算机系统中存在着多个进程，它们彼此之间往往会存在一定的联系。
一方面，进程间互相竞争系统中的有限资源；另一方面，有些进程之间必须相互协作，共同完成某一任务。这种进程间的既竞争又合作的关系表现为进程的互斥与同步关系。

[图片：内存地址空间分布图，显示主存和CPU]

（一）互斥
互斥可以简单理解为，当某个进程访问某个共享资源时，如果有其他进程正在访问，则必须等待其他进程释放该资源后才可以访问。
【例】倘若当前有两个进程 A 和 B 分别负责打印 408 真题和考研英语真题，若不采用互斥共享方式使用打印机，则可能一张 A4 纸上又有 408 真题，又有英语真题！

[图片：打印机打印出混杂内容的示意图]

为此，当进程 A 访问某个资源时，必须先提出请求，若此时该资源空闲，则系统便将之分配给 A 使用，此后有其他进程也要访问该资源时（只要 A 未用完）就必须等待。
仅当 A 访问完并释放该资源后，才允许另一个进程对该资源进行访问。

[图片：进程A和进程B访问打印机的互斥流程图]

这种资源共享方式称为互斥共享，进程 A 和进程 B 的关系为互斥关系，也叫间接制约关系。
（二）同步
进程同步是指多个进程间发生的事件存在某种时序关系，例如先后关系，必须协同动作、相互配合来共同完成一个任务。
【例】一项工作由 A、B、C 三个进程来完成，进程 A 用于输入数据，进程 B 对输入的数据进行处理，进程 C 输出结果。

[图片：进程A、B、C顺序处理数据的同步流程图]

--- Page 102 ---
显然，当 A 进程完成数据输入后，B 进程才可运行，B 进程计算出结果后，C 进程才可输出结果。此时，进程之间的这种制约关系称作同步关系，又叫直接制约关系。

（三）临界资源与临界区（理解并记忆）
进程的互斥是由某些共享资源引起的，若不对某些共享资源互斥访问，则会导致执行结果不一致。

【例】有两个进程 P1 和 P2，它们共享一个变量 n，程序代码描述如下所示。请问当两个进程并发执行时的输出结果是怎么样的？

```c
int n=0;//n 为共享变量
P1(){
    while (n<5)
    {
        n++;
    }
    printf("A=%d\n",n);
}
P2(){
    while (n<5)
    {
        n++;
    }
    printf("B=%d\n",n);
}
```

解：由于两个进程并发执行时，很可能 P1 执行了某个语句后跳转到 P2 执行，因此处理这种问题时我们最好先给各个语句标号，这样方便后续描述。标号结果如下所示。

```c
int n=0;//n 为共享变量
P1(){
    ①while (n<5)
    {
        ②n++;
    }
    ③printf("A=%d\n",n);
}
P2(){
    ④while (n<5)
    {
        ⑤n++;
    }
    ⑥printf("B=%d\n",n);
}
```

1.倘若 P1 执行完语句③后切换到 P2 执行，则输出结果为：
A=5
B=5

2.倘若 P2 执行完语句⑥后切换到 P1 执行，则输出结果为：
B=5
A=5

3.倘若按照①②①②①②①②的顺序执行，此时 n=4;
接着执行①，由于 n<5 成立，接下来将要执行②;
若即将执行②时发生进程切换，执行④，由于 n<5 成立，接下来执行⑤，此时 n=5;
接着执行⑥后切换到 P1，继续执行②，此时 n=6;
接着执行①，由于 n<5 不成立，接下来执行③，最终输出结果为：
B=5
A=6

显然，并发执行下，输出结果有多种可能性。

【练习】有两个并发进程 P1 和 P2，其程序代码如下：
```c
P1(){
    x=1;    //A1
    y=2;
    z=x+y;
    print z;    //A2
}
P2(){
    x=-3;   //B1
    c=x*x;
    print c;    //B2
}
```
可能打印出的 z 值有（ ），可能打印出的 c 值有（ ）（其中 x 为 P1，P2 的共享变量）
A.z=1,-3; c=-1;9      B.z=-1,3; c=1,9      C.z=-1,3,1; c=9      D.z=3; c=1,9
答案：B

由于进程的并发执行可能会导致错误的运行结果，因此需要相应的规则来加以约束进程对共享资源的使用。因此，我们引入了临界资源和临界区的概念。
临界资源指一次仅允许一个进程使用的资源，既可以是硬件资源，也可以是软件资源，比如打印机、变量、队列等。在上述例子中，进程 P1 与 P2 的临界资源是变量 n。在每个程序中，对临界资源访问的那段程序称为临界区。

```c
int n=0; // n 为共享变量
P1(){
    ①while (n<5)
    {
        ②临界区
    }
    ③printf("A=%d\n",n);
}
```

若能保证一次仅允许一个进程使用临界资源，即保证只有一个进程进入临界区，就不会产生错误的结果。因此，为了保证临界资源的正确使用，可将临界资源的访问过程分成 4 个部分：

```c
while (true) {
    进入区
    临界区
    退出区
    剩余区
}
```

①进入区（entry section）
每个进程在进入临界区之前，应先对欲访问的临界资源进行检查，看它是否正在被访问。如果此刻该临界资源未被访问，进程便可进入临界区对该资源进行访问，并将访问标志设置为“正被访问”；如果此刻该临界资源正在被某进程访问，则本进程不能进入临界区。
②临界区（critical section）
进程中访问临界资源的那段代码，又称临界段。
③退出区（exit section）
将正在访问临界区的标志清除。
④剩余区（remainder section）
代码中的其余部分。

【练习】以下（ ）属于临界资源。
A.磁盘存储介质      B.公用队列      C.私用数据      D.可重入的程序代码
答案：B

（一）临界区互斥的准则
可以采用多种方法实现进程互斥地进入自己的临界区，但是，不论用什么办法都要遵循以下解决临界区互斥的准则：
1.空闲让进
当无进程处于临界区时，表明临界资源处于空闲状态，应允许一个请求进入临界区的进程立即进入自己的临界区，以有效地利用临界资源。
[图片：进程A请求打印机，打印机空闲]
2.忙则等待
当已有进程进入临界区时，表明临界资源正在被访问，因而其他试图进入临界区的进程必须等待，以保证对临界资源的互斥访问。
[图片：进程A占用打印机，进程B请求，打印机很忙]
3.有限等待
对于要求访问临界资源的进程，应保证其在有限时间内能进入自己的临界区，以免陷入“死等”状态。
4.让权等待（原则上应遵循，但非必须）
进程不能进入自己的临界区时，应立即释放处理机，以免进程陷入“忙等”状态。
进程互斥、同步的解决方法可以被划分为软件方法和硬件方法。其中，信号量机制、管程比较特殊，自成一类。
（二）软件方法
软件方法的基本思路是在进入临界区时通过检查标志的状态决定是否进入，在退出临界区时修改标志。
```c
while (true) {
    进入区
    临界区
    退出区
    剩余区
}
```
其中的主要问题是设置什么标志和如何检查标志。下面讨论几种用软件方法实现的互斥方法。
1.单标志法
该算法设置一个公用整型变量 turn，指示允许进入临界区的进程编号，当 turn=0 时，表示允许 P0 进入临界区；当 turn=1 时，表示允许 P1 进入临界区。
进程退出临界区时将临界区的使用权赋给另一个进程，当 Pi 退出临界区时，将 turn 置为 j（i=0、j=1 或 i=1、j=0）。
```c
进程 P0:
while (turn!=0);        //进入区
critical section;       //临界区
turn=1;                 //退出区
remainder section;      //剩余区
```
```c
进程 P1:
while (turn!=1);        //进入区
critical section;       //临界区
turn=0;                 //退出区
remainder section;      //剩余区
```
[图片：P0和P1指向打印机，旁边有CPU]

2.4 同步与互斥

该算法可以实现每次只允许一个进程进入临界区。
但两个进程必须交替进入临界区，若某个进程不再进入临界区，则另一个进程也将无法进入临界区（违背“空闲让进”准则）。
这样很容易造成资源利用不充分。
例如，若 P0 顺利进入临界区并从临界区离开，则此时临界区是空闲的，但 P1 并没有进入临界区的打算，而 turn=1 一直成立，则 P0 就无法再次进入临界区。

**2.双标志先检查法**
该算法设置一个布尔型数组 flag[2]，用来标记各个进程想进入临界区的意愿，flag[i]=true 表示 Pi 想要进入临界区（i=0 或 1）。Pi 进入临界区前，先检查对方是否想进入临界区，若想，则等待；否则，将 flag[i] 置为 true 后，再进入临界区；当 Pi 退出临界区时，将 flag[i] 置为 false。

```c
进程 P0:
while(flag[1]);      ①      //进入区
flag[0]=true;        ③      //进入区
critical section;            //临界区
flag[0]=false;               //退出区
remainder section;           //剩余区

进程 P1:
while(flag[0]);      ②      //进入区
flag[1]=true;        ④      //进入区
critical section;            //临界区
flag[1]=false;               //退出区
remainder section;           //剩余区
```

[图片：P0和P1指向打印机，旁边有CPU]

优点：不用交替进入，可连续使用。
缺点：P0 和 P1 可能同时进入临界区。
例如，按序列①②③④执行时，即检查对方标志后和设置自己的标志前可能发生进程切换，结果双方都检查通过，会同时进入临界区（违背“忙则等待”准则）。原因在于检查和设置操作不是一气呵成的。

**3.双标志后检查法**
双标志先检查法的问题在于：先检查对方的标志，再设置自己的标志，但这两个操作又无法一气呵成，于是使得两个进程同时进入临界区的问题。
因此，想到先设置后检查法来避免上述问题。
双标志后检查法中先设置自己的标志，再检查对方的标志，若对方的标志为 true，则等待；否则，进入临界区。

```c
进程 P0:
flag[0]=true;        ①      //进入区
while(flag[1]);      ③      //进入区
critical section;            //临界区
flag[0]=false;               //退出区
remainder section;           //剩余区

进程 P1:
flag[1]=true;        ②      //进入区
while(flag[0]);      ④      //进入区
critical section;            //临界区
flag[1]=false;               //退出区
remainder section;           //剩余区
```

按序列①②③④执行时，即两个进程依次设置自己的标志，并依次检查对方的标志，发现对方也想进入临界区，双方都争着进入临界区，结果谁也进不了（违背“空闲让进”准则），于是因各个进程都长期无法访问临界区而导致“饥饿”现象（违背“有限等待”准则）。

**4.Peterson 算法**
Peterson 算法结合了单标志法和双标志后检查法的思想，使用 flag[]表达自己的意愿，变量 turn 表示哪个进程可以进入临界区，即如果 turn==i，那么允许进程 Pi 进入临界区内执行。
程序代码如下所示：

进程 P0:
```c
flag[0]=true;             //进入区
turn=1;                   //进入区
while(flag[1]&&turn==1);  //进入区
critical section;         //临界区
flag[0]=false;            //退出区
remainder section;        //剩余区
```

进程 P1:
```c
flag[1]=true;             //进入区
turn=0;                   //进入区
while(flag[0]&&turn==0);  //进入区
critical section;         //临界区
flag[1]=false;            //退出区
remainder section;        //剩余区
```

[图片：P0和P1进程指向打印机，旁边有CPU]

若要证明 Peterson 解决方案是正确的，则须证明该方案满足解决临界区问题的 3 个准则：忙则等待、空闲让进、有限等待。
PS：在早期的解决方案中均未对“让权等待”做出要求，因为这样的做法虽会影响系统效率，但不影响临界区问题的解决。
证明如下：
假设 P0 准备进入临界区，则先将 flag[0]置为 true，表达自己的访问意愿，接着将 turn 置为 1，表示优先让对方 P1 进入临界区。
之后，再不断检测 flag[1]和 turn 的值。若 P1 的 flag 值为 flase 或 turn==0，则 P0 不会陷入 while 循环，可以进入临界区，满足“空闲让进”准则。
若在 P0 访问临界区的过程中发生进程切换，切换到进程 P1，由于 flag[0]的值在 P0 访问临界资源结束前始终为 true，turn 的值在 P0 访问临界资源结束前始终为 0，因此 P1 始终陷入 while 循环，满足“忙则等待”准则。只有当 P0 访问完临界资源，进入退出区，将 flag[0]置为 false，P1 才可进入临界区，满足“有限等待”准则。
由此可见，Peterson 算法很好地遵循了“空闲让进”“忙则等待”“有限等待”三个准则，但依然未遵循“让权等待”准则。因为进程即使进入不了临界区，也会一直运行 while 循环占用处理器资源。

【练习】若将 P0 进程和 P1 进程改造成不断循环访问临界区，则并发执行下，是否会出现饥饿现象？

进程 P0:
```c
While(true){
    flag[0]=true;             //进入区
    turn=1;                   //进入区
    while(flag[1]&&turn==1);  //进入区
    critical section;         //临界区
    flag[0]=false;            //退出区
    remainder section;        //剩余区
}
```

进程 P1:
```c
While(true){
    flag[1]=true;             //进入区
    turn=0;                   //进入区
    while(flag[0]&&turn==0);  //进入区
    critical section;         //临界区
    flag[1]=false;            //退出区
    remainder section;        //剩余区
}
```

答：不会，若 P0 先访问临界区，P1 卡在 while(flag[0]&&turn==0)时，进程 P1 无法再修改 flag[1]以及 turn 的值，这就意味着他在表达自身想使用临界资源的基础上，又无法做到把资源让给进程 P0，因此一定会轮到 P1 执行。

上述软件方法总结如下表：

| 方法 | 简述 | 典型缺点 |
| :--- | :--- | :--- |
| 单标志法 | 引入 turn,循环检查直到 turn=本进程编号 | 只能交替进入、违背“空闲让进” |
| 双标志先检查法 | 引入 flag[2]，先检查对方是否进入临界区 | 可能会同时进入，违背“忙则等待” |
| 双标志后检查法 | 引入 flag[2]，先设置自身 flag，再检查 | 同时希望进入时，违背“空闲让进”、“有限等待” |
| Peterson 算法 | 引入 turn 和 flag[2]，进入临界区前检查两次 | 违背“让权等待” |

注：典型缺点仅代表某个方法最明显的缺点，可能还有其他缺点

（三）硬件方法
由于机器指令在执行过程中不会受到其他指令的干扰，也不会被中断。因此，可以利用机器指令来实现进程互斥。能够实现这种功能的机器指令有三种：关/开中断指令、测试与设置指令和交换指令。

1. 关中断
在单处理器环境中，并发进程之间只能交替运行，当进程进入临界区之前禁止中断后，所有外部中断都被屏蔽，这样，进程在临界区执行期间，计算机系统不响应中断，从而不会引发调度，也就不会发生进程或线程切换。关中断方法的一般流程如下所示。
```
...
关中断;
临界区;
开中断;
...
```

[图片：P0和P1进程指向打印机，旁边有一个CPU框]

然而，关中断的方法有很多缺点：
① 滥用关中断权可能会产生严重后果，不应将这一权限交给用户进程。
② 并发环境中关中断会限制程序交替运行，导致 CPU 和 I/O 不能并行，影响系统执行效率。
③ 不适用于多 CPU 系统，因为关闭中断仅对执行该指令的处理器起作用，而其他处理器照常运行，也不能保证对临界资源的互斥使用。

[图片：P0和P1进程指向打印机，旁边有两个CPU框]

2. TS (Test And Set Lock, 测试并上锁) 指令
TS/TSL 指令是一条特殊的机器指令，其特殊之处在于用一条指令连贯地执行了多个操作，即原子操作，其描述如下。
```c
bool TSL(bool *Plock) { // Plock 指向 lock 变量
    bool old = *Plock; // 记录锁的状态
    *Plock = TRUE; // 上锁
    return old;
}
```
lock 变量代表锁，其值为 FALSE 代表未上锁，其值为 TRUE 则对应上锁。若 lock 为 false，则代表未上锁，记录完锁的状态后上锁。若 lock 为 true，则执行 *Plock=true 后，保持上锁状态。

```c
void Swap (bool *a, bool *b) {
```

1.信号量的类型（不同教材有所出入）
（1）整型信号量
整型信号量代表一个用于表示资源数目的整数 S，S 指代的是操作系统中某资源的数量。
除初始化外，仅能通过两个标准的原子操作来访问，即 wait(S)和 signal(S)操作。这两个操作又被分别称为 P 操作和 V 操作，说明如下。
① wait(S)原语实现 S 值的减 1，减 1 代表申请并获取一个资源。如果 S≤0，操作系统无法为其分配资源，该进程就会陷入 while 循环，直到 S>0，方可结束等待，描述如下。
```c
wait ( S ) {
    while (S <= 0); // 不符合条件则陷入“忙等”
    S--;
}
```
② signal(S)原语实现 S 数值的加 1，加 1 相当于使用结束并释放一个资源，描述如下。
```c
signal (S) {
    S++;
}
```
wait(S)和 signal(S)是两个原子操作，因此，它们在执行时是不可中断的。亦即，当一个进程在修改某信号量时，没有其他进程可同时对该信号量进行修改。
注意，这里只是用伪代码描述指令，不代表指令等于函数。

生产者-消费者问题可以描述为一组生产者和一群消费者一起工作，通过一个大小为n的有限缓冲区进行生产和消费。一个缓冲区可以容纳n个产品，其中生产者负责投放产品，消费者负责消费产品。

冲区拿取产品。
PV 大题的解题思路如下。
① 找出各类进程和各类资源。
② 分析各类进程之间的同步、互斥关系，分析产生死锁的情况。
③ 定义信号量，确定信号量初值。
④ 写出伪代码。
⑤ 检查是否 PV 成对，检查是否有死锁，是否符合题意。若有问题，则继续优化。
注意，“找同步关系”的核心是找事件之间的关系，某个事件导致某个事件，而不是找“某个进程导致某个进程”，直接找进程之间的关系往往无从下手。
解：
① 两类进程：生成者进程和消费者进程。一类资源：大小为 n 的缓冲区。
② 互斥关系：生产者和消费者均需要互斥访问缓冲区。同步关系：缓冲区有产品，消费者才能取，缓冲区有空位，生产者才能生产。
③ semaphore mutex=1; //互斥使用缓冲区
semaphore full=0; //表示缓冲区的产品数量（也可以看作非空闲区数量）
semaphore empty=n; //表示缓冲区的空闲区数量
④ 生产者-消费者问题的程序实现如下。
```c
semaphore mutex = 1, empty = n, full = 0;
void producer () {
    while (TRUE) {
        生产一个新的产品;
        P(empty);         //判断是否有空闲的缓冲区，若无则加入等待队列
        P(mutex);         //对缓冲区上锁
        将产品放入空闲缓冲区;
        V(mutex);         //对缓冲区解锁
        V(full);          //产品数量加 1，并唤醒等待队列中的消费者
    }
}//生产、消费都是种不断重复的动作，所以需要 while 循环
void consumer () {
    while (TRUE) {
        P(full);          //判断是否有产品，若无则加入等待队列
        P(mutex);         //对缓冲区上锁
        从缓冲区取出一个产品;
        V(mutex);         //对缓冲区解锁
        V(empty);         //空闲区数量加 1，并唤醒等待队列中的生产者
        生产该产品;
    }
}
```
⑤ PV 成对，无死锁，符合题意。
（五）管程（理解并记忆部分重点）
1.基本定义
虽然信号量机制可以解决进程的同步与互斥问题，但每个希望访问临界资源的进程都必须分别定义自己的 P/V 操作，使得众多同步操作分散在各个进程中。这不仅使系统难以管理，还可能会因为误用同步操作，导致代码中存在死锁的概率增加。因此，提出了管程这种同步机制，下面举例说明。
【例】自助餐厅有一些所有人都可以用的资源，比如调料瓶。
[图片：自助餐厅的调料台] [图片：一组调料瓶]

--- Page 113 ---
在信号量机制下，餐厅里的顾客需要自己协调，以确保不会同时争抢餐具或调料瓶，确保一次只有一个人使用共享资源。
而在管程机制下，餐厅里有一个服务员来分配和管理所有资源。顾客不需要自己协调，他们只需要向服务员请求资源。服务员会确保资源被正确地分配和回收。
管程是一种同步机制，它封装了共享资源和对资源的操作，简化了并发编程，集中管理了对共享资源的所有访问。（管程就是个服务员，共享资源登记在他这，任何进程（客人）想要访问共享资源都得找他）
管程由以下部分组成：
① 管程的名称。
② 局部于管程内部的共享数据结构说明。
③ 对局部于管程内部的共享数据设置初始值的语句。
④ 对该数据结构进行操作的一组过程（或函数）。

管程可以看作一个类（不理解“类”的宝子也没事儿，就当作这是个可以定义函数的结构体变量）。

```c
}
}
```
条件变量和信号量的相似点为，条件变量的 wait/signal 操作类似于信号量的 P/V 操作，可以实现进程的阻塞/唤醒。两者的区别在于，条件变量是“没有值”的，仅实现了“排队等待”功能，而信号量是“有值”的。信号量的值反映了剩余资源数，而在管程中，剩余资源数用共享数据结构记录。

在读者一写者问题中，有多个进程访问一个共享文件。允许多个进程（读者）同时读取 F 的信息，但只允许一个进程（写者）修改 F 中的信息。当一个进程正在读取时，不允许其他进程写入（但可以读取）；当一个进程正在写入时，其他进程的任何读写操作均不被允许。

[图片：包含“写”、“读”、“读”、“读”的示意图]

读者一写者问题有三种典型的算法，即读者优先算法、写者优先算法和读写公平算法，这里仅讨论读者优先问题。在该算法中，只要有一个读进程在读，读进程就能继续获得对临界区的控制权，但可能会导致写进程饥饿。算法描述如下：
① 当一个读进程正在读取时，写进程必须等待。
② 在写进程等待时，随后的读进程仍可以进入临界区读取。
③ 只有在没有进程访问文件时，写进程才可以写入。
解：
1.两类进程，分为读者进程，写者进程。一类资源，文件。
2.同步互斥关系，一个写进程与其他进程。（复杂问题中，同步和互斥往往有交叉，就不去细分了）
3.定义信号量

```c
        V(wmutex);          // 对 wmutex 执行 V 操作，允许他人访问文件
    } while(TRUE);
```

【补充】自旋锁与互斥锁（理解即可）
Linux 有多种同步机制，这里仅补充自旋锁和互斥锁两种机制。
1.自旋锁
Linux 内核中最常用的锁是自旋锁（spin lock），其最初的设计目的是在多处理机系统中提供对共享数据的保护。

[图片：CPU A、V、CPU、CPU 的示意图]

自旋锁的设计思想为，在多处理机之间设置一个全局变量 V，表示锁。当 V=1 时为锁定状态，V=0 时为解锁状态。自旋锁属于“忙等”机制。
如果处理机 A 上的代码要进入临界区，就要先读取 V 的值。如果 V!=0（即锁定状态），则表明有其他处理机上的代码正在对共享数据进行访问，那么此时处理机 A 就会进入忙等状态（即自旋状态）；如果 V=0，则表明当前没有其他处理机上的代码进入临界区，此时处理机 A 可以访问该临界资源。然后把 V 设置为 1，再进入临界区，访问完毕后离开临界区时将 V 设置为 0。注意，必须确保处理机 A “读取 V、判断 V 的值、更新 V”这一操作是一个原子操作。
自旋锁的实现与体系结构相关，其基本使用形式如下：
```c
1 DEFINE_SPINLOCK(mr_lock);
2 spin_lock(&mr_lock);
3 /* 临界区 */
4 spin_unlock(&mr_lock);
```
2.互斥锁
新版 Linux 内核中，引入了互斥锁（mutex）这个数据类型。互斥锁与使用计数为 1 的信号量类似，但操作接口更简单，实现更高效，而且使用限制更强。
对互斥锁进行锁定和解锁的操作如下：
```c
1 mutex_lock(&name);
2 /* 临界区 */
3 mutex_unlock(&name);
```
基于上述介绍可知，互斥锁就是一个简化版的信号量，因为它不再需要使用任何计数。

【补充】利用管程解决生产者消费者问题（时间紧张可跳过）
在利用管程来解决生产者-消费者问题时，首先须建立一个管程，并将其命名为 producerconsumer（简称 PC），其中包括以下两个过程。
（1）put(x)过程：生产者利用该过程将自己生产的产品投放到缓冲池中，并用整型变量 count 来表示在缓冲池中已有的产品数目，当 count>=N 时，表示缓冲池已满，生产者应等待。
（2）get(x)过程：消费者利用该过程从缓冲池中取出一个产品，当 count<=0 时，表示缓冲池中已无可取用的产品，消费者应等待。

--- Page 116 ---
对于条件变量 full 和 empty，有以下两个过程可对它们进行操作。
(1) cwait (condition) 过程：如果正在调用管程的进程因 x 条件而被阻塞，则调用 cwait 将自己插入到 x 条件的等待队列上，并释放管程。
(2) csignal (condition) 过程：唤醒阻塞在条件 condition 队列上的某个进程。
程序描述如下：
```c
monitor producerconsumer {          // 定义生产者-消费者管程
    item buffer[N];                 // 缓冲区（固定大小 N）
    int in, out;                    // 缓冲区写入指针(in)和读取指针(out)
    condition full, empty;          // 条件变量，用于阻塞时排队
    int count;                      // 当前缓冲区中的产品数目
    public:
        void put(item x) {          // 生产者放入缓冲区
            if (count >= N) cwait(full); // 缓冲区已满时等待
            buffer[in] = x;         // 写入缓冲区
            in = (in + 1) % N;      // 移动写入指针（循环缓冲区）
            count++;                // 更新产品数目
            csignal(empty);         // 唤醒等待的消费者
        }
        void get(item x) {          // 消费者取出产品
            if (count <= 0) cwait(empty);   // 缓冲区为空时等待
            x = buffer[out];        // 读取缓冲区
            out = (out + 1) % N;    // 移动读取指针（循环缓冲区）
            count--;                // 更新产品数目
            csignal(full);          // 唤醒等待的生产者
        }
        {
            in = 0; out = 0; count = 0;     // 初始化
        }
} PC;                               // 实例化管程对象 PC
void producer() {                   // 生产者
    item x;
    while (TRUE) {
        生产一个产品;
        PC.put(x);                  // 调用管程的 put 方法存入缓冲区
    }
}

void consumer() {                   // 消费者
    item x;
    while (TRUE) {
        PC.get(x);                  // 调用管程的 get 方法取走一个产品
        消费产品;
    }
}
```
