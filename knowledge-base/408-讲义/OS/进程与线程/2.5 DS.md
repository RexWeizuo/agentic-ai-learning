## 先从"死锁"的根源说起

回忆一下你笔记中死锁的四个必要条件：

1. **互斥条件** —— 一个资源同时只能被一个进程占用
2. **请求和保持条件** —— 进程占着资源不放，还请求别的资源
3. **不可抢占条件** —— 资源不能强行抢走
4. **循环等待条件** —— ABC 互相等对方的资源，形成环

> **只要打破其中一个，死锁就发生不了。**

---

## 现在来看"破坏互斥条件"这条思路

### 核心思想就一句话：**如果只有一个进程在访问某个资源，那就不存在"多个进程抢资源"的问题，死锁自然就没了。**

但是问题来了——很多资源（如打印机）天生就是**互斥的**：一次只能打印一份文档，不能两个进程同时往打印机上扔数据，否则打印出来的东西会乱七八糟混在一起。

那怎么办呢？

### 思路：找一个"代理进程"

> 不为每个用户进程直接使用打印机，而是专门安排一个"打印管理进程"（代理），所有其他进程想用打印机？**告诉代理就行了，别自己亲自上。**

这样：

- 直接碰打印机的进程 **只有这一个代理进程**
- 其他进程不直接访问打印机，所以不参与竞争
- 互斥条件被"绕过去"了——不是没有互斥，而是**不需要互斥**了，因为只有一个进程在用

---

## 具体到 SPOOLing 技术（最经典的例子）

**SPOOLing = Simultaneous Peripheral Operation On-Line**（联机同时外围操作），俗称"假脱机技术"。

### 传统打印的问题（死锁风险）

假设进程 A 和 B 都想打印：

- A 拿到了打印机，正在打印
- B 也想打印，但打印机被 A 占着，B 阻塞等待
- 如果 A 还占着别的资源不放，B 也占着什么资源不放，可能就死锁了

### SPOOLing 怎么解决？

SPOOLing 系统中有一个**守护进程（daemon）** 专门管理打印机，叫"打印服务进程"。

工作流程：

```
进程A ------> [把要打印的数据写入磁盘] ------> 打印服务进程（唯一能碰打印机）-----> 打印机
进程B ------> [把要打印的数据写入磁盘] ------>
进程C ------> [把要打印的数据写入磁盘] ------>
```

**详细步骤：**

1. **用户进程要打印** → 不是直接往打印机发数据，而是把打印内容**写入磁盘上的一个"输出井"（缓冲区）**
2. **打印服务进程**（唯一的代理）从输出井里**按顺序**取出打印任务，**由它一个人去操作物理打印机**
3. 用户进程写完就完事了，完全不阻塞，可以继续干别的

### 关键变化：打印机从"独占设备"变成了"共享设备"

|对比|传统方式|SPOOLing 方式|
| :-----------------| :-------------------| :---------------------------------------|
|谁碰打印机|多个用户进程直接抢|只有1个打印服务进程|
|用户进程感觉|独占，互斥|好像每个人都有自己的打印机（逻辑共享）|
|死锁风险|有（多个进程竞争）|**无**（只有一个进程访问）|
|用户进程是否阻塞|阻塞等待打印机空闲|不阻塞，写入磁盘即可|

### 为什么消除了死锁？

因为死锁的前提是**多个进程竞争资源**。SPOOLing 把所有对打印机的请求都**集中**到一个进程处理，其他进程根本不碰打印机，所以：

- **互斥条件** → 只有一个进程访问，无所谓互不互斥了 ✅（被"绕过去"）
- 其他三个条件自然也不会成立

- **厕所管理员**一个一个按顺序喊人进去
- 张三不用等，他的"请求"被存下来了

SPOOLing 的做法是：**不让进程直接碰打印机，而是让它们把要打印的内容交给"代理"** 。

```

                                              ↓
                                    [代理进程（SPOOLing守护进程）]
                                              ↓
                                          🖨️ 打印机（只有代理碰它）
```

### 具体流程：

1. **你的进程想打印** → 不是直接抢打印机，而是把打印数据**写到一个磁盘文件**（叫"输出井"）
2. **你的进程直接返回**，继续干别的事，毫秒级完成 ✅
3. **有一个专门的代理进程**（SPOOLing进程）排队逐一从"输出井"取数据，真正往打印机上送

### 为什么不会死锁？

看关键点：

|传统模式|SPOOLing模式|
| --------------------------| ----------------------------------|
|多个进程抢打印机|只有一个进程（代理）碰打印机|
|需要互斥访问 → 可能死锁|**不需要互斥 → 破坏了"互斥条件"**|
|进程拿着打印机还被阻塞|进程写磁盘就完事，从不持有打印机|

> 你原话"加了一个代理协调，代理给打印机请求排队"——**完全正确！**  代理成了**唯一访问打印机的人**，其他进程都跟磁盘打交道（磁盘可共享），互斥条件被破坏，死锁自然就不可能发生了。

```
① 请求磁带机 + 磁盘
       ↓
    OS分配成功 ✅  （这两个设备当前空闲，立刻拿到）
       ↓

       ↓
⑤ 请求打印机
```

**所以并不是"不等待数据复制"，而是：**

- **请求阶段**（①）：只是跟OS说"我要用这两个设备"，OS说"好，给你"，**瞬间完成**
- **使用阶段**（②）：拿到设备后，才真正发起 I/O 操作（读磁带→写磁盘），**这里确实要等**

## 🔄 为什么不可完全简化 = 死锁？

因为"简化"本质上是在模拟：

>  **"如果这个进程现在能拿到它需要的所有资源，让它运行完释放，会发生什么？"**

如果一个进程能被简化（即**不阻塞**），说明它**当前请求的资源有空闲的**，它可以继续推进。推进完了释放资源，就可能解开其他人的阻塞。

但如果**所有进程都阻塞了**，说明：

- 每个人手上的资源都被别人需要
- 每个人需要的资源都在别人手上
- 没有一个人能向前推进

这就是**死锁的核心定义**：一组进程都在互相等待对方手里的资源，谁也动不了。

    一、基本概念（理解即可）  
    死锁指一组并发进程中的每一个进程都等待其余进程中的某个进程拥有的资源，且这些并发进程在得到自己需要的资源之前不会释放自己拥有的资源，从而造成所有并发进程都想得到资源而又都得不到资源，均不能继续向前推进的状态。

    [图片：两只猫分别拿着筷子和碗，对话框显示“我要吃饭！”]

    死锁、饥饿、死循环都是进程无法顺利向前推进的现象，但有以下区别：  
    ① 死锁是 n 个进程 (n≥2) 循环等待对方手里的资源，导致 n 个进程都被阻塞，均无法向前推进的现象。  
    ② 饥饿是进程由于长期未获得想要的资源，从而无法向前推进的现象。  
    “资源”可以是 CPU——饥饿进程长期处于就绪态，也可以是 I/O 设备——饥饿进程长期处于阻塞态。  
    ③ 死循环是进程在执行过程中，一直进行某个循环，而不跳出的现象。  
    ④ 死锁和饥饿是由于资源的不合理分配引起，而死循环是由于程序的设计错误引起。

    [图片：内存分配示意图，显示主存中有不同进程占用不同内存块，旁边有CPU]

    二、死锁产生的原因（理解即可）  
    死锁的起因通常包含以下三个要素，分别是对不可抢占资源进行争夺、对可消耗资源进行争夺以及进程推进顺序不当。  
    （一）竞争不可抢占资源引起死锁  
    不可抢占资源是指，一旦系统把这类资源分配给某进程后，就不能将它强行收回，而只能在进程用完后等待其自行释放。  
    【例】当一个进程已开始刻录光盘时，如果突然将刻录机分配给另一个进程，则结果必然会损坏正在刻录的光盘，因此只能等刻录好光盘后，由进程自己释放刻录机。

    [图片：光驱]

    另外，磁带机、打印机等也都属于不可抢占资源。

    --- Page 118 ---  
    [图片：打印机图片]

    通常，系统中所拥有的不可抢占资源的数量不足以满足多个进程运行的需要，这使得进程在运行过程中，会因争夺资源而陷入僵局。

    【例】系统中有两个进程 P1 和 P2，他们都准备写两个文件 F1 和 F2，这两个文件都属于不可抢占资源。进程 P1 先打开文件 F1，后打开文件 F2；进程 P2 先打开文件 F2，后打开文件 F1，下面给出了对应的代码。

    ```
    P1                  P2
    ......              ......
    open (F1,w);        open (F2, w);
    open (F2, w);       open (F1,w);
    ```
    [图片：进程资源分配图，左侧显示P1请求F1和F2；右侧显示P1持有F1请求F2，P2持有F2请求F1，形成死锁]

    两个进程 P1 和 P2 在并发执行时，如果 P1 先打开 F1 和 F2，然后 P2 才去打开 F2，则由于文件 F2 已被 P1 打开，故 P2 会被阻塞。

    当 P1 写完文件 F1、F2 并将其关闭时，P2 会由阻塞状态转为就绪状态，并被调度执行后实现打开文件 F1、F2。在这种情况下，P1 和 P2 都能正常运行下去。

    但如果在 P1 打开 F1 的同时，P2 去打开 F2，每个进程都占有一个打开的文件，此时就可能会出现问题。因为当 P1 试图去打开 F2，而 P2 试图去打开 F1 时，这两个进程都会因文件已被打开而阻塞，它们都希望对方关闭自己所需要的文件，但谁都无法进行该操作，因此这两个进程将会无限期地等待下去，进而形成死锁。

    （二）竞争可消耗资源引起死锁

    可消耗资源指进程运行期间动态创建和消耗的资源，通常是由生产者进程创建、由消费者进程消耗的，最典型的可消耗资源就是用于进程间通信的消息。

    【例】假设 P1、P2、P3 三个进程通过消息机制通信，如下所示。

    [图片：进程通信循环图，P1发送m1给P2，P2发送m2给P3，P3发送m3给P1]

    P1 产生消息 m1，利用 send (P2, m1) 原语将它发送给 P2。  
    P2 产生消息 m2，利用 send (P3, m2) 原语将它发送给 P3。  
    P3 产生消息 m3，利用 send (P1, m3) 原语将它发送给 P1。  
    三个进程分别利用 receive(Pi,mj)原语接收消息。

    如果这 3 个进程间的消息通信按下列步骤进行：  
    P1 : ...send (P2, m1); receive (P3, m3); ...  
    P2 : ...send (P3, m2); receive (P1, m1); ...  
    P3 : ...send (P1, m3); receive (P2, m2); ...  
    那么，这3个进程都可以先将消息发送给下一个进程，也都能够接收到从上一个进程发来的消息。因此，这3个进程可以顺利地运行下去，而不会发生死锁。  
    但若改成3个进程都先执行 receive 操作，后执行 send 操作，即按下列步骤进行：  
    P1 : ...receive (P3, m3); send (P2, m1); ...  
    P2 : ...receive (P1, m1); send (P3, m2); ...  
    P3 : ...receive (P2, m2); send (P1, m3); ...  
    那么，这3个进程就会永远阻塞在它们的 receive 操作上，等待一条永远不会发出的消息，于是发生死锁。  
    （三）进程推进顺序不当引起死锁  
    不难发现，进程推进顺序不当，也是产生死锁的一个重要因素，上述的两个例子中都体现了该因素。  
    【练习】假设系统中有 m (m≥1) 个进程，每个进程都需要 k (k≥1) 个相同类型的临界资源，那么确保系统不出现死锁的最小资源数是多少？  
    解：考虑极端情况：假设每个进程被分配了(k-1)个资源，若此时系统资源耗尽，将会发生死锁。  
    但只需要一个额外资源，就可以帮助其中某个进程完成任务，并释放其占有的资源，以便其他进程运行所需。  
    因此，确保系统不出现死锁的最小资源数是(k-1)m+1。

    三、死锁产生的必要条件（理解并记忆）  
    死锁的产生必须同时满足以下四个必要条件，只要其中一个条件不满足，死锁就不会发生。  
    （一）互斥条件  
    进程要求对其分配的资源独占使用，即一个资源在一定时间内只能被一个进程占有，如果其他进程请求该资源，只能等到占有该资源的进程用完后释放。  
    （二）请求和保持条件  
    进程已经占有至少一个资源，但又提出了新的资源请求，而该资源已被其他进程占有，此时进程需阻塞，且不会释放已占有的资源。  
    （三）不可抢占条件  
    一旦一个进程占有一个资源（即一旦它的资源请求被批准），那么该资源就不能被强行抢占，只能由该进程主动释放。

    [图片：两只猫争夺筷子和碗，气泡文字“我要吃饭！”]

    （四）循环等待条件  
    系统发生死锁时，一定存在一条进程请求资源的循环等待链。链上的每个进程都在等待使用相邻进程所占有的资源，形成循环等待。  
    注意，上述四个必要条件并不是相互孤立的，只是每个条件所强调的内容不同。例如，循环等待条件包含了前三个条件。  
    注意，满足这些条件不一定发生死锁，但不满足这些条件一定发生不了死锁。

    四、死锁的处理方法（理解并记忆）  
    一般来说，有四种处理死锁的方法：死锁预防和死锁避免，即系统不允许出现死锁。死锁检测和死锁解除，即当检测到死锁时，通过中止一个进程或抢占一些资源来处理死锁。  
    （一）死锁预防  
    死锁预防是一种静态的死锁解决方法。在产生死锁的四个必要条件均满足时，系统才可能进入死锁状态，因此可以通过破坏四个必要条件中的一个或多个来防止死锁的发生。  
    1.破坏互斥条件  
    多个进程共享临界资源必须保证互斥访问，反过来说，如果只有一个进程访问临界资源就没必要保证互斥访问。对于某些资源，可以为其建立一个专门的进程，用来代理其他进程对这种资源的访问。其他进程想要访问这种资源，就必须告知代理进程。访问这类资源的进程只有代理进程一个，自然不会产生死锁。例如，SPOOLing 技术就能将打印机改为逻辑上的共享设备，达到死锁预防的效果。

    [图片：P1、P2请求打印机，以及通过代理进程请求逻辑打印机的流程图]

    这一方式的缺点在于条件严苛，只有少数资源（如打印机）适合这一方式。因此，预防死锁主要是破坏产生死锁的后三个条件。  
    2.破坏请求和保持条件  
    为了能够破坏“请求和保持”条件，系统必须保证做到：当一个进程在请求资源时，它不能持有不可抢占资源。该保证可通过以下两个不同的协议实现。  
    （1）第一种协议 （开始之前就申请好全部资源 一种不满足就等待）  
    该协议规定，所有进程在开始运行之前，必须一次性地申请其在整个运行过程中所需的全部资源。这样，该进程在整个运行期间便不会再提出资源要求，从而破坏了“请求”条件。  
    系统在分配资源时，只要有一种资源不能满足进程的要求，则即使所需的其他资源都空闲也不分配给该进程，而是让其等待。由于该进程在等待期间未占有任何资源，从而破坏了“保持”条件。

    [图片：硬盘和打印机]

    第一种协议的优点是简单、易行、安全。但缺点也极其明显。  
    ① 资源被严重浪费，严重地降低了资源利用率。进程在开始运行时，就会一次性地占用整个运行过程所需的全部资源，其中有些资源可能仅在运行初期或运行快结束时才会使用。  
    ② 进程经常会发生饥饿现象。因为仅当进程在获得其所需的全部资源后才能开始运行，所以可能由于个别资源长期被其他进程占用，等待该资源的进程迟迟不能开始运行。  
    (2) 第二种协议 （只申请初期需要资源 边运行边释放边继续申请）  
    该协议是对第一种协议的改进，它允许一个进程只获得运行初期所需的资源后，便开始运行。进程运行过程中再逐步释放已分配给自己的且已用毕的全部资源，然后再请求新的所需资源。  
    下面通过一个具体例子来说明第二种协议比第一种协议要好。  
    【例】例如有一个进程，它所要完成的任务是，先将数据从磁带上复制到磁盘上，然后对数据进行处理，最后把结果打印出来。

    [图片：数据从磁带复制到磁盘，再复制到打印机]

    攥着资源等 vs 空手等

    ① 在采用第一种协议时，进程必须在开始时就请求磁带机、磁盘和打印机，然而打印机仅在最后才会用到，所以刚开始就请求占用打印机，既影响其利用率，还会影响其他进程的运行。此外，若刚开始时磁带机和磁盘空闲，但因打印机被很多进程所需而分配给了其他进程，则该进程还需要等待。  
    ② 在采用第二种协议时，进程在开始时只须请求磁带机、磁盘，然后即可运行。等到磁带上的全部数据已复制到磁盘中并已处理好后，便可将磁带机和磁盘释放掉，再去请求打印机。这不仅能使进程更快地完成任务，提高设备的利用率，还可减少进程发生饥饿现象的概率。  
    3.破坏不可抢占条件 （资源申请失败 就释放已保持的左右资源）  
    为了能破坏“不可抢占”条件，协议中规定，当一个已经保持了某些不可抢占资源的进程提出新的资源请求而不能得到满足时，它必须释放已经保持的所有资源，待以后需要时再重新申请。这意味着进程已占有的资源会被暂时释放，从而破坏了“不可抢占”条件。  
    该方法实现起来比较复杂，且需要付出很大的代价。因为一个不可抢占的资源（如打印机、刻录机等）在使用一段时间后被抢占，可能会造成进程前一阶段工作的失效。

    [图片：光盘/刻录机]

    4.破坏循环等待条件   
    一个能保证“循环等待”条件不成立的方法是，对系统的所有资源进行编号，并要求进程只能按资源编号严格递增或递减的顺序请求资源。  
    例如，进程在申请资源时，如果一个资源的编号小于进程已获得的某个资源的编号，则进程不能请求该资源。  
    【例】假设在系统中有 10 个资源，可以把它们编号为 R1,R2,...,R10。假设进程 P1 当前已经申请了资源 R1、R3 和 R6。此后，它就不能再申请 R6 之前的资源了，只能申请 R6 以后的资源。

    [图片：资源编号 R1 到 R10，箭头指向 P0]

      
    可以证明，该方案下，死锁无法发生。  
    【例】假设 P0、P1、P2、P3 分别占用 R0、R1、R2、R3，P0、P1、P2 依次申请 R1、R2、R3。

    [图片：资源分配图，显示进程P0、P1、P2、P3与资源R0、R1、R2、R3之间的分配与请求关系]

    若要实现循环等待链，则 P3 必须申请 P0 占用的 R0，显然，由于 R0 编号小于 R3，因此 P3 不可能有申请 R0 的请求，如果其需要 R0，则必须在申请 R3 前就申请 R0。因此，无法形成循环等待链。  
    这种预防死锁策略和前两种策略相比，其资源利用率和系统吞吐量都有比较明显的改善，但也存在下列问题（了解即可）。  
    ① 为系统中各类资源规定的序号必须相对稳定，这限制了新类型设备的增加。  
    ② 尽管在为资源的类型分配序号时已经考虑到了大多数作业在实际使用这些资源时的顺序，但也经常会发生这种情况：作业使用各类资源的顺序与系统规定的顺序不同，造成对资源的浪费。  
    第三，为了方便用户，系统对用户在编程时所施加的限制条件应尽量少，然而这种按规定次序申请资源的方法必然会限制用户进行简单、自主的编程。

    【练习】一次分配所有资源的方法可以预防死锁的发生，它破坏死锁 4 个必要条件中的（ ）。  
    A.互斥  
    B.占有并请求  
    C.非剥夺  
    D.循环等待  
    答案：B 严格来说是请求

    5.哲学家进餐问题  
    如图所示，有五个哲学家，他们共用一张圆桌，围坐在五把椅子上，桌上有五个碗和五只筷子，每根筷子只能互斥使用。

    [图片：哲学家进餐示意图，五只猫代表哲学家围坐在圆桌旁，桌上有碗和筷子]

    当哲学家饥饿的时候，他会尝试拿自己左右两边的筷子，只有当他手上有两根筷子的时候才能吃饭。饭后，他放下筷子，继续思考。为了实现对筷子的互斥使用，可以用 1 个信号量表示 1 根筷子，由这 5 根筷子对应的 5 个信号量构成信号量数组，描述如下。

    ```c
    semaphore chopstick[5]={1,1,1,1,1}; //初始化信号量
    //其中第i位哲学家的活动可描述如下：
    while (TRUE) {
        P(chopstick[i]);          //取左边筷子
        P(chopstick[(i+1) % 5]);  //取右边筷子
        ...
        吃饭;
        ...
        V(chopstick[i]);          //放下左边筷子
        V(chopstick[(i+1) % 5]);  //放下右边筷子
        ...
        思考:
        ...
    }
    ```
    在上述描述中，哲学家饥饿时总是会先去拿他左边的筷子，即执行 wait(chopstick[i])；成功后，再去拿他右边的筷子，即执行 wait(chopstick [(i+1)%5])；又成功后便可进餐。进餐完毕，先放下他左边的筷子，再放下他右边的筷子。

    上述解法有可能引起死锁。假如 5 位哲学家同时饥饿而各自拿起左边的筷子，这时就会使 5 个信号量 chopstick 均为 0；当他们再试图去拿右边的筷子时，都将会因无筷子可拿而进行无限期的等待。对于这样的死锁问题，可采取以下 3 种方法进行解决。  
    （1）至多只允许有 4 位哲学家同时去拿左边的筷子，最终能保证至少有 1 位哲学家能够进餐，并在进餐毕时能释放出他用过的 2 根筷子，从而使更多的哲学家能够进餐。  
    （2）仅当哲学家的左右两根筷子均可用时，才允许他拿起筷子进餐。  
    （3）规定奇数号哲学家先拿他左边的筷子，然后再去拿他右边的筷子；而偶数号哲学家则相反。按此规定，5 位哲学家都先竞争奇数号筷子，获得后，再去竞争偶数号筷子，最后总会有一位哲学家能获得两根筷子而进餐。

    基于方法（1）的代码如下。

    ```c
    semaphore chopstick[5]={1,1,1,1,1}; //初始化信号量
    semaphore priority=4;               //设置允许拿筷子的哲学家数量
    //其中第 i 位哲学家的活动可描述如下：
    while (TRUE) {
        P(priority);
        P(chopstick[i]);                //取左边筷子
        P(chopstick[(i+1) % 5]);        //取右边筷子
        ...
        吃饭;
        ...
        V(chopstick[i]);                //放下左边筷子
        V(chopstick[(i+1) % 5]);        //放下右边筷子
        V(priority);
        ...
        思考:
        ...
    }
    ```
    【练习】在哲学家就餐问题中，若同时存在左撇子和右撇子（将先拿起左边筷子的人称为左撇子，而将先拿起右边筷子的人称为右撇子），则不会发生死锁，因为破坏了（ ）。  
    A.互斥条件        B.请求与保持条件        C.不剥夺条件        D.循环等待条件  
    答案：D

    （二）死锁避免（理解即可）  
    1.基本概念  
    死锁预防是静态的，在系统运行前限定了进程的运行规则、破坏了死锁产生的必要条件中的至少一个。<u>死锁避免是一种</u>​<u>**运行时的检查方法**</u>​<u>，在进程请求资源时判断是否可能导致死锁；如果可能，就将其拒绝。</u>  
    下面基于红绿灯和交警叔叔的例子说明死锁预防和死锁避免的区别。  
    【例】假设有一个十字路口，如果要发生车辆相撞的话，显然，需要满足不同方向的车同时往十字路口中心开。  
    [图片：十字路口交通示意图，带有红绿灯和箭头指示]

    死锁预防：设置红绿灯破坏发生相撞的条件，由于各个车辆都遵守“红灯停、绿灯行”的规则，因此，不可能出现两辆不同方向的车同时往十字路口中心开。

    [图片：左侧为十字路口示意图，中间为红绿灯，右侧为十字路口示意图]

    死锁避免：若到了下班高峰期，有大量沿着绿色箭头方向移动的车。由于红绿灯对每个方向的车流的绿灯时间都是60s，而红色方向的车辆并不需要那么长时间绿灯，只需要30s，而绿色方向却需要90s的绿灯时间。如果继续按照红绿灯规则的话，非常容易导致绿色方向的车辆拥堵。因此可以安交警叔叔动态指挥。

    [图片：左侧为十字路口示意图，中间为交警照片，右侧为十字路口示意图]

    彦祖蜀黍根据各个方向的车流量动态指挥，比如绿色方向运行车辆通过的时间从60s变成90s，红色的变成30s，只要确保不发生死锁就行。  
    因此，死锁避免相比死锁预防，施加了更精确的约束，并带来了更好的性能。

    2.安全状态 （安全的运行序列）  
    死锁主要是由不安全的进程推进顺序引起的，可以将系统的状态分为安全状态和不安全状态。安全状态指系统能按某种推进顺序为进程分配其所需资源，最终使每个进程都可顺利完成。如果系统无法找到这样一个安全序列，则称系统处于不安全状态。只要系统能一直保持在安全状态，就可以避免死锁发生。

    3.银行家算法（理解并记忆）（优先选择可满足其资源缺口的进程）  
    最具代表性的死锁避免算法是银行家算法。该算法中，操作系统是银行，系统资源是资金，进程相当于向银行申请贷款的客户。不妨假装自己是一个银行家，如果有两个老板同时找你贷款，银行只能贷出 50w，而 A 老板需要 30w，B 老板需要 100w，那显然借给 A 老板更安全。因为借给 B 老板后，其资金缺口仍有 50w，万一他借不到更多的钱，项目就会烂尾。  
    [图片：大楼]  
    因此，操作系统分配资源时，优先选择可以满足其资源缺口的进程，进而保证系统不会进入不安全状态，以避免死锁。  
    为实现银行家算法，每一个新进程在进入系统时，它必须申明在运行过程中，可能需要每种资源类型的最大单元数目，其数目不应超过系统所拥有的资源总量。  
    当进程请求资源时，系统必须确定是否有足够的资源分配给该进程，以及进程申请资源数是否超过其所申报的数目。若资源足够且未超过申报数目，再计算在将这些资源分配给进程后，是否会使系统处于不安全状态。如果不会，才将资源分配给它，否则让进程等待。

    (3) 银行家算法  
    理解了系统是如何进行安全性检测之后，再来学习一下银行家算法，整体流程如下。  
    设 Requesti 为进程 Pi 的请求向量，如果 Requesti[j]=K，则说明进程 Pi 需要 K 个 Rj 类型的资源。  
    ① 每次有进程申请资源时，都先判断  
    a. 进程本次申请的资源数是否超过尚需的需求量，即是否满足 Requesti ≤ Need[i]。若不满足则请求失败，进程所需要的资源数量已经超过了它所申报的最大需求数量。  
    b. 系统可用资源是否能满足申请的资源数目，即是否满足 Requesti ≤ Available，若不满足说明系统还没有足够的可用资源，进程 Pi 必须等待；

    --- Page 128 ---  
    ② 若以上判断都通过后，则系统会试探着为进程 Pi 分配资源，即更新以下数据结构。

    Allocation[i] = Allocation[i]+Requesti（更新 Pi 得到的资源数量）  
    Need[i] = Need[i]-Requesti（更新 Pi 还需要的资源数量）  
    ③ 系统通过安全性算法检查为该进程分配资源后，是否仍处于安全状态。若安全，资源就被正式分配给进程 Pi；否则，将本次的试探分配作废，恢复为原来的资源分配状态，让进程 Pi 等待。

    （四）死锁解除（理解即可）  
    如果利用死锁检测算法检测出在系统中已发生了死锁，则应立即采取相应的处理措施来解除死锁。最简单的方法就是通过人工方法解除死锁。

    [图片：两只猫抢筷子和碗的图，配文“我要吃饭！”]

    另一种方法则是利用死锁解除算法，把系统从死锁状态中解脱出来。  
    通常采用的解除死锁的两种方法是：  
    1.抢占资源，从一个或多个进程中抢占足够数量的资源，然后将它们分配给死锁进程，以解除死锁状态。  
    2.终止死锁进程，即终止系统中的一个或多个死锁进程，直至打破循环等待，使系统从死锁状态中解脱出来。终止死锁进程的方法有以下两种（了解即可）：  
    （1）终止所有死锁进程  
    终止所有死锁进程是一种最简单的方法，但由此付出的代价可能会很大，因为其中有些进程可能已经运行了很长时间并已接近结束。  
    （2）逐个终止死锁进程  
    相比于终止所有死锁进程，稍微温和一点的方法是：按照某种顺序逐个地终止死锁进程，直至有足够的资源来打破循环等待，把系统从死锁状态中解脱出来。  
    但该方法所付出的代价也可能会很大，因为每终止一个进程，都需要用死锁检测算法来确定系统死锁是否已被解除，若未被解除，则需再终止另一个进程。
