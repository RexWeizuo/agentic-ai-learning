# 六、死锁检测与解除

## （一）死锁检测

为了能对系统中是否已发生了死锁进行检测，在系统中必须：

① 保存有关资源的请求和分配信息

② 提供一种算法，它利用这些信息来检测系统是否已进入死锁状态。

### 资源分配图

可利用资源分配图来描述资源的请求和分配信息。

[图片：资源分配图，包含进程P1、P2和资源R1、R2的节点及箭头]

[1]

如图所示，用圆圈代表一个进程，用方框代表一类资源。由于一种类型的资源可能有多个，用方框中的一个点代表一类资源中的一个资源。此时，请求边是由进程指向方框中的Rj，而分配边则应始于方框中的一个点。其中，P1 进程已经分得了两个 R1 资源，并又请求一个 R2 资源；P2 进程分得了一个 R1 和一个 R2 资源，并又请求 R1 资源。

[1] 汤小丹,王红玲,姜华,汤子瀛.计算机操作系统（慕课版）[M].北京:人民邮电出版社,2021.

### 死锁定理

可以基于把资源分配图加以简化的方法，来检测系统是否为死锁状态。简化方法如下：

① 在资源分配图中，找出一个既不阻塞又非独立的进程结点 Pi。在顺利的情况下，Pi 可获得所需资源而继续运行，直至运行完毕，再释放其所占有的全部资源，这相当于消去 Pi 的请求边和分配边，使之成为孤立的结点。在图（a）中，将 P1 的两个分配边和一个请求边消去，便形成图（b）所示的情况。

[图片：资源分配图简化过程(b)，显示P1, P2, R1, R2及其连接关系]

② P1 释放资源后，便可使 P2 获得资源而继续运行，直至 P2 完成后又释放出它所占有的全部资源，形成图（c）所示的情况，即将 P2 的两条请求边和一条分配边消去。

[图片：资源分配图简化过程(c)，显示P1, P2, R1, R2及其连接关系]

③ 在进行一系列的简化后，若能消去图中所有的边，使所有的进程结点都成为孤立结点，则称该图是可完全简化的；若不能通过任何过程使该图完全简化，则称该图是不可完全简化的。若资源分配图是不可完全简化的，则发生了死锁。

## （二）死锁解除

如果利用死锁检测算法检测出在系统中已发生了死锁，则应立即采取相应的处理措施来解除死锁。最简单的方法就是通过人工方法解除死锁。

[图片：两只猫抢筷子和碗的图，配文“我要吃饭！”]

另一种方法则是利用死锁解除算法，把系统从死锁状态中解脱出来。

通常采用的解除死锁的两种方法是：

1.抢占资源，从一个或多个进程中抢占足够数量的资源，然后将它们分配给死锁进程，以解除死锁状态。

2.终止死锁进程，即终止系统中的一个或多个死锁进程，直至打破循环等待，使系统从死锁状态中解脱出来。终止死锁进程的方法有以下两种（了解即可）：

（1）终止所有死锁进程

终止所有死锁进程是一种最简单的方法，但由此付出的代价可能会很大，因为其中有些进程可能已经运行了很长时间并已接近结束。

（2）逐个终止死锁进程

相比于终止所有死锁进程，稍微温和一点的方法是：按照某种顺序逐个地终止死锁进程，直至有足够的资源来打破循环等待，把系统从死锁状态中解脱出来。

但该方法所付出的代价也可能会很大，因为每终止一个进程，都需要用死锁检测算法来确定系统死锁是否已被解除，若未被解除，则需再终止另一个进程。

### 🧠 深度理解：资源图简化与死锁检测（来自 2.5 DS）

> **Q：**  为什么资源图不可完全简化就是发生了死锁？能不能给个例子？

> **A：**  简化资源图 = 模拟进程执行顺序。找既不阻塞又非孤立的进程，假设它执行完释放资源，把资源分给在等的进程，消掉它的边。重复直到：

> - 所有边都能消掉 → **无死锁**

> - 剩一些边消不掉 → **有死锁**

> 消不掉的边恰恰构成的就是循环等待环——每个进程都在等别人释放，但没人能释放。

> **Q：**  资源图简化不也类似银行家算法吗？能消掉所有边 = 能找到安全序列？

> **A：**  底层逻辑一致，但不是一回事：

> | | 资源图简化（死锁检测） | 安全性算法（死锁避免） |

> | :--- | :--- | :--- |

> | 时机 | 事后，检测是否已死锁 | 事前，分配前预防 |

> | 过程 | 模拟释放+重新分配（消边）| 计算 Need ≤ Available |

> | 可简化 = ? | 无死锁 | 存在安全序列 |

> | 不可简化 = ? | 有死锁 | 拒绝分配 |

> 一个**事后验尸**，一个**事前审批**，算的都是同一件事——"按这个顺序能不能走完"。

### 三、简答题

**16.**  简述资源分配图的组成要素。如何从图中判断一个进程是“既不阻塞又非孤立”的？该进程当前请求的资源数目 ≤ 系统可分配给它的数目（即没有因为资源不足而阻塞），且该进程还有边连接（非孤立结点），可以继续运行。

**17.**  简述资源分配图简化方法的基本步骤，并解释为什么不可完全简化意味着发生了死锁。

<span data-type="text" style="color: var(--b3-font-color8);">① 在图中找到既不阻塞又非孤立的进程 Pi。
② 假设 Pi 获得所需资源并运行完毕，释放其占用的全部资源，消去 Pi 的所有请求边和分配边，使 Pi 成为孤立结点。
③ 重复上述步骤，直到无法继续简化。
若所有边都能消去，图可完全简化，无死锁。
若无法消去所有边，剩余的部分必然形成循环等待环——每个剩余进程都在等待别人释放资源，而没有人能运行完成，这就是死锁。</span>

**18.**  比较资源分配图简化与银行家算法安全性检测的异同点。

相同点：两者本质上都在寻找一个能让所有进程顺利完成的推进序列。  
不同点：

- 时机：资源图简化是**事后检测**死锁是否已发生；银行家算法安全性检测是​**事前避免**，在分配资源前判断。
- 过程：资源图简化模拟进程释放资源并消边；安全性算法计算每个进程的 Need 与 Available，寻找安全序列。
- 可完成等价：资源图可完全简化 ⇔ 系统无死锁（已发生）；安全序列存在 ⇔ 系统处于安全状态（当前及未来）。

**19.**  文档中提到解除死锁的两种常用方法是什么？逐个终止死锁进程的优缺点是什么？

缺点是每终止一个都需要重新检测死锁是否解除，代价可能较大，且难以决定终止顺序。

**20.**  在一个资源分配图中，进程 P1 持有资源 R1，等待资源 R2；进程 P2 持有资源 R2，等待资源 R1。请画出该资源分配图，并通过简化过程判断是否会发生死锁。

- 分配边：R1 的点 → P1，R2 的点 → P2
- 请求边：P1 → R2 的点，P2 → R1 的点

题10 修定为：  
A. 死锁检测算法利用资源请求和分配信息判断系统是否进入死锁  
B. 资源分配图不可完全简化是系统处于不安全状态的必要条件  
C. 最简单解除死锁的方法是人工干预

### 简答题要点

**16.**   
资源分配图组成：圆圈代表进程，方框代表一类资源，方框中的点代表该类资源的一个实例。请求边从进程指向方框中的点，分配边从方框中的点指向进程。  
“既不阻塞又非孤立”的进程：该进程当前请求的资源数目 ≤ 系统可分配给它的数目（即没有因为资源不足而阻塞），且该进程还有边连接（非孤立结点），可以继续运行。

**17.**   
简化步骤：  
① 在图中找到既不阻塞又非孤立的进程 Pi。  
② 假设 Pi 获得所需资源并运行完毕，释放其占用的全部资源，消去 Pi 的所有请求边和分配边，使 Pi 成为孤立结点。  
③ 重复上述步骤，直到无法继续简化。  
若所有边都能消去，图可完全简化，无死锁。  
若无法消去所有边，剩余的部分必然形成循环等待环——每个剩余进程都在等待别人释放资源，而没有人能运行完成，这就是死锁。

**18.**   
相同点：两者本质上都在寻找一个能让所有进程顺利完成的推进序列。  
不同点：

- 时机：资源图简化是**事后检测**死锁是否已发生；银行家算法安全性检测是**事前避免**，在分配资源前判断。
- 过程：资源图简化模拟进程释放资源并消边；安全性算法计算每个进程的 Need 与 Available，寻找安全序列。
- 可完成等价：资源图可完全简化 ⇔ 系统无死锁（已发生）；安全序列存在 ⇔ 系统处于安全状态（当前及未来）。

**19.**   
两种方法：① 抢占资源，从一个或多个进程中强行抢占足够资源分配给死锁进程；② 终止死锁进程，打破循环等待。  
逐个终止的优缺点：优点是相比全部终止更温和，可能只终止少数进程就解除死锁；缺点是每终止一个都需要重新检测死锁是否解除，代价可能较大，且难以决定终止顺序。

**20.**   
画图：

- 两个圆圈 P1、P2
- 两个方框 R1、R2，各有一个点
- 分配边：R1 的点 → P1，R2 的点 → P2
- 请求边：P1 → R2 的点，P2 → R1 的点

简化判断：  
P1 请求 R2，但 R2 已被 P2 占用；P2 请求 R1，但 R1 已被 P1 占用。两个进程都阻塞，无法找到“既不阻塞又非孤立”的进程。因此图不可简化，死锁已发生。
