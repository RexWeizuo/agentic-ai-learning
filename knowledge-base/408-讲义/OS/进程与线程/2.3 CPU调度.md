# 2.3 CPU调度

‍2.3 CPU 调度
处理器调度就是合理有效地把 CPU 分配给作业（进程）。
在多道程序设计系统中，如果只有一个 CPU，有两个或两个以上的进程或线程处于就绪状态，当 CPU 空闲时，它们就会竞争 CPU，必须把 CPU 分配给其中一个进程或线程来运行。在操作系统中，完成这项任务的程序称为调度程序（scheduler），该程序使用的算法称为调度算法。

[图片：内存、CPU与调度程序示意图，包含主存、CPU、scheduler等模块]

一、分级调度（理解即可）
（一）基本概念
作业是用户要求计算机完成的关于一次业务处理的全部工作，包括作业的提交、执行、输出等过程。
作业是早期批处理系统引入的一个概念，例如在单道批处理系统中，监督程序每次会从外存中调入一个作业处理（除了大型机之外，在微机、智能手机以及嵌入式计算机系统中没有作业的概念）。

[图片：单道批处理系统工作流程图，展示了从纸带机到打印机的数据流向，包含监督程序、外存、主机等环节]

作业从提交到完成，往往要经过多级调度才能实现。下面以批处理系统为例展示典型的三级调度。
【例】一个作业从提交给计算机系统到执行结束，一般都要经历提交、收容、执行和完成 4 个状态。

[图片：作业状态转换与三级调度示意图，展示了提交状态、收容状态、后备作业、就绪队列、阻塞队列、外存、内存、CPU之间的调度关系]

--- Page 82 ---
一个作业从输入设备进入外存的过程称为提交状态。
若一个作业的全部信息已输入到外存中，在它还未被作业调度程序选取并执行之前，该作业处于收容状态（后备状态）。
作业调度程序会从后备作业中选取若干作业到内存中投入运行。它为被选中的作业分配内存、I/O设备等必要的资源，并建立相应的进程。
若只有一个CPU，就绪队列中有多个就绪态的进程，则进程调度程序需要选择一个进程使用处理机。
当作业运行完毕，但它所占用的资源尚未全部被系统回收时，该作业处于完成状态，系统需要做回收资源等善后处理工作。
另外，为了提高资源的利用率，有些OS会把部分在内存中处于就绪状态或等待状态而在短时期内又得不到执行的进程换出内存。这样，在外存中，除了处于后备状态的作业外，还有处于等待内存资源的进程。
显然，通过上述过程可以得知，一个作业从被提交到运行结束并不是一帆风顺的，中间需要经过多次调度，分别是作业调度、交换调度和进程调度（不同OS的调度层级不同）。
（二）作业调度
作业调度又称长程调度，它的调度对象是作业。其主要功能是决定后备队列中的哪几个作业调入内存，为它们创建进程、分配必要的资源，并将它们放入就绪队列。
作业调度主要用于多道批处理系统中，在分时和实时系统中不设置作业调度。
（三）进程调度
进程调度又称短程调度，其所调度的对象是进程（或内核级线程）。其主要功能是决定就绪队列中的哪个进程应获得处理机。
进程调度是最基本的一种调度，在多道批处理、分时和实时三种类型的OS中，都必须配置这级调度。
（四）交换调度
交换调度又称内存调度。其主要任务是外存中的就绪挂起态或等待挂起态的进程调入内存，或把处于就绪状态或等待状态的进程交换到外存中。被交换到外存的进程的状态为挂起态。
引入挂起态后，一个进程有如下七种状态。

[图片：进程状态转换图，展示了内存和外存区域。内存中包含创建态、就绪态、执行态、阻塞态、终止态；外存中包含就绪挂起态、阻塞挂起态。箭头表示状态转换，如时间片用尽、I/O请求、换出内存、换入内存等]

在上述三种调度中，进程调度的运行频率最高，分时系统中大概10~100ms一次；作业调度的周期较长，大约几分钟一次；交换调度的运行频率一般介于上述两种调度之间。
作业调度、交换调度、进程调度又分别被称为高级调度、中级调度、低级调度。

二、调度目标与性能指标（理解即可）
当后备队列中有多个作业时，需要按照某种规则（算法）选出某个作业进行调度；
当就绪队列中有多个就绪态的进程时，也需要按照某种规则（算法）选出某个进程进行调度。

[图片：示意图展示了等待队列和就绪队列。等待队列中有文档图标指向队首；就绪队列中有“进程”圆圈指向队首，最终指向CPU]

不同的调度目标下有不同的调度算法，例如，在批处理系统、分时系统和实时系统中，通常都采用不同的调度方式和算法。具体的调度算法一会再讲，这里先介绍调度目标。
（一）处理机调度算法的共同目标
不同处理机调度算法的共同目标主要有以下三个。
1.资源利用率
为提高系统的资源利用率，应使系统中的处理机和其它所有资源都尽可能地保持忙碌状态，其中最重要的处理机利用率可用以下方法计算：
$$ \text{CPU 利用率} = \frac{\text{CPU 有效工作时间}}{\text{CPU 有效工作时间} + \text{CPU 空闲时间}} $$
2.公平性（了解即可）
公平性是指应使诸进程都获得合理的 CPU 时间，不会发生进程饥饿现象（饥饿指长时间接收不到 CPU 服务）。
3.平衡性（了解即可）
由于在系统中可能具有多种类型的进程，有的属于计算型（CPU 繁忙型），有的属于 I/O 型。为使系统中的 CPU 和各种外部设备都能经常处于忙碌状态，调度算法应尽可能保持系统资源使用的平衡性。
若某个进程在它的整个执行过程中，大部分时间都处于运行和就绪状态，这样的进程就称为 CPU 繁忙的进程；反之，如果一个进程，它的大部分时间都处于阻塞状态，这样的进程就称为 I/O 繁忙的进程。

（二）批处理系统的目标（理解并记忆）
1.平均周转时间短
从作业的角度看，周转时间指从作业被提交给系统开始，到作业完成为止的这段时间间隔（称为作业周转时间）。

[图片：作业状态转换图，展示了从输入设备到外存（收容状态、等待队列、就绪挂起队列、阻塞挂起队列），经过高级调度进入内存（就绪队列、阻塞队列），再经过低级调度进入CPU的过程，以及中级调度在外存和内存之间的调度]

周转时间计算公式如下：
$$T = T_{finish} - T_{arrive}$$
其中，Tfinish表示作业完成时间，Tarrive表示作业到达时间。
作业的周转时间短意味着用户可以更快地拿到结果，从操作系统的角度考虑，会希望平均周转时间最短，因为这能直接反映出操作系统更高的效率。
平均周转时间指所有作业的周转时间的平均值。
$$\overline{T} = \frac{1}{n} \left[ \sum_{i=1}^{n} T_i \right]$$
其中 n 表示作业总数，Ti 表示作业 i 的周转时间。
为了更清晰地描述作业在其周转时间中，等待和执行时间的具体分配状况，往往使用带权周转时间，即作业的周转时间 T 与系统为它提供服务的时间 Ts 之比，计算公式如下：
$$W = \frac{T}{t_s}$$
平均带权周转时间计算公式如下：
$$\overline{W} = \frac{1}{n} \left[ \sum_{i=1}^{n} W_i \right] = \frac{1}{n} \left[ \sum_{i=1}^{n} \frac{T_i}{t_i} \right]$$
其中，Ti 表示作业 i 的周转时间，ti 表示操作系统为作业 i 提供的服务时间。

【练习】设有 4 个作业同时到达，每个作业的执行时间均为 2h，它们在一台处理器上按单道式运行，则平均周转时间为（ ）。
A.1h
B.5h

从进程的角度看，周转时间指从进程进入就绪队列到最终执行完毕的时间。

2.3 CPU 调度

2.系统吞吐量高
吞吐量是指在单位时间内系统所完成的作业数，它与批处理作业的平均长度有关。如果单纯是为了获得高的系统吞吐量，就应尽量多地选择短作业运行。
【例】假设有一个批处理系统，在1小时（60分钟）内需要处理多个作业，作业分为两类：
短作业：每个耗时1分钟；
长作业：每个耗时10分钟；
显然，如果1小时内全部执行短作业，则系统吞吐量最高。
3.处理机利用率高
对于大、中型计算机，CPU价格十分昂贵，致使处理机的利用率成为衡量系统性能的十分重要的指标。如果单纯是为使处理机利用率高，应尽量多地选择计算量大的作业运行。
（三）分时系统的目标
1.响应时间快
响应时间是从用户通过键盘提交一个请求开始，直到屏幕上显示出处理结果为止的一段时间间隔。
[图片：终端界面截图，显示命令 ./hello 输出 hello world!]
2.均衡性（了解即可）
用户对响应时间的要求并非完全相同，通常用户对较复杂任务的响应时间允许较长，而对较简单任务的响应时间则要短。因此，均衡性指系统响应时间的快慢应与用户所请求服务的复杂性相适应。
（四）等待时间
从进程的角度看，等待时间指进程在就绪队列中的等待时间。
从作业的角度看，如果作业i提交给系统的时刻是 $S_i$，被作业调度程序选中时刻是 $K_i$，那么该作业的等待时间为 $W_i=K_i-S_i$。

三、调度算法（理解并记忆）
接下来讨论各种常用的进程调度算法和作业调度算法。
（一）先来先服务调度算法
先来先服务调度算法（FCFS）也称为先进先出算法（FIFO），可以适用于作业调度、进程调度。
在作业调度时，系统会按照作业放入外存队列的顺序，依次将其调入内存执行。在进程调度时，系统会从就绪队列中选择一个最先进入该队列的进程。

（二）短进程（作业）优先调度算法
短进程（作业）优先调度算法（Shortest Job/Process First），可以适用于作业调度、进程调度。
在进程调度时，称为短进程优先调度算法，会根据就绪队列中排队的进程所预估的处理器使用时间，每次调度选择预估剩余处理器使用时间最短的进程。
在作业调度时，称为短作业优先调度算法，会根据外存队列中作业所要求的执行时间来调度，每次调度选择所要求的执行时间最短的作业。
短进程优先算法有两种实现方案：
1.不可抢占方式
当前进程正在运行时，即使来了一个比它更短的进程，也不会被打断，只有当它运行完毕或是被阻塞时，才会让出 CPU，进行新的调度。
2.可抢占方式
如果一个新的短进程到来，其运行时间小于当前正在运行的进程的剩余时间，则立即抢占 CPU 去运行。这种方法也称为最短剩余时间优先（Shortest Remaining Time First, SRTF）。

SPF 调度算法较之 FCFS 算法有了明显的改进，但仍然存在不容忽视的缺点：
①在采用这种算法时，要先知道每个作业/进程的运行时间，即使是程序员也很难准确估计，该算法在现实中难以实现。
② 对长作业非常不利，长作业的周转时间会明显地增长。更严重的是，该算法完全忽视作业的等待时间，可能使作业等待时间过长，出现饥饿现象。

③ 该调度算法完全未考虑作业的紧迫程度，故不能保证紧迫性作业能得到及时处理。
在分析题目时，注意题目对于该调度算法是否可抢占的描述。若未说明，一般默认为非抢占式。
（三）优先级调度算法
优先级进程调度算法下，把处理机分配给就绪队列中优先级最高的进程。
优先级可以用一个整数来表示，例如 0~255 中的某一整数，又把该整数称为优先数。如果进程 1 和进程 2 的优先数分别为 5 和 2，若优先数越大优先级越高，则进程 1 的优先级高于进程 2。
该算法可分成如下两种。
1.非抢占式优先级调度算法
该算法规定，一旦把处理机分配给优先级最高的进程后，该进程便一直执行下去直至完成，或者因该进程发生某事件而放弃处理机时，才可以将处理机重新分配给另一进程。
[图片：就绪队列指向CPU的示意图]
2.抢占式优先级调度算法
如果在进程运行过程中，就绪队列中出现了更高优先级的进程，则这个拥有更高优先级的进程会抢占处理器资源。抢占式优先级调度算法常用于对实时性要求较高的系统中。
根据优先级是否可以动态变化，分为静态优先级和动态优先级。
（1）静态优先级
静态优先级指各个进程的优先级在调度程序运行之初就已经确定好，整个运行期间不会改变。

设计优先级一般有以下几个依据：
① 进程类型
通常系统进程的优先级高于一般用户进程的优先级。
交互式进程的优先级高于非交互式进程。
I/O 型进程的优先级高于计算型进程。
由于 I/O 设备（如打印机）的处理速度要比 CPU 慢得多，因此若将 I/O 型进程的优先级设置得更高，就更可能让 I/O 设备尽早开始工作，进而提升整体效率。
② 进程对资源的需求：对资源要求少的进程应赋予较高的优先级。
③ 用户要求：根据进程的紧迫程度确定优先级。
该算法系统开销小，但可能会出现优先级低的进程长期没有被调度的情况。
(2) 动态优先级
动态优先级是指在创建进程之初，先赋予其一个优先级，然后其值随进程的推进或等待时间的增加而改变，以便获得更好的调度性能。
【例】可以规定在就绪队列中的进程随其等待时间的增长，使其优先级相应提高。

[图片：进程调度示意图，显示进程进入就绪队列，箭头指向CPU]

若所有的进程都具有相同优先级初值，则最先进入就绪队列的进程会因其优先级变得最高，而优先获得处理机，这相当于 FCFS 算法。
若所有的就绪进程具有各不相同的优先级初值，那么对于优先级初值低的进程，在等待了足够的时间后，也可以获得处理机。
当采用抢占式调度方式时，若再规定当前进程的优先级随运行时间的推移而下降，则可防止一个长作业长期地垄断处理机。
(四) 高响应比优先调度算法
高响应比优先调度算法综合考虑进程（作业）的等待时间和运行时间来调度进程。
该算法根据以下公式来确定进程（作业）的优先级。
$$优先级 = \frac{等待时间 + 要求服务时间}{要求服务时间} = \frac{等待时间}{要求服务时间} + 1$$
【练习】一个作业 8:00 到达系统，估计运行时间为 1h。若 10:00 开始执行该作业，其响应比是（ ）。
A.2
B.1
C.3
D.0.5
答案：C
由于等待时间是变化的，则优先级也是动态变化的，随着进程的等待时间增加，该进程的优先级就会变高，更容易被调度。
【例】假设有 2 个进程，它们进入就绪队列的时刻和预计运行时间如下表所示。（单位：ms）

| 进程名 | P1 | P2 | P3 |
| :--- | :---: | :---: | :---: |
| 到达时刻 | 0 | 1 | 2 |
| 执行时间 | 2 | 3 | 1 |

如果采用不可抢占的高响应比优先调度算法，t=2 时，应该调度谁？
解：初始时，P1 到达，开始执行。
t=2 时，P1 执行结束，此时有两个就绪态进程，分别是 P2、P3，其优先级分别为 4/3 以及 1，因此接下来调度 P2。
不难发现，当两个进程要求时间相同时，则优先级由等待时间决定，这符合 FCFS 算法。
当两个进程的等待时间相同时，短进程会有更高的优先级。因此，该算法兼顾长短作业。
此外，这种算法不会发生饥饿现象，长进程只要等待足够长的时间，优先级就会高于短进程。

（五）时间片轮转调度算法（Round Robin,RR）
该算法下，处理器使用时间被划分成一个个时间片，公平地分给每一个等待处理器资源的就绪进程。
[图片：时间片划分示意图（10ms, 10ms...）及就绪队列进程轮转进入CPU的示意图]
时钟中断发生时，系统会修改当前进程在时间片内的剩余时间，当一个进程分配的时间片耗尽后，其会重新进入就绪队列等待下一次分配时间片。若一个时间片未耗尽，但进程已完成工作，或因阻塞等原因放弃处理器资源，调度程序会选择下一个就绪进程进行调度。
该算法适用于分时操作系统中的进程调度，每个时间片结束后操作系统都会调度下一个就绪进程运行，有利于实现人机交互。
另外，时间片不能太长或太短。当时间片很小时，操作系统需要频繁地进行进程调度，大大增加系统开销。当时间片足够大时，使得每个进程只用一个时间片就可以完成任务时，该算法就退化成了 FCFS 算法，无法满足短作业和交互式用户的需求。

（六）多级队列调度算法（理解即可）
如前所述的各种调度算法，尤其在应用于进程调度时，由于系统中仅设置一个进程的就绪队列，调度算法是固定的、单一的，无法满足系统中不同用户对进程调度策略的不同要求。
[图片：单就绪队列示意图，进程依次进入CPU]
由此，多级队列调度算法能够在一定程度上弥补这一缺点。
该算法将系统中的进程就绪队列从一个拆分为若干个，将不同类型或性质的进程固定分配在不同的就绪队列，不同的就绪队列采用不同的调度算法。
[图片：多级队列示意图，包含交互进程队列（时间片轮转法）、批处理进程队列（FCFS）、短进程队列（SPF），统一指向CPU]
因此，系统针对不同用户进程的需求，很容易提供多种调度策略。
在多处理机系统中，该算法由于安排了多个就绪队列，因此，很方便为每个处理机设置一个单独的就绪队列。

(七) 多级反馈队列调度算法 (理解即可)
如果未指明进程长度，则短进程优先算法将无法使用，而多级反馈队列调度算法则不必事先知道各种进程所需的执行时间，还可以较好地满足各种类型进程的需要，它是目前公认的一种较好的进程调度算法。
多级反馈队列调度算法的调度机制可描述如下：
1.设置多个就绪队列
在系统中设置多个就绪队列，并为每个队列赋予不同的优先级。

[图片：多级反馈队列调度示意图，展示了从就绪队列1到就绪队列n的多个层级，每个队列包含若干进程，时间片分别为S1, S2, S3...Sn，箭头指示进程流向CPU及反馈回路]

(时间片：S1<S2<S3...<Sn)
第一个队列的优先级最高，第二个次之，其余队列的优先级逐个降低。
该算法为不同队列中的进程所赋予的执行时间片的大小也各不相同，在优先级越高的队列中，其时间片就越小。例如第二个队列的时间片要比第一个的时间片长一倍，……，第 i+1 个队列的时间片要比第 i 个的时间片长一倍。
2.每个队列都采用 FCFS 算法
当新进程进入内存后，首先将它放入第一队列的末尾，按 FCFS 原则等待调度。
当轮到该进程执行时，如它能在该时间片内完成，便可撤离系统，否则，调度程序将其转入第二队列的末尾等待调度；如果它在第二队列中运行一个时间片后仍未完成，再将它放入第三队列，……，依此类推。
当进程最后被降到第 n 队列后，在第 n 队列中便采取按时间片轮转调度算法 (RR) 运行。
3.按队列优先级调度
调度程序首先调度最高优先级队列中的诸进程运行，仅当第一队列空闲时才调度第二队列中的进程运行，仅当第 1~(i-1)所有队列均空时，才会调度第 i 队列中的进程运行。
如果处理机正在第 i 队列中为某进程服务时又有新进程进入任一优先级较高的队列了，此时须立即把正在运行的进程放回到第 i 队列的末尾，而把处理机分配给新到的高优先级进程。

```c
void schedule(void)//具体实现不在考纲范围，不用去看懂
{
    while(1) {//选中下一个任务
        c = -1;  next = 0;   i = NR_TASKS;
        p = &task[NR_TASKS];
        while(--i) {
            if((*p)->state == TASK_RUNNING && (*p)->counter > c)
                c = (*p)->counter;    next = i;
        }
        if(c) break;
        for(p = &LAST_TASK; p > &FIRST_TASK; --p)
            (*p)->counter = ((*p)->counter >> 1) + (*p)->priority;
    }
    switch_to(next);  // 切换到选中任务
}[2]
```

通过上述代码，我们不难发现，调度程序主要负责根据调度算法选中下一个进程，随后通过 swtich_to 将当前的运行环境切换成目标进程的运行环境。

【例】假设进程 1 在执行的过程中收到一个时钟中断，中断响应后，CPU 从用户态切换到内核态，中断处理程序接管 CPU。若进程 1 的时间片到了，需要调度，则会调用调度程序从就绪队列中选中一个进程 2，随后将当前的运行上下文保存到进程 1 的 PCB 中，根据进程 2 的 PCB 恢复其上下文。随后通过中断返回到进程 2 中。

2.在一个进程运行完毕时，需要执行调度。
3.当一个进程由于 I/O 操作、信号量或其他某个原因被阻塞时，需要执行调度。

(三) 调度的方式(理解即可)
调度的方式分为非抢占式调度和抢占式调度,主要区别在于操作系统能否“违背”一个进程的意愿,主动打断其运行并将处理器资源分配给其他进程。
1.非抢占式调度
在非抢占式调度中,操作系统选择一个进程并让其运行,直到其发生阻塞或者完成任务释放资源为止。
优点:调度算法设计更简单,减少了调度的频次,进而减少调度成本。
缺点:仅适用于早期的批处理系统,不能用于分时系统和大多数的实时系统。
注意,调度是一个“白白”消耗系统资源的行为,因为调度程序占用处理器资源这一行为对作业的完成是没有任何贡献的。

[图片：进程流转示意图，显示三个“进程”圆圈指向“CPU”方框，下方标注“就绪队列”]

2.抢占式调度
这种调度方式允许调度程序根据某种原则,去暂停某个正在执行的进程,将已分配给该进程的处理机重新分配给另一进程。在现代OS中广泛采用抢占方式,这是因为:
对于批处理机系统,可以防止一个长进程长时间地占用处理机,以确保处理机能为所有进程提供更为公平的服务。在分时系统中,只有采用抢占方式才有可能实现人机交互。在实时系统中,抢占方式能满足实时任务的需求。
但抢占方式比较复杂,所需付出的系统开销也较大。另外,“抢占”不是一种任意性行为,必须遵循一定的原则。主要原则有:
① 优先权原则
优先级高的进程可以抢占优先级低的进程。
例如,当有新进程被加入就绪队列中,如果它拥有比当前运行进程更高的优先级,则可以抢占。
② 短进程优先原则
请求处理器时间短的进程,可以抢占请求时间长的进程。
③ 时间片轮转原则
每个进程依次占用处理器资源,一个进程耗尽当前分配的时间片后,其他就绪进程可以在进程调度发生时进行抢占。
(四) 闲逛进程(了解即可)
闲逛进程是一种没有明确工作,用于解决调度中可能发生的特殊情况的进程。

【补充】基于公平原则的调度算法（理解即可）

（一）保证调度算法
保证调度算法向用户所做出的保证并不是优先运行，而是明确的性能保证，该算法可以做到调度的公平性。一种比较容易实现的性能保证是处理机分配的公平性。

[图片：进程 -> 进程 -> 进程 -> CPU，下方标注“就绪队列”]

如果在系统中有 n 个相同类型的进程同时运行，为公平起见，须保证每个进程都获得相同的处理机时间 1/n。在实施公平调度算法时系统中必须具有这样一些功能：
（1）跟踪计算每个进程自创建以来已经执行的处理时间。
（2）计算每个进程应获得的处理机时间，即自创建以来的时间除以 n。
（3）计算进程获得处理机时间的比率，即进程实际执行的处理时间和应获得的处理机时间之比。
（4）比较各进程获得处理机时间的比率。如进程 A 的比率最低，为 0.5，而进程 B 的比率为 0.8，进程 C 的比率为 1.2 等。
（5）调度程序应选择比率最小的进程将处理机分配给它，并让该进程一直运行，直到超过最接近它的进程比率为止。

（二）公平分享调度算法
分配给每个进程相同的处理机时间体现了一定程度的公平，但如果各个用户所拥有的进程数不同，就会发生对用户的不公平问题。
【例】若系统中仅有两个用户，用户 1 启动了 4 个进程，用户 2 只启动 1 个进程，采用轮转法让每个进程轮流运行一个时间片时间，对进程而言很公平，但用户 1 和用户 2 得到的处理机时间分别为 80%和 20%，显然对用户 2 而言就有失公平。
在该调度算法中，调度的公平性主要是针对用户而言，使所有用户能获得相同的处理机时间，或所要求的时间比例。为此，必须考虑到每一个用户所拥有的进程数目。
例如，系统中有两个用户，用户 1 有 4 个进程 A、B、C、D，用户 2 只有 1 个进程 E。为保证两个用户能获得相同的处理机时间，则必须执行如下所示的强制调度序列：

【补充】多处理机调度（建议二轮学习）
一、多处理机的基本概念
多处理机指的是在某个计算机系统内有多个 CPU。在多处理机调度问题中，多处理机一般指的是共享存储器多处理机，其两个或更多的 CPU 全部共享一个公用的 RAM。

[图片：四个CPU（0, 1, 2, 3）连接到一个RAM]

根据系统中所用处理器的相同与否，可将多处理器系统（MPS）分为如下两类：
① 对称多处理系统 SMPS（Symmetric Multiprocessor System）
在系统中所包含的各处理器单元，在功能和结构上都是相同的。
② 非对称多处理系统 ASMPS（Asymmetric Multiprocessor System）
在系统中有多种类型的处理单元，它们的功能和结构各不相同。系统中只有一个主处理器，有多个从处理器。

[图片：CPU 0为主，CPU 1、2、3为从，连接到RAM]

二、进程分配方式
假设有多个进程，我们需要将其分配到不同的 CPU 上，可以采取以下的方案。
1.非对称 MPS 中的进程分配方式
对于非对称 MPS，其 OS 大多采用主-从式 OS，即 OS 的核心部分驻留在一台主机上，而从机上只是用户程序，进程调度只由主机执行。

[图片：Process 1-4 分配到 CPU 0(主) 和 CPU 1,2,3(从)，连接到RAM]

调度过程如下：
① 每当从机空闲时，便向主机发送一索求进程的信号，然后，便等待主机为它分配进程。
② 在主机中保持有一个就绪队列，只要就绪队列不空，主机便从其队首摘下一进程分配给请求的从机。
③ 从机接收到分配的进程后便运行该进程，该进程结束后从机又向主机发出请求。
在非对称 MPS 中，主从式的进程分配方式的优点是：系统处理比较简单，因为所有的进程分配都由一台主机独自处理。但由一台主机控制一切，也存在问题：①主机一旦出现故障，将会导致整个系统瘫痪。②很容易因主机太忙，来不及处理而形成系统瓶颈。

--- Page 97 ---
2.对称 MPS 中的进程分配方式
在 SMP 系统中，所有的处理机都是相同的，可以将任何一个进程分配任何一个处理机。对于这种进程分配，可采用以下三种方式。
①静态分配方式
该方式下，一个进程从开始执行直至其完成，都被固定地分配到一个处理器上去执行。

[图片：静态分配示意图，显示CPU 0、CPU 1、CPU 2、CPU 3各自拥有独立的就绪队列，下方连接RAM]

此时，须为每一处理器设置一专用的就绪队列，该队列中的就绪进程先后都被分配到该处理器上执行。在进程阻塞后再次就绪时，也仍被挂在这个就绪队列中，因而下次它仍在此处理器上执行。其优点是进程调度的开销小；缺点是会使各处理器的忙闲不均。换言之，可能有些处理机的就绪队列很快就变成空队列，使处理器处于空闲状态，而另一些处理器则可能一直忙碌。
②动态分配方式

[图片：动态分配示意图，显示一个公共的就绪队列，下方连接CPU 0、CPU 1、CPU 2、CPU 3及RAM]

为了防止系统中的多个处理器忙闲不均，可以在系统中仅设置一个公共的就绪队列，系统中的所有就绪进程都被放入该队列中。分配进程时，可将进程分配到任何一个处理器上。
对一个进程的整个运行过程而言，在每次被调度执行时，都是随机地被分配到当时是空闲的某一处理器上去执行。例如，某进程一开始是被分配到处理器 0 上去执行，后来因阻塞而放弃处理器 0。当它又恢复为就绪状态后，就被挂到公共的就绪队列上，在下次被调度时，就可能被分配到处理器 1 上去执行，也有可能被分配到处理器 2 或处理器 3 上去执行。动态分配方式的主要优点是消除了各处理器忙闲不均的现象。
然而，该方案下不容易实现处理器亲和性。处理器亲和性指的是绑定某一进程（或线程）到特定的 CPU（或 CPU 集合），从而使得该进程（或线程）只能运行在绑定的 CPU（或 CPU 集合）上。
CPU 亲和性利用了这样一个事实：进程上一次运行后的残余信息会保留在 CPU 的状态中（也就是指 CPU 的缓存）。如果下一次仍然将该进程调度到同一个 CPU 上，就能避免缓存未命中等对 CPU 处理性能不利的情况，从而使得进程的运行更加高效。
CPU 亲和性分为两种：软亲和性和硬亲和性。
软亲和性主要由操作系统来实现，操作系统的调度器会倾向于保持一个进程不会频繁地在多个 CPU 之间迁移，通常情况下调度器都会根据各个 CPU 的负载情况合理地调度运行中的进程，以减轻繁忙 CPU 的压力，提高所有进程的整体性能。
硬亲和性指用户可以通过调用系统 API 实现自定义进程运行在哪个 CPU 上，从而满足特定进程的特殊性能需求。

③混合方式
[图片：混合调度方式示意图，上方为全局队列，下方为CPU 0-3各自的本地队列及RAM]

这种方式下，同时使用本地队列和全局队列。本地队列是每个 CPU 所独享的，而全局队列是所有 CPU 所共享的。全局队列用于保证负载均衡，即每个 CPU 都有公平的进程份额去执行。本地队列减少了进程调度开销的影响，保证了亲和性。
除了全局队列，还有两种实现负载均衡的技术：
① 基于推的迁移（Push Migration）
一个特定的任务周期性检查每个处理器的负载平衡。如果发现不平衡，那么通过将进程从超载处理器推到空闲或不太忙的处理器上来实现负载平衡。
② 基于拉的迁移（PULL Migration）
空闲处理器从一个忙的处理器上拉一些就绪进程到自己的就绪队列中。

三、进程（线程）调度方式
进（线）程调度方式有：自调度方式、成组调度方式等，这里仅讲解两种较为典型的调度方式。
1.自调度方式
自调度方式是最简单的一种调度方式。在系统中设置有一个公共的进程或线程就绪队列，所有的处理器在空闲时，都可自己到该队列中取得一个进程（或线程）来运行。在自调度方式中，可采用在单处理机环境下所用的调度算法，如先来先服务（FCFS）调度算法等。
[图片：自调度方式示意图，上方为公共就绪队列，下方为CPU 0-3及RAM]

自调度方式的主要优点表现为：只要公共就绪队列不空，就不会出现处理机空闲的情况，也不会发生处理机忙闲不均的现象，因而有利于提高处理机的利用率。
然而，由于在整个系统中只设置一个就绪队列，这些处理器必须互斥地访问该队列，这很容易形成系统瓶颈。另外，通常一个应用中的多个线程都属于相互合作型的，但在采用自调度方式时，这些线程很难同时获得处理器而同时运行，这将会使某些线程因其合作线程未获得处理器运行而阻塞，进而被切换下来，导致线程切换频繁。
2.成组调度方式
为了解决在自调度方式中线程被频繁切换的问题，产生了成组调度方式。该方式将一个进程中的一组线程分配到一组处理器上去执行。在成组调度时，如何为应用程序分配处理器时间，可考虑采用以下两种
