# Peterson算法为何未遵循"让权等待"准则

## 一、Peterson算法回顾

首先，我们来看Peterson算法的核心代码（以P0和P1两个进程为例）：

**进程 P0：**

```c
While(true){
    flag[0]=true;             // 进入区：表达访问意愿
    turn=1;                   // 进入区：谦让，让对方先进入
    while(flag[1]&&turn==1);  // 进入区：检查对方
    critical section;         // 临界区
    flag[0]=false;            // 退出区
    remainder section;        // 剩余区
}
```

**进程 P1：**

```c
While(true){
    flag[1]=true;             // 进入区
    turn=0;                   // 进入区：谦让
    while(flag[0]&&turn==0);  // 进入区
    critical section;         // 临界区
    flag[1]=false;            // 退出区
    remainder section;        // 剩余区
}
```

---

## 二、结合两个进程分析"让权等待"准则

### 什么是"让权等待"？

 **"让权等待"准则要求：**   当进程无法进入临界区时，应立即释放处理机（CPU），让出处理机给其他进程使用，而不是占用CPU空转。

### Peterson算法的问题所在

假设如下场景：

**场景：P0先进入临界区，P1被阻塞**

1. **P0执行：**   `flag[0]=true;`​  → `turn=1;`​  → `while(flag[1]&&turn==1);`​ → 此时flag[1]为false，条件不成立 → **P0成功进入临界区** ✅
2. **发生进程切换** → CPU切换到P1
3. **P1执行：**   `flag[1]=true;`​  → `turn=0;`​  → `while(flag[0]&&turn==0);` → 此时：

   - ​`flag[0]`​ = **true**（P0尚未退出临界区）
   - ​`turn`​ = **0**（刚被P1设为0）
   - 条件成立 → **P1陷入while循环！**   ❌

### 关键问题：P1此时在做什么？

P1陷入 `while(flag[0]&&turn==0);`​ 后，会**不断地、无休止地**进行条件测试，即：

```
while(flag[0]&&turn==0);  // CPU不断执行这条语句
```

这被称为  **"忙等"（busy waiting）**  ——进程虽然无法进入临界区，但依然**霸占着CPU**，不断循环检查条件，不释放处理机。

**这就是违背"让权等待"准则的核心原因：**

> 进程即使进入不了临界区，也会一直运行 **while 循环** 占用处理器资源，而不会主动让出CPU去睡眠/阻塞。

---

## 三、P0退出后发生了什么？

当P0完成临界区操作，执行退出区：

```c
flag[0]=false;  // P0释放资源
```

此时P1的while条件 `flag[0]&&turn==0`​ 中的 `flag[0]`​ 变为 `false`，条件不成立，P1跳出循环，成功进入临界区。但在此之前，P1一直在"空转"浪费CPU。

---

## 四、为什么说"不影响临界区问题的解决"？

从文档中可以看到一句重要的说明：

> **PS：在早期的解决方案中均未对"让权等待"做出要求，因为这样的做法虽会影响系统效率，但不影响临界区问题的解决。**

Peterson算法虽然违背了"让权等待"，但它完美满足了其他三个准则：

|准则|Peterson满足情况|说明|
| :---| :---------------| :-------------------------------------------------------------|
|✅ **空闲让进**|满足|当临界区空闲时，P0或P1可以进入|
|✅ **忙则等待**|满足|P0在临界区时，P1被while拦住|
|✅ **有限等待**|满足|P0退出后，P1一定能进入，不会无限等待（因为turn机制保证了谦让）|
|❌ **让权等待**|**不满足**|P1在while循环中忙等，占用CPU不释放|

---

## 五、对比：哪些方法满足了"让权等待"？

**信号量机制**（记录型信号量）中的 `wait(S)` 操作可以满足"让权等待"：

```c
wait(semaphore S) {
    S.s--;
    if(S.s < 0) 
        阻塞该进程并将其放入阻塞队列中;  // ✨ 主动让出CPU，进入阻塞状态
}
```

当进程无法获取资源时，会被阻塞并放入等待队列，**主动让出CPU**，而不是像Peterson算法那样在while循环中空转。

---

## 总结一句话

**Peterson算法未遵循"让权等待"是因为：**   当P1想进入临界区但P0正在使用时，P1只能在 `while(flag[0]&&turn==1)`​ 循环中**忙等**，不断占用CPU进行条件测试，而不会主动让出处理机进入阻塞状态，导致CPU资源的浪费。

## ② 并发环境中关中断会限制程序交替运行，导致 CPU 和 I/O 不能并行，影响系统执行效率

### 核心问题

关中断的本质是**禁止CPU响应外部中断**。在单CPU系统中，操作系统正是通过**时钟中断**来实现进程调度的——每次时钟中断触发，OS才有机会切换进程。关中断后，时钟中断被屏蔽，CPU就无法切换到其他进程。

### 直观例子：两个进程交替运行

假设有两个进程并发执行：

```
进程 P0:
    关中断;
    临界区操作（读写共享数据）;
    开中断;
    去访问磁盘（I/O操作，很慢）;

进程 P1:
    从磁盘读取数据（I/O操作）;
    关中断;
    临界区操作;
    开中断;
```

**正常情况下（不开中断）**  ，并发执行的理想流程应该是：

```
时间线 →
P0: [临界区] → [读磁盘...等待...] → [继续运行]
P1:                     → [从磁盘读数据] → [临界区]
```

P0 在做磁盘 I/O（慢操作、不需要CPU）时，CPU可以切换给P1使用，**让CPU和I/O并行工作**。

**如果关中断的范围过大**（比如把临界区外的操作也包进关中断中）：

```
P0: [关中断] → [临界区] → [读磁盘...CPU空转等待...] → [开中断]
P1:                                                         → [从磁盘读数据] → [临界区]
```

由于 `关中断`​ → `开中断`​ 之间时钟中断被屏蔽，即使P0已经在等待慢速的磁盘I/O（**不需要CPU**），CPU也无法切换给P1。结果就是：

- **CPU在空转等待**（本可以给P1用）
- **P1在空闲等待**（本可以运行）
- **CPU和I/O完全串行化**，浪费了系统资源

### 更具体的场景：生产者和消费者

```
生产者进程:        消费者进程:
关中断;            关中断;
while(缓冲区满);   while(缓冲区空);
放入数据;          取出数据;
开中断;            开中断;
```

如果生产者在关中断期间发现**缓冲区满了**，就会在while循环中空转等待。但消费者的临界区也需要开中断才能执行，而中断已被关闭——**生产者进不去临界区，消费者也进不去临界区，双方死等**（虽然不会死锁，但CPU被白白浪费）。

---

### 总结

|正常并发|关中断导致的问题|
| ----------------------------------| --------------------------------------|
|P0做I/O时，CPU可切换给P1|P0关中断做I/O时，CPU无法切换，空转等待|
|CPU和I/O设备可并行工作|CPU和I/O串行化，效率降低|
|时钟中断触发调度，实现进程交替运行|时钟中断被关，进程无法被调度|

简单一句话：**关中断把CPU的调度权锁死了，即使当前进程不需要CPU（如等待I/O），别的进程也用不上，导致CPU资源浪费。**

## lock 变量是什么？

​`lock`​ 是一个**全局布尔变量**，用来表示临界资源是否被占用：

|lock 的值|含义|
| ---------| ------------------------|
|**FALSE（0/false）**|**未上锁** → 临界区空闲，可以进入|
|**TRUE（1/true）**|**已上锁** → 临界区被占用，请等待|

---

## 🚪 生活类比：厕所门锁

想象一个**公共厕所**，只有一间房（临界资源），门口有一个**锁**（lock 变量）：

|场景|lock 状态|含义|
| ------------| -----------------| ------------------------|
|厕所没人|lock = FALSE（0）|门是开的，可以进|
|有人进去了|lock = TRUE（1）|门锁上了，外面的人得等着|
|里面的人出来|lock = FALSE（0）|门开了，下一个人可以进|

**关键**：TSL 指令就是"**先看看门有没有锁，同时把门锁上**"这个一气呵成的动作。

## 💡 为什么 TSL 要用硬件实现原子性？

「检查 lock → 上锁」这两个步骤如果分开，就可能出问题：

### ❌ 非原子操作（会出问题）

```
P0: 检查 lock → 发现是 FALSE（空闲）→【**被切换**】→ P1: 检查 lock → 发现是 FALSE → 上锁 → 进入临界区
                                                                              ↑
                                                                         两个都进了！
P0: 上锁 → 进入临界区
```

### ✅ TSL 原子操作（正确）

```
P0: TSL(&lock) → 检查+上锁一气呵成 → lock=TRUE → 进入临界区
P1: TSL(&lock) → 检查+上锁一气呵成 → lock=TRUE → while循环等待
```

**TSL 指令在硬件层面保证：**   读取 lock 的值和设置 lock = TRUE 之间**不可能被中断**，其他进程无法插进来。

---

## ⚠️ 硬伤：违背"让权等待"

虽然 TSL + lock 解决了互斥问题，但**不满足"让权等待"准则**：

```
P1 进不去临界区时：
  while(TSL(&lock));   ← CPU一直在空转（自旋），占用处理器资源却不做有用的事
```

P1 本可以把CPU让给其他进程运行，但它却**死循环空转**，这就是"忙等"（busy waiting），也是自旋锁（spin lock）名字的由来——进不去就"自旋"。

## 核心原因：**原子性**

TSL 必须保证   **"读取旧值 + 上锁"**   这两个操作在硬件层面**一气呵成、不可分割**。

### ❌ 如果分开写（你提的方案）：

```c
// 伪代码，不是原子操作
if (*Plock == FALSE) {   // ① 先检查
    *Plock = TRUE;        // ② 再上锁
    return FALSE;
}
```

**问题场景**：P0 和 P1 同时想进入临界区

```
P0: ① 检查 *Plock → 发现是 FALSE（空闲！）
                            ↓
                        【此时发生进程切换！】
                            ↓
P1: ① 检查 *Plock → 发现是 FALSE（空闲！）
P1: ② *Plock = TRUE → 进入临界区
                            ↓
                        【切换回 P0】
                            ↓
P0: ② *Plock = TRUE → 也进入临界区 ❌
```

**结果：两个进程都进了临界区，互斥失败！**   💥

---

### ✅ TSL 的正确做法：

```c
bool TSL(bool *Plock) {
    bool old = *Plock;   // ① 记录"锁原来的状态"
    *Plock = TRUE;       // ② 立刻上锁（管你原来是什么状态，先锁上再说！）
    return old;          // ③ 返回"原来的状态"
}
```

**关键**：①和②是**一条硬件指令**，CPU执行时**不可中断**！

```
P0: 执行 TSL → ①记录old=FALSE → ②上锁(TRUE) → ③返回FALSE → 进入临界区 ✅
                  ↓
               【不可分割！P1插不进来】
```

```
P1: 执行 TSL → ①记录old=TRUE → ②上锁(TRUE) → ③返回TRUE → while循环等待
```

---

## 💡 换个角度理解

可以把 TSL 想象成一个**拿钥匙换锁**的动作：

|步骤|说明|
| ----| ------------------------------------|
|① `old = *Plock`|看一眼门上挂的牌子是"空闲"还是"占用"|
|② `*Plock = TRUE`|不管三七二十一，**立刻把牌子翻成"占用"**|
|③ `return old`|告诉调用者：**刚才的牌子是什么状态**|

**为什么不能先翻牌子再看？**

```c
// ❌ 错误做法
*Plock = TRUE;        // 先翻成占用
return *Plock;        // 返回 TRUE → 调用者以为"原来就被占用了"
                      // 但实际上原来可能是空闲的，只是被你翻了！
```

这样的话，即使原来锁是空闲的，TSL也会返回TRUE，调用者就会错误地**继续等待**，永远进不去临界区！

---

## 一句话总结

|分开操作（❌）|TSL（✅）|
| :-------------------------| :-----------|
|检查和上锁之间可能被切换|检查和上锁是**一条硬件指令**|
|两个进程可能同时进入临界区|保证互斥|
|不安全 💀|安全 ✅|

​**​**​**​`old`​**​**​**​ **的作用就是帮你记住：在执行 TSL 之前，锁到底是什么状态。**   如果 old == FALSE，说明锁本来空闲，你成功抢到了锁；如果 old == TRUE，说明本来就被锁着，你得继续等。

## 二、读者优先算法是怎么解决的？

### 核心思想

### 解决思路分三步

#### ① 识别两类进程和资源

|进程类型|资源|
| --------| ----------|
|读者进程|共享文件 F|

#### ② 分析关系

|关系|需要的机制|
| -----------------| --------------------------------------------------|
|读者-写者互斥|需要一个互斥信号量 `wmutex`，保证读和写不能同时进行|
|写者-写者互斥|同一个 `wmutex` 也保证写者之间互斥|
|读者-读者可同时读|需要特殊处理——不能让第一个读者之后的人又被 `wmutex` 拦住|

#### ③ 关键问题：怎么让多个读者同时读？

如果所有读者和写者都用同一个 `wmutex`：

```
读者1: P(wmutex) → 读 → V(wmutex)
读者2: P(wmutex) → 被拦住，进不去 ❌
```

这就变成"读者之间也互斥"了，不满足题意。

**解决办法**：引入一个计数器 `readcount`，记录当前有几个读者在读

```
读者1 进来时：readcount从0变成1，此时才需要P(wmutex)上锁
读者2 进来时：readcount从1变成2，直接进，不再P(wmutex)
读者1 出去时：readcount从2变成1，不操作wmutex
读者2 出去时：readcount从1变成0，此时才V(wmutex)解锁，让写者进来
```

**核心规律**：`readcount`​ 的临界区保护用 `rmutex` 信号量

```c

semaphore wmutex = 1;   // 读者和写者之间的互斥信号量

    do {
        P(rmutex);                 // ① 我要修改readcount了，别抢
        if (readcount == 0)        // ② 我是第一个读者吗？
            P(wmutex);             // ③ 是→上锁，禁止写者写
        readcount++;               // ④ 读者数量+1
        V(rmutex);                 // ⑤ 修改完了，放开readcount

        读文件;                     // ⑥ 读文件（多个读者可以同时读）

        P(rmutex);                 // ⑦ 又要修改readcount了
        readcount--;               // ⑧ 读者数量-1
        if (readcount == 0)        // ⑨ 我是最后一个读者吗？
            V(wmutex);             // ⑩ 是→解锁，允许写者写了
        V(rmutex);                 // ⑪ 修改完了
    } while(TRUE);
}

void writer() {
    do {
        P(wmutex);                 // 申请写锁
        写文件;                     // 写文件（只有一个写者）
        V(wmutex);                 // 释放写锁
    } while(TRUE);
}

临界资源指一次仅允许一个进程使用的资源，既可以是硬件资源，也可以是软件资源，比如打印机、变量、队列等。在上述例子中，进程 P1 与 P2 的临界资源是变量 n。在每个程序中，对临界资源访问的那段程序称为临界区。

```c
int n=0; // n 为共享变量
P1(){
    ①while (n<5)
    {
        ②临界区
    }
    ③printf("A=%d\n",n);
}
```

```c
while (true) {
    进入区
    临界区
    退出区
    剩余区
}
```
①进入区（entry section）  
每个进程在进入临界区之前，应先对欲访问的临界资源进行检查，看它是否正在被访问。如果此刻该临界资源未被访问，进程便可进入临界区对该资源进行访问，并将访问标志设置为“正被访问”；如果此刻该临界资源正在被某进程访问，则本进程不能进入临界区。  
②临界区（critical section）

③退出区（exit section）

# 二、进程互斥、同步的实现（理解并记忆）

## （一）临界区互斥的准则

```c
while (true) {
    进入区
    临界区
    退出区
    剩余区
}
```
### 1.单标志法

```c
进程 P0:
while (turn!=0);        //进入区
critical section;       //临界区
turn=1;                 //退出区
remainder section;      //剩余区
```
```c
进程 P1:
while (turn!=1);        //进入区
critical section;       //临界区
turn=0;                 //退出区
remainder section;      //剩余区
```
[图片：P0和P1指向打印机，旁边有CPU]

该算法可以实现每次只允许一个进程进入临界区。  
但两个进程必须交替进入临界区，若某个进程不再进入临界区，则另一个进程也将无法进入临界区（违背“空闲让进”准则）。  
这样很容易造成资源利用不充分。  
例如，若 P0 顺利进入临界区并从临界区离开，则此时临界区是空闲的，但 P1 并没有进入临界区的打算，而 turn=1 一直成立，则 P0 就无法再次进入临界区。

### 2.双标志先检查法

```c
进程 P0:
while(flag[1]);      (1)      //进入区
flag[0]=true;        (3)     //进入区
critical section;            //临界区
flag[0]=false;               //退出区
remainder section;           //剩余区

进程 P1:
while(flag[0]);      (2)      //进入区
flag[1]=true;        (5)      //进入区
critical section;            //临界区
flag[1]=false;               //退出区
remainder section;           //剩余区
```
[图片：P0和P1指向打印机，旁边有CPU]

优点：不用交替进入，可连续使用。  
缺点：P0 和 P1 可能同时进入临界区。  
例如，按序列①②③④执行时，即检查对方标志后和设置自己的标志前可能发生进程切换，结果双方都检查通过，会同时进入临界区（违背“忙则等待”准则）。原因在于检查和设置操作不是一气呵成的。

### 初始条件

- ​`flag[0]`​ 和 `flag[1]`​ 初始均为 `false`，表示两个进程都不在临界区。

### 问题场景（按题目中的序列①②③④执行）

1. ​**P0 执行 (1)** ​：`while(flag[1]);`​  
   此时 `flag[1]`​ 为 `false`​，循环条件不成立，P0 通过检查，​**准备设置自己的标志**。
2. ​**P1 执行 (2)** ​：`while(flag[0]);`​  
   此时 `flag[0]`​ 仍为 `false`​（因为 P0 尚未执行 `flag[0]=true`​），P1 也通过检查，​**准备设置自己的标志**。
3. ​**P0 执行 (3)** ​：`flag[0]=true;`  
   P0 设置了自己的标志，表明它将要进入临界区。
4. ​**P1 执行 (5)** ​：`flag[1]=true;`  
   P1 设置了自己的标志，表明它将要进入临界区。

此时，两个进程都已经分别完成了“检查对方标志”和“设置自己标志”的操作，并且都认为对方不在临界区（因为检查时对方标志为 `false`​）。因此，​**两个进程会同时进入临界区**，违反了“忙则等待”准则（即最多只有一个进程处于临界区）。

### 根本原因

- ​**检查与设置不是原子操作**：每个进程的进入区实际上包含两个步骤：先检查对方标志，再设置自己的标志。这两个步骤之间可能被调度切换，导致竞争条件。
- ​**缺乏互斥机制**：该算法没有保证两个进程的“检查-设置”操作不会被交错执行，从而产生一个时间窗口，使得双方都以为对方不在临界区。

remainder section;           //剩余区
```
按序列①②③④执行时，即两个进程依次设置自己的标志，并依次检查对方的标志，发现对方也想进入临界区，双方都争着进入临界区，结果谁也进不了（违背“空闲让进”准则），于是因各个进程都长期无法访问临界区而导致“饥饿”现象（违背“有限等待”准则）。

1. **P0 执行 (1)** ：`flag[0]=true;`​  
   此时 `flag[1]`​ 为 `false`​，循环条件不成立，P0 通过检查，**准备设置自己的标志**。
2. ​**P1 执行 (2)** ​：`while(flag[0]);`​  
   此时 `flag[0]`​ 仍为 `false`​（因为 P0 尚未执行 `flag[0]=true`​），P1 也通过检查，​**准备设置自己的标志**。
3. ​**P0 执行 (3)** ​：`flag[0]=true;`  
   P0 设置了自己的标志，表明它将要进入临界区。
4. ​**P1 执行 (5)** ​：`flag[1]=true;`  
   P1 设置了自己的标志，表明它将要进入临界区。

### 4.Peterson 算法

进程 P0:

```c
flag[0]=true;             //进入区
turn=1;                   //进入区
while(flag[1]&&turn==1);  //进入区
critical section;         //临界区
flag[0]=false;            //退出区
remainder section;        //剩余区
```
进程 P1:

```c
flag[1]=true;             //进入区
turn=0;                   //进入区
while(flag[0]&&turn==0);  //进入区
critical section;         //临界区
flag[1]=false;            //退出区
remainder section;        //剩余区
```
[图片：P0和P1进程指向打印机，旁边有CPU]

若要证明 Peterson 解决方案是正确的，则须证明该方案满足解决临界区问题的 3 个准则：忙则等待、空闲让进、有限等待。  
PS：在早期的解决方案中均未对“让权等待”做出要求，因为这样的做法虽会影响系统效率，但不影响临界区问题的解决。  
证明如下：  
假设 P0 准备进入临界区，则先将 flag[0]置为 true，表达自己的访问意愿，接着将 turn 置为 1，表示优先让对方 P1 进入临界区。  
之后，再不断检测 flag[1]和 turn 的值。若 P1 的 flag 值为 flase 或 turn==0，则 P0 不会陷入 while 循环，可以进入临界区，满足“空闲让进”准则。  
若在 P0 访问临界区的过程中发生进程切换，切换到进程 P1，由于 flag[0]的值在 P0 访问临界资源结束前始终为 true，turn 的值在 P0 访问临界资源结束前始终为 0，因此 P1 始终陷入 while 循环，满足“忙则等待”准则。只有当 P0 访问完临界资源，进入退出区，将 flag[0]置为 false，P1 才可进入临界区，满足“有限等待”准则。  
由此可见，Peterson 算法很好地遵循了“空闲让进”“忙则等待”“有限等待”三个准则，但依然未遵循“让权等待”准则。因为进程即使进入不了临界区，也会一直运行 while 循环占用处理器资源。

### 什么是"让权等待"？

 **"让权等待"准则要求：**  当进程无法进入临界区时，应立即释放处理机（CPU），让出处理机给其他进程使用，而不是占用CPU空转。

### Peterson算法的问题所在

假设如下场景：

**场景：P0先进入临界区，P1被阻塞**

1. **P0执行：**  `flag[0]=true;`​  → `turn=1;`​  → `while(flag[1]&&turn==1);`​ → 此时flag[1]为false，条件不成立 → **P0成功进入临界区** ✅
2. **发生进程切换** → CPU切换到P1
3. **P1执行：**  `flag[1]=true;`​  → `turn=0;`​  → `while(flag[0]&&turn==0);` → 此时：

   - ​`flag[0]`​ = ​**true**（P0尚未退出临界区）
   - ​`turn`​ = ​**0**（刚被P1设为0）
   - 条件成立 → **P1陷入while循环！**  ❌

### 关键问题：P1此时在做什么？

P1陷入 `while(flag[0]&&turn==0);`​ 后，会**不断地、无休止地**进行条件测试，即：

```hljs
while(flag[0]&&turn==0);  // CPU不断执行这条语句
```
这被称为\*\*"忙等"（busy waiting） **——进程虽然无法进入临界区，但依然**霸占着CPU\*\*，不断循环检查条件，不释放处理机。

**这就是违背"让权等待"准则的核心原因：**

> 进程即使进入不了临界区，也会一直运行 **while 循环** 占用处理器资源，而不会主动让出CPU去睡眠/阻塞。
>

## 三、P0退出后发生了什么？

当P0完成临界区操作，执行退出区：

```c
flag[0]=false;  // P0释放资源
```
此时P1的while条件 `flag[0]&&turn==0`​ 中的 `flag[0]`​ 变为 `false`，条件不成立，P1跳出循环，成功进入临界区。但在此之前，P1一直在"空转"浪费CPU。

‍

> 【练习】若将 P0 进程和 P1 进程改造成不断循环访问临界区，则并发执行下，是否会出现饥饿现象？
>
> 答：不会，若 P0 先访问临界区，P1 卡在 while(flag[0]&&turn==0)时，进程 P1 无法再修改 flag[1]以及 turn 的值，这就意味着他在表达自身想使用临界资源的基础上，又无法做到把资源让给进程 P0，因此一定会轮到 P1 执行。
>

进程 P0:

```c
While(true){
    flag[0]=true;             //进入区
    turn=1;                   //进入区
    while(flag[1]&&turn==1);  //进入区
    critical section;         //临界区
    flag[0]=false;            //退出区
    remainder section;        //剩余区
}
```
进程 P1:

```c
While(true){
    flag[1]=true;             //进入区
    turn=0;                   //进入区
    while(flag[0]&&turn==0);  //进入区
    critical section;         //临界区
    flag[1]=false;            //退出区
    remainder section;        //剩余区
}
```
上述软件方法总结如下表：

|方法|简述|典型缺点|
| :---------------| :-------------------------------------------| :-----------------------------------------------|
|单标志法|引入 turn,循环检查直到 turn=本进程编号|只能交替进入、违背“空闲让进”|
|双标志先检查法|引入 flag[2]，先检查对方是否进入临界区|可能会同时进入，违背“忙则等待”|
|双标志后检查法|引入 flag[2]，先设置自身 flag，再检查|同时希望进入时，违背“空闲让进”、“有限等待”|
|Peterson 算法|引入 turn 和 flag[2]，进入临界区前检查两次|违背“让权等待”|

注：典型缺点仅代表某个方法最明显的缺点，可能还有其他缺点

## （三）硬件方法

由于机器指令在执行过程中不会受到其他指令的干扰，也不会被中断。因此，可以利用机器指令来实现进程互斥。能够实现这种功能的机器指令有三种：关/开中断指令、测试与设置指令和交换指令。

### 1.关中断

在单处理器环境中，并发进程之间只能交替运行，当进程进入临界区之前禁止中断后，所有外部中断都被屏蔽，这样，进程在临界区执行期间，计算机系统不响应中断，从而不会引发调度，也就不会发生进程或线程切换。关中断方法的一般流程如下所示。

```
...
关中断;
临界区;
开中断;
...
```
[图片：P0和P1进程指向打印机，旁边有一个CPU框]

然而，关中断的方法有很多缺点：  
① 滥用关中断权可能会产生严重后果，不应将这一权限交给用户进程。  
② 并发环境中关中断会限制程序交替运行，导致 CPU 和 I/O 不能并行，影响系统执行效率。  
③ 不适用于多 CPU 系统，因为关闭中断仅对执行该指令的处理器起作用，而其他处理器照常运行，也不能保证对临界资源的互斥使用。

## ② 并发环境中关中断会限制程序交替运行，导致 CPU 和 I/O 不能并行，影响系统执行效率

### 核心问题

关中断的本质是​**禁止CPU响应外部中断**​。在单CPU系统中，操作系统正是通过**时钟中断**来实现进程调度的——每次时钟中断触发，OS才有机会切换进程。关中断后，时钟中断被屏蔽，CPU就无法切换到其他进程。

### 直观例子：两个进程交替运行

假设有两个进程并发执行：

```hljs
进程 P0:
    关中断;
    临界区操作（读写共享数据）;
    开中断;
    去访问磁盘（I/O操作，很慢）;

进程 P1:
    从磁盘读取数据（I/O操作）;
    关中断;
    临界区操作;
    开中断;
```
​**正常情况下（不开中断）** ，并发执行的理想流程应该是：

```hljs
时间线 →
P0: [临界区] → [读磁盘...等待...] → [继续运行]
P1:                     → [从磁盘读数据] → [临界区]
```
P0 在做磁盘 I/O（慢操作、不需要CPU）时，CPU可以切换给P1使用，​**让CPU和I/O并行工作**。

​**如果关中断的范围过大**（比如把临界区外的操作也包进关中断中）：

```hljs
P0: [关中断] → [临界区] → [读磁盘...CPU空转等待...] → [开中断]
P1:                                                         → [从磁盘读数据] → [临界区]
```
由于 `关中断`​ → `开中断`​ 之间时钟中断被屏蔽，即使P0已经在等待慢速的磁盘I/O（​**不需要CPU**），CPU也无法切换给P1。结果就是：

- ​**CPU在空转等待**（本可以给P1用）
- ​**P1在空闲等待**（本可以运行）
- ​**CPU和I/O完全串行化**，浪费了系统资源

### 更具体的场景：生产者和消费者

```hljs
生产者进程:        消费者进程:
关中断;            关中断;
while(缓冲区满);   while(缓冲区空);
放入数据;          取出数据;
开中断;            开中断;
```
如果生产者在关中断期间发现​**缓冲区满了**​，就会在while循环中空转等待。但消费者的临界区也需要开中断才能执行，而中断已被关闭——​**生产者进不去临界区，消费者也进不去临界区，双方死等**（虽然不会死锁，但CPU被白白浪费）。

[图片：P0和P1进程指向打印机，旁边有两个CPU框]

### 2.TS (Test And Set Lock, 测试并上锁) 指令

TS/TSL 指令是一条特殊的机器指令，其特殊之处在于用一条指令连贯地执行了多个操作，即原子操作，其描述如下。

```c
bool TSL(bool *Plock) { // Plock 指向 lock 变量
    bool old = *Plock; // 记录锁的状态
    *Plock = TRUE; // 上锁
    return old;
}
```
lock 变量代表锁，其值为 FALSE 代表未上锁，其值为 TRUE 则对应上锁。若 lock 为 false，则代表未上锁，记录完锁的状态后上锁。若 lock 为 true，则执行 *Plock=true 后，保持上锁状态。

[图片：lock]

若进程希望进入临界区，应测试 lock 变量的值是否为 FALSE，若为 FALSE（检测到原先锁时开的）则立刻上锁，进程就能进入临界区。若 lock 为 TRUE（检测到锁是关的），则该进程需要等待持有锁的进程解锁。

```c
While (TRUE) {
    ...
    while (TSL(&lock));
    临界区;
    Lock = FALSE;
    剩余区;
}
```
P0

```c
While (TRUE) {
    ...
    while (TSL(&lock));
    临界区;
    Lock = FALSE;
    剩余区;
}
```
P1

[图片：P0和P1进程通过CPU访问打印机的示意图]

具体来说，当 lock 为 FALSE 时，TSL 函数才会返回 FALSE，进程可以离开 while 循环体，并进入临界区，同时将 lock 设为 TRUE。如果有其他进程申请已经上锁，TSL 函数的返回值为 TRUE，该进程需自旋等待。只有当进入临界区的进程退出时，将 lock 设为 FALSE（开锁），其他进程才能进入临界区。

当临界区空闲时，lock 为 FALSE，TSL 指令会返回 FALSE，满足“空闲让进”。由于 TSL 是硬件原子指令，其执行操作一气呵成，不会有多个进程同时访问 lock 变量，只有一个进程可以顺利离开 while 循环体，满足“忙则等待”。被拦在临界区外的进程会不断测试 while 语句，该方法违反“让权等待”。

相较于关中断方法，由于 lock 是共享的，TS 指令可以在多核处理器上使用。

​**关键**​：TSL 指令就是"​**先看看门有没有锁，同时把门锁上**"这个一气呵成的动作。

### 3.Swap 指令

该指令又称 exchange 指令，用于交换两个变量的内容，描述如下。

```c
void Swap (bool *a, bool *b) {
    bool temp = *a;
    *a = *b;
    *b = temp;
}
```
Swap 指令为每个临界资源设置一个全局布尔变量 lock，其初值为 FALSE；在每个进程中还会设置一个局部布尔变量 key，其初值为 TRUE。

[图片：lock, P0(key), P1(key) 的变量示意图]

全局变量 lock 的值被初始化为 FALSE，两个变量 key 被初始化为 TRUE。

若 P0 想要访问临界资源，则使用 swap 指令将 key 与 lock 交换，随后循环判断 key。只有当 key 为 FALSE 时，进程才可以进行临界区操作。此后，其余进程再使用 Swap 指令去交换自己的 key 与 lock 时，会得到 key 为 TRUE，因此无法访问临界资源。

[图片：P0和P1进程通过CPU访问打印机的示意图]

```c
while (TRUE) {
    key = TRUE;                        // 每次循环都把 key 重置为 TRUE
    while (key != FALSE) {             // 只要 key 不是 FALSE，就继续交换
        Swap(&lock, &key);             // 交换 lock 和 key
    }
    // key == FALSE 时跳出循环
    临界区;       // 使用打印机
    lock = FALSE; // 用完释放锁
    剩余区;
}
```
P0

```c
while (TRUE){
    key = TRUE;
    while (key != FALSE){
        Swap(&lock, &key);
    }
    临界区;
    lock = FALSE;
    剩余区;
}
```
P1

只有当 P0 退出临界区，lock 才会重置为 FALSE，允许新的进程进入临界区。

## 场景模拟：P0 和 P1 争夺打印机

​**图示**：

  → 锁变成(TRUE) → 别人进不来了
```

### Step 3：P1 也想用打印机（P0 正在用）

Swap 执行前：lock=TRUE（P0锁着呢）,  key1=TRUE
                ↓

          lock = key1 = TRUE

                ↓

```hljs

                ↓
          temp = lock = TRUE
          lock = key1 = TRUE
          key1 = temp = TRUE
                ↓
再次 Swap 执行后：lock=TRUE,   key1=TRUE  （一模一样，死循环了！）
```
**P1 陷入死循环：不断 Swap，但 key 永远是 TRUE，永远进不去！⏳**

```

P1 的下一次 `Swap(&lock, &key1)`：

                ↓
          temp = lock = FALSE
          lock = key1 = TRUE
          key1 = temp = FALSE
                ↓

```
P1 检查 `while (key1 != FALSE)`​ → `key1 = FALSE`​ → **跳出循环，进入临界区！✅**

利用上述硬件指令能有效地实现进程互斥。但当临界资源被访问时，其他访问进程必须不断地进行测试，即处于一种忙等状态，这不符合“让权等待”的原则，因而会造成处理机时间的浪费。

PS：如果系统资源分配不当，将会出现若干个进程相互等待被对方占有的资源，而无法运行下去，系统产生死锁。

## （四）信号量

前文所讲的软件硬件方法都无法满足“让权等待”准则。  
信号灯是人类社会中应用于交通管理等领域的一种设备，人们可以利用信号灯的状态（颜色）来规范相关活动，如十字路口的交通管理等。信号量机制类似于信号灯，通过在几个进程之间使用简单的变量来实现同步。

[图片：红绿灯]

在长期的应用中，信号量机制得到了很大的发展，它从整型信号量、记录型信号量、AND 型信号量，最终发展为信号量集。本小节重点学习整型信号量和记录型信号量。  
1.信号量的类型（不同教材有所出入）  
（1）整型信号量  
整型信号量代表一个用于表示资源数目的整数 S，S 指代的是操作系统中某资源的数量。  
除初始化外，仅能通过两个标准的原子操作来访问，即 wait(S)和 signal(S)操作。这两个操作又被分别称为 P 操作和 V 操作，说明如下。  
① wait(S)原语实现 S 值的减 1，减 1 代表申请并获取一个资源。如果 S≤0，操作系统无法为其分配资源，该进程就会陷入 while 循环，直到 S>0，方可结束等待，描述如下。

```c
wait ( S ) {
    while (S <= 0); // 不符合条件则陷入“忙等”
    S--;
}
```
② signal(S)原语实现 S 数值的加 1，加 1 相当于使用结束并释放一个资源，描述如下。

```c
signal (S) {
    S++;
}
```
wait(S)和 signal(S)是两个原子操作，因此，它们在执行时是不可中断的。亦即，当一个进程在修改某信号量时，没有其他进程可同时对该信号量进行修改。  
注意，这里只是用伪代码描述指令，不代表指令等于函数。

  
（2）记录型信号量  
整型信号量中，如果 S≤0 时，wait(S)操作会陷入“忙等”状态，违反了“让权等待”准则。为了实现“让权等待”准则，记录型信号量在整型信号量的基础上，增加了一个被阻塞进程的队列，用于链接所有等待相应资源的进程，定义如下。

```c
typedef struct semaphore {
    int s; // 整型信号量.
    struct process *queue; // 用于记录被阻塞进程的队列
} semaphore ;
```
可以把记录型信号量机制看作是对上述 semaphore 结构体定义了三种操作，能且仅能通过以下操作对信号量进行访问：  
① 初始化 semaphore 结构体。  
② 通过 wait(S)原语实现对 S.s 的值减 1，如果 S.s<0，则阻塞该进程，并将其放入 S 的阻塞队列。

```c
wait(semaphore S) {
	S.s--;
	if(S.s<0) 阻塞该进程并将其放入阻塞队列中;
}
```
[图片：流程图显示进程P1请求打印机资源，CPU。下方显示S.s=-3，进程进入阻塞队列，队列中有P2、P3、P4]

若信号量的值小于0，说明有进程在等待这种资源，信号量的值的绝对值就是等待该资源的进程数；若信号量的值等于0，说明该资源恰好被分配完。  
③ 可以通过 signal(S)原语实现 S.s 的加 1，如果 S.s≤0，则从阻塞队列中唤醒一个进程。

```c
signal (semaphore S) {
    S.s++;
    if(S.s <= 0) 从阻塞队列中选择并唤醒一个进程;
}
```
[图片：流程图显示进程P2请求打印机资源，CPU。下方显示S.s=-2，进程进入阻塞队列，队列中有P3、P4]

注意，一般来说，若要创建一个记录型信号量，按照整型信号量进行声明和定义即可。  
【例】semaphore printer = 8; // 声明一个记录系统中打印机资源的信号量  
      semaphore mutex = 1; // 声明一个互斥信号量  
其中，一个信号量对应一种资源，信号量的值等于该资源的剩余数量。互斥信号量的取值一般为 1。  
2.信号量的应用  
(1) 利用信号量实现进程互斥  
一般而言，用于实现进程互斥的信号量命名为 mutex。当多个进程需要互斥访问某一临界资源时，每个进程在访问临界区前需对互斥量 mutex 执行 P 操作，在退出临界区后需对互斥量 mutex 执行 V 操作。  
利用信号量实现进程互斥的伪代码实现如下(以两个进程为例)：

```c
semaphore mute x = 1; //互斥量的初始值为1
Process_A {
    while(ture) {
        P(mutex);    //加锁
        访问临界区;
        V(mutex);    //解锁
        剩余区;
    }
}
Process_B {
    while(ture) {
        P(mutex); //加锁
        访问临界区;
        V(mutex); //解锁
        剩余区;
    }
}
```
[图片：流程图显示进程PA和PB都指向打印机，打印机指向CPU]

(2) 利用信号量实现进程同步  
信号量可以实现两个进程 PA 和 PB 的同步关系，即 PA 和 PB 的前后关系。例如，PA 执行结束后才能执行 PB。利用信号量实现进程同步的伪代码如下。

```c
Semaphore S = 0; // 同步信号量的初始值为0
PA {
    修改文件;
    V(S);//表示修改完毕
}
PB {
    P(S); //判断是否修改完成
    修改文件;
}
```
[图片：流程图显示PA通过V(S)操作指向PB]

同步是协作关系：多个协作进程在工作中相互等待或通信的制约关系。

互斥是竞争关系：当一个进程进入临界区访问临界资源时，另一个进程必须等待，当访问临界资源的进程退出临界区后，另一个进程才被允许访问该临界资源。进程互斥是进程同步的一种特殊情况。  
(3) 利用信号量实现多进程的前驱后继关系（同步关系的一种）  
【例】假设有6个进程，进程与进程之间存在着如图所示的前驱后继关系。图中两个节点间的箭头表示两个节点的前驱后继关系，例如 S1→S2 表示 S2 需要在 S1 进程之后运行。

[图片：进程前驱图，S1指向S2和S3，S2指向S4和S5，S3指向S6，S4和S5指向S6]

② 为每个关系设置一个信号量，并初始化为0。  
③ 编写伪代码，每个进程都需要检查前驱进程是否运行完毕（使用P操作），然后执行本进程的任务，最后告知后继进程，本进程已运行完毕（使用V操作）。

}
```

## 分析思路

**关键点：信号量 S 的值 = 当前剩余的资源数**

### Step 1：确定初始值

> 每次只允许 **2个** 进程进入
>

所以初始时，资源数是 ​**2**：

```c
semaphore S = 2;   // 一开始有2个"入场名额"
```
### Step 2：想象3个进程依次请求

假设三个进程 P1、P2、P3 都想进入：

|操作|S 变化|说明|
| -----------| --------| -------------------------------------|
|初始状态|**S = 2**|两个名额空闲|
|P1 执行`P(S)`|**S = 1**|P1进去了，还剩1个名额|
|P2 执行`P(S)`|**S = 0**|P2进去了，名额刚好用完|
|P3 执行`P(S)`|**S = -1**|⛔ 没名额了，P3被阻塞，进入等待队列|

**S 的取值范围：2, 1, 0, -1**

### Step 3：检查极限情况

|场景|S 的值|
| ------------------------------------| ----------|
|最多空闲时（没人进）|S = 2|
|有人进有人出，动态变化|S = 1, 0|
|最挤时（2个名额已满，还有1个在等）|S = -1|

**所以答案是 A：2, 1, 0, -1** ✅

3.PV 大题  
408 真题往往以**大题形式**考察信号量相关的经典问题，大题并非一轮重点，因此仅以“生产者-消费者问题”讲解 PV 大题的基本思路，“读者写者”问题见补充部分，“哲学家进餐”问题见下一小节，其他大题放二轮学习。  
生产者-消费者问题可以描述为一组生产者和一群消费者一起工作，通过一个大小为n的有限缓冲区进行生产和消费。一个缓冲区可以容纳n个产品，其中生产者负责投放产品，消费者负责消费产品。

[图片：生产者消费者缓冲区示意图，环形缓冲区，in指针指向放入，out指针指向取出]

由于生产者和消费者共享的缓冲区的大小为n，缓冲区最多可以容纳n个产品。如果缓冲区已满，生产者不能继续生产，否则会使产品溢出。如果缓冲区为空，消费者不能继续消费，否则会导致从空缓冲区拿取产品。  
PV 大题的解题思路如下。  
① 找出各类进程和各类资源。  
② 分析各类进程之间的同步、互斥关系，分析产生死锁的情况。  
③ 定义信号量，确定信号量初值。  
④ 写出伪代码。  
⑤ 检查是否 PV 成对，检查是否有死锁，是否符合题意。若有问题，则继续优化。  
注意，“找同步关系”的核心是找事件之间的关系，某个事件导致某个事件，而不是找“某个进程导致某个进程”，直接找进程之间的关系往往无从下手。  
解：  
① 两类进程：生成者进程和消费者进程。一类资源：大小为 n 的缓冲区。  
② 互斥关系：生产者和消费者均需要互斥访问缓冲区。同步关系：缓冲区有产品，消费者才能取，缓冲区有空位，生产者才能生产。  
③ semaphore mutex=1; //互斥使用缓冲区  
semaphore full=0; //表示缓冲区的产品数量（也可以看作非空闲区数量）  
semaphore empty=n; //表示缓冲区的空闲区数量  
④ 生产者-消费者问题的程序实现如下。

```c
semaphore mutex = 1, empty = n, full = 0;
void producer () {
    while (TRUE) {
        生产一个新的产品;
        P(empty);         //判断是否有空闲的缓冲区，若无则加入等待队列
        P(mutex);         //对缓冲区上锁
        将产品放入空闲缓冲区;
        V(mutex);         //对缓冲区解锁
        V(full);          //产品数量加 1，并唤醒等待队列中的消费者
    }
}//生产、消费都是种不断重复的动作，所以需要 while 循环
void consumer () {
    while (TRUE) {
        P(full);          //判断是否有产品，若无则加入等待队列
        P(mutex);         //对缓冲区上锁
        从缓冲区取出一个产品;
        V(mutex);         //对缓冲区解锁
        V(empty);         //空闲区数量加 1，并唤醒等待队列中的生产者
        生产该产品;
    }
}
```
⑤ PV 成对，无死锁，符合题意。  
（五）管程（理解并记忆部分重点）  
1.基本定义  
虽然信号量机制可以解决进程的同步与互斥问题，但每个希望访问临界资源的进程都必须分别定义自己的 P/V 操作，使得众多同步操作分散在各个进程中。这不仅使系统难以管理，还可能会因为误用同步操作，导致代码中存在死锁的概率增加。因此，提出了管程这种同步机制，下面举例说明。  
【例】自助餐厅有一些所有人都可以用的资源，比如调料瓶。  
[图片：自助餐厅的调料台] [图片：一组调料瓶]

在信号量机制下，餐厅里的顾客需要自己协调，以确保不会同时争抢餐具或调料瓶，确保一次只有一个人使用共享资源。  
而在管程机制下，餐厅里有一个服务员来分配和管理所有资源。顾客不需要自己协调，他们只需要向服务员请求资源。服务员会确保资源被正确地分配和回收。  
管程是一种同步机制，它封装了共享资源和对资源的操作，简化了并发编程，集中管理了对共享资源的所有访问。（管程就是个服务员，共享资源登记在他这，任何进程（客人）想要访问共享资源都得找他）  
管程由以下部分组成：  
① 管程的名称。  
② 局部于管程内部的共享数据结构说明。  
③ 对局部于管程内部的共享数据设置初始值的语句。  
④ 对该数据结构进行操作的一组过程（或函数）。  
一个典型的管程定义如下所示。

```c
monitor Demo{          // 定义一个名称为 Demo 的管程
    共享数据结构 S;    // 定义共享数据结构，对应系统中的某种共享资源
    //对共享数据结构初始化的语句;
    init(){
        S=5;           //初始资源数量为 5
    }
    //过程 1：申请一个资源
    take_away( ){
        ...
        S--;           //可用资源数-1
        ...
    }
    //过程 2：归还一个资源
    give_back( ){
        ...
        S++;           //可用资源数+1
        ...
    }
}
```
管程可以看作一个类（不理解“类”的宝子也没事儿，就当作这是个可以定义函数的结构体变量）。  
一方面，管程将对共享资源的操作封装起来，管程内的共享数据结构只能被管程内的过程所访问。一个进程只有通过调用管程内的过程才能进入管程访问共享资源。例如，外部进程只能通过调用 take_away() 过程来申请一个资源；归还资源也类似。另一方面，每次仅允许一个进程进入管程，从而实现进程互斥。  
2.条件变量  
当一个进程进入管程后，若未能得到自己想要的资源，则会被阻塞，插入阻塞队列中。由于阻塞的原因不同，需要设置不同的条件变量，每个条件变量保存了一个等待队列，用于记录被阻塞的进程，对条件变量只能进行两种操作，即 wait 和 signal。  
x.wait：当 x 对应的条件不满足时，正在调用管程的进程调用 x.wait 将自己插入 x 条件的等待队列，并释放管程。此时其他进程可以使用该管程。  
x.signal：x 对应的条件发生了变化，则调用 x.signal，唤醒一个因 x 条件而阻塞的进程。  
下面给出条件变量的定义和使用：

```c
monitor Demo{          // 定义一个名称为 Demo 的管程
    共享数据结构 S;    // 定义共享数据结构，对应系统中的某种共享资源
    condition x;       // 定义一个条件变量 x
    init(){
        ......
    }
    take_away( ){
        if(S<=0) x.wait();    //资源不够，在条件变量上阻塞等待
        资源足够，分配资源，做一系列响应处理;
	...
	}
	give_back( ){ //
	    归还资源，做一系列处理;
	    if(有进程在等待) x.signal(); //唤醒一个阻塞进程
	}
}
```
条件变量和信号量的相似点为，条件变量的 wait/signal 操作类似于信号量的 P/V 操作，可以实现进程的阻塞/唤醒。两者的区别在于，条件变量是“没有值”的，仅实现了“排队等待”功能，而信号量是“有值”的。信号量的值反映了剩余资源数，而在管程中，剩余资源数用共享数据结构记录。

【补充】读者一写者问题（理解并记忆，时间紧张、目标分数不高的宝子可跳过）  
在读者一写者问题中，有多个进程访问一个共享文件。允许多个进程（读者）同时读取 F 的信息，但只允许一个进程（写者）修改 F 中的信息。当一个进程正在读取时，不允许其他进程写入（但可以读取）；当一个进程正在写入时，其他进程的任何读写操作均不被允许。

[图片：包含“写”、“读”、“读”、“读”的示意图]

读者一写者问题有三种典型的算法，即读者优先算法、写者优先算法和读写公平算法，这里仅讨论读者优先问题。在该算法中，只要有一个读进程在读，读进程就能继续获得对临界区的控制权，但可能会导致写进程饥饿。算法描述如下：  
① 当一个读进程正在读取时，写进程必须等待。  
② 在写进程等待时，随后的读进程仍可以进入临界区读取。  
③ 只有在没有进程访问文件时，写进程才可以写入。  
解：  
1.两类进程，分为读者进程，写者进程。一类资源，文件。  
2.同步互斥关系，一个写进程与其他进程。（复杂问题中，同步和互斥往往有交叉，就不去细分了）  
3.定义信号量

```c
semaphore rmutex = 1;       // 用于实现 readcount 的互斥访问
semaphore wmutex = 1;       // 用于实现进程间的互斥
int readcount = 0;          // 用于记录正在访问临界区的读者数量，初始值为 0
```
4.伪代码如下。控制多个读进程与写进程互斥的核心在于第一个读进程上锁，最后一个解锁。

```c
void reader() {             //读进程
    do {
        P(rmutex);          // 对 rmutex 执行 P 操作，防止他人访问
        if (readcount == 0)
            P(wmutex);      // 如果自己是第一个读者，禁止写者写
        readcount++;        // 读者数量加 1
        V(rmutex);          // 对 rmutex 执行 V 操作，允许他人访问进行读操作
        读文件;
        P(rmutex);          // 对 rmutex 执行 P 操作，防止他人访问
        readcount--;        // 读者数量减 1
        if (readcount == 0)
            V(wmutex);      // 如果自己是最后一个读者，允许写者写
        V(rmutex);          // 对 rmutex 执行 v 操作，允许他人访问
    } while(TRUE);
}
void writer() {             // 写进程
    do {
        P(wmutex);          // 对 wmutex 执行 P 操作，防止他人访问文件进行写操作
        写文件;
        V(wmutex);          // 对 wmutex 执行 V 操作，允许他人访问文件
    } while(TRUE);
}
```
定义一个初始值为 1 的互斥信号量 wmutex，第一个进入的读者和所有进入的写者在对共享文件 F 进行操作之前，必须先执行 P(wmutex)操作，在结束对 F 的操作后，必须执行 V(wmutex)操作。  
定义一个整型变量 readcount，用于表示当前正在读取 F 的进程数量。读进程在读取 F 之前，必须执行 readcount 加 1 的操作；而在读取 F 之后，必须执行 readcount 减 1 的操作。  
readcount 变量是一种临界资源，因此定义一个 rmutex 信号量，实现所有的读者进程对 readcount 变量的互斥访问。  
5. PV 成对，无死锁，符合题意。

【补充】自旋锁与互斥锁（理解即可）  
Linux 有多种同步机制，这里仅补充自旋锁和互斥锁两种机制。  
1.自旋锁  
Linux 内核中最常用的锁是自旋锁（spin lock），其最初的设计目的是在多处理机系统中提供对共享数据的保护。

[图片：CPU A、V、CPU、CPU 的示意图]

自旋锁的设计思想为，在多处理机之间设置一个全局变量 V，表示锁。当 V=1 时为锁定状态，V=0 时为解锁状态。自旋锁属于“忙等”机制。  
如果处理机 A 上的代码要进入临界区，就要先读取 V 的值。如果 V!=0（即锁定状态），则表明有其他处理机上的代码正在对共享数据进行访问，那么此时处理机 A 就会进入忙等状态（即自旋状态）；如果 V=0，则表明当前没有其他处理机上的代码进入临界区，此时处理机 A 可以访问该临界资源。然后把 V 设置为 1，再进入临界区，访问完毕后离开临界区时将 V 设置为 0。注意，必须确保处理机 A “读取 V、判断 V 的值、更新 V”这一操作是一个原子操作。  
自旋锁的实现与体系结构相关，其基本使用形式如下：

```c
1 DEFINE_SPINLOCK(mr_lock);
2 spin_lock(&mr_lock);
3 /* 临界区 */
4 spin_unlock(&mr_lock);
```
2.互斥锁  
新版 Linux 内核中，引入了互斥锁（mutex）这个数据类型。互斥锁与使用计数为 1 的信号量类似，但操作接口更简单，实现更高效，而且使用限制更强。  
对互斥锁进行锁定和解锁的操作如下：

```c
1 mutex_lock(&name);
2 /* 临界区 */
3 mutex_unlock(&name);
```
基于上述介绍可知，互斥锁就是一个简化版的信号量，因为它不再需要使用任何计数。

【补充】利用管程解决生产者消费者问题（时间紧张可跳过）  
在利用管程来解决生产者-消费者问题时，首先须建立一个管程，并将其命名为 producerconsumer（简称 PC），其中包括以下两个过程。  
（1）put(x)过程：生产者利用该过程将自己生产的产品投放到缓冲池中，并用整型变量 count 来表示在缓冲池中已有的产品数目，当 count>=N 时，表示缓冲池已满，生产者应等待。  
（2）get(x)过程：消费者利用该过程从缓冲池中取出一个产品，当 count<=0 时，表示缓冲池中已无可取用的产品，消费者应等待。  
对于条件变量 full 和 empty，有以下两个过程可对它们进行操作。  
(1) cwait (condition) 过程：如果正在调用管程的进程因 x 条件而被阻塞，则调用 cwait 将自己插入到 x 条件的等待队列上，并释放管程。  
(2) csignal (condition) 过程：唤醒阻塞在条件 condition 队列上的某个进程。  
程序描述如下：

```c
monitor producerconsumer {          // 定义生产者-消费者管程
    item buffer[N];                 // 缓冲区（固定大小 N）
    int in, out;                    // 缓冲区写入指针(in)和读取指针(out)
    condition full, empty;          // 条件变量，用于阻塞时排队
    int count;                      // 当前缓冲区中的产品数目
    public:
        void put(item x) {          // 生产者放入缓冲区
            if (count >= N) cwait(full); // 缓冲区已满时等待
            buffer[in] = x;         // 写入缓冲区
            in = (in + 1) % N;      // 移动写入指针（循环缓冲区）
            count++;                // 更新产品数目
            csignal(empty);         // 唤醒等待的消费者
        }
        void get(item x) {          // 消费者取出产品
            if (count <= 0) cwait(empty);   // 缓冲区为空时等待
            x = buffer[out];        // 读取缓冲区
            out = (out + 1) % N;    // 移动读取指针（循环缓冲区）
            count--;                // 更新产品数目
            csignal(full);          // 唤醒等待的生产者
        }
        {
            in = 0; out = 0; count = 0;     // 初始化
        }
} PC;                               // 实例化管程对象 PC
void producer() {                   // 生产者
    item x;
    while (TRUE) {
        生产一个产品;
        PC.put(x);                  // 调用管程的 put 方法存入缓冲区
    }
}

void consumer() {                   // 消费者
    item x;
    while (TRUE) {
        PC.get(x);                  // 调用管程的 get 方法取走一个产品
        消费产品;
    }
}
```
