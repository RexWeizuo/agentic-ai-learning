# 七、管程

## （一）为什么需要管程？

虽然信号量机制可以解决进程的同步与互斥问题，但每个希望访问临界资源的进程都必须分别定义自己的 P/V 操作，使得众多同步操作分散在各个进程中。这不仅使系统难以管理，还可能会因为误用同步操作，导致代码中存在死锁的概率增加。因此，提出了**管程**这种同步机制。

**类比：信号量 vs 管程**

|场景|信号量机制|管程机制|
| :-------| :-----------------------------------------| :-----------------------------------------------------------|
|自助餐厅|顾客自己协调，确保不会同时争抢餐具或调料瓶|有一个服务员来分配和管理所有资源，顾客只需要向服务员请求资源|

**管程**就是一种同步机制，它封装了共享资源和对资源的操作，简化了并发编程，集中管理了对共享资源的所有访问。
（管程就是个服务员，共享资源登记在他这，任何进程想要访问共享资源都得找他）

## （二）管程的定义与组成部分

### 管程的四个组成部分

1. 管程的名称
2. 局部于管程内部的共享数据结构说明
3. 对局部于管程内部的共享数据设置初始值的语句
4. 对该数据结构进行操作的一组过程（或函数）

### 典型定义

```c
monitor Demo{          // 定义一个名称为 Demo 的管程
    共享数据结构 S;    // 定义共享数据结构，对应系统中的某种共享资源
    //对共享数据结构初始化的语句;
    init(){
        S=5;           //初始资源数量为 5
    }
    //过程 1：申请一个资源
    take_away( ){
        ...
        S--;           //可用资源数-1
        ...
    }
    //过程 2：归还一个资源
    give_back( ){
        ...
        S++;           //可用资源数+1
        ...
    }
}
```

### 核心特性

- 管程可以看作一个**类**（或者可以定义函数的结构体变量）
- 管程将对共享资源的操作**封装**起来，管程内的共享数据结构只能被管程内的过程所访问
- 一个进程只有通过调用管程内的过程才能进入管程访问共享资源
- **每次仅允许一个进程进入管程**，从而实现进程互斥

## （三）条件变量

### 为什么需要条件变量

当一个进程进入管程后，若未能得到自己想要的资源，则会被阻塞，插入阻塞队列中。由于阻塞的原因不同，需要设置不同的**条件变量**，每个条件变量保存了一个等待队列，用于记录被阻塞的进程。

### 对条件变量的操作

对条件变量只能进行两种操作：

- **x.wait**：当 x 对应的条件不满足时，正在调用管程的进程调用 x.wait 将自己插入 x 条件的等待队列，并释放管程。此时其他进程可以使用该管程。
- **x.signal**：x 对应的条件发生了变化，则调用 x.signal，唤醒一个因 x 条件而阻塞的进程。

### 条件变量的使用示例

```c
monitor Demo{          // 定义一个名称为 Demo 的管程
    共享数据结构 S;    // 定义共享数据结构，对应系统中的某种共享资源
    condition x;       // 定义一个条件变量 x
    init(){
        ......
    }
    take_away( ){
        if(S<=0) x.wait();    //资源不够，在条件变量上阻塞等待
        资源足够，分配资源，做一系列响应处理;
        ...
    }
    give_back( ){
        归还资源，做一系列处理;
        if(有进程在等待) x.signal(); //唤醒一个阻塞进程
    }
}
```

### 条件变量 vs 信号量的区别

条件变量和信号量的相似点为，条件变量的 wait/signal 操作类似于信号量的 P/V 操作，可以实现进程的阻塞/唤醒。

两者的区别在于：

- **条件变量是"没有值"的**，仅实现了"排队等待"功能
- **信号量是"有值"的**，信号量的值反映了剩余资源数
- 在管程中，剩余资源数用**共享数据结构**记录，而非条件变量

## （四）补充：读者-写者问题（理解并记忆，时间紧张可跳过）

### 问题描述

有多个进程访问一个共享文件。**允许多个进程（读者）同时读取** F 的信息，但**只允许一个进程（写者）修改** F 中的信息。

- 当一个进程正在读取时，不允许其他进程写入（但可以读取）
- 当一个进程正在写入时，其他进程的任何读写操作均不被允许

### 读者优先算法

这里仅讨论**读者优先**算法。在该算法中，只要有一个读进程在读，读进程就能继续获得对临界区的控制权，但可能会导致**写进程饥饿**。

#### 算法规则

1. 当一个读进程正在读取时，写进程必须等待
2. 在写进程等待时，随后的读进程仍可以进入临界区读取
3. 只有在没有进程访问文件时，写进程才可以写入

#### 解题步骤

**1. 两类进程**：读者进程，写者进程。一类资源：文件。

**2. 同步互斥关系**：一个写进程与其他进程。（复杂问题中，同步和互斥往往有交叉，就不去细分了）

**3. 定义信号量**

```c
semaphore rmutex = 1;       // 用于实现 readcount 的互斥访问
semaphore wmutex = 1;       // 用于实现进程间的互斥
int readcount = 0;          // 用于记录正在访问临界区的读者数量，初始值为 0
```

**4. 伪代码**

控制多个读进程与写进程互斥的核心在于**第一个读进程上锁，最后一个解锁**。

```c
void reader() {             // 读进程
    do {
        P(rmutex);          // 对 rmutex 执行 P 操作，防止他人访问
        if (readcount == 0)
            P(wmutex);      // 如果自己是第一个读者，禁止写者写
        readcount++;        // 读者数量加 1
        V(rmutex);          // 对 rmutex 执行 V 操作，允许他人访问进行读操作
        读文件;
        P(rmutex);          // 对 rmutex 执行 P 操作，防止他人访问
        readcount--;        // 读者数量减 1
        if (readcount == 0)
            V(wmutex);      // 如果自己是最后一个读者，允许写者写
        V(rmutex);          // 对 rmutex 执行 v 操作，允许他人访问
    } while(TRUE);
}

void writer() {             // 写进程
    do {
        P(wmutex);          // 对 wmutex 执行 P 操作，防止他人访问文件进行写操作
        写文件;
        V(wmutex);          // 对 wmutex 执行 V 操作，允许他人访问文件
    } while(TRUE);
}
```

**5. 检查**：PV 成对，无死锁，符合题意。

#### 读者优先的调度效果

```
原始排队顺序（按到达时间）：
  读者A → 读者B → 写者C → 读者D → 写者E

实际执行顺序（读者优先）：
  读者A ✅ → 读者B ✅ → 读者D ✅（插队！）→ ...直到最后一个读者读完
                                                          ↓
                                                    写者C ✅ → 写者E ✅
```

## （五）补充：自旋锁与互斥锁（理解即可）

Linux 有多种同步机制，这里仅补充**自旋锁**和**互斥锁**两种机制。

### 自旋锁（spin lock）

Linux 内核中最常用的锁是自旋锁，其最初的设计目的是在多处理机系统中提供对共享数据的保护。

**设计思想**：在多处理机之间设置一个全局变量 V，表示锁。当 V=1 时为锁定状态，V=0 时为解锁状态。自旋锁属于"忙等"机制。

基本使用形式：

```c
DEFINE_SPINLOCK(mr_lock);
spin_lock(&mr_lock);
/* 临界区 */
spin_unlock(&mr_lock);
```

### 互斥锁（mutex）

新版 Linux 内核中，引入了互斥锁这个数据类型。互斥锁与使用计数为 1 的信号量类似，但操作接口更简单，实现更高效，而且使用限制更强。

基本使用形式：

```c
mutex_lock(&name);
/* 临界区 */
mutex_unlock(&name);
```

基于上述介绍可知，互斥锁就是一个**简化版的信号量**，因为它不再需要使用任何计数。

## （六）补充：利用管程解决生产者-消费者问题（时间紧张可跳过）

### 管程定义

建立一个管程，命名为 producerconsumer（简称 PC），包括两个过程：

- **put(x) 过程**：生产者利用该过程将自己生产的产品投放到缓冲池中，用整型变量 count 表示缓冲池中已有的产品数目。当 count >= N 时，表示缓冲池已满，生产者应等待
- **get(x) 过程**：消费者利用该过程从缓冲池中取出一个产品。当 count <= 0 时，表示缓冲池中已无可取用的产品，消费者应等待

对于条件变量 full 和 empty，有以下两个过程可对它们进行操作：

- **cwait(condition) 过程**：如果正在调用管程的进程因 x 条件而被阻塞，则调用 cwait 将自己插入到 x 条件的等待队列上，并释放管程
- **csignal(condition) 过程**：唤醒阻塞在条件 condition 队列上的某个进程

### 完整代码

```c
monitor producerconsumer {          // 定义生产者-消费者管程
    item buffer[N];                 // 缓冲区（固定大小 N）
    int in, out;                    // 缓冲区写入指针(in)和读取指针(out)
    condition full, empty;          // 条件变量，用于阻塞时排队
    int count;                      // 当前缓冲区中的产品数目
    public:
        void put(item x) {          // 生产者放入缓冲区
            if (count >= N) cwait(full); // 缓冲区已满时等待
            buffer[in] = x;         // 写入缓冲区
            in = (in + 1) % N;      // 移动写入指针（循环缓冲区）
            count++;                // 更新产品数目
            csignal(empty);         // 唤醒等待的消费者
        }
        void get(item x) {          // 消费者取出产品
            if (count <= 0) cwait(empty);   // 缓冲区为空时等待
            x = buffer[out];        // 读取缓冲区
            out = (out + 1) % N;    // 移动读取指针（循环缓冲区）
            count--;                // 更新产品数目
            csignal(full);          // 唤醒等待的生产者
        }
        {
            in = 0; out = 0; count = 0;     // 初始化
        }
} PC;                               // 实例化管程对象 PC

void producer() {                   // 生产者
    item x;
    while (TRUE) {
        生产一个产品;
        PC.put(x);                  // 调用管程的 put 方法存入缓冲区
    }
}

void consumer() {                   // 消费者
    item x;
    while (TRUE) {
        PC.get(x);                  // 调用管程的 get 方法取走一个产品
        消费产品;
    }
}
```

### 对比：管程 vs 信号量解决生产者-消费者

|对比维度|信号量方案|管程方案|
| :--------| :--------------------| :---------------------------|
|互斥实现|P(mutex) / V(mutex)|管程本身保证一次只进一个进程|
|等待/唤醒|信号量的 P/V 自动处理|条件变量的 cwait / csignal|
|同步关系|empty / full 信号量|empty / full 条件变量|
|代码结构|同步分散在进程中|同步封装在管程内|

## 🧠 深度理解：读者-写者问题的关系分析（来自 DS 问答）

### 关系分析

|关系对|互斥关系|同步关系|
| :-----| :----------------| :------------------------------------------|
|**读者 ↔ 读者**|❌ **不需要互斥**，可以同时读|❌ 不需要同步|
|**读者 ↔ 写者**|✅ **必须互斥**，不能同时访问|✅ 有同步关系（写完才能读，读完才能写）|
|**写者 ↔ 写者**|✅ **必须互斥**，不能同时访问|❌ 严格来说是互斥（竞争同一把锁），不是同步|

### 读者优先算法的核心思路

 **"第一个读进程负责上锁，最后一个读进程负责解锁。"**

如果所有读者和写者都用同一个 `wmutex`：

```
读者1: P(wmutex) → 读 → V(wmutex)
读者2: P(wmutex) → 被拦住，进不去 ❌  ← 这就变成读者也互斥了
```

**解决办法**：引入一个计数器 `readcount`，记录当前有几个读者在读

```
读者1 进来时：readcount从0变成1，此时才需要P(wmutex)上锁
读者2 进来时：readcount从1变成2，直接进，不再P(wmutex)
读者1 出去时：readcount从2变成1，不操作wmutex
读者2 出去时：readcount从1变成0，此时才V(wmutex)解锁，让写者进来
```
