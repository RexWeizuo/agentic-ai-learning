2. 在操作系统中，P、V操作是一种（ ）。IV  
   A. 机器指令  
   B. 系统调用命令  
   C. 作业控制命令  
   D. 低级进程通信原语

   **正确答案：D. 低级进程通信原语**

   **解释：**   
   P、V操作是荷兰计算机科学家 **Dijkstra** 提出的用于解决进程同步与互斥问题的机制。它们是对**信号量**进行操作的两个**原语**（Primitive，即不可分割的原子操作），属于**低级进程通信**方式（因为传递的信息量少，仅传递控制信息，而不传递大量数据）。

   ---

   ### 各选项举例说明

   #### ❌ A. 机器指令

   **定义**：CPU能直接识别和执行的指令，由操作码和操作数组成，如 `MOV`​、`ADD`​、`JMP`。

   **举例**：

   ```assembly
   MOV AX, 5      ; 将5送入寄存器AX
   ADD AX, BX     ; AX = AX + BX
   JMP 0x1234     ; 跳转到地址0x1234
   ```

   > P、V操作不是硬件直接执行的机器指令，而是**由操作系统实现的软件机制**。
   >

   #### ❌ B. 系统调用命令

   **定义**：应用程序请求操作系统内核服务的接口，如文件操作、进程管理。

   **举例**：

   ```c
   #include <unistd.h>
   #include <sys/wait.h>

   pid_t pid = fork();       // 创建进程（系统调用）
   int fd = open("file.txt", O_RDONLY);  // 打开文件（系统调用）
   write(fd, "hello", 5);   // 写入文件（系统调用）
   ```

   > 虽然P、V操作**可能通过系统调用实现**（如Linux的 `semop()`​），但它们的本质定义是"**通信原语**"，而非系统调用本身。系统调用是更广泛的概念。
   >

   #### ❌ C. 作业控制命令

   **定义**：用户通过命令控制作业的执行状态（提交、暂停、恢复、终止等）。

   **举例**：

   ```bash
   # 在Shell中控制作业
   sleep 100 &        # 后台运行作业（提交）
   jobs               # 查看作业列表
   fg %1              # 将作业1切换到前台
   kill %2            # 终止作业2
   Ctrl+Z             # 暂停当前前台作业
   ```

   > P、V操作与作业控制完全无关，作业控制是**高级用户界面**层面的事情。
   >

   #### ✅ D. 低级进程通信原语

   **定义**：**原语**（Primitive）是不可中断的原子操作，**低级通信**指传递控制信息（信号量值），而非大量数据。

   **举例**：

   ```c
   // 经典的哲学家就餐问题 - 使用P、V原语实现互斥

   semaphore chopstick[5] = {1, 1, 1, 1, 1};  // 5根筷子信号量

   void philosopher(int i) {
       while (true) {
           think();
           
           P(chopstick[i]);           // 拿左边筷子（P操作：信号量-1）
           P(chopstick[(i + 1) % 5]); // 拿右边筷子（P操作：信号量-1）
           
           eat();
           
           V(chopstick[i]);           // 放左边筷子（V操作：信号量+1）
           V(chopstick[(i + 1) % 5]); // 放右边筷子（V操作：信号量+1）
       }
   }
   ```

   > **P操作**（Proberen，荷兰语"尝试"）：信号量值减1，若结果为负则阻塞；  
   > **V操作**（Verhogen，荷兰语"增加"）：信号量值加1，若有等待进程则唤醒。
   >
   > 它们是**最基础的进程同步工具**，故称"低级原语"。
   >

   ---

   ### 总结

   |选项|分类|与PV操作的关系|
   | -----------------| --------------| :--------------: |
   |A. 机器指令|硬件层面|❌ 无关|
   |B. 系统调用命令|操作系统接口|❌ 概念不同|
   |C. 作业控制命令|用户命令层面|❌ 无关|
   |**D. 低级进程通信原语**|**进程同步机制**| **✅ 正确**|

   **所以选 D。**

   ### 生产者和消费者问题的本质

   生产者-消费者问题（Producer-Consumer Problem）是 **Dijkstra** 提出的经典同步问题，核心要解决的是 **同步** 和 **互斥** 两方面：

   |问题|说明|在生产者-消费者问题中的体现|
   | :----: | :-----------------------------| :----------------------------------------------------------------------------|
   |**互斥（Mutual Exclusion）**|多个进程不能同时访问共享资源|生产者和消费者**不能同时操作缓冲区**，需要用互斥信号量（mutex）保护|
   |**同步（Synchronization）**|进程之间的执行顺序需要协调|缓冲区**满**时生产者必须等待（full信号量）<br />缓冲区**空**时消费者必须等待（empty信号量）|

   **经典实现举例：**

   ```c
   semaphore mutex = 1;    // 互斥信号量 - 保护缓冲区
   semaphore empty = n;    // 同步信号量 - 缓冲区空位数
   semaphore full = 0;     // 同步信号量 - 缓冲区中产品数

   void producer() {
       while (true) {
           produce_item();
           P(empty);       // 同步：检查是否有空位
           P(mutex);       // 互斥：进入临界区
           put_item();
           V(mutex);       // 互斥：离开临界区
           V(full);        // 同步：通知消费者有产品
       }
   }

   void consumer() {
       while (true) {
           P(full);        // 同步：检查是否有产品
           P(mutex);       // 互斥：进入临界区
           get_item();
           V(mutex);       // 互斥：离开临界区
           V(empty);       // 同步：通知生产者有空位
           consume_item();
       }
   }
   ```

   ---

   |选项|分析|判断|
   | :------------------------------------: | :-----------------------------------------------------------------------------------------------------------------------| :--------: |
   |**A**. 多个并发进程共享一个数据对象的问题|❌ 共享数据对象是问题的**场景**，但不是要解决的**核心问题**。共享数据对象带来的核心问题是同步与互斥|不准确|
   |**B**. 多个进程之间的同步和互斥问题|✅ 生产者-消费者问题是经典的**同步与互斥**案例，展示了如何利用信号量同时解决同步（empty/full）和互斥（mutex）两个问题|**正确**|
   |**C**. 多个进程共享资源的死锁与饥饿问题|❌ 生产者-消费者问题如果**实现不当**确实可能引入死锁（如P(mutex)和P(empty)顺序颠倒），但问题的**初衷**不是解决死锁和饥饿，而是解决同步互斥|不准确|
   |**D**. 利用信号量实现多个进程并发的问题|❌ 信号量是**工具**，生产者-消费者问题是**案例**。该问题的目的是展示如何用信号量**正确协调**并发进程，而非单纯"实现并发"|过于宽泛|

   ---

   > **生产者和消费者问题 = 同步问题（缓冲区满/空约束）+ 互斥问题（缓冲区互斥访问）**

**14. 【2016】**  使用TSL (Test and Set Lock) 指令实现进程互斥的伪代码如下所示。

```c
do{
  ...
  while(TSL(&lock));
  critical section;
  lock=FALSE;
  ...
}while(TRUE);
```

下列与该实现机制相关的叙述中，正确的是（ ）。II  
A. 退出临界区的进程负责唤醒阻塞态进程  
B. 等待进入临界区的进程不会主动放弃CPU  
C. 上述伪代码满足“让权等待”的同步准则  
D. while(TSL(&lock))语句应在关中断状态下执行

---

#### ❌ A. 退出临界区的进程负责唤醒阻塞态进程

TSL实现的是**自旋锁（Spinlock）** ，等待的进程通过`while(TSL(&lock))`​进行**忙等（Busy Waiting）** ，不会被阻塞进入阻塞态，而是一直占用CPU不断循环测试。因此不存在"唤醒阻塞进程"的机制。

**举例说明忙等现象：**

```
进程P1进入临界区 → lock=TRUE
进程P2想进入 → while(TSL(&lock)) 一直循环...
               （P2一直在CPU上跑循环，不放弃CPU）
进程P1退出 → lock=FALSE
进程P2终于跳出循环，进入临界区
```

#### ✅ B. 等待进入临界区的进程不会主动放弃CPU

**正确！**  TSL自旋锁的典型特征就是：

- 进程在`while(TSL(&lock))`​中不断**自旋**（spin）
- 不会主动调用`yield()`​、`sleep()`等来放弃CPU
- 一直占用CPU直到时间片用完或被抢占

这被称为 **"忙等"（Busy Waiting）** ，是自旋锁的核心特点。

#### ❌ C. 上述伪代码满足"让权等待"的同步准则

 **"让权等待"**  准则要求：当进程不能进入临界区时，应立即**释放处理机**，避免忙等。而TSL实现显然**不满足**这一准则，因为它让进程一直自旋等待。

**对比正确实现（满足让权等待）—— 信号量阻塞机制：**

```c
// 使用信号量（正确实现让权等待）
void P(semaphore *S) {
    S->value--;
    if (S->value < 0) {
        block(S->list);   // 将自己加入等待队列并阻塞（让出CPU）
    }
}
```

#### ❌ D. while(TSL(&lock))语句应在关中断状态下执行

TSL指令本身就是**硬件原子指令**，`Test and Set`操作在硬件层面是不可分割的，不需要关中断来保证原子性。

此外，如果在`while`循环期间长时间关中断：

- 系统无法响应外部中断（时钟中断、IO中断等）
- 其他进程无法获得CPU时间片
- 严重影响系统性能

**TSL指令硬件实现原理（原子性由硬件保证）：**

```assembly
TSL:        ; Test and Set Lock - 原子操作
    MOV  AX, #1      ; 将1送入寄存器AX
    XCHG AX, lock    ; 交换AX与lock的值（硬件保证不可中断）
    RET
```

执行过程中不会被时钟中断打断，本身就具有原子性，无需额外关中断。

---

### 总结

|选项|判断|原因|
| :----: | :----: | :----------------------------------|
|A|❌|TSL是忙等，没有阻塞态进程需要唤醒|
|**B**| **✅**|**忙等（自旋）是TSL的核心特征，不主动放弃CPU**|
|C|❌|忙等违背了"让权等待"准则|
|D|❌|TSL硬件原子指令，无需关中断|

**所以选 B。**

**16. 【2018】**  属于同一进程的两个线程thread1和thread2并发执行，共享初值为0的全局变量x。thread1和thread2实现对全局变量x加1的机器级代码描述如下。

|thread1|thread2|
| :---------------------| :---------------------|
|​`mov R1, x` // (x) → R1|​`mov R2, x` // (x) → R2|
|​`inc R1` // (R1)+1 → R1|​`inc R2` // (R2)+1 → R2|
|​`mov x, R1` // (R1) → x|​`mov x, R2` // (R2) → x|

在所有可能的指令执行序列中，使x的值为2的序列个数是（ ）。II  
A. 1  
B. 2  
C. 3  
D. 4

必须要t1执行完三条x=1 再执行t2或者反过来x才可能是2

正确答案是 **B. 2**。

两个线程各自执行三条指令（读、加、写），所有可能的交错序列共有 \( \binom{6}{3} = 20 \) 种。要使最终 \( x = 2 \)，必须一个线程的所有指令完全在另一个线程之前执行（即顺序执行），只有两种顺序：线程1全部先于线程2，或线程2全部先于线程1。其他任何交错都会导致一个线程的写入被覆盖，最终 \( x = 1 \)。因此，使 \( x = 2 \) 的序列个数为 **2**。

**17. 【2018】**  若x是管程内的条件变量，则当进程执行x.wait()时所做的工作是（ ）。I IV

- C 错误：条件变量没有“值”，wait 直接阻塞进程，不依赖判断。
- D 正确：阻塞进程并插入条件变量的阻塞队列。

A. 仅 I、IV

---

实现临界区互斥机制需要遵循以下三条**基本准则**（由 Dijkstra 提出）：

|准则|含义|是否必须|
| :----: | :-------------------------------------------------------| :-------------------------------------: |
|**I. 互斥**|两个进程不能同时进入临界区|✅ **必须**|
|**II. 空闲让进**|当临界区空闲时，允许一个进程立即进入|✅ **必须**|
|**III. 有限等待**|进程从申请进入到获准进入的时间是有限的（不会无限等待）|✅ **必须**|
|**IV. 让权等待**|不能进入临界区的进程应释放 CPU（即避免忙等）|❌ **可选**（好但非必须，例如自旋锁就不满足）|

**选项分析：**

- **A. 仅 I、IV** ❌ 缺少 II 和 III，且 IV 不是必须的
- **B. 仅 II、III** ❌ 缺少 I（互斥是最基本的）
- **C. 仅 I、II、III** ✅ 正确，正是三条必须准则
- **D. 仅 I、III、IV** ❌ 缺少 II，且 IV 不是必须的

**补充说明：**

- 让权等待（IV）是为了提高 CPU 利用率，但不是实现互斥的必要条件。例如 TSL 指令实现的自旋锁（忙等）依然是正确的互斥机制，只是不满足让权等待。
- 因此，必须遵循的是 I、II、III。

**故选择 C。**
