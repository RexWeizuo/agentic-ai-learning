# 4.3 I/O 软件的层次结构

本小节从具体例子出发，逐步讲解 I/O 软件的层次结构。

---

## 一、I/O 控制的代码实现（理解即可）

### （一）程序查询方式实例

要在打印机上打印字符串 S="ABCDEFGH"（保存在内核缓冲区，指针 p），采用统一编址方式：

```c
for(int i = 0; i < count; i++) {
    while(*printer_status_reg != READY);  // 忙等直到打印机就绪
    *printer_data_reg = p[i];             // 将数据写入数据寄存器
}
```

CPU 不断轮询状态寄存器，直到 Ready 才写一个字符，然后继续轮询下一个。

### （二）程序中断方式实例

同样打印 "ABCDEFGH"，但 S 在进程 A 的**用户缓冲区**中，采用程序中断方式：

**用户程序**（用户态）：
```c
print(buffer, strlen(buffer));  // 调用系统调用
```

**系统调用 print()**（内核态）：
```c
copy_from_user(buffer, p, count);       // 用户空间 → 内核缓冲区
enable_interrupts();                     // 打开中断
while(*printer_status_reg != READY);     // 等待打印机就绪
i = 0;
*printer_data_reg = p[i];               // 放入第一个字符
scheduler();                             // 切换到进程 B，自己阻塞
```

**中断处理程序**（每次打印机打完一个字符触发）：
```c
if(i >= count - 1)       // 所有字符打印完毕
    unblock_user();       // 唤醒进程 A
else {
    i = i + 1;
    *printer_data_reg = p[i];  // 放入下一个字符
}
acknowledge_interrupt();
return_from_interrupt();  // 返回进程 B
```

**关键理解**：print() 只放了第一个字符就阻塞了，剩余字符都在中断处理程序中逐个发送。

## 二、I/O 软件的层次结构（理解并记忆）

### （一）为什么需要分层

打印字符串的底层实现非常复杂——需要理解进程切换、知道端口作用和访问方式、熟悉 I/O 控制流程。不同设备端口数量和控制方式不同，程序员不可能手动编码每个设备。

→ 将这些底层实现**封装**，提供 API 给程序员使用。

### （二）四层结构（由上到下）

| 层次 | 位置 | 职责 |
|------|------|------|
| **用户层 I/O 软件** | 用户空间 | 提供库函数（如 scanf/printf），调用系统调用 |
| **设备独立性软件** | 内核空间 | 提供统一接口、设备命名、保护、缓冲、分配回收 |
| **设备驱动程序** | 内核空间 | 将抽象请求转为具体命令，控制设备控制器 |
| **中断处理程序** | 内核空间 | I/O 完成后处理善后，唤醒阻塞进程 |

> 低层（驱动程序 + 中断处理）与硬件打交道；高层（设备独立性软件 + 用户层）独立于硬件。
> 设备驱动程序和中断处理程序并非严格分层调用，实际有交叉。

### （三）逐层详解

#### 1. 用户层 I/O 软件

```c
void main() {
    int x;
    scanf("%d", &x);  // C 库函数，用户态
}
```

scanf() 是 C 库函数，属于用户层 I/O 软件。负责调用系统调用（如 read()），本身不直接操作硬件。

#### 2. 设备独立性软件

```c
int sys_read(......) {
    int devID = get_devID(......);          // 获取设备号
    chrdev_sw[devID].read(......);          // 调用字符设备接口函数
}
```

通过**字符设备开关表**找到对应的驱动程序：

| 设备号 | 设备类型 | read 函数地址 | write 函数地址 |
|--------|----------|---------------|----------------|
| 1 | 键盘 | 3000 | NULL |
| 2 | 鼠标 | 6000 | NULL |

```c
struct chrdev_switch {
    int (*read)(......);
    int (*write)(......);
};

// 驱动程序初始化时注册
chrdev_sw[1].read = keyboard_read;    // 1号设备 = 键盘
chrdev_sw[2].read = mouse_read;       // 2号设备 = 鼠标
```

三类标准接口：
- **流设备接口（字符设备）**：`read(devNum, buffer, size)`, `write(devNum, buffer, size)`
- **块设备接口**：`read(devNum, devAddr, buffer)`, `write(devNum, devAddr, buffer)`, `seek(devNum, devAddr)`
- **网络通信接口**

#### 3. 设备驱动程序

将上层抽象 I/O 请求转换为针对 I/O 设备的具体命令和参数，装入设备控制器的寄存器中。由设备厂商提供。

**驱动程序子程序**：
1. 初始化函数
2. 卸载函数
3. 申请设备函数
4. 释放设备函数
5. I/O 操作函数
6. 中断处理函数

**设备驱动程序执行过程**（重点顺序）：
1. 将抽象要求转换为具体要求（如：盘块号 → 柱面号/磁头号/扇区号）
2. 校验服务请求（如：不能从打印机"读"数据）
3. 检查设备状态（读状态寄存器）
4. 传送必要的参数（速率、字符长度等）
5. 启动 I/O 设备（向命令寄存器写控制命令）

> **2013 真题**：计算柱面号/磁头号/扇区号的程序是——**设备驱动程序**（答案 C）
> **2010 真题**：用户通过键盘登录，首先获得键盘输入信息的程序是——**中断处理程序**（答案 B）

#### 4. 中断处理程序

设备 I/O 完成时向 CPU 发中断后调用，进行善后处理：找到等待 I/O 的阻塞进程 → 将其就绪。若 I/O 请求队列还有后续请求，启动下一个。

## 三、SPOOLing 技术（假脱机技术）（理解即可）

### （一）从脱机到假脱机

**联机批处理**：CPU 直接控制输入/输出全程 → CPU 与 I/O 速度不匹配，效率低。

**脱机批处理**：用卫星机（外围机）专门负责 I/O，主机只负责计算。但需要额外硬件。

**SPOOLing（假脱机）**：在多道程序环境下，用一道程序模拟脱机输入的外围机，用另一道模拟脱机输出的外围机，**不需要额外硬件**。

### （二）SPOOLing 打印系统实例

当用户进程请求打印时：

1. **假脱机管理进程**：
   - 在磁盘缓冲区申请空闲盘块，将打印数据存入
   - 为用户进程申请"请求打印表"，填入打印要求，挂到假脱机文件队列
   - 对用户进程而言 → **打印已完成**（实际没打）

2. **假脱机打印进程**（打印机空闲时）：
   - 从假脱机文件队列队首取请求打印表
   - 将数据从磁盘缓冲区 → 内存缓冲区 → 打印机
   - 一个任务完成，继续下一个；队列空则阻塞

**SPOOLing 特点**：
- 提高了 I/O 速度（低速设备 → 磁盘存取）
- 将独占设备改造为共享设备
- 实现了虚拟设备功能（每个进程以为独占一台打印机）

### （三）SPOOLing 系统组成

| 组成部分 | 位置 | 作用 |
|----------|------|------|
| 输入井/输出井 | 磁盘 | 收容输入/输出数据 |
| 输入/输出缓冲区 | 内存 | 暂存设备与井之间的数据 |
| 输入/输出进程 | 内存 | 模拟外围机，管理数据传送 |
| 井管理程序 | 内存 | 控制作业与磁盘井之间的信息交换 |
