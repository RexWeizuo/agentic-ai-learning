为了完成这个任务，操作系统（内核）需要把 S 的 8 个字符一个接一个地送到打印机控制器的数据端口中，随后控制器会自动把这些字符送到打印机上去打印。指针 printer_data_reg 保存了数据端口地址，指针 printer_status_reg 保存了状态端口地址。

```c

    // 忙等直到打印机就绪（轮询状态寄存器）

    // 将数据写入打印机数据寄存器
    *printer_data_reg = p[i];
}
```
count 表示需要打印的字符个数，p[i]表示当前需要打印的字符。在 for 循环中，第一条语句用来判断打印机是否已经就绪。如果尚未就绪，就在该处循环等待；如果已经就绪，则退出循环检测语句，并且把第 i 个字符复制到打印机的数据寄存器当中。

```c

```
对于系统调用函数 print()（内核函数），它的部分代码如下。
```c

enable_interrupts( );
while( *printer_status_reg != READY); // 轮询状态端口，直到设备返回 READY 状态
i = 0;
*printer_data_reg = p[i];
```

这段代码的说明如下：
① 把 buffer 缓冲区中的字符串复制到 p 指向的内核缓冲区中，通过 P[i]访问不同字符。
② 打开中断，这样，CPU 就能够处理中断。
③ 轮询状态端口，判断打印机的当前状态是否空闲。如果不空闲，就一直循环等待。否则，就把循环变量 i 初始化为 0，并把需要打印的 P[0]放入数据端口中。注意，这里仅仅是把 P[0]放入到数据寄存器中，然后就不管了。
④ 接着，调用进程调度程序，切换到另一个进程 B 去执行，而自己则会被阻塞起来，并且挂到相应的阻塞队列中。

用户程序
```c

}
```

库函数
```c
int scanf(char *format[, argument, ...]) {
    // ... 其他逻辑
    read(......); //调用 read 系统调用
    ......
}
```

scanf()函数是 C 语言的库函数，属于用户层 I/O 软件（处于用户空间）。用户层软件用于提供与用户互动的接口，用户可直接调用该层所提供的与 I/O 操作有关的库函数对设备进行操作。
这些库函数在用户态运行，往往不会做太多的事情，主要负责调用操作系统提供的系统调用（API），例如，open()、read()、write()等。因此，scanf()函数在具体实现时需要启动系统调用，让操作系统来完成此次 I/O 工作，即调用 read()。
当执行 main()和 scanf()函数时，CPU 都处于用户态，一旦启动了系统调用，就会发生状态的切换，CPU 就会从用户态切换为内核态，然后去执行操作系统内核的代码，如下所示。

系统调用
```c

    int devID=get_devID(......);//获取设备号

    ......
}
```

Q：I/O 控制的具体实现是由设备驱动程序和中断处理程序来完成的，似乎不太需要设备独立性软件，那它的作用是什么呢？
A：简单来说，设备驱动程序解决了能不能用的问题，而设备独立性软件解决了如何更好用，用起来更方便的问题。下面举例说明。

（一）基本原理
1.联机/脱机批处理
单道批处理分为联机批处理和脱机批处理两种类型。
（1）联机批处理
联机批处理的工作方式如下：

--- Page 220 ---
① 若干个程序员把自己的作业（卡片或纸带）交给操作员。
② 操作员把一批作业装到输入设备上，在监督程序的控制下纸带信息送到外存上。
③ 监督程序从外存上将作业调入内存运行，并将结果保存到外存上，再通过打印设备打印。
④ 当一个作业处理完毕后，再处理下一个作业，直到该批作业全部处理完毕。

[图片：联机批处理流程图，展示了程序A/B通过操作员装入输入设备，经读纸带机进入外存，监督程序控制主机调入作业，处理后保存结果至外存，最后经打印机打印]

联机批处理下，作业从输入到打印的整个过程都在 CPU 直接控制下进行。由于 CPU 运算速度不断提高，CPU 和 I/O 设备之间的速度越来越不匹配，让 CPU 直接控制 I/O 设备已经严重地影响了系统的效率。因此，脱机操作方式就出现了。

(2) 脱机批处理
脱机指输入输出操作脱离主机控制。脱机批处理系统由主机和卫星机组成，卫星机又称为外围机，它只与 I/O 设备打交道，即卫星机专门负责输入/输出，主机专门负责计算。
该技术下，利用专门的外围控制机预先将低速 I/O 设备上的数据传送到外存上，这样，当 CPU 需要输入数据时，便可以直接从磁盘中读取数据，极大地提高了输入速度。反之，在 CPU 需要输出数据时，也能以很快的速度把数据先输出到外存上，然后 CPU 便可去做自己的事情了。

[图片：脱机批处理流程图，展示了纸带机连接卫星机，卫星机连接外存，外存连接主机（调入作业），主机连接外存（输出结果），外存连接卫星机，卫星机连接打印机]

2.假脱机技术
(1) 基本原理
当系统中引入多道程序技术后，系统便完全可以利用其中的一道程序来模拟脱机输入时的外围控制机功能，把低速 I/O 设备上的数据传送到高速磁盘上；再用另一道程序模拟脱机输出时外围控制机的功能，把数据从磁盘传送到低速输出设备上。该技术称为假脱机技术（Spooling 技术），属于用户层软件，下面举例说明。

【例】打印机属于独占设备。利用假脱机技术可将它改造为一台共享设备，从而提高了设备的利用率，也方便了用户使用。

假脱机打印机系统主要由磁盘缓冲区、打印缓冲区以及假脱机管理进程、假脱机打印进程构成。磁盘缓冲区用于暂存用户进程的输出数据，该缓冲区可以设置几个盘块队列，如空盘块队列、满盘块队列等。
每当用户进程发出打印输出请求时，假脱机打印机系统并不会立即把打印机分配给该用户进程，而是会由假脱机管理进程完成两项工作：
① 在磁盘缓冲区中申请一个空闲盘块，并将要打印的数据送入其中暂存。
② 为用户进程申请一张空白的用户请求打印表，并将用户的打印要求填入其中，再将该表挂到假脱机文件队列上。

[图片：假脱机打印机系统工作流程图，展示了假脱机文件队列、磁盘缓冲区（含空盘块队列、满盘块队列）、打印机、用户进程、内存、磁盘以及假脱机打印进程和假脱机管理进程之间的关系]

在这两项工作完成后，虽然还没有进行实际的打印输出，但对用户进程而言，其打印请求已经得到满足，打印输出任务已经完成。
真正的打印输出是假脱机打印进程负责的，当打印机空闲时，该进程首先从假脱机文件队列的队首插取一张请求打印表，然后根据表中的要求将要打印的数据由磁盘缓冲区传送到内存缓冲区，再交付打印机进行打印。

[图片：假脱机打印机系统数据流向图，展示了数据从磁盘缓冲区经内存缓冲区最终到达打印机的过程]

一个打印任务完成后，假脱机打印进程会再次查看假脱机文件队列，若队列非空，则重复上述工作，直至队列为空。此后，假脱机打印进程会将自己阻塞起来，当再次有打印请求时，其会被重新唤醒运行。
由此可见，对每个用户而言，系统并非即时执行其程序输出数据的真实打印操作，而只是即时将数据输出到缓冲区，让用户感觉系统已为他打印；真正的打印操作，是在打印机空闲且该打印任务在等待队列中已排到队首时进行的；而且，打印操作本身也是利用 CPU 的一个时间片，没有使用专门的外围机。以上过程是对用户屏蔽的，即用户是不可见的。假脱机系统的特点如下。
① 提高了 I/O 速度。这里对数据所执行的 I/O 操作，已从对低速 I/O 设备执行的 I/O 操作演变为对磁盘缓冲区中的数据进行存取操作。
② 将独占设备改造为共享设备。
③ 实现了虚拟设备功能。虽然只有一台独占设备，但每个进程都认为自己独占了一个设备。当然，该设备只是逻辑上的设备。

（二）SPOOLing 的组成
SPOOLing 技术的具体实现有所不同，下面给出一个通用模型，主要由以下四部分构成。
1.输入井和输出井
在磁盘上开辟出的两个存储区域。输入井用于收容 I/O 设备输入的数据，输出井用于收容用户程序的输出数据。输入/输出井中的数据一般以文件的形式组织管理，称为井文件。
一个文件仅存放某一个进程的输入（或者输出）数据，所有进程的数据输入（或输出）文件链接成为一个输入（或输出）队列。
2.输入缓冲区和输出缓冲区
在内存中开辟的两个缓冲区，输入缓冲区用于暂存由输入设备传送的数据，之后再传送到输入井。输出缓冲区用于暂存从输出井传送的数据，之后再传送到输出设备。

[图片：SPOOLing系统工作流程图，展示了输入进程、输入设备、输入缓冲区、输入井、输出井、输出缓冲区、输出设备、输出进程以及井管理程序之间的数据流向]

3.输入进程和输出进程
输入进程也称为预输入进程，用于模拟脱机输入时的外围控制机，将用户要求的数据从输入设备传送到输入缓冲区，再存放到输入井。当 CPU 需要输入数据时，直接从输入井读入内存。
输出进程也称为缓输出进程，用于模拟脱机输出时的外围控制机，待输出设备空闲时再将输出井中的数据经过输出缓冲区输出至输出设备上。
4.井管理程序
用于控制作业与磁盘井之间信息的交换。当作业执行过程中向某台设备发出启动输入或输出操作请求时，由操作系统调用井管理程序，由其控制从输入井读取信息或将信息输出至输出井。

【补充】设备驱动程序的执行过程（理解即可）
设备驱动程序的主要任务是启动指定设备，完成上层软件指定的 I/O 工作。但在启动设备之前，应先完成必要的准备工作，然后才向设备控制器发送一条启动命令。以下是设备驱动程序的执行过程。
1.将抽象要求转换为具体要求
由于上层软件对设备控制器的具体情况毫无了解，因而只能发出命令（抽象要求），这些命令是无法传送给设备控制器的。因此，就需要将这些抽象要求转换为具体要求。例如，将抽象要求中的盘块号转换为磁盘的盘面号、磁道号及扇区号。
2.校验服务请求
设备驱动程序在启动 I/O 设备之前，必须先检查该用户的 I/O 请求是不是该设备能够执行的。若用户试图请求从一台打印机读入数据，则设备驱动程序可能认为此次 I/O 请求非法，它将报告 I/O 请求出错。
3.检查设备的状态
设备驱动程序在启动设备之前，要先把状态寄存器中的内容读入 CPU 的某个寄存器中，通过测试寄存器中的不同位来了解设备的状态。
[图片：设备状态寄存器位定义图，展示了D7-D0各位的含义，如DSR、SYN DET、FE等，以及对应的中文解释如发送就绪、接受就绪、发送器空等]
4.传送必要的参数
在确定设备处于接收（发送）就绪状态后，便可向设备控制器的相应寄存器传送数据以及本次数据传送有关的参数，包括本次传送数据的速率、发送的字符长度等数据。
5.启动 I/O 设备
在完成上述各项准备工作后，驱动程序便可向设备控制器中的命令寄存器传送相应的控制命令。

**4.4 设备独立性软件**

设备独立性（device independence）的含义是上层软件所用的设备不局限于使用某个具体的物理设备。为了实现设备独立性，必须再在设备驱动程序之上设置设备独立性软件。

设备独立性软件的主要功能包括：用户程序与设备驱动程序的统一接口、设备命名、设备保护以及设备的分配与释放等，同时为设备管理和数据传输提供必要的存储空间。

**一、用户程序与设备驱动程序的统一接口（理解即可）**

**（一）用户程序的统一接口**

用户不会直接与底层的硬件设备打交道，而是与操作系统打交道，操作系统会提供 API 让编程人员来调用。例如，scanf()函数只是一个空的外壳，在具体实现时，它们会把传给它们的参数再往下传递给相应的系统调用函数 open()、read()、write()等。

Q：用户希望操作系统提供什么样的接口？这些接口函数应该具备什么样的一些特点？
A：显然，越简单越好，使用越方便越好。具体来说，有如下三个方面。

1.设备独立性
用户在编写程序、访问各种 I/O 设备时，无需事先指定特定的设备类型。例如，假设需要访问一个文件，该文件可能来自硬盘、U 盘或光盘，显然，这些设备都是各不相同的，希望各种类型的设备之间的差异由操作系统来处理，对用户来说是透明的，不必关心。

2.统一命名
用简单的字符串或整数的方式来命名一个文件或设备。例如，所有的文件和设备都采用路径名命名。“A:\1.txt”可能是软盘上的一个文件，“C:\2.txt”可能是硬盘上的一个文件。无论是访问什么类型的对象，使用的命名方式都是统一的路径名的方式。这样，程序使用起来就很方便。下图展示了基于路径名运行可执行程序 hello。

```bash
leon@leon-virtual-machine:~/OS$ /home/leon/OS/hello
hello world!
leon@leon-virtual-machine:~/OS$ 
```
