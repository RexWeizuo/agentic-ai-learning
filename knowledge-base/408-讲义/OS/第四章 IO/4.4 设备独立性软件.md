# 4.4 设备独立性软件

设备独立性：上层软件所用的设备不局限于使用某个具体的物理设备。为此必须在驱动程序之上设置设备独立性软件。

主要功能：统一接口、设备命名、设备保护、分配与释放、缓冲区管理。

---

## 一、用户程序与设备驱动程序的统一接口（理解即可）

### （一）用户程序的统一接口

三个设计目标：

**1. 设备独立性**：访问文件无需指定设备类型（硬盘/U盘/光盘差异由 OS 处理，对用户透明）。

**2. 统一命名**：所有文件和设备都用路径名命名。

| 设备路径 | 类型 | 说明 |
|----------|------|------|
| /dev/sda | 块设备 | 第一块 SATA/SCSI 硬盘 |
| /dev/sda1 | 块设备 | 硬盘第一个分区 |
| /dev/input/mice | 字符设备 | 鼠标输入设备 |
| /dev/fb0 | 字符设备 | 帧缓冲设备（显存直接访问） |

**3. 阻塞与非阻塞 I/O**

| | 阻塞 I/O | 非阻塞 I/O |
|---|---|---|
| 行为 | 进程被阻塞，直到 I/O 完成 | I/O 启动后立即返回 |
| 进程切换 | 发生切换 | 不切换，继续执行 |
| 难度 | 简单 | 涉及多线程，较难 |
| 适用 | 小并发 | 高并发，需协调并发线程 |

### （二）设备驱动程序的统一接口

设备独立性软件要求每个驱动程序与 OS 之间有相同的接口。添加新设备时只需按接口实现驱动程序即可。

## 二、缓冲区管理（理解并记忆）

### 为什么需要缓冲区

没有缓冲区也行（进程内存直接与硬盘交互），但引入缓冲区的好处：

1. **统一管理**：所有块设备数据统一管理，OS 设计更方便
2. **效率更高**：缓冲区的**共享**是关键——A 进程读入缓冲区的数据，B/C/D 进程可直接从缓冲区读（内存-内存速度是内存-硬盘的 100 倍），不必重复读盘

> 现代 OS 中几乎所有 I/O 设备都使用缓冲区。

### （一）单缓冲

| 场景 | 处理每块数据的时间 |
|------|-------------------|
| T > C（输入比处理慢） | M + T |
| T < C（处理比输入慢） | M + C |
| **通用公式** | **Max(C, T) + M** |

> T = 设备 → 缓冲区时间，M = 缓冲区 → 工作区时间，C = CPU 处理时间

**缺点**：将缓冲区数据写入工作区时，I/O 设备无法访问缓冲区，需等待。

### （二）双缓冲

两个缓冲区交替使用：当设备往 Buffer1 输入时，CPU 可从 Buffer2 取数据。

| 场景 | 处理每块数据的时间 |
|------|-------------------|
| T > C+M | T |
| T < C+M | C+M |
| **通用公式** | **Max(C+M, T)** |

> 双机通信：单缓冲只能单方向传输；双缓冲（各配发送+接收缓冲）才能实现双向同时传输。

**练习**：T=80μs, M=40μs, C=30μs，用单缓冲区，处理一块数据总时间？  
解：Max(30, 80) + 40 = **120μs**（答案 A）

### （三）循环缓冲

当生产者/消费者速度差距大时，双缓冲不够。引入多缓冲 → 组织成环形：

- **in 指针**：指向第一个空缓冲区（输入时移动）
- **out 指针**：指向第一个装满数据的缓冲区（输出时移动）

### （四）缓冲池

多个缓冲区组成池，可被多个进程共享（类似管程）。区别于缓冲区链表：缓冲池内含操作函数。

**三个队列**：

| 队列 | 内容 |
|------|------|
| 空缓冲队列 | 空缓冲区，用于写入 |
| 输入队列 | 装满输入数据的缓冲区，供读取 |
| 输出队列 | 装满输出数据的缓冲区，供输出 |

**四种工作方式**：

| 方式 | 操作 |
|------|------|
| 收容输入 | 从空队列取缓冲区 → 装输入数据 → 挂到输入队列 |
| 提取输入 | 从输入队列取缓冲区 → 提取数据 → 挂到空队列 |
| 收容输出 | 从空队列取缓冲区 → 装输出数据 → 挂到输出队列 |
| 提取输出 | 从输出队列取缓冲区 → 提取数据 → 挂到空队列 |

**四种缓冲机制对比**：

| 机制 | 处理时间公式 | 特点 |
|------|-------------|------|
| 单缓冲 | Max(C,T)+M | 简单，效率一般 |
| 双缓冲 | Max(C+M,T) | 可连续输入/双向通信 |
| 循环缓冲 | — | 多缓冲环形组织 |
| 缓冲池 | — | 可被多个进程共享 |

## 三、设备分配与回收（理解即可）

### （一）逻辑设备名 vs 物理设备名

**物理设备名**：早期 OS 直接用物理名。若指定设备忙，即使有同类空闲设备也无法分配——不灵活。

**逻辑设备名**（如 /dev/printer）：抽象设备名，系统自动查找该类设备中第一台空闲的。只要该类有一台空闲，进程就不会因申请失败而阻塞。

**映射**：逻辑设备表（LUT）完成逻辑名 → 物理名的转换。

### （二）设备分配的数据结构

| 结构 | 数量 | 关键字段 |
|------|------|----------|
| 系统设备表（SDT） | 系统唯一 | 设备类、设备标识符、DCT 指针、驱动入口 |
| 设备控制表（DCT） | 每设备一张 | 设备类型、标识符、状态（忙/闲）、控制器指针、等待队列 |
| 控制器控制表（COCT） | 每控制器一张 | 控制器标识符、状态、通道指针、等待队列 |
| 通道控制表（CHCT） | 每通道一张 | 通道标识符、状态、控制器表首址、等待队列 |

### （三）设备分配步骤

1. **分配设备**：查 SDT → 找 DCT → 设备空闲？是则分配；否则进程挂设备等待队列
2. **分配控制器**：查 COCT → 控制器空闲？是则分配；否则挂控制器等待队列
3. **分配通道**：查 CHCT → 通道空闲？是则分配；否则挂通道等待队列

只有**设备、控制器、通道三者都分配成功**，这次分配才算成功。

> 如果用**逻辑设备名**：先查第一个该类设备的 DCT，忙则查第二个……全部忙才挂等待队列。

### （四）设备分配考虑因素

**1. 设备固有属性**：

| 属性 | 分配策略 |
|------|----------|
| 独占设备 | 分配给一进程后独占至释放 |
| 共享设备 | 可同时分配给多进程，需调度访问次序 |
| 虚拟设备 | 可共享，同时分配给多进程 |

**2. 分配算法**：先来先服务 / 优先级高者优先。

**3. 安全性**：

| | 安全分配 | 不安全分配 |
|---|---|---|
| I/O 请求后进程状态 | 阻塞至 I/O 完成 | 继续运行不阻塞 |
| 死锁风险 | 无（摒弃"请求和保持"） | 有（可用银行家算法避免） |
| CPU 与 I/O | 串行工作 | 可并行（进程可操作多设备） |

### （五）逻辑设备表（LUT）

两种设置方式：

| | 整个系统一张 LUT | 每用户一张 LUT |
|---|---|---|
| 逻辑设备名 | 不可重复 | 各用户独立 |
| 适用 | 单用户系统 | 多用户系统 |
| 存放位置 | 系统全局 | 进程 PCB 中 |
