# 1.2 运行环境

‍1.2 操作系统的运行环境
通过 1.1 的学习，已知操作系统是个大型程序。既然是程序，就得依靠 CPU 等硬件设备执行。硬件性能决定了操作系统的流畅度以及功能。例如，同样是一条加法指令，50 年前的 CPU 的执行时间比今天的 CPU 的执行时间久得多！

[图片：应用软件、操作系统、硬件设备的层级示意图]

因此，需要重点研究硬件给操作系统提供了哪些支持（功能），即操作系统的运行环境是什么样的。一般认为，硬件需要提供给操作系统以下支持：中断机制、特权指令、系统调用、内存保护、输入输出系统。本小节主要讨论中断机制、特权指令以及系统调用，其他内容在后续的章节中深入。

一、中断机制（理解并记忆）
中断指 CPU 暂停正在执行的程序，转去执行处理中断事件的中断服务程序，待执行完中断服务程序再返回到原被中断的程序继续执行。

[图片：中断流程示意图，显示主程序、中断申请、中断服务程序、返回]

注：“中断机制”这部分内容已在计组中详细讲过
计算机中能够打断当前程序正常执行的事件大体可以分为两大类：内部异常和外部中断。
（一）外部中断
1. 基本概念
外部中断（简称中断）指程序执行过程中，若外设完成任务或发生某些特殊事件，例如，打印机缺纸、键盘缓冲区已满等，外设会向 CPU 发出中断请求，要求 CPU 对这些情况进行处理。

[图片：CPU 与打印机、键盘的交互示意图]

通常，每条指令执行后，CPU 都会主动去查询有没有中断请求，有的话，则将下条指令地址作为断点保存，然后转到用来处理相应中断事件的“中断服务程序”执行，结束后回到断点继续执行。

[图片：中断处理详细流程图，显示被中断程序、中断响应、中断处理程序、返回到下一条指令执行]

Intel 将外部中断分成可屏蔽中断（maskable interrupt）和不可屏蔽中断（nonmaskable interrupt, NMI）。
(1) 可屏蔽中断
可屏蔽中断是指通过可屏蔽中断请求线 INTR 向 CPU 进行请求的中断，主要是来自 I/O 设备的中断请求。CPU 可以屏蔽这些中断设备的请求。

[图片：CPU与中断请求示意图。左侧为CPU，上方有一个小人举手说“我拒绝”。中间箭头标示 INTR 指向 IO设备（键盘、鼠标、打印机、网卡）。下方箭头标示 NMI 指向 灾难性错误（电源掉电、内存读写错误）。]

若某个 I/O 设备的中断请求被屏蔽，则它的中断请求信号将不会被送到 CPU。
(2) 不可屏蔽中断
不可屏蔽中断通常是非常紧急的硬件故障，通过专门的不可屏蔽中断请求线 NMI 向 CPU 发出中断请求。如：电源掉电，硬件线路故障等。
这类中断请求信号一旦产生，任何情况下它都不可以被屏蔽，因此一定会被送到 CPU，以便让 CPU 快速处理这类紧急事件。通常，这种情况下，中断服务程序会尽快保存系统重要信息，然后在屏幕上显示相应的消息或直接重启系统。

[图片：Windows 蓝屏界面，显示“你的设备遇到问题，需要重启。我们只收集某些错误信息，然后你可以重新启动。0% 完成”]

(1) 中断响应
CPU 中断响应的过程可分为关中断、保存断点和程序状态、识别异常和中断并转到相应的处理程序。
① 保存断点和程序状态
为了能在中断处理后正确返回到被中断的程序继续执行，必须将程序的断点（返回地址）送到栈或特定寄存器中。例如，x86 中需要保存将返回地址压入栈中，MIPS 需要保存 PC 至 EPC（异常指令地址寄存器）中。注：为了便于理解，这里先讲了“保存断点和程序状态”，不代表这是中断响应的第一步。

中断处理后还要回到被中断的程序继续执行，被中断时的程序状态字寄存器 PSW 的内容也需要保存在栈或特定寄存器中，中断返回时恢复到 PSW 中（有些 CPU 架构下不需要保存 PSW，仅需保存断点）。

② 关中断

若在保存断点和程序状态期间产生了新的中断，则可能破坏程序断点和程序状态的保存。因此，为了不被新的中断打断，要禁止响应新的中断，即关中断。

[图片：CPU响应中断示意图，显示CPU拒绝INTR（IO设备），响应NMI（灾难性错误）]

通常通过设置“中断允许位”（中断允许触发器）来实现中断的打开或关闭。例如，IA-32 中的“中断允许位”就是 EFLAG 寄存器中的中断标志位 IF。若 IF 置为 1，则为开中断，表示允许响应中断；若 IF 置为 0，则表示关中断，表示不允许响应中断。

[图片：EFLAGS寄存器标志位示意图，显示溢出标志、方向标志、中断使能标志等]

在中断响应过程中，通常由硬件将中断允许位清 0，以进行关中断操作。此外，也可以通过执行指令 sti 或 cli，将标志寄存器 EFLAGS 中的位置 1 或清 0，以使 CPU 处在开中断或关中断状态。

③ 判别中断源，转向中断服务程序

中断识别的任务是确定中断是由哪个中断源发出的，识别出中断源后还需要获取中断服务程序入口地址，这样才能执行中断服务程序。中断识别分为非向量中断和向量中断两种，这里我们仅讨论较为常见的向量中断。

向量中断中，每一个中断源都有一个唯一的中断编号与之对应，称为中断号。中断号由计算机系统统一分配，通常是固定不变的，如 x86 计算机中就包含 0~255 共 256 个中断号，其中，键盘的中断号为 09H，打印机的中断号为 0FH。

当某个设备进行中断请求时，由中断控制器（负责管理中断的硬件）通过 INTR 向 CPU 发送中断请求，

(2) 执行中断服务程序
① 单重中断 vs 多重中断
单重中断：指 CPU 执行中断服务程序过程中不能被其他中断请求打断的中断。
多重中断：指 CPU 执行中断服务程序过程中可以去响应其他中断请求的中断，又称为中断嵌套。

[图片：中断处理流程图。左侧展示了简单的中断申请、主程序、中断服务程序、返回流程。右侧展示了多重中断（中断嵌套）流程：主程序执行中响应中断1进入外设1中断服务程序，期间响应中断2进入外设2中断服务程序，最后依次通过IRET指令返回。]

② 单重中断的中断处理程序
通常，单重中断下的中断处理程序一般包含以下内容：
① 保护现场
② 执行中断处理逻辑相关的指令
③ 恢复现场
④ 开中断
⑤ 执行中断返回指令，该指令会恢复断点和部分寄存器，返回原程序执行。

（二）内部异常
内部异常（也称“内中断”）指处理器执行一条指令时，由处理器在其内部检测到的、与正在执行的指令相关的事件。Intel 将内部异常分为三类：故障（fault）、陷阱（trap）和终止（abort）。

1. 故障
故障是在引起故障的指令被启动后但未执行结束时 CPU 检测到的一类与指令执行相关的意外事件。例如，指令译码时出现“非法操作码”，执行除法指令时发现“除数为 0”等。
这种意外事件有些可以恢复，有些则不能恢复。例如，溢出、非法操作码、整数除 0 等故障无法通过异常处理程序恢复，因此不能回到被中断的程序继续执行。通常异常处理程序在屏幕上显示一个对话框告知发生了某种故障，然后终止发生故障的当前程序。

2. 陷阱
陷阱也称为自陷或陷入，与“故障”等其他异常事件不同，是预先安排的一种“异常”事件，就像预先设定的“陷阱”一样。
当执行到陷阱指令（也称为自陷指令）时，CPU 就调出特定的程序进行相应的处理，处理结束后返回到陷阱指令的下一条指令执行。其处理过程如下所示。

在 IA-32 中，用于“断点设置”的陷阱指令为 int 3，若想在被调试程序某处设置断点，就在该处加入一条 int 3 指令。
```assembly
_start:
; 初始化寄存器
mov eax, 5      ; 将 eax 寄存器设置为 5
mov ebx, 3      ; 将 ebx 寄存器设置为 3
; 设置断点
int 3           ; 插入断点指令，当执行到这里时，触发调试异常
; 简单加法操作
add eax, ebx    ; 将 ebx 的值加到 eax 中，结果存储在 eax 中
......
```
当 CPU 执行到该指令时，就会暂停当前被调试程序的运行，并发出异常，从而调出相应的调试程序来执行，执行结束后再回到被调试程序执行。
此外，也可以利用同样的方式实现系统调用（下文会详细解释）。

二、特权指令以及非特权指令（理解并记忆）
考研范围内，可以把程序简单地分为两类，一类是操作系统程序，另一类是用户程序。

程序是由一条条指令构成的，可以把指令分为两类，一类是特权指令，一类是非特权指令。
特权指令指那些只有操作系统才有权使用的指令。
例如，对输入输出设备进行直接访问的指令是特权指令，一般的用户程序都不能直接去访问磁盘、打印机等输入输出设备。

三、处理器状态（理解并记忆）
【例】假设小八同学在写程序 A 时，不小心在程序 A 中写入了几条特权指令，显然，这是不允许的！只有操作系统才有资格执行特权指令！因此，CPU 必须具备区分普通程序和操作系统程序执行权限的能力！如果 CPU 在执行一个用户程序时，发现其使用了特权指令，就必须马上击毙它！

[图片：一只拿着枪的猫，配文“鸡哔你”]

为了实现 CPU 区分程序执行权限的能力，CPU 厂商们给 CPU 设置了不同的执行状态～具体来说，可以根据运行程序对资源和机器指令的使用权限，把处理器设置为不同的状态。当运行操作系统程序的时候，CPU 处于某种执行状态；当运行应用程序的时候，CPU 处于另外一种执行状态。
多数系统把处理器的工作状态划分为管态和目态两种。

[图片：两个奥特曼，左边标注“管态”，右边标注“目态”]

（一）管态
管态指 CPU 执行操作系统程序时的状态，又称为特权态、系统态、内核态、核心态。当处理器处于管态时，它可以执行所有的指令，包括各种特权指令，也可以使用所有的资源，访问所有寄存器和存储区域，并具有改变处理器状态的能力。（某些教材下写的是除了“访管”以外的全部指令）

（二）目态
目态指 CPU 执行用户程序时的状态，又称为普通态或用户态。当处理器处于目态时，其执行能力受限。这种状态下，处理器不能使用特权指令，不能直接使用系统资源，也不能改变 CPU 的工作状态，并且只能访问这个用户程序自己的存储空间。

四、系统调用（理解并记忆）
（一）基本定义
【例】若用户程序A想在显示器上输出字符串“hello,world!”，则需要执行I/O指令，将字符串的编码写入显卡中。然而，I/O指令属于特权指令，用户程序A不能执行特权指令，这可怎么办呢？
虽然用户程序无法直接执行I/O指令，但是操作系统可以！因此，小A灵机一动：我无法变成迪迦，但我可以召唤迪迦！我可以让迪迦（操作系统）帮我完成访问显卡的工作~
那应该怎么召唤迪迦呢？小A轻蔑一笑，我可以用“陷入指令”触发异常，让迪迦接管CPU，这样不就召唤出来了！
因此，系统调用是操作系统提供给用户程序的相关过程（函数），为用户程序实现一些服务。这些服务的实现往往需要执行一些特权指令，访问一些核心资源，是用户程序无法自己完成的。
[图片：迪迦奥特曼图片，配文“你相信光么”]
（二）实现原理
系统调用的实现原理为：用户程序通过访管指令（陷阱指令、trap指令、系统调用指令）主动陷入异常，切换到操作系统程序来为其实现某种功能。具体过程如下：
①当CPU执行到一条访管指令时，会引起一个中断（异常），称为访管中断。
②中断响应的过程中，处理器会保存程序状态字和程序计数器，将CPU工作状态由目态切换到管态。从目态到管态的转换，不是通过指令来修改CPU的状态标志位，而是由CPU在中断时硬件自动完成的。

③接着执行相应的中断处理程序，调用相应的系统服务。
④当中断处理结束后，CPU 会恢复被中断程序的上下文环境，将工作状态由管态恢复为目态，回到中断点继续执行。

[图片：CPU状态切换流程图。左侧为“CPU处于目态 用户程序”，包含指令“8048400: 陷阱指令”和“8048402: mov1...”。中间通过“中断响应”进入右侧“CPU处于管态”。右侧为“OS的陷阱处理程序（中断服务程序）”，向下执行“根据用户程序的需求调用不同的系统服务”，右侧标注“系统服务”。最后通过“返回”箭头回到左侧“返回到下一条指令执行”。]

注：以上过程也可以简化为中断响应后，执行中断服务程序
因此，如果用户程序想要调用一个系统服务，如从键盘输入数据或在屏幕上显示数据等，就执行一条陷阱指令陷入中断。此时 CPU 会从目态升级为管态，跳转到中断处理程序去执行，在中断处理过程中再去调用相应的系统服务程序。当这些事情做完以后，CPU 的状态从管态恢复为目态，回到用户程序继续执行。
总之，通过系统调用这种方式，一方面支持了特权指令，保护了系统的安全；另一方面普通的用户程序也能间接地执行特权指令。

（三）从目态转成管态的情况
在下列情况下，CPU 的工作状态可由用户态转变为核心态。
1.用户程序执行系统调用时。
2.当 I/O 设备产生中断事件或 CPU 内部出现异常（比如程序运行出现除零错误或非法执行特权指令等），当前程序被中断，CPU 转向中断（异常）处理程序。
这两种情况都通过中断机制发生，可以说，中断和异常是用户态到核心态转换的途径。
另外，C 语言的许多标准库函数（如 printf、fopen）通过调用系统调用来实现功能，但并非所有库函数都依赖系统调用。例如，sin()这样的数学函数完全在用户态实现。

2.时钟管理

系统调用中，调用程序是运行在用户态，而被调用程序是运行在系统态。

[图片：系统调用与一般过程调用的状态转换示意图，左侧为CPU处于目态/用户程序，右侧为CPU处于管态/OS的陷阱处理程序，中间通过中断响应和返回进行交互]

2.状态的转换
一般的过程调用并不涉及到系统状态的转换，但在运行系统调用时，需要通过中断机制，由用户态转换为系统态。
3.返回主程序的方式不同。函数调用返回与中断服务程序返回使用的指令不同，如 x86 中函数使用 ret 指令返回，中断返回则是 iret 指令。
call 指令的作用：①将 IP/PC 旧值压栈保存。②设置 IP 新值，无条件转移至被调用函数的第一条指令。
ret 指令的作用：从函数的栈帧顶部找到 IP 旧值，将其出栈并放入 IP 寄存器中。
在 x86 架构中，CPU 响应中断后，硬件会自动将 PSW 和 PC 的值压栈保存。中断处理结束后，通过执行 iret 指令恢复 PC 和 PSW。因此，系统调用发生时需要保存 PSW 和 PC 的值。而一般的过程调用，无需保存 PSW 的值（除非特殊需求）。

系统调用指令通常是以汇编语言代码的形式存在的，例如 x86 指令系统中的 int 0x80 指令。
Q：既然系统调用如此重要、如此常见，那我们平时编程的时候为何又从未碰到过呢？
A：我们平时一般使用高级程序设计语言编程。在这种情形下，为了编程的方便，一般不会直接去启动系统调用，而是通过各种库函数来间接地启动系统调用。

```c
    x=x+1;
    printf("%d",x);
    return 0;
}
```

scanf()和 printf()都是标准的 C 语言的库函数。
以 print()函数为例，它自己并不能完成在屏幕上打印数据的功能，因为这需要用到特权指令，只能由操作系统来完成。
因此，printf()函数的工作主要是一些辅助工作，如根据格式控制字符串来创建待打印的字符串，并根据系统服务的编号来准备参数，然后就启动系统调用指令，由操作系统来完成剩余的工作。
等系统调用结束以后，CPU 的状态又从内核态恢复为用户态，并去执行 printf()函数后面的语句。
当 printf()函数结束后，又会回到 main()函数继续往下执行。

【补充】系统调用分类（一轮可以直接跳，二轮再看，了解即可）
这些系统调用按功能大致可分为如下几类：
1.设备管理。完成设备的请求或释放，以及设备启动等功能。
2.文件管理。完成文件的读、写、创建及删除等功能。例如 open、read 系统调用，用于打开文件、读文件。
3.进程控制。完成进程的创建、撤销、阻塞及唤醒等功能。例如 fork 系统调用，可以创建一个子进程。
4.进程通信。完成进程之间的消息传递或信号传递等功能。
5.内存管理。完成内存的分配、回收等功能。
