# 1.4 操作系统引导结构

‍1.4 操作系统引导、结构以及虚拟机
一、操作系统的引导（理解即可）
操作系统是个大型程序。程序不执行的时候被保存在外存中，保存在外存中的这些 0/1 序列被称为文件。程序运行前，需要从外存把程序文件给复制到主存中，然后 CPU 不断从主存中取出指令并执行。

[图片：计算机硬件连接示意图，展示了主机（CPU）、主存储器、I/O设备（打印机、键盘、外存）之间的地址、控制、数据连接]

为了更方便地连接各个部件，往往会通过主板将 CPU、主存、I/O 设备相连。

[图片：主板实物图]

主存包含了 RAM（DRAM）和 ROM 两部分，ROM 位于主板上，空间有限，仅保存用于引导操作系统的部分程序，其他程序执行时需要从外存拉到 RAM 中。大部分情况下，“主存”指的是其 RAM 区域。
操作系统负责把待执行的用户程序调入 RAM，然后修改 PC 的值，跳转到该程序执行。但是计算机刚启动的时候，RAM 里还是空的（易失性），不存在任何程序，那是谁把操作系统给拉入 RAM，即操作系统到底是如何引导的？下面举例说明（传统的 BIOS 方式）。
【例】假设 CPU 有 20 根地址线，地址范围为 00000H~FFFFFH，按字节编址，采用统一编址，CPU 通过不同的地址访问不同的存储设备，包括 RAM、ROM 以及外设。

[图片：内存地址空间分配示意图，上方为ROM（包含64KB BIOS代码，主要用于外设），下方为RAM（其他）]

当按下电源键启动计算机时，RAM 里空空如也，但 ROM 里却可以保存程序（非易失性）！具体来说，ROM 的 64KB 空间里保存了 BIOS（Basic Input and Output System）的程序，它是操作系统引导的核心。说明如下。

在开机的一瞬间，PC 被强制初始化为 FFFF0H，此地址是 BIOS 的入口地址，这里有一条无条件转移指令。

[图片：BIOS内存映射及跳转示意图，显示PC指向FFFF0H处的jmp指令，跳转到FE05BH处的主体代码]

接着执行无条件转移指令，跳转到 BIOS 的主体代码部分开始执行。具体来说，BIOS 需要完成加电自检，包括检查 CPU、内存、主板和外设等的状态。当检测通过后，并初始化好硬件后（例如激活显卡），开始在内存 0x00000~0x003FF 处建立中断向量表，在 0x0E05B~0x0FFFE 加载中断服务程序并填写中断例程（可以简单理解为创建一张表格、填写中断服务程序的入口地址）。

[图片：内存布局图，显示中断向量表（00000H-003FFH）及中断服务程序（0E05BH-0FFFFEH）的位置关系]

BIOS 填写的中断例程是一些对硬件的 I/O 操作。由于 BIOS 只有 64KB 大小的空间，不可能把所有硬件的 I/O 操作都实现，所以仅挑一些能保证计算机能运行的那些硬件的基本 I/O 操作，例如屏幕显示、键盘输入。这就是 BIOS 称为基本输入输出系统的原因。

按照生活经验，如果在开机的时候马上按下某个特殊的键（例如 F10），屏幕上会显示一个 BIOS 画面，可以在里面设置启动设备（操作系统程序所在的设备）。

[图片：CMOS Setup Utility界面截图，显示Hard Disk Boot Priority设置选项]

硬盘、软盘、U 盘等设备都可以作为启动盘，BIOS 会按照配置顺序检测各个设备，直至找到第一个可启动的设备，接着把控制权交给该存储设备。

具体来说，是将该存储设备的主引导记录 MBR（Master Boot Record）加载到内存，MBR 是段程序，记录了存储设备的存储结构。若当前读取的是磁盘的 MBR，其中记录了各个磁盘分区的大小以及地址，其中保存了操作系统的分区被称为活动分区。（MBR 一般保存在磁盘的首个扇区）

注：“CMOS”是主板上的一块小型芯片，用于存储用户通过 BIOS 界面修改的配置数据（如启动顺序、日期时间等），其中就包括了系统启动时的设备检测顺序。

每个磁盘分区还有自己的分区引导记录（Partition Boot Record, PBR），PBR 也是段程序，负责加载该分区的操作系统引导程序（如 Windows 的 BOOTMGR 或 Linux 的 GRUB）。若当前的活动分区是 C 盘，则 MBR 会把 C 盘的 PBR 程序加载到内存。

最后，启动管理器负责加载操作系统内核。至此，就把操作系统这个大家伙给拉进内存啦！

二、操作系统的体系结构（理解即可）
操作系统是一个十分复杂、庞大的系统软件，由许多系统程序组成，例如，Windows2000 的代码超过 2900 万行。这些系统程序组织起来的关系则称为操作系统的结构。操作系统的结构随着技术发展也在不断地演变。

（一）无结构（早期）（了解即可）
在早期开发操作系统时，设计者只关注具体功能的实现，缺乏整体设计的视角。此时的 OS 是许多过程（函数）的集合，每个过程可以任意调用其他过程，致使操作系统内部既复杂又混乱，因此，这种 OS 是无结构的。
当 OS 不太大时，系统运行问题还不是太大，但随着系统的不断扩大，操作系统就会变得既庞大又杂乱。一方面会使程序错误很多，难以调试。另一方面也使程序难以阅读和理解，增加了维护人员的负担。

（二）模块化结构
20 世纪 60 年代出现了模块化程序设计技术，使用该技术设计的操作系统不再是众多过程的集合，而是按照功能划分为彼此独立的模块，并规定好各模块之间交互的接口，各模块还可以进一步细分为更小的功能子模块。这种设计方法称为模块—接口法。

在模块-接口法中，需要注意两个关键问题：模块的划分和模块的独立性。
1.模块的划分（了解即可）
① 如果模块划分得太大，则每个模块内部会变得复杂，内部联系增加，不利于编程。

② 如果模块划分得太小，虽然模块本身相对简单，但是模块间的联系会增多，系统会变得复杂。
2.模块的独立性（了解即可）
模块的独立性越高，各模块间的交互就越少，系统结构也就相对更简单。
衡量模块独立性的标准主要是内聚性和耦合度。内聚性是指模块内部各部分间联系的紧密程度。耦合度是指模块间相互联系和相互影响的程度。内聚性越高，耦合度越低，模块独立性越强。下面举例说明。
【例】假设你正在厨房做饭，厨房可以看作一个“系统”，而厨房中的不同功能区域（比如洗菜区、切菜区、炒菜区）可以看作“模块”。
[图片：厨房功能区示意图，包含洗菜、切菜、炒菜三个区域]
在厨房中，内聚性可以类比为某个功能区域的专注程度。
切菜区负责食材的切割和腌制等工作，因此，所有操作都围绕切菜区的核心功能展开，功能高度集中，这就是高内聚性的体现。例如，切菜需要使用刀具、砧板等工具，但如果切菜区还负责烧水或洗碗，那就破坏了内聚性，因为这些操作与切菜无关。
在厨房中，耦合度可以类比为不同功能区域之间的协作方式。
切菜区和炒菜区之间的交互是通过“传递盘子”的方式完成的。例如，切好的菜被整齐地放在一个盘子里，传递到烹饪区。切菜区不需要干预炒菜区的操作，炒菜区也不需要进入切菜区去拿菜。这种通过“接口”（盘子）的交互方式降低了耦合度。
如果把切好的菜直接扔到炒锅里，或者直接进入切菜区拿菜，会导致切菜区和炒菜区直接相互影响，导致系统变得混乱。
通过这种设计，厨房的各个功能区域可以独立运作，即使某个区域出现问题（比如切菜区的刀具坏了），也不会影响其他区域（比如烹饪区的锅具）。
优点：相较于无结构 OS，利用模块-接口法开发的 OS 具有以下优点。
① 提高 OS 设计的正确性、可理解性、可维护性、可适应性。
② 加速 OS 的开发过程。
缺点：在 OS 设计时，对各模块间的接口规定很难满足实际需求，各模块的设计很难找到一个可靠的顺序。
（三）分层结构
分层式结构指把操作系统的所有功能模块分别排列成若干层，各层之间的模块只能是单向依赖或单向调用关系，即低层为高层服务，高层可以调用低层的功能，反之则不能。
【例】第一个分层式系统是 1968 年开发的 THE 系统。该系统共有 6 层，如下所示。
THE 操作系统的结构
| 层号 | 功能 |
| :--- | :--- |
| 5 | 操作员 |
| 4 | 用户程序 |
| 3 | 输入/输出管理 |
| 2 | 操作系统-控制台通信 |
| 1 | 内存和磁鼓管理 |
| 0 | 处理器分配和多道程序设计 |
第 0 层负责处理器的分配。
第 1 层执行内存和磁鼓管理。
第 2 层处理操作系统与控制台之间的通信。
第 3 层对连接到计算机上的所有 I/O 设备进行管理。
第 4 层是用户程序层，第 5 层是用户，即操作员。

这样系统结构不但清晰，而且不会构成循环调用。这种分层的概念后来体现在 MULTICS 系统中，但它不是层，而是一系列同心环，且内层环比外层环有更多的权利。

[图片：同心环结构图，从外到内依次为第N层：用户接口层...第1层，第0层：硬件层]

当外环中的过程想调用内环过程时，它必须通过一个特殊的接口（比如系统调用）才能访问。
自底向上的分层设计的基本原则是：每一步设计都建立在可靠的基础上。为此规定每一层仅能使用其底层所提供的功能和服务，这样可使系统的调试和验证都变得更容易。
例如，在调试第 0 层时，由于不依赖于其他层，因此可以经过详细测试来确保其正确性。在第 1 层设计时，其仅依赖于第 0 层，第 0 层已经验证正确，因此比较容易验证第 1 层。同理，如此一层一层地自底向上增添软件层，每一层都实现若干功能，最后构成一个能满足需要的 OS。
优点：分层结构容易保证系统的正确性，方便扩充和维护。因为每层都建立在较为可靠的基础上，且每层相对独立，只要不改变各层次间的接口，各层次的修改不会相互影响。
缺点：分层结构的主要缺点是系统效率降低。由于层次结构是分层单向依赖的，必须在每层之间都建立层次间的通信机制，OS 每执行一个功能，通常要自上而下地穿越多个层次，这无疑会增加系统的通信开销，从而导致系统效率的降低。

（四）宏内核
宏内核是一种传统的内核结构，它将进程管理、内存管理、文件系统等各项功能都放到内核中。
优点：效率高。
缺点：随着内核的增大，内核代码难以维护，系统稳定性差（考研范围内）。

（五）微内核
1 客户/服务器模式（了解即可）
客户/服务器模式（Client/Server Model）是一种网络架构，它将网络功能分为两个部分：客户端（Client）和服务器（Server）。在这种模式下，客户端是请求服务的一方，而服务器是提供服务的一方。
客户端负责发起请求，它通常是用户直接交互的界面，负责处理用户输入、显示输出结果，以及与服务器进行通信，可以是网页浏览器、移动应用、桌面软件等。

[图片：客户端（猫）通过浏览器向服务器发送请求并接收响应的示意图]

服务器负责处理客户端的请求，执行任务，如数据存储、处理、转发等，并将结果返回给客户端。服务器通常拥有较高的计算能力、存储容量，并且能够同时处理来自多个客户端的请求。

客户/服务器模式工作流程如下：
① 请求：客户端发送请求到服务器。
② 处理：服务器接收请求，并进行相应的处理。
③ 响应：服务器将处理结果发送回客户端。
④ 显示：客户端接收到结果后，将其展示给用户。

2.定义
微内核构架，是指将最基本的功能保留在内核，包含中断机制、时钟管理等，那些不需要在核心态执行的功能将被移到用户态执行，以服务程序（服务器）的形式存在，如 I/O 设备服务器。
因此，微内核结构将操作系统划分为两大部分：微内核和多个服务器。
服务器的执行相互独立，交互则都借助于微内核进行通信。（暂时先将“进程”简单看作“程序”，我们第二章中会深入学习）

[图片：表格展示了微内核架构，上方包括客户进程、进程服务器、I/O设备服务器、文件服务器、虚拟存储器服务器，下方是微内核]

客户与服务器之间的交互也是借助微内核来实现的。

[图片：流程图展示了客户机应用通过微内核向文件服务器发送请求并接收返回]

微内核结构中，一个模块中的错误只会使这个模块崩溃，而不会使整个系统崩溃。
例如，文件服务代码运行时出了问题。宏内核中，该服务是运行在内核态的，会导致系统崩溃；而微内核的该服务是运行在用户态的，只要将该服务强行停止，然后重启，就可以继续使用，系统不会崩溃。

3.机制与策略分离（了解即可）
微内核结构通常利用“机制与策略分离”的原理来构造 OS。机制指实现某一功能的具体执行机构。策略指在机制的基础上借助某些参数和算法来实现该功能的优化，或达到不同的功能目标。
【例】假设厨房是一个微内核操作系统。厨房的核心设备（炉灶、冰箱、水槽）是机制部分，这些设备是固定的，它们的功能是做饭、冷藏食物、清洗餐具。厨师做菜的方式是策略部分。比如今天做红烧肉还是清蒸鱼，先切菜还是先烧水。这些策略可以根据你的需求灵活调整。
微内核操作系统将进程管理、存储器管理以及 I/O 管理这些功能一分为二，属于机制的很小一部分放入微内核，而绝大部分属于策略的工作放入微内核外的各种服务器实现，大多数服务器都要比微内核大。
因此，在采用客户/服务器模式时，能将微内核做得很小。

缺点：由于需要频繁地在核心态和用户态之间进行切换，操作系统的执行开销偏大。具体来说，在早期的 OS 中，用户进程在请求取得 OS 服务时，一般只需进行两次状态切换：一次是在执行系统调用后由用户态转向系统态时；另一次是在系统完成用户请求的服务后，由系统态返回用户态时。

[图片：客户机应用与OS内核交互示意图，包含“请求”和“返回”箭头]

在微内核 OS 中，由于客户和服务器、服务器和服务器之间的通信都需通过微内核，致使同样的服务请求至少需要进行四次上下文切换。
第一次是发生在客户发送请求消息给内核，以请求取得某服务器特定的服务时；
第二次是发生在由内核把客户的请求消息发往服务器时；
第三次是当服务器完成客户请求后，把响应消息发送到内核时；
第四次是在内核将响应消息发送给客户时。

[图片：客户机应用、文件服务器与微内核交互示意图]

为了改善运行效率，可以将那些频繁使用的系统服务移回内核，从而保证系统性能，但这又会使微内核的容量明显地增大。
OS 的微内核结构是 20 世纪 80 年代后期发展起来的。由于它能有效地支持多处理机运行，故非常适用于分布式系统环境。当前比较流行的、能支持多处理机运行的 OS，几乎全部都采用了微内核结构，例如 Windows 2000/XP 系统采用了微内核结构。虽然 Linux 内核是公认的宏内核架构，但其近期也开始融合了微内核架构中的用户态驱动模型。

第一类虚拟机管理程序直接运行在裸机上，拥有最高权限，为在其上运行的虚拟机精确模拟出整套计算机硬件。每个虚拟机可以运行不同的操作系统。如下所示，虚拟机管理程序上运行了多台虚拟机，分别安装了 LinuxOS 和 WindowsOS。

（二）第二类虚拟机管理程序
第二类与第一类虚拟机管理程序最大的区别在于，它需要利用宿主操作系统提供的创建进程、分配资源等服务；而第一类虚拟机管理程序没有底层操作系统的支持，只能自行实现这些功能。
