### 第 40 行：为什么写 `dict(INITIAL_STATE)` 而不是 `= INITIAL_STATE`

你对两个变量的理解完全正确。`dict()` 的问题才是关键。

```python
# 如果这样写：
GAME_STATE = INITIAL_STATE
# 它们指向的是同一个对象：
GAME_STATE["hp"] = 50
print(INITIAL_STATE["hp"])  # → 50  ← 初始模板被污染了！
```

在 Python 里 `=` 不复制，只绑定。`dict(INITIAL_STATE)` 创建一个浅拷贝——新字典，内容相同但独立。之后你砍 `GAME_STATE["hp"]` 到 0，`INITIAL_STATE["hp"]` 还是 180。

**一句话：保留一个干净的"出厂设置"，随时可以重来。**

### 第 43-48 行：`_sync_derived()`——为什么不直接存

```python
def _sync_derived() -> None:
    a = GAME_STATE["attributes"]
    GAME_STATE["hp_max"] = 100 + a["VIT"] * 10    # ← 由 VIT 计算
    GAME_STATE["mp_max"] = a["INT"] * 5            # ← 由 INT 计算
    GAME_STATE["sp_max"] = a["SPI"] * 8            # ← 由 SPI 计算
    GAME_STATE["exp_to_next"] = 200 + GAME_STATE["level"] * 20
```

hp_max / mp_max / sp_max / exp_to_next 是从属性和等级派生出来的。它们不是独立数据。如果直接存，每次 VIT 加 1 你都得记得手动改 hp_max。忘了改 → 显示 180 max 但计算公式说应该是 190 → 数据不一致。

`_sync_derived()` 是一个门卫：在任何展示或计算之前跑一次，保证 max 值永远和属性一致。这叫派生值不复存原则——真理只有一个来源（属性+公式），max 只是展示层。

### 第 27 行：INT=10 最高，RES=4 最低

你的理解正是设计意图。穿越者的设定就是：只有你从现实穿越到埃特尼亚。力量来源不是和 NPC 打交道，而是在现实中学习。所以：

- INT=10（最高）——学习/理解力是唯一的力量来源。你的编程、数学、读书直接转化
- RES=4（最低）——刚穿越，对这个世界的魔法/毒素/精神攻击毫无抗性
- VIT=8——普通人体质，不高不低
- CRT=5——还没经历过真正的战斗，暴击能力几乎为零

这是叙事和游戏机制的统一：初始属性分配讲了一个故事——"一个靠脑子生存的外来者，脆弱但对世界充满理解力。"

**继续走读 NPC 部分（57-71 行）**：

**第一，为什么 NPC_ROSTER 是模块级常量，不直接写在 GAME_STATE 里面？**

NPC 的名字、职业、等级、背景故事是静态数据——伊修卡永远是剑圣 Lv.26，被龙诅咒。但 `social_links`（好感度）是动态数据——你今天跟伊修卡聊了天，好感度从 0 涨到 5。把静态和动态分开：`NPC_ROSTER` 是只读模板，`social_links` 是可写状态。这就是关注点分离——和 `INITIAL_STATE` vs `GAME_STATE` 是同一个设计思路。

**第二，`get_full_status()` 里又出现了 `dict(GAME_STATE)`。为什么？**

和 INITIAL_STATE 那边的逻辑一样——防外部污染。如果直接 `return GAME_STATE`，调用方拿到引用后可以随便改，等于绕过了所有规则。返回拷贝意味着：你只能通过工具来改状态，不能直接捅进去。

**排名公式（92-113 行）**：

```python
score = level * avg_attr * 0.5          # 第 100 行
pct = min(score / 50 * 100, 95)         # 第 101 行
rank = max(int(pool * (1 - pct / 100)), 1)  # 第 102 行
```

拿 Lv.1, 属性均值 ~6.67 代进去：
```
score = 1 × 6.67 × 0.5 = 3.33
pct   = min(3.33/50×100, 95) = min(6.67, 95) = 6.67%
rank  = max(int(1000 × (1 - 0.0667)), 1) = max(933, 1) = 933
```

所以排位 933/1000，百分比 6.7%——和之前验证跑出来的完全一致。

**`min(pct, 95)`——为什么排名百分比最高只到 95%，不到 100%？**

答案在游戏设计心理学里：永远留一个够不着的天花板。即使你满级全属性 max，排位也只会显示 95%。这意味着公会里总有 50 个"传说中的冒险者"排在你前面——他们的名字你不会在 NPC roster 里看到，但你知道他们存在。这是 RPG 的惯用技法：排行榜顶部的不可见存在感。

**`COMPUTE_RANK_SALARY = compute_rank_and_salary`（第 116 行）**：给外部 import 一个稳定的名字。如果以后 `compute_rank_and_salary` 重构拆成了两个函数，`tools.py` 里 `from .game_engine import COMPUTE_RANK_SALARY` 不用改。一个小保险。

**SYSTEM_PROMPT（123-146 行）**：Ch.04 的核心概念——稳定前缀（stable prefix）。它在会话一开始就写死进 `messages[0]`，之后整个对话中不再变。Qwen 的自动前缀缓存会把它冻结在 KV cache 里，每轮对话不需要重复计算。

问：如果我没把 SYSTEM_PROMPT 放在 messages[0]，而是放在 messages[5]，前缀缓存还有效吗？
答：没有。前缀缓存只看从头开始的连续不变部分。messages[0] 改了，缓存全废。

**AI**：

`tools.py` 全文 250 行，四个层次：

### 第一层：Tool 数据类（20-31 行）

```python
@dataclass
class Tool:
    name: str
    description: str
    input_schema: dict[str, Any]
    handler: Callable
    read_only: bool = False        # ← 全部默认 False
    destructive: bool = False
    concurrency_safe: bool = False
    idempotent: bool = False
    open_world: bool = True         # ← 唯一默认 True
```

注意这个默认值的不对称：五个 flag 里四个默认 False，只有 `open_world` 默认 True。这是 Ch.03 的 fail-closed（故障时关门）原则。

因为工具的"安全假设"是：不知道的时候，就当它不安全。写一个新工具时如果你忘了标 `read_only=True`，默认 False 意味着 loop 会把它当可写工具处理——最多是过度保守，不会造成破坏。反过来如果默认 True，忘了标 → 一个破坏性工具被当成只读 → 事故。

`open_world=True` 同理：默认假设每次调用结果可能不同（安全假设），去重时保留全部调用。只有你明确知道某个工具是幂等的、结果不变的，才手动关掉。

### 第二层：build_tool 工厂（43-55 行）

```python
def build_tool(name, description, input_schema, handler, **overrides):
    defaults = {..., read_only: False, destructive: False, ...}
    defaults.update(overrides)
    return Tool(**defaults)
```

这就是 Claude Code 源码里的模式。它做的事很简单：给你一个"默认安全"的底子，你只覆盖真正需要的那几个 flag。看第 192 行怎么用的：

```python
build_tool(..., read_only=True, concurrency_safe=True, idempotent=True, open_world=True)
```

四个参数覆盖，其余 6 个继承默认值。

### 第三层：5 个工具（69-172 行）

- **`get_status`**：代理层，调用 `game_engine.get_full_status()`。多包一层的目的：以后可以在这里加权限检查、字段过滤、日志。
- **`get_npc`**：分两路——传了名字就精确查，没传就全列。合并了动态的 `social_link_level` 到静态 NPC 数据里——关注点分离在消费侧的体现。
- **`get_ranking`**：纯计算，委托给 `COMPUTE_RANK_SALARY`。
- **`write_memory`**：Ch.07 的落地。安全过滤器黑名单（`ignore previous`、`you are now`、`system prompt`、`<system>`、`<admin>`、`execute the following`）防止提示注入。原子写入（`mkstemp` + `os.replace`）保证文件要么完整要么不变，永不出半截状态。
- **`final_answer`**：最简单的工具，但 loop 里对它做了特殊处理（直接 return）。这是 Ch.02 四种停止条件里的"工具驱动停止"。

### 第四层：TOOLS 注册表（179-249 行）

五个 `build_tool(...)` 调用。元数据标记差异：

| 工具 | read_only | idempotent | concurrency_safe |
|------|-----------|------------|------------------|
| get_status | ✅ | ✅ | ✅ |
| get_npc | ✅ | ✅ | ✅ |
| get_ranking | ✅ | ✅ | ✅ |

## 工具调用 ID + 缺口补全

**学生**：每一个工具调用都有自己的id是吗 这样是为了方便溯源？模型知道是哪一个工具调用除了问题？举个例子讲讲id的作用；把缺口补上 为什么是存在GAME_STATE中 GAME_STATE包含的东西是不是太多了要单独存文件或者怎么存储？

**AI**：

### 1. ID 的作用

不是"方便溯源"——比这更基础。没有 id，模型分不清哪个结果对应哪个调用。

假设模型在同一轮发了三个工具调用：

```
turn 3, assistant:
  tool_call id=call_001  →  get_status()
  tool_call id=call_002  →  get_npc(name="伊修卡")
  tool_call id=call_003  →  get_ranking()
```

你的代码执行完，返回三个结果。如果不带 id 回去，模型只能靠顺序猜。顺序错了 → 它以为排名结果是状态、状态结果是 NPC → 后面的推理全乱。

带 id 回去：
```
tool_call_id=call_001: {hp: 180, mp: 50, ...}
tool_call_id=call_002: {found: true, npc: 伊修卡...}
tool_call_id=call_003: {rank: 933, salary: 6200}
```

模型内部知道 "call_001 是我发起的 get_status，这个结果就是它的返回值"。一对一绑定，不存在歧义。

类比：你同时给三个人发了微信，三个人都回了，但回话里没引用你的原消息——你分得清哪条回哪条吗？id 就是"回复: 关于...那件事"。

### 2. 缺口补全实现

你说得完全对：GAME_STATE 不该存这个。GAME_STATE 是游戏机制层——属性、等级、好感度。结果缓存是基础设施层——和游戏世界观毫无关系。混在一起就是"上帝类"反模式。

正确的分层：
```
GAME_STATE          → 游戏数据（属性、等级、NPC...）
RESULT_STASH        → 基础设施数据（被截断的原始结果...）
```

**数据流**：
```
工具返回 5000 字结果
    ↓
_clip() 发现超过 2000 字
    ↓
stash_result() 把完整 5000 字存进 RESULT_STASH（内存字典）
    ↓
模型看到：前 1000 字 + "省略标记(stash_003)" + 后 1000 字
    ↓
模型决定需要完整数据 → 调 fetch_full_result("stash_003")
    ↓
取回完整 5000 字 → stash 自动清除（一次性）
```

**为什么用内存字典而不用文件？** 因为截断结果只需要活到当前会话结束。模型取回一次就够了（one-shot），不需要持久化到磁盘。RESULT_STASH 是临时缓存，不是数据库。

**为什么 RESULT_STASH 放 tools.py？** 它是 loop 和 tools 的共享数据——loop 往里写，工具往里读。放 tools.py 最合理：tools.py 已经被 loop 导入，不存在循环依赖。

### 3. 三文件修改

- `tools.py`：新增 `RESULT_STASH` 字典 + `stash_result()` 函数 + `fetch_full_result` 工具（one-shot 取后即删），工具从 5 个增至 6 个
- `loop.py`：`_clip()` 返回值从 `(text, was_clipped)` 扩展为 `(text, was_clipped, stash_id_or_none)`，超限时自动 stash
- `game_engine.py`：SYSTEM_PROMPT 新增 Rule 6，引导模型使用 fetch_full_result
