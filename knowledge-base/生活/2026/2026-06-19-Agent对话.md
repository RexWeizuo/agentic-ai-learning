# 2026-06-19 周五 · Agent 逐问答记录

## tool_call_id 机制深挖 + RESULT_STASH 持久化

**学生**：tool_call_id应该更像是微信人id吧 区分谁给我回的消息 如果模型执行复杂任务需要根据之前工具调用的结果往下执行怎么办 同一个工具调用还是继续同一个id嘛（同一个id可以看到之前工具调用的结果） 举例讲讲；RESULT_STASH不用保存吗 不是有log保存所有的完整信息吗 agent下次要检索这一块的内容但是RESULT_STASH里面没有了怎么办

**AI**：

### 1. tool_call_id：你的类比更准确

上次说"三人同时回微信"——不够精确。更准确的说法：**tool_call_id 是"快递单号"**。你（模型）同时下了三单外卖，每单一个单号。外卖到了（tool result），你靠单号知道哪个袋子是你的汉堡、哪个是你的奶茶。

但核心问题是：模型执行复杂任务，需要根据之前工具调用的结果往下执行——同一个工具调用还是继续同一个 id 吗？

**不是。每次都换新 id。**

```python
# ❌ 你的直觉（不对）：
# 模型想"看上一次 get_status 的结果"→ 用同一个 id 查
```

实际情况是：**模型不看 id 查历史，它看 messages 数组。**

举个例子——模型要完成一个三段式任务：

```
Turn 1 ─────────────────────────────────────
模型: "我先看看状态"
  → tool_call id=call_A: get_status()
  ← tool_result id=call_A: {hp: 180, level: 1, INT: 10}

Turn 2 ─────────────────────────────────────
模型: "INT=10 的话，查查有没有适合的智力型 NPC"
  → tool_call id=call_B: get_npc(name="凡")   ← 新 id！
  ← tool_result id=call_B: {name: "凡", class: "剑圣", level: 52}

Turn 3 ─────────────────────────────────────
模型: "Lv.52 的凡，Lv.1 的我…差距太大了。写个叙事吧。"
  → tool_call id=call_C: final_answer(text="凡瞥了你一眼...")  ← 又一个新 id！
```

关键点：**Turn 3 的模型怎么知道 "hp=180" 和 "凡 Lv.52" 这两个信息？**

不是靠 id 去查。是靠 **messages 数组**——所有历史消息都堆在里面：

```python
messages = [
    {"role": "system", "content": SYSTEM_PROMPT},
    {"role": "user", "content": "我刚刚在旅店醒来..."},
    {"role": "assistant", "tool_calls": [{"id": "call_A", ...}]},    # Turn 1
    {"role": "tool", "tool_call_id": "call_A", "content": "{hp:180...}"},  # ← 结果在这儿
    {"role": "assistant", "tool_calls": [{"id": "call_B", ...}]},    # Turn 2
    {"role": "tool", "tool_call_id": "call_B", "content": "{凡...}"},     # ← 结果在这儿
    # Turn 3 的 assistant 消息会追加到这里...
]
```

模型每轮都读**整个 messages 数组**。它不需要用 id 去翻旧账——旧账全在上下文里。

**所以 tool_call_id 的职责只有一个**：在当前 turn 内把结果和请求配对。跨 turn 不靠它。

类比修正：

| 我之前说的 | 更准确的说法 |
|-----------|------------|
| 三人回微信，靠备注名区分 | 同一个 turn 内，靠单号区分三个并发调用的返回值 |
| 跨 turn 追溯 | 跨 turn 靠 messages 数组，不是靠 id |

### 2. RESULT_STASH 丢失问题

你的问题本质是：**log 文件保存了完整信息，但 RESULT_STASH 在会话结束后没了——下次会话的 agent 怎么办？**

先分清三个东西的职责：

```
RESULT_STASH    → 当前会话内，模型临时取回被截断的原始结果
                 （生命周期：一次 fetch_full_result 调用，或会话结束）

logs/           → 所有对话的完整记录，给人看和调试用
                 （生命周期：永久，存在磁盘上）

write_memory    → 模型认定的"长期有价值知识"
                 （生命周期：跨会话，curated）
```

**你的担心是对的——但场景不同。**

RESULT_STASH 解决的场景是：
> 同一个 turn 内，结果太长被截断了，模型需要完整数据才能回答。

不是：
> 三天后的会话，agent 想查"上次那个被截断的结果是什么"。

后者应该走两条路：
- **如果是重要的知识** → `write_memory` 存下来
- **如果是原始数据** → 重新调用原工具（`get_status`、`get_npc` 等），模型会拿到最新结果

打个比方：
- RESULT_STASH = 你看网页时按 Ctrl+F 搜索——用完就关
- logs = 浏览器历史记录——想查上个月的可以翻
- write_memory = 你收藏的书签——"这个重要，以后还要看"

**但如果你的意思是"下次会话启动时，agent 应该能看到上一次被截断的完整结果"——那确实不该丢。**

要不要加一个：在 `_clip()` 截断时，除了 stash 到内存，也写一份到 log 目录的 `stash/` 子目录？这样 log 里就保留了完整原始结果，下次会话的 agent 如果有文件读取能力可以翻到。

要不要补这个？

## Ch.01 Provider-specific Knobs 六项对照

**学生**：暂时不用吧那就；这一段有没有用到

（选中 course/01-function-calling.md 第 108-119 行：Provider-specific knobs worth knowing）

**AI**：

你选的这段是 Ch.01 最后的 "Provider-specific knobs"——六个生产环境才需要关心的开关。逐个对照我们代码：

### 1. `tool_choice` — 没用到

```python
# loop.py 第 160 行，我们的调用：
response = client.chat.completions.create(
    model=model, messages=messages, tools=tool_schemas, temperature=0.7,
)
# 没有 tool_choice 参数
```

没设 `tool_choice`，就是默认行为——模型自己决定调不调工具。讲义说的三种模式：
- `"none"` → 纯文本，不调用工具
- `"auto"` → 模型自己判断（我们现在）
- `"required"` / `{"type": "function", "function": {"name": "X"}}` → 强制调某个工具

什么时候该加？比如：你已经知道用户问题需要查状态，可以设 `tool_choice={"type": "function", "function": {"name": "get_status"}}`，省一轮推理。但我们目前的消息流简单，不需要。

### 2. Parallel tool calls — 隐式兼容，没显式控制

```python
# loop.py 第 188 行：
for tc in msg.tool_calls:   # ← 遍历所有 tool_call
```

我们没设 `parallel_tool_calls: false`，但 Qwen 支持并行调用。如果模型同一轮发了 3 个 tool_call，我们的 `for` 循环会依次处理——能接住。

### 3. Strict schema mode — 没用到，靠手动防御

```python
# loop.py 第 207-215 行：
try: args = json.loads(args_str)
except json.JSONDecodeError:
    cleaned = re.sub(r"[\x00-\x1f\x7f-\x9f]", "", args_str)
    try: args = json.loads(cleaned)
    except json.JSONDecodeError as e:
        result = _make_error_envelope(f"Parse error: {e}")
```

我们没有 `strict: true`——如果开了，这段 try/except 就基本不需要了。但 Qwen 的 strict 模式支持不如 OpenAI，所以我们的防御层反而更可靠。

### 4. Structured outputs — 没用到

我们用 tool calls（动作），不用 `response_format`（结构化回答）。final_answer 的文本是自由格式的奇幻叙事，不适合 JSON schema。

### 5. Hosted tools — 没用到

全部 6 个工具都是我们自己写的 handler。没有用云厂商的内置搜索/代码执行。

### 6. Refusals — 没处理

```python
# loop.py 第 164 行：
msg = response.choices[0].message

# 第 171 行：
if not msg.tool_calls:
    final_text = msg.content or "(silence)"
```

如果模型因为安全原因拒绝回答（refusal），我们的代码会把它当普通文本返回——不会特殊标记"这是拒绝"，也不会给用户提示。Ch.18 才深入讲这个威胁模型，目前先知道存在。

### 总结

| 六项 | 状态 |
|------|------|
| tool_choice | ❌ 未用 |
| parallel_tool_calls | ⚠️ 隐式兼容，未显式开关 |
| strict schema | ❌ 未用，靠手动 JSON 防御 |
| structured outputs | ❌ 未用 |
| hosted tools | ❌ 未用 |
| refusals | ❌ 未处理 |

全部六项都是"生产环境才需要"的开关。我们目前是学习阶段的最小可行循环，一个都没用是正常的——等从 Ch.08 往后做到多 agent 协作和部署时，至少有 4 个会自然出现。
