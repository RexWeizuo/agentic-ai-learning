# 5.2 文件基础（二）· Socratic 问答

## Q1 · 打开文件的六步流程

> 用户调用 `fd = open("/home/documents/example.txt", READ)` 后，OS 内部做了什么？按顺序说出六个步骤。

**学生回答**：① 通过路径找到文件 ②-⑤ 检查访问权限，添加到进程打开表中，添加指针下次不用路径，共享计数加一 ⑥ 返回 fd（进程打开文件表的数组下标）

**AI 批改** ⚠️ 框架对，漏了关键：查系统打开文件表 + 首次/非首次判断。

**完整六步**：

| 步骤 | 动作 | 关键 |
|------|------|------|
| ① | 路径解析，逐层查目录，找到目录项 | 从根目录开始 |
| ② | 检查访问权限 | 不合法→返回错误 |
| ③ | 查**系统打开文件表**，看文件是否已打开 | 区分首次/非首次 |
| ④ | 首次→读 FCB 到系统表；非首次→共享计数+1 | **FCB 只读一次** |
| ⑤ | 在**进程打开文件表**中新增一项，填访问方式+读写指针+指向系统表项 | 每个进程独立 |
| ⑥ | 返回 fd（进程表的数组下标）给用户 | fd = file descriptor |

> fd 是 file descriptor（文件描述符），不是 fb。

## Q1-真题 · 首次打开文件

> 【讲义练习】在一个文件被用户进程首次打开的过程中，操作系统需做的是（ ）。
> A. 将文件内容读到内存中
> B. 将文件控制块读到内存中
> C. 修改文件控制块中的读写权限
> D. 将文件的数据缓冲区首指针返回给用户进程

**学生回答**：B

**AI 批改** ✅ 5/5

选 B。首次打开只读 **FCB（文件属性）**，不读文件内容。文件内容等到 `read()` 才读入——打开文件和读内容是两步。完整过程就是 Q1 的六步流程。

## Q2 · 三张表的关系

> 外存的 FCB、内存的系统打开文件表、进程打开文件表——三张表的关系是什么？为什么需要两张内存表而不是一张？共享计数值存在哪张表里？

**学生回答**：系统打开文件表记录系统所有文件，进程表记录专属进程的可以更详细；共享计数存在系统打开文件表；各进程读写指针独立。

**AI 批改** ⚠️ 三点都对，但"为什么"不够精确：

- **系统表**：FCB 只读入一次，所有进程共享同一个表项 → **省磁盘 I/O**
- **进程表**：每个进程有自己的读写位置和打开方式，互不干扰 → **隔离状态**

> 一句话：**系统表省 I/O，进程表隔离状态。** 一张表不行——要么 FCB 反复读（浪费），要么进程指针打架（混乱）。

## Q3 · 关闭文件 + 共享计数

> P1 和 P2 都打开了文件 F。P1 close → 系统表项会删除吗？P2 close → 这次会删除吗？还可能触发什么？

**学生回答**：① 不会，共享计数-1 ② 如果只有两个进程，都 close 就删除，共享计数清零。

**AI 批改** ✅ 方向全对，补一个细节：

共享计数归零后，如果文件内容或 FCB 被修改过，还要**写回外存**——把更新后的 FCB 和元数据写回磁盘的目录结构。

## Q3-真题 · 多进程共享文件（2020 真题）

> 【2020年真题】若多个进程共享同一个文件 F，则下列叙述中，正确的是（ ）。
> A. 各进程只能用"读"方式打开文件 F
> B. 在系统打开文件表中仅有一个表项包含 F 的属性
> C. 各进程的用户打开文件表中关于 F 的表项内容相同
> D. 进程关闭 F 时，系统删除 F 在系统打开文件表中的表项

**学生回答**：B

**AI 批改** ✅ 5/5

| 选项 | 对错 | 原因 |
|------|------|------|
| A | ❌ | 可用不同方式打开——P1 只读，P2 读写 |
| B | ✅ | FCB 在系统表中只有一份，共享计数记录有几个进程在用 |
| C | ❌ | 各进程的读写指针、打开方式在各自进程表里，独立 |
| D | ❌ | 共享计数 >1 时只减一，归零才删除（这是 Q3 考过的） |

## Q4 · read 的完整路径

> `read(fd, buf, 1024)` → OS 怎么从 fd 一步步定位到磁盘物理块？把链路说出来。

**学生回答**：fd 查进程打开表，再查系统打开文件表拿到 FCB，FCB 记录逻辑块对应物理块地址，1024 数据存到 buf。

**AI 批改** ⚠️ 方向对，漏了一步。

完整链路：
```
fd
→ 进程打开文件表[fd]          ← 拿到读写指针 + 指向系统表的指针
→ 读写指针位置 → 算出逻辑块号  ← 【漏了这步】
→ 系统打开文件表               ← 拿到 FCB
→ FCB → 逻辑块号→物理块号转换  ← 5.1 的 Q12 同样的操作
→ 读磁盘物理块 → buf
```

## Q5 · 精简访问控制列表

> 权限位 `rwxr-x---`（即 111 101 000）：所有者/同组/其他各能做什么？改为 `rw-rw-r--` 后数字表示是多少？

**学生回答**：所有者读写执行，同组读执行，其他什么都做不了。110 110 100 = 664。

**AI 批改** ✅ 5/5

- 所有者 rwx → 读+写+执行 ✅
- 同组 r-x → 读+执行 ✅
- 其他 --- → 无权限 ✅
- `rw-`=6, `rw-`=6, `r--`=4 → **664** ✅

## Q6 · 口令 vs 密码

> "口令"和"密码"在 OS 文件保护中是两个不同的概念。区别是什么？

**学生回答**：口令=访问文件要输入的字符串，相当于 QQ 密码；密码=加密算法，乱码+密钥解码出真实信息。

**AI 批改** ✅ 5/5

- **口令**：明文字符串，存在 FCB 里，验证就是比对——简单但不安全
- **密码**：加密算法，数据变成乱码，需要密钥才能还原——安全但有加解密开销

> 考研 OS 不考公钥私钥（非对称加密），讲义只涉及对称加密。

## Q7 · 文件逻辑结构分类

> 画出分类树。定长和变长对查找效率的影响？

**学生回答**：有结构-定长/变长，无结构-流式。定长方便能随机访问，变长不好查找。组织方式分为顺序、索引、索引顺序。

**AI 批改** ⚠️ 3/5 —— 漏了散列文件（Hash）

完整分类树：
```
文件逻辑结构
├── 有结构文件（记录式）
│   ├── 按记录长度：定长 / 变长
│   └── 按组织方式：
│       ├── 顺序文件（串结构 / 顺序结构）
│       ├── 索引文件
│       ├── 索引顺序文件（先索引定组，组内顺序找）
│       └── 散列文件（Hash，key→地址，一次直达）
└── 无结构文件（流式文件）—— 字节流
```

| | 定长 | 变长 |
|---|---|---|
| 查找第 i 条 | 直接算：首地址 + i×L | 必须从第一条顺序找 |
| 随机访问 | ✅ | ❌ |

## Q8 · 索引顺序文件 + √N 公式

> 10000 条记录，最优分组？平均查找次数？

**学生回答**：100 组每组 100 条。平均 100/2 + 100/2 = 100 次。索引表大了找索引也慢。

**AI 批改** ✅ 5/5

- 最优：√10000 = **100 组，每组 100 条**
- 索引查找：~50 次 + 组内查找：~50 次 = **~100 次**

| | 普通顺序文件 | 索引顺序文件 |
|---|---|---|
| 平均查找 | 5000 次 | **100 次** |

> 分组越细，索引表越大 → 查索引变慢，和组内查找的节省抵消。平衡点在 √N。

## Q9 · 记录成组分解（真题型计算）

> 逻辑记录 100B，盘块 512B，隐式链接。目录项已在内存。记录不能跨块。
> （1）一个盘块放几条记录？
> （2）修改第 22 号逻辑记录（从 0 开始），共启动几次磁盘？

**学生回答**（初答）：5 条记录；22 次磁盘（从第 0 块开始依次往后读）。
**学生纠错后**：懂了，记录号≠盘块号。

**AI 批改** ❌→✅ 初答错了，纠错后明白。

正确解法：
1. 512/100 = **5 条**记录/块 ✅
2. 第 22 号记录 → 22/5 = **4...2**，在第 **4** 个盘块
3. 隐式链接要读块 4，必须先读 0→1→2→3（找指针）→ 块 4：**5 次读**
4. 修改后写回：**1 次写**
5. 共 **6 次磁盘启动**（选项 D）

> 错误原因：22 是记录号不是盘块号。和 5.1 的 Q4、Q12 同一个考点——先算逻辑块号，再算访问路径。

## Q10 · 提前读与延迟写

> "提前读"和"延迟写"分别解决什么问题？利用了什么特性？C→D→E 第二次拷贝为什么更快？

**学生回答**：提前读对顺序文件有效，提前知道下一块先放到缓冲区；延迟写数据先放缓冲区不写回，后续访问直接读。第二次快因为延迟写。

**AI 批改** ⚠️ 4/5 —— 概念对，第二问错了。

- **提前读** ✅：利用顺序访问可预测性——读第 N 块时预读第 N+1 块
- **延迟写** ✅：利用时间局部性——改了先不写回，可能马上又被访问

**第二问** ❌ 不是延迟写，是**磁盘高速缓存（buffer cache）**：
- C→D 拷贝时，文件内容经过内存缓冲区
- 拷贝完后数据还在缓冲区里没清掉
- C→E 再拷贝，直接从内存缓冲区读，不走磁盘 I/O

> 延迟写是"改了先不写回"（写操作），拷贝加速是"读了走缓存"（读操作）。两个都用到内存缓冲区，但机制不同。
