### Q1 · 三种 I/O 方式 CPU 参与度

> 程序查询、程序中断、DMA 三种方式，CPU 在数据传送阶段的参与程度分别是怎样的？用一句话概括每种方式的 CPU 在"等数据"时在干什么。

**学生回答**：程序查询 CPU 参与程度很高——一种是 CPU 一直查询 I/O 信号自己什么都不做了等 I/O 传过来数据 CPU 再继续工作，另一种是 CPU 隔一段时间看有没有 I/O 结果没有就先执行其他程序有结果就中断当前操作去处理 I/O 的结果；程序中断方式 CPU 启动完外设就去忙自己的事了 I/O 准备好了就主动给 CPU 发中断请求然后 CPU 响应中断请求响应外设；DMA 可以理解成一个暂存通路 I/O 的数据先给 DMA，DMA 一个字一个字接，接完一个数据块告诉 CPU（发送中断）CPU 此时处理 DMA 的中断（就是程序中断多了一个代理）

**AI 反馈 ⚠️ 概念混了**

你把程序查询的两种子类型和三种大方式混在一起了。正确的三层递进关系：

| 方式 | 数据传送 CPU 参与 | CPU "等数据"时在干什么 |
|------|-------------------|----------------------|
| 程序查询（独占） | 100%，每条数据 CPU 亲自搬 | **忙等**（busy wait）——死循环查状态寄存器 |
| 程序查询（定时） | 同上，但间隔检查 | 切换到其他进程，定时中断唤醒检查 |
| 程序中断 | 中等，每条数据 CPU 在中断服务程序里搬 | 执行其他进程，I/O 就绪 → 中断通知 CPU |
| DMA | 极低，整块数据由 DMAC 搬 | 执行其他进程，完全不管数据传送，传完才发中断 |

**纠正**：你说"DMA 是程序中断多了一个代理"——这个比喻基本对。DMA 把数据传送工作从 CPU 手中接过来（代理），只在最初（初始化）和最后（结束中断）才找 CPU。但注意 DMA 请求的是**总线控制权**，不是 CPU 时间，这点和中断不同。

### Q2 · 独占查询 vs 定时查询

> 独占查询和定时查询的根本区别在哪？在讲义键盘输入的例子中，若用户每隔 10 秒敲一次键，哪种查询方式更合理？如果每秒 10000 次按键，哪种更合理？

**学生回答**：一个是 CPU 时时监控另一个是 CPU 时不时监控一下。10 秒肯定是定时好些不然 CPU 的工作时间都在等待键盘；10000 次那就独占查询。

**AI 反馈 ✅ 正确（原反馈有误，已纠正）**

"时时监控" vs "时不时监控"抓住了核心。

你的场景判断 **全部正确**：
- **10 秒/次** → 定时查询 ✅
- **10,000 次/秒** → 独占查询 ✅

---

**学生追 Q**：高频场景换进程的开销不还是很大吗？越高频不就越应该独占查询？中断方式本身也有换进程的开销，10000 次/秒每次开销都很大，那么独占查询时不时优于中断或 DMA？

**AI 纠正自己**：你说得对。关键在于**每种方式都有开销，需要比较的是"开销 / 数据间隔"的比值**，而不是抽象地说"高频该升级"。

| 开销来源 | 典型大小 |
|----------|----------|
| 独占查询（一次） | ~10-20 个时钟周期（读状态寄存器 + 条件跳转） |
| 中断（一次） | ~400 个时钟周期（保存/恢复现场 + ISR 执行） |
| DMA 启动（一次）| ~500 个时钟周期（分摊到整块数据） |
| 上下文切换（一次） | 几十~上百个时钟周期 |

**盈亏平衡点思考**：

```
数据间隔 > 中断/DMA 开销？
  ├─ 是 → 用中断/DMA（CPU 可以做别的事）
  └─ 否 → 中断/DMA 跟不上
           ├─ 数据是批量大块 → DMA（启动开销被整块分摊）
           └─ 数据是连续高频小块 → 独占查询（零切换开销，延迟最低）
```

讲义自己给了例子：40MB/s 数据率、32 位端口 → 每 0.1μs 一批数据，但中断处理要 0.8μs，根本跟不上 → **中断不适合**。但如果 DMA 的块太小（比如每 1KB 就中断一次），DMA 的启动开销也不划算。此时独占查询反而是唯一能跟上的方案——没有切换开销，死循环读寄存器就是最快路径。

**纠正后的结论**：选择 I/O 方式不是看"谁先进"，而是算**单次开销 vs 数据间隔**。高频不一定该升到中断/DMA，取决于每条数据到达的间隔是否大于中断/DMA 单次开销。你的直觉是对的。

### Q3 · DMA 阶段与 CPU 介入 + 访存优先级

> DMA 方式从预处理到后处理，CPU 在哪些阶段必须介入？哪些阶段完全不参与？如果 DMA 和 CPU 同时请求访存，谁优先？为什么？

**学生回答**：预处理 CPU 要初始化 DMA 中的寄存器，I/O 设备向 DMA 发出请求 CPU 不用参与，DMA 向 CPU 发出总线请求 CPU 响应时参与，数据传送过程中 CPU 不参与，传送完了向 CPU 发出中断请求，后处理阶段 CPU 校验数据等。不知道谁优先。为什么 DMA 会请求访存，DMA 不是只会请求中断吗？

**AI 反馈 ⚠️ 阶段描述全对，但缺了两个关键概念**

你的阶段梳理**完全正确**：
| 阶段 | CPU 介入 |
|------|----------|
| 预处理（初始化 DMAC 寄存器） | ✅ 必须 |
| I/O → DMAC（DMA 请求） | ❌ |
| DMAC → CPU（总线请求 HRQ） | ✅ 响应 HLDA |
| 数据传送（逐字传输） | ❌ 完全不参与 |
| DMAC → CPU（结束中断） | ✅ 后处理/校验 |

**两个关键问题**：

**① DMA 为什么请求"访存"？DMA 不是只发中断吗？**

DMA 工作中有**两种请求**，你混淆了：
- **总线请求（HRQ）**：数据传送阶段，DMAC 需要占用**系统总线**来在 I/O 设备和内存之间搬运数据。总线 = CPU 和内存之间的高速公路，DMAC 要借用这条路就得请求 CPU 让出总线。
- **中断请求**：仅在整块数据传完后发一次，通知 CPU "我干完了，来善后"。

所以 DMA 既要请求**总线**（传数据用），也要请求**中断**（通知完成用）。

**② DMA 和 CPU 同时请求访存，谁优先？**

**DMA 优先**。原因：I/O 设备不立即访存就可能**丢失数据**（数据到了缓冲区如果不及时读走，下一批数据会覆盖），CPU 等一个存储周期不会丢数据。

### Q4 · 周期挪用

> DMA 三种传送方式中哪种是实际主流？周期挪用"挪用"的具体是什么？为什么它可以发生在"任一机器周期结束时刻"而中断只能发生在"指令执行后"？

**学生回答**：周期挪用。挪用一个 CPU 的存储周期。（存储周期是多久，一个时钟周期吗）对啊为什么可以发生在"任一机器周期结束时刻"而中断只能发生在"指令执行后"

**AI 反馈 ✅✅ 答案对，追问好，解释如下**

**存储周期 ≠ 时钟周期**：存储周期是完成一次内存读/写操作的时间，通常由多个时钟周期组成（比如 4 个时钟周期 = 1 个存储周期）。

**"为什么 DMA 可以随时插入，中断只能等指令结束？"**——这是本节最核心的区别：

| | DMA（周期挪用） | 中断 |
|---|---|---|
| 要占用什么 | **总线**（用于内存和 I/O 之间搬数据） | **CPU**（执行中断服务程序） |
| 插入时机 | 任一**总线/机器周期**结束时 | 每条**指令**执行后 |
| 原因 | 只借用总线一个周期，CPU 只是暂停一拍，**内部状态不变**（寄存器、PC 全保留） | 要切换程序执行流，必须等当前指令完整结束才能安全保存现场（PC、寄存器、PSW） |

**一句话**：DMA 偷的是路（总线），CPU 可以原地停顿一拍继续走；中断偷的是人（CPU 执行流），换人必须在完成手头这一步之后。

> 补充：周期挪用具体"挪用"的是一个**存储周期**——这一拍本该 CPU 访存，DMAC 插队用一下总线往内存写一个字，CPU 这一拍不访存但继续执行内部操作（如运算）。

### Q5 · DMA vs 中断三维对比 + 结合使用

> 从请求对象、响应时机、是否改变 CPU 现场三个维度，对比中断方式和 DMA 方式。如果一台设备既需要传输大批数据，又需要处理异常情况（如传输错误），应该怎么结合使用两种方式？

**学生回答**：请求对象——中断方式 I/O 设备请求的是 CPU 而 DMA 方式请求的是 DMA。CPU 不关中断原子操作等可以响应，DMA 随时响应。一个是当即改变 CPU 现场另一个是 DMA 全部数据接收了才改变。怎么结合使用呢？

**AI 反馈 ⚠️ 三维对比需修正**

| 维度 | 中断方式 | DMA 方式 |
|------|----------|----------|
| 请求对象 | 请求 **CPU 时间**（让 CPU 暂停当前程序执行 ISR） | 请求**总线控制权**（HOLD/HLDA），不是请求 DMA 控制器 |
| 响应时机 | 每条**指令执行后**检查中断请求 | 每个**总线周期/机器周期**结束时即可响应 |
| 改变 CPU 现场 | **每次中断都改变**（保存现场→执行 ISR→恢复现场） | 数据传送期间**完全不改变** CPU 现场；只有最后发结束中断时才改变 |

**纠正**：你说的"DMA 全部数据接收了才改变"—这个对，但补充：中间数据传送过程 CPU 现场一分不变，CPU 在正常执行自己的程序。

**怎么结合？**

正常模式：**大批数据传输用 DMA**（CPU 完全不参与，效率最高）

但如果传输中途出错（比如磁盘坏扇区），DMA 控制器可以发出**异常中断**通知 CPU → CPU 执行中断服务程序处理错误（如重试、标记坏块）。这就是两者结合的方式：

> **DMA 负责搬运，中断负责异常告警 + 完成通知。**

### Q6 · 定时查询例

> 假设计算机的主频为 500MHz，CPI 为 4，某外设的数据率为 2MB/s，I/O 接口中有一个 32 位数据缓冲寄存器，采用定时查询方式，每次 I/O 都执行 10 条指令。外设最多间隔多长时间查询一次才能不丢失数据？CPU 用于外设 I/O 的时间占 CPU 总时间的百分比至少是多少？

**学生回答**：
32bit(4B)/2MB/s=2×10⁻⁶s 最多隔这么久就要查询一次
每次I/O的时间 4×10×(1/(500×10⁶))=0.08×10⁻⁶s 这个就是用于I/O的时间吗
CPU总时间是什么（两次定时查询之间的时间间隔吗）
0.08/2=4%

**AI 反馈 ✅ 计算全对，澄清两个概念**

**Q: "0.08μs 就是用于 I/O 的时间吗？"**
是的。4(CPI)×10(条指令)×2ns(每时钟周期)=80ns=0.08μs。这就是 CPU 完成**一次**查询的耗时——执行 10 条轮询指令的纯 CPU 时间。

**Q: "CPU总时间是什么？"**
"CPU 总时间"= 把 CPU 看作一个资源池，每秒钟它总共能工作 1 秒。占比 = 每秒花在 I/O 上的时间 / 1 秒。

两种等价的算法：
- **按比例**：每次耗时/时间间隔 = 0.08μs/2μs = **4%** ✅
- **按秒算**：每秒查询 500000 次 × 每次 80ns = 40ms → 40ms/1s = **4%**

你的 0.08/2 用的是比例法，结果是 4%，完全正确。

### Q7 · 中断方式例

> 假定计算机的主频为 500MHz，CPI 为 4，某外设的数据率为 40MB/s，I/O 接口中有一个 32 位数据缓冲寄存器。在中断 I/O 方式下，若每次中断响应和中断处理的总时钟周期数至少为 400，则该外设能否采用中断 I/O 方式？为什么？如果数据率降到多少就能用中断了？

**学生回答**：
数据缓冲寄存器刷新时间 4B(32bit)/40MB/s=0.1×10⁻⁶s
中断响应处理时间 400×(1/(500×10⁶))=0.8×10⁻⁶s > 刷新时间 数据都刷新了中断还没处理完
数据率降到 4B/speed=0.8×10⁻⁶s → speed=5MB/s

**AI 反馈 ✅ 全部正确**

| 步骤 | 计算 | 结果 |
|------|------|------|
| 数据刷新间隔 | 4B ÷ 40MB/s | **0.1μs** |
| 中断处理耗时 | 400 × 2ns(主频倒数) | **0.8μs** |
| 判断 | 0.8μs > 0.1μs | **跟不上，不能用中断** |
| 临界数据率 | 4B ÷ 0.8μs | **5MB/s**（≤5MB/s 才能用中断） |

**两种等价视角**：

**时间对比法**（你用的）：中断处理时间 vs 数据到达间隔。0.8μs > 0.1μs → 来不及。

**频率法**（你没写的）：
- 每秒中断请求次数 = 40MB/s ÷ 4B = **10,000,000 次**
- CPU 每秒能处理的中断上限 = 500MHz ÷ 400 = **1,250,000 次**
- 1,250,000 < 10,000,000 → 跟不上
- 临界：1,250,000 次 × 4B = **5MB/s**

两种算法殊途同归。频率法在比较"CPU 的处理能力上限"和"I/O 设备的请求速率"。 |

结论干净利落。这道题本质上就是 Q2 那个"开销 vs 数据间隔"公式的数值验证。

### Q8 · DMA 计算例

> 假定计算机的主频为 500MHz，CPI 为 4，某外设的数据率为 40MB/s，I/O 接口中的数据端口为 32 位，采用 DMA 方式，每次 DMA 传送块大小为 1000B，且 DMA 预处理和后处理的总时钟周期数为 500。则 CPU 用于该外设 I/O 的时间占 CPU 总时间的百分比是多少？——对比 Q7 的中断结论和 4.1 定时查询的 4%，这三个"4%"含义相同吗？

**学生回答**：
数据缓冲寄存器刷新时间 4B/40MB/s=0.1×10⁻⁶s（在这里没用？）
DMA传送数据的时间：1000B/40MB/s=25×10⁻⁶s（CPU的总时间？但实际上CPU没参与）
CPU用于该外设I/O的时间：500×(1/500×10⁶)=1×10⁻⁶s（就是DMA预处理和后处理的总时钟周期数？）
DMA中CPU 96%的时间都没参与，中断哪有4%

**AI 反馈 ✅ 直觉全对，补上最后一步计算**

你的每一步判断都正确：
- "缓冲寄存器刷新时间在这里没用" ✅ —— DMA 按块传，不按 32 位逐字传
- "25μs 是 DMA 传输时间，CPU 没参与" ✅ —— 完全正确
- "1μs 是 CPU 的预处理+后处理时间" ✅ —— 500 周期 × 2ns = 1μs，每块一次

**占比计算（你漏的最后一步）**：

**按秒算**（刚才补的）：
- 每秒 DMA 块数 = 40MB/s ÷ 1000B/块 = **40000 块/秒**
- CPU 每秒花在 I/O 上的时间 = 40000 × 1μs = 40000μs = 0.04s
- 占比 = 0.04s / 1s = **4%**

**比例法**（你没写的）：
- 每块 CPU 时间 = 500 × (1/500MHz) = **1μs**
- 块到达间隔 = 1000B ÷ 40MB/s = **25μs**
- 占比 = 1μs ÷ 25μs = **4%**

比例法更直观：DMA 每搬 25μs 的数据只需要 CPU 1μs 的启动/收尾时间，剩下 24μs CPU 完全自由。

---

**三个"4%"含义完全不同**（你的追问切中要害）：

| 来源 | 4% 的含义 | CPU 这 4% 在干什么 |
|------|-----------|-------------------|
| 4.1 定时查询 | CPU 4% 时间在**轮询** | 跑查询程序，反复读状态寄存器 |
| Q7 中断 | **根本不能用于 40MB/s**，不能用所以没有占比 | — |
| Q8 DMA | CPU 4% 时间在**预处理+后处理** | 初始化 DMAC + 校验完成数据 |

> 你说"DMA 中 CPU 96% 时间都没参与"——完全正确。这 4% 只花在每块的启动和收尾上，数据传送的 96% 时间 CPU 完全自由（跑其他进程）。中断方式连机会都没有（0.1μs 间隔来不及响应），更谈不上占比。

---

**学生追 Q**：这两个 4% 一个在轮询一个在预处理后处理，好像 CPU 占用率都不高，感觉 DMA 也没比定时查询好到哪去啊？

**AI 回应**：关键在——你对比的不是同一台设备。

| | 定时查询（Q6） | DMA（Q8） |
|---|---|---|
| 数据率 | **2 MB/s** | **40 MB/s** |
| CPU 占比 | 4% | 4% |

同样的 4%，DMA 搬的数据量是定时查询的 **20 倍**。把定时查询那台设备也升到 40MB/s 看看：

**定时查询 × 40MB/s**：
- 数据间隔 = 4B ÷ 40MB/s = **0.1μs**
- 每次查询 = **0.08μs**
- 占比 = 0.08 ÷ 0.1 = **80%**

CPU 80% 时间在轮询。而且 0.08μs 查询时间紧贴 0.1μs 到达间隔，稍有不慎就丢数据。

**DMA × 40MB/s**：还是 **4%**，因为每 1000B 才找 CPU 一次。

| 数据率 | 定时查询 CPU 占比 | DMA CPU 占比 |
|--------|-------------------|-------------|
| 2 MB/s | 4% | <1% |
| 40 MB/s | **80%**（基本不可用） | **4%** |
| 400 MB/s | 跟不上 | 4%（块调大即可） |

**DMA 的真正优势**：CPU 开销和**数据量无关**，只和**块数**有关。同样 40MB/s 数据率，把块从 1000B 调到 4000B，占比直接从 4% → 1%。而定时查询的 CPU 开销和**数据率成正比**——越快越忙，最终必然跟不上。两个 4% 放到同一数据率下，是 80% vs 4%。

### Q9 · 2011 真题

> 某计算机处理器主频为 50MHz，采用定时查询方式控制设备 A 的 I/O，查询程序运行一次所用的时钟周期数至少为 500。在设备 A 工作期间，为保证数据不丢失，每秒需对其查询至少 200 次，则 CPU 用于设备 A 的 I/O 的时间占整个 CPU 时间的百分比至少是？
> A. 0.02%  B. 0.05%  C. 0.20%  D. 0.50%

**学生回答**：查询时间 500 × (1/500×10⁵) = 2×10⁻⁵s，数据刷新时间 1/200=0.5×10⁻²s，2/500=0.004=0.4% 答案没有？

**AI 反馈 ⚠️ 方法对，计算差一倍**

思路完全正确。错误在计算——50MHz = 5×10⁷Hz。周期 = 1/50MHz = 20ns。500 × 20ns = 10μs（1×10⁻⁵s），你写成了 2×10⁻⁵s。

正确：

**比例法**：一次查询 = 500 × 20ns = **10μs**，查询间隔 = 1/200 = **5ms**，占比 = 10μs/5ms = 10/5000 = **0.20%**

**按秒算**：每秒 200 次 × 10μs = 2000μs = 2ms，占比 = 2ms/1s = **0.20%**

答案 **C**。你的 0.4% 刚好两倍——500/500×10⁵ = 1×10⁻⁵，你是否把 500/500 误算成了 2？

### Q10 · 2014 真题

> 若某设备中断请求的响应和处理时间为 100ns，每 400ns 发出一次中断请求，中断响应所允许的最长延迟时间为 50ns，则在该设备持续工作过程中，CPU 用于该设备的 I/O 时间占整个 CPU 时间的百分比至少是？
> A. 12.5%  B. 25%  C. 37.5%  D. 50%

**学生回答**：100/400 = 25%

**AI 反馈 ✅ 正确**

100ns ÷ 400ns = **25%**。答案 **B**。

注意"至少"取最短处理时间（不含延迟）。如果问"最多"：加 50ns 延迟 → (100+50)/400 = 37.5%。"至少/最多"在 OS 题常见，看清取处理时间的下限还是上限。

### Q11 · 2023 真题

> 下列关于 I/O 控制方式的叙述中，错误的是（ ）。
> A. 查询方式下，通过 CPU 执行查询程序进行 I/O 操作
> B. 中断方式下，通过 CPU 执行中断服务程序进行 I/O 操作
> C. DMA 方式下，通过 CPU 执行 DMA 传送程序进行 I/O 操作
> D. 对于 SSD、网络适配器等高速设备，采用 DMA 方式输入/输出

**学生回答**：C

**AI 反馈 ✅ 正确**

C 错：DMA 数据传送由 **DMAC 硬件**完成，CPU 不执行 DMA 传送程序。CPU 只在预处理（初始化寄存器）和后处理（结束中断）介入。

### Q12 · DMA vs 中断多选题

> 下列说法中，错误的是（ ）。
> I. 程序中断过程是由硬件和中断服务程序共同完成的
> II. 在每条指令的执行过程中，每个总线周期要检查一次有无中断请求
> III. 检测有无 DMA 请求，一般安排在一条指令执行过程的末尾
> IV. 中断服务程序的最后指令是无条件转移指令
> A. III、IV  B. II、III、IV  C. II、IV  D. I、II、III、IV

**学生回答**：I 没问题，II 不是总线周期是指令执行周期（指令执行的最后一步），III 错 每个总线周期检测 DMA 请求，IV 不是我记得是中断返回还是什么的 → **B**

**AI 反馈 ✅ 全对，逐项分析也准**

| | 说法 | 判断 |
|---|---|---|
| I | 中断过程由硬件+中断服务程序共同完成 | ✅ 正确 |
| II | 每个总线周期检查一次有无中断请求 | ❌ 指令执行后查 |
| III | 检测DMA请求安排在指令执行末尾 | ❌ 每个总线周期结束可检测 |
| IV | 中断服务程序最后指令是无条件转移 | ❌ 是中断返回指令(IRET) |

II、III、IV 全错 → 答案 **B**。

### Q13 · DMA 请求归属

> 在 DMA 传送方式中，由（ ）发出 DMA 请求，在传送期间总线控制权由（ ）掌握。DMA 请求触发器的作用是什么？传送长度计数器为什么是递减的？

**学生回答**：外设发出 DMA 请求，传送期间 DMAC 掌控总线。接受外设的 DMA 请求。因为是总字数，传完一个字递减→归零表示这批数据传完了

**AI 反馈 ✅ 全部正确**

| 问题 | 答案 |
|------|------|
| 谁发 DMA 请求 | **外部设备**（I/O 设备→DMAC） |
| 总线控制权 | **DMA 控制器（DMAC）** |
| 触发器作用 | I/O 设备数据就绪时置位，向 DMAC 表示"有数据要传" |
| 为什么递减 | 预设总字数，传一字减 1，减到 0 自动置零标志位 |

**补充**：递减比递增省硬件——减到 0 零标志自动触发，无需额外比较器。递增则每步都要和总字数比较"到了没"，多一个比较器。
